Problem Analysis

At its core, this problem asks me to impose a total order on poker hands - a rule that, given any two hands, decides unambiguously which one is greater. A total order is a comparison relation under which every pair of elements can be ranked relative to each other with no ties (the problem guarantees a clear winner). The technique is to map each hand to a comparison key that captures both its category (pair, flush, full house, and so on) and the cascading tie-breaker data (which pair first, then which kicker, then the next). Once each hand is reduced to such a key, choosing the winner is a single comparison.

The key observation is that every poker category can be detected with two cheap pieces of derived data: a frequency count of how often each card value appears, plus two boolean flags - whether all five suits match (a flush) and whether the five distinct values form a consecutive run (a straight). Four-of-a-kind, full house, three-of-a-kind, two pairs, and one pair are distinguished purely by the shape of the frequency counts; flush and straight come from the two flags; straight flush and royal flush are their combination. Because a hand is always exactly five cards, every per-hand operation - counting, sorting, flag checks - is O(1). Processing N lines from the file is therefore O(N) overall, and here N is just 1000.

Solution Approach

Canonical Hand Evaluation with a Sorted Frequency Table

Both the Python (p0054_s0.py) and C (p0054_s0.c) solutions implement the same algorithm and differ only in language idioms. For each line, the ten whitespace-separated tokens are split into Player 1's first five cards and Player 2's last five. Each hand is evaluated independently into a (rank, tie-breakers) structure, and the two structures are then compared.

My first design decision is how to turn a card character into a number. I encode the value ordering as the string "23456789TJQKA" and take each card's integer rank from its position in that string, so '2' maps to 0, 'T' to 8, and 'A' to 12. In Python this is VALUES.index(c); in C it is pointer arithmetic via strchr. I prefer this over a dictionary or a switch because the ordering is implicit in the string and the data stays collocated and readable.

Next I compute the two flags. is_flush is true when all five suits are identical. is_straight is true when the five values are all distinct and max - min == 4 - a compact test for "five consecutive values" that avoids explicitly sorting and walking the run. If both flags hold, the hand is a straight flush; if additionally the lowest value is 8 (the run T-J-Q-K-A), it is a royal flush. Settling these top categories first keeps the high-end cases short.

For everything else, I build a frequency table of (count, value) pairs and sort it by count descending, then value descending. This single sort into canonical form is the heart of the classification: the dominant group always lands first, so I only need to inspect the first one or two entries. A leading count of 4 is four-of-a-kind; 3 followed by 2 is a full house; a lone 3 is three-of-a-kind; two consecutive 2s is two pairs; a single 2 is one pair; otherwise high card. Crucially, I build the tie-breaker sequence in priority order at the same time - the dominant value first, then kickers in descending order - so that comparing two hands of equal rank becomes a plain lexicographic comparison of the tie-breaker tuples with no extra branching.

In Python the result is a HandRank dataclass with a custom __gt__; the hand itself is a PokerHand dataclass whose hand_rank is a functools.cached_property, and the categories are an IntEnum so they compare as integers. The C version mirrors this with a HandRank struct holding the rank enum plus a fixed five-element tie-breaker array and a count, and a hand_rank_gt function that returns on the first differing element. The C code also defends against blank and short lines, using strtok_r for reentrant tokenisation.

Programming Techniques

Encoding an ordered domain as a string index

Representing the card ranks as positions in "23456789TJQKA" turns a mapping problem into a search-and-subtract. This works whenever you have a small, fixed, ordered alphabet of symbols and want a consistent numeric ordering without maintaining a separate lookup table. It keeps the data in one place and makes the ordering self-evident. Reach for it any time you would otherwise write a tiny dictionary mapping characters to consecutive integers.

Sorting into canonical form to simplify classification

By sorting the frequency table once (count descending, then value descending) into canonical form, I replace a tangle of "search the counts for a 3, then search for a 2" logic with a simple cascade of leading-entry checks. The general principle - normalise data into a canonical shape so downstream code can read it sequentially - recurs in anagram detection, histogram problems, and interval scheduling. Pay a small one-time sorting cost to make every later inspection trivial.

Comparison keys with most-significant-field-first ordering

I build the tie-breaker tuple so that the most discriminating value comes first. Python tuples already compare lexicographically, so the tuple comparison does exactly the right thing; in C, hand_rank_gt walks the arrays and returns at the first difference. The insight is that if the key is constructed correctly, the comparison itself needs no logic. This lexicographic approach applies to any multi-level ranking: encode all priority information into the key, then compare keys.

Separating classification from comparison

Evaluating a hand (what category and key is it?) is kept entirely separate from comparing two hands (which is greater?). This keeps each piece small and independently testable, and it means the comparison code is identical for every category. When a problem has a "compute a property" phase and a "use the property" phase, splitting them almost always pays off.

Lazy, cached derived attributes (Python)

The hand_rank is a functools.cached_property: it is computed on first access and then stored on the instance. This documents that the value is a pure function of the hand and avoids recomputation if accessed more than once. Use a cached_property for any derived attribute that is side-effect-free and potentially costly.

Generator pipeline for streaming line processing (Python)

The solve function builds a generator of (player1, player2) pairs, using the walrus operator to bind the split tokens inside the comprehension's filter, and feeds it straight into sum. Nothing is materialised in memory all at once. For 1000 lines this is negligible, but the habit scales gracefully to far larger files.

Fixed-size stack arrays and inline sorts (C)

Because a hand is exactly five cards, every intermediate buffer in the C code is a stack-allocated fixed-size array, and the small sorts use inline nested loops rather than qsort. For arrays of five elements, the per-comparison function-pointer overhead of qsort would dominate, so a hand-rolled insertion-style sort is both faster and clearer. For small fixed-size collections, an inline sort usually beats a library sort.

Key Takeaways

The unlocking insight is that a complex-looking ranking problem collapses once you reduce each hand to a single ordered comparison key. Two cheap derivations - a sorted frequency table plus two boolean flags - classify any hand, and building the tie-breakers in priority order makes the final comparison a one-line lexicographic check with no special cases.

Both solutions return the correct answer of 376. The C implementation runs in about 0.0004 s and the Python in about 0.0079 s - roughly a 20x difference. The gap is the usual per-object Python overhead (dataclass construction, property access, generator stepping) accumulating across thousands of iterations, while C keeps everything on the stack with minimal indirection. For 1000 hands either is instant; the comparison simply illustrates the constant-factor cost of high-level abstractions.

  • Reduce a comparison problem to a single ordered key, then let lexicographic comparison do the work.
  • Encode small, fixed, ordered domains as string or array positions instead of dictionaries.
  • Sort data into canonical form once so all later inspection is cheap and sequential.
  • Put the most significant discriminator first in any comparison key.
  • Separate "compute the property" from "use the property" to keep code small and testable.
  • Use lazy cached attributes for pure, potentially-costly derived values.
  • For tiny fixed-size collections, prefer inline sorts over general-purpose library sorts.

Problem 54: Poker Hands [Level 3]

Problem Statement

In the card game poker, a hand consists of five cards and are ranked, from lowest to highest, in the following way:

  • High Card : Highest value card.
  • One Pair : Two cards of the same value.
  • Two Pairs : Two different pairs.
  • Three of a Kind : Three cards of the same value.
  • Straight : All cards are consecutive values.
  • Flush : All cards of the same suit.
  • Full House : Three of a kind and a pair.
  • Four of a Kind : Four cards of the same value.
  • Straight Flush : All cards are consecutive values of same suit.
  • Royal Flush : Ten, Jack, Queen, King, Ace, in same suit.

The cards are valued in the order:
2, 3, 4, 5, 6, 7, 8, 9, 10, Jack, Queen, King, Ace.

If two players have the same ranked hands then the rank made up of the highest value wins; for example, a pair of eights beats a pair of fives (see example 1 below). But if two ranks tie, for example, both players have a pair of queens, then highest cards in each hand are compared (see example 4 below); if the highest cards tie then the next highest cards are compared, and so on.

Mathematical Complexity

This problem is an exercise in discrete classification and total ordering rather than number theory or combinatorics in the traditional sense. The core challenge is defining a strict total order over poker hands, which requires understanding a hierarchy of ten ranked categories and, within each category, a lexicographic tie-breaking rule that resolves any draw completely. Each hand must be mapped to a comparable representation that encodes both its category rank and an ordered sequence of card values sufficient to break any tie. The problem guarantees a clear winner on every line, so only strict greater-than comparison is needed.

Processing the input involves evaluating each of the 1,000 lines exactly once. Classifying a single five-card hand takes O(1) time since the hand size is fixed at five cards - all loops over the hand are bounded by a constant. The overall solution therefore runs in O(N) time where N is the number of lines, making the asymptotic complexity trivial. All the interesting work lies in getting the constant-factor logic correct: mapping face characters to numeric values, counting duplicates, and assembling tie-breaker sequences that respect the full hand ranking rules.

Solution Approach

Sorted Count-Pair Classification with a Category Integer and Tie-Breaker Sequence (Python and C)

Both the Python and C solutions implement the same algorithm. I modelled each hand by extracting two parallel sequences - numeric card values mapped via the string VALUES = "23456789TJQKA" so that index position directly encodes rank, and suit characters. Classification first computes two boolean flags, is_flush and is_straight, to handle the top-tier hands. For the straight check I required both that all five values are distinct and that the span is exactly four; both conditions together are necessary and sufficient for a standard straight. For the remaining hand types I built a list of (count, value) pairs sorted descending by count then value, so quads sort before trips before pairs and the highest-count group's value surfaces automatically. Each evaluated hand is returned as a primary category integer paired with a tie-breaker sequence of up to five integers. Comparison first checks the category, then falls through to a lexicographic comparison of the tie-breaker sequence.

In the Python solution I used a PokerRank IntEnum so that category comparison reduces to plain integer comparison, a HandRank dataclass with a custom __gt__ method, and a cached_property on PokerHand to avoid recomputing the rank if accessed more than once. The final answer is a sum over a generator expression that pairs each line's two hands and applies the comparison. In the C solution I implemented the same logic procedurally: evaluate_hand returns a HandRank struct containing a fixed-size tie_breakers array and a count field, and hand_rank_gt performs the two-level comparison by iterating over the tie-breaker array element by element. The C version avoids all dynamic allocation during classification, keeping every working array on the stack.

Learnings

The key insight that made this problem tractable was realising I could encode the entire comparison problem into exactly two values - a category integer and a tie-breaker sequence - and then let the language's built-in ordering or a simple loop handle the rest. Once I had that framing, the classification code became a straightforward chain of if-elif tests with no backtracking. The (count, value) sorted-pair trick is particularly clean: sorting descending by count automatically groups quads before trips before pairs, and the secondary sort by value ensures the highest-count group's value appears first, which is precisely what tie-breaking needs without any additional logic.

A subtlety I handled carefully was two-pair tie-breaking: after identifying both pair values I explicitly sort them to guarantee the higher pair is always in position zero, rather than relying on the order they happened to appear in the count-pair list. Another subtle point was the Royal Flush case - I stored the suit order as the tie-breaker for completeness even though the problem guarantees no draws, because leaving it undefined would be a silent correctness hole.

Since there is only one algorithm index here, the Python and C files are directly comparable implementations of the same approach. The C solution runs in approximately 0.0007 s versus 0.0129 s for Python - a roughly 18x speedup - but both are fast enough that the choice has no practical consequence for this input size. The Python version is considerably more readable thanks to IntEnum, dataclasses, and tuple unpacking. For future problems involving structured multi-attribute comparison I will remember the pattern of separating a primary integer rank from a secondary sequence tie-breaker, combined with an IntEnum for the category in Python. This pattern keeps comparison logic clean, extends naturally to deeper tie-breaking chains without custom recursion, and translates directly into a plain struct in C with minimal adaptation.

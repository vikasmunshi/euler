Problem 58: Spiral Primes [Level 2]

Problem Statement

Starting with $1$ and spiralling anticlockwise in the following way, a square spiral with side length $7$ is formed.

37 36 35 34 33 32 31
38 17 16 15 14 13 30
39 18 5 4 3 12 29
40 19  6  1  2 11 28
41 20 7 8  9 10 27
42 21 22 23 24 25 26
43 44 45 46 47 48 49

It is interesting to note that the odd squares lie along the bottom right diagonal, but what is more interesting is that $8$ out of the $13$ numbers lying along both diagonals are prime; that is, a ratio of $8/13 \approx 62\%$.

If one complete new layer is wrapped around the spiral above, a square spiral with side length $9$ will be formed. If this process is continued, what is the side length of the square spiral for which the ratio of primes along both diagonals first falls below $10\%$?

Mathematical Complexity

The key insight is that the four diagonal corners produced by each new spiral layer can be derived algebraically from the side length alone. For a layer that produces a square of side length s = 2k + 1, the bottom-right corner is always s² (a perfect square, never prime for s > 1), and the remaining three corners are obtained by subtracting successive multiples of s - 1: the values are s² - (s-1), s² - 2(s-1), and s² - 3(s-1). This means no spiral simulation is needed at all - all four corners per layer are O(1) arithmetic. Primality testing each corner via trial division costs O(sqrt(corner)) per call, and since corner values grow as O(s²), each individual test costs O(s). Summing this cost across all layers up to the answer at side length 26241 gives roughly O(S²) total trial divisions, which is consistent with the observed ~3-second Python runtime.

The problem has no fixed upper bound, so termination must be guaranteed analytically. Because the bottom-right corner is always a perfect square (contributing zero primes), at most three of the four new corners per layer can be prime. The total count of diagonal elements grows by exactly 4 per layer, so the denominator always increases by 4 while the numerator increases by at most 3. This means the ratio is eventually forced below any positive threshold, guaranteeing the loop always terminates regardless of the chosen threshold value.

Solution Approach

Layer-by-Layer Corner Generator with Trial Division (Python and C)

I implemented the same algorithm in both Python and C. The core idea is to avoid all grid simulation by computing corner values directly from the side length. In the Python version I factored the corner generation into a dedicated generator, generator_spiral_corners, that yields the side length and all four corner values for each successive layer. The main solve function accumulates a running count of diagonal elements (initialised to 1 for the centre) and a running count of prime corners, then checks the ratio after each layer. Primality is tested with a straightforward trial division routine: short-circuit on even numbers, then test odd divisors up to the square root. Since the bottom-right corner is always s², it fails the primality check almost immediately, so in practice only up to three meaningful trial-division calls are made per layer. Separating the generator from the ratio logic kept the Python code readable and allowed the threshold to be passed in as a parameter, making the solution easy to test with different values.

The C translation follows exactly the same algorithm, replacing the generator with a plain for loop and using long long for corner values to avoid overflow as the spiral grows. The threshold is read from argv[1] via atof, and the ratio comparison uses an explicit cast to double. There are no algorithmic differences between the two versions - the speedup of roughly 27x (0.121 s versus 3.236 s) is attributable entirely to native compilation and the absence of Python interpreter overhead on the inner trial division loop.

Learnings

The key insight that unlocked this solution was recognising that no spiral walk is needed at all. Once I noticed that the step between consecutive corners on the same layer is a constant s - 1, the closed-form corner formulas followed immediately and the generator wrote itself in a few lines. A secondary observation - that the bottom-right corner is always s² and therefore always composite for s > 1 - means that corner could be skipped as an optimisation, though treating all four corners uniformly keeps the code cleaner with negligible cost.

Since both solution files implement the same algorithm, the comparison is purely about language overhead. The benchmark shows C running at 0.121 s versus Python at 3.236 s, a factor of about 27x, which is typical for tight arithmetic loops with no memory-intensive data structures. For much larger spirals, switching to a Miller-Rabin primality test would reduce the per-corner cost from O(sqrt(n)) to O(log²(n)) and would matter for very deep searches, but trial division is perfectly adequate for the side lengths encountered here.

  • Closed-form corner formulas eliminate all grid simulation overhead and make the generator trivially simple to implement.
  • A generator-based design in Python lets the stopping condition live in the caller, keeping concerns cleanly separated.
  • Parameterising the threshold rather than hardcoding 0.1 made the solution easy to test and reuse with different inputs.
  • Guaranteeing termination analytically - the ratio must eventually fall below any positive threshold - removes the need for a hard iteration cap and makes the loop provably correct.
  • For problems with quadratically growing numbers and trial division as the inner loop, C provides a large constant-factor speedup that is worth having when Python runtimes approach several seconds.

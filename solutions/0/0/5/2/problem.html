Problem 52: Permuted Multiples [Level 0]

Problem Statement

It can be seen that the number, $125874$, and its double, $251748$, contain exactly the same digits, but in a different order.

Find the smallest positive integer, $x$, such that $2x$, $3x$, $4x$, $5x$, and $6x$, contain the same digits.

Mathematical Complexity

The core concept is digit-multiset equivalence: two integers are permutations of each other's digits if and only if sorting their decimal digit sequences produces identical strings. The problem asks for the smallest positive integer x such that all six values x, 2x, 3x, 4x, 5x, 6x belong to the same digit-permutation equivalence class. No closed-form characterisation of such numbers is known, so the solution reduces to a linear exhaustive scan. Checking a single candidate costs O(m * d log d) where m is the number of multiples and d is the digit count (the sort dominates), giving overall complexity O(N * m * d log d) with N = 142857 for the six-multiple case.

The answer 142857 is the cyclic number arising from the decimal expansion of 1/7. Multiplying it by any integer from 1 to 6 cyclically permutes its digits, a consequence of 7 being a full-reptend prime. I did not need that number-theoretic fact to find the answer - the brute-force search discovers it naturally - but it explains why the answer is what it is, and why multiples 3 through 6 all share the same answer while the two-multiple case already terminates much earlier at 125874.

Solution Approach

Linear Scan with Sorted-Digit Fingerprinting (s0 - Python and C)

Both the Python and C implementations share the same algorithm. I iterate over every positive integer starting from 1 and, for each candidate, compute a canonical fingerprint - the sorted decimal digit string - for each required multiple. If every fingerprint is identical, the candidate is the answer and the loop terminates.

In Python I build a set comprehension over all fingerprints and check len(...) == 1. This is a clean one-liner that scales naturally with the number of multiples without any pairwise comparison loop. I parameterised the number of multiples so I could verify intermediate results: passing multiples=2 reproduces the 125874 example from the problem statement, and higher values all converge to 142857. The upper bound on the loop is sys.maxsize // multiples to guard against integer overflow on any platform, though the loop terminates far sooner in practice.

In C I use a helper function sorted_digits that extracts digits into a temporary buffer, sorts them with qsort, and null-terminates the result. I then compare each multiple's fingerprint against the fingerprint of i itself using strcmp, breaking early on the first mismatch. One language-specific difference is that the multiples parameter is read from argv rather than a keyword argument, and the guard is a simple range check with a fprintf error message. The early break on mismatch is a minor optimisation not present in the Python set comprehension, though it matters little given how quickly matching candidates are found.

Learnings

The key insight was recognising that sorted digit strings are a cheap, perfect fingerprint for digit-permutation equivalence. Collecting all fingerprints into a set and checking its length is more elegant than any pairwise comparison loop and generalises immediately to any number of objects being compared. Parameterising the number of multiples was also valuable: it let me confirm the two-multiple example before running the full six-multiple search, and revealed the interesting structure that multiples 3 through 6 all share the same answer while 2 alone terminates earlier.

The benchmark results confirm that a single algorithm is sufficient here. The Python solution solves the main case in about 0.42 s and the C translation solves it in about 0.029 s - roughly a 15x speedup - which is the expected payoff of eliminating interpreter overhead for a tight inner loop with many short string operations. For a problem at this difficulty level the Python solution is perfectly adequate, but the C version demonstrates the consistent gain from removing that overhead when digit-string comparisons dominate the runtime.

  • Sorted digit strings as a digit-multiset fingerprint are broadly reusable across digit-manipulation problems.
  • Using len(set_of_fingerprints) == 1 is cleaner than explicit pairwise equality checks and generalises to any number of values being compared.
  • Parameterising the search and verifying against known partial results is a reliable way to build confidence before committing to the full run.
  • The cyclic number 142857 - arising from the decimal expansion of 1/7 - surfaces in several digit-permutation contexts and is the canonical example of a full-reptend prime generating a cyclic number; worth keeping in mind for future problems.
  • Early termination on the first fingerprint mismatch is a worthwhile micro-optimisation in the C version, though the problem is easy enough that it does not change the order of magnitude of the runtime.

Problem Analysis

The mathematical core of this problem is the notion of a digit multiset: two integers are permutations of one another exactly when they contain the same decimal digits with the same multiplicities, irrespective of order. The challenge is to test this relation cheaply. My tool is a canonical form - a single, unique representative chosen for each class of digit-permutations. The obvious choice is the string of the number's decimal digits sorted into ascending order. I call this a fingerprint. Two numbers are digit-permutations of each other if and only if their fingerprints are identical, so the awkward question "do these share the same digits?" collapses into a plain string equality test.

The second observation bounds the search and proves that brute force is safe. If x has d decimal digits, any permutation of x must also have exactly d digits. Therefore x, 2x, ..., 6x must all have the same digit count. Since 6x is the largest, this forces x into the lower sixth of its digit-length band (roughly x < 10^d / 6); otherwise 6x would carry into the next band and gain a digit. Consequently the answer is never far from the start of any band, and a simple upward linear scan finds 142857 after about 140,000 iterations. The total work is O(N * M * D) where N is the answer, M the number of multiples (at most 6), and D the digit count (at most 7) - trivially small.

Solution Approach

Linear Scan with Sorted-Digit Fingerprinting (Python and C)

Both files implement the same algorithm. I iterate over candidates x = 1, 2, 3, .... For each x I compute the fingerprint of every required multiple and check whether they all agree with the fingerprint of x itself. The first x for which they all coincide is the answer.

In Python I express the test with a set comprehension: {"".join(sorted(str(i * multiple))) for multiple in multiples_range}. Building a set collapses duplicate fingerprints, so if every multiple shares the same digits the set has exactly one element. The condition len(...) == 1 therefore reads as "all fingerprints are equal." I bound the loop at sys.maxsize // multiples as a conceptual overflow guard, though Python integers are unbounded in practice.

In C I implement fingerprinting by hand. The sorted_digits function extracts digits by repeatedly taking n % 10 and dividing by 10, writing them into a buffer, then calls qsort to sort that buffer and null-terminates it. I compare fingerprints with strcmp. Crucially I use an early exit: in the inner loop over multiples 2..M, the moment a multiple fails to match I set all_match = 0 and break. Because the vast majority of candidates fail at the very first multiple (m = 2), most iterations do only a single comparison.

Both versions take the number of multiples as a parameter (a command-line argument in C, a positional argument in Python), so the same code verifies the simpler 2-multiple case (answer 125874, given in the statement) as well as the full 6-multiple case.

Programming Techniques

Canonical Fingerprinting via Sorted Digits

A fingerprint maps an object to a compact canonical form that is identical for every member of its equivalence class. Here, sorting the decimal digits yields a string that uniquely identifies the digit-multiset, so "are these permutations?" becomes "are these fingerprints equal?". This pattern recurs constantly: whenever you must group items by an equivalence relation, compute a cheap canonical representative and compare those instead of doing expensive pairwise checks. In Python it is the one-liner "".join(sorted(str(n))); in C it is a digit-extraction loop plus qsort.

Set-Cardinality Test for Universal Equality (Python)

Rather than store a reference fingerprint and compare each subsequent one against it, I build a set of all fingerprints and test len(set) == 1. A set keeps only distinct elements, so a cardinality of one means they were all equal. This removes bookkeeping and is highly readable. The tradeoff is that it always evaluates all elements (no short-circuit) and pays a small hashing cost, but for a collection of at most six items that is negligible. Reach for it whenever you need to assert that all elements of a small iterable coincide.

Short-Circuit Early Exit in the Inner Loop (C)

The C inner loop breaks the instant a multiple's fingerprint mismatches. Because most candidates fail immediately at m = 2, this turns the typical iteration into a single comparison rather than five. This is the standard optimisation for linear scans guarded by a conjunction of conditions: order the checks so the most-likely-to-fail one runs first, and bail out as soon as it fails. The Python set comprehension deliberately forgoes this for readability, which is acceptable when the per-item check is cheap.

Digit Extraction by Division and Modulo (C)

Instead of formatting numbers into strings with sprintf, the C code extracts digits arithmetically with repeated % 10 and /= 10. This avoids string-formatting overhead and a separate parsing pass. The digits emerge least-significant first, but that is irrelevant here because they are immediately sorted. Use this technique whenever you need to process digits individually and either the order does not matter or it will be corrected downstream.

Parameterising the Problem Shape

Both solutions take the number of multiples as an argument rather than hard-coding 6. This separates the algorithm from the specific instance: it lets the test suite verify correctness on the known 2-multiple case, supports benchmarking across difficulty levels, and even reveals structure (the answer stabilises at 142857 once three or more multiples are required). Making the instance a parameter is a good habit for any algorithm you expect to test or reuse.

Key Takeaways

The unlocking insight is the fingerprint: reducing each number to its sorted-digit string turns a complex "same digits?" relation into a uniform string comparison. The brute-force scan is justified by the digit-length constraint, which confines the answer to a small range within each band.

Both files implement the identical algorithm, so the benchmark differences are purely language overhead. On the main 6-multiple case the C solution runs in about 0.0211 s versus 0.3361 s for Python - roughly 16x faster. The gap comes from Python allocating many small string and set objects on each of about 140,000 iterations, plus interpreted dispatch. The C version is the fastest; the Python version is the more elegant and readable. For a one-off answer either is more than fast enough.

  • When testing equivalence among several values, compute a canonical fingerprint for each and compare those instead of doing pairwise comparisons.
  • Sorting the characters or digits of a value is the standard fingerprint for "same multiset" problems.
  • len(set(values)) == 1 is a clean Python idiom for "all elements are equal."
  • Put the likeliest-to-fail check first and short-circuit out of inner loops; most candidates fail early.
  • Extract digits arithmetically with % 10 and / 10 when string formatting is unnecessary overhead.
  • Parameterise the shape of a problem rather than hard-coding constants - it aids testing and reveals structure.
  • Digit-count constraints often implicitly bound a search space; spotting them early confirms that brute force is safe.

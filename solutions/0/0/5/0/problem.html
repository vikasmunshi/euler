Problem 50: Consecutive Prime Sum [Level 2]

Problem Statement

The prime $41$, can be written as the sum of six consecutive primes:

$$41 = 2 + 3 + 5 + 7 + 11 + 13.$$

This is the longest sum of consecutive primes that adds to a prime below one-hundred.

The longest sum of consecutive primes below one-thousand that adds to a prime, contains $21$ terms, and is equal to $953$.

Which prime, below one-million, can be written as the sum of the most consecutive primes?

Mathematical Complexity

The problem sits at the intersection of prime generation and contiguous subarray search. Generating all primes below a limit n with a sieve runs in roughly O(n log log n) time and O(n) space. The harder part is searching over all windows of consecutive primes whose sum stays below n and is itself prime. If there are approximately pi(n) ~ n / ln(n) primes below n by the prime number theorem, a naive double loop over all start and end index pairs is O(pi(n)^2) in the worst case - on the order of tens of millions of iterations for n = 10^6. In practice two pruning conditions cut this down sharply: prefix sums grow quickly so many windows exceed the limit early and trigger a break, and the inner loop starts only at a position that guarantees any candidate window is strictly longer than the current best, so shorter windows are never evaluated once a good one is found.

The key mathematical tool is the prefix sum array. Building the cumulative sum of the prime sequence means any contiguous window sum reduces to a single subtraction: prefix[i] - prefix[j] gives the sum of exactly i - j consecutive primes starting at index j. Checking whether that value is prime is O(1) with a boolean sieve array or a hash set. So each candidate evaluation costs O(1), and the total cost is dominated by how many candidates are actually evaluated, which the pruning keeps well below the theoretical quadratic bound.

Solution Approach

Sundaram Sieve with Prefix-Sum Window Search (s0 - Python and C)

I chose the Sieve of Sundaram to generate all primes below max_limit. It operates over odd numbers only, so its working array is roughly half the size of a standard Eratosthenes sieve and the implementation stays compact. After sieving I store the primes in two structures: an ordered sequence for building the prefix sum array, and a fast lookup structure - a hash set in Python, a flat unsigned char boolean array in C - for O(1) primality tests.

I build the prefix sum array with a leading zero element so that prime_sums[i] - prime_sums[j] equals the sum of exactly i - j consecutive primes starting at position j. The outer loop advances the right endpoint i forward through the prefix array. For each i the inner loop starts j at i - best_length - 1 rather than at i - 1; every window evaluated is therefore guaranteed to be strictly longer than the current best, meaning no work is wasted on shorter windows that could not improve the answer. The inner loop breaks early if the running sum exceeds max_limit. When a candidate is in range and is prime I update both the recorded best length and the best prime value.

The Python and C implementations follow exactly the same algorithm. The main language-level differences are that C uses a flat unsigned char array indexed directly by value for primality checking, while Python uses a hash set, and C uses long long for the prefix sum array to avoid integer overflow during accumulation. Both produce correct results across all test cases.

Learnings

The key insight that unlocked the solution was recognising that finding the longest consecutive-prime window whose sum is prime and below a limit is a subarray-sum problem in disguise. Once I framed it that way, prefix sums made per-candidate evaluation O(1) and the search became straightforward. The second insight was the pruning start index for the inner loop: initialising j at i - best_length - 1 means only windows that can actually improve on the current best are ever evaluated, turning a theoretically quadratic scan into something that completes in well under a second even at n = 10^6.

Since I only implemented one algorithm (index s0), there is no algorithmic comparison to make. The benchmark results do illustrate the language difference clearly: the C version solves the main case in about 5 ms compared to 272 ms for Python - roughly a 50x speedup - which is typical for tight nested loops where Python interpreter overhead accumulates over many iterations. Both are fast enough at this scale, but for larger limits the C version would remain practical for much longer.

  • Prefix sums convert any contiguous-range sum query into a single O(1) subtraction, a pattern worth reaching for whenever a problem involves sums over sliding or arbitrary windows in a sequence.
  • Starting the inner loop at a position that guarantees improvement over the current best - rather than scanning all shorter windows - is a simple but powerful pruning technique that transforms a quadratic worst case into something much more manageable in practice.
  • Storing primes in both a sorted sequence (for accumulation) and a fast-lookup structure (hash set in Python, boolean array in C) is a useful dual representation that avoids redundant primality-test overhead.
  • The Sundaram sieve is a compact, low-memory alternative to the Eratosthenes sieve and worth keeping in the toolkit when a small self-contained sieve implementation is needed.
  • Early termination on exceeding the limit and the "must beat current best" start index are both necessary; neither alone is sufficient to keep the search fast across all outer-loop iterations.

Problem Analysis

This problem asks for the maximum digit sum - the sum of the decimal digits of a number - over all values of the form ab with a and b both below 100. The digit sum of 1234, for instance, is 1+2+3+4 = 10. There is no closed-form shortcut here: the digit sum is a property of the decimal representation, and the decimal representation of a power has no simple analytic structure. The problem is therefore inherently computational. Its one genuine obstacle is size: ab for a and b near 99 has roughly b * log10(a) ~= 99 * 2 ~= 196 decimal digits, far beyond what a 64-bit integer can hold. We need arbitrary-precision arithmetic (commonly called "bignum" arithmetic) to compute these numbers exactly, because a single rounding error anywhere in the high digits would corrupt the digit sum.

The observation that makes the problem tractable is that the search space is minuscule. There are at most 99 * 99 = 9801 pairs (a, b), and even after generalising the bound to 10num_digits the count stays modest. So a brute-force search suffices: enumerate every pair, compute ab exactly, sum its digits, keep a running maximum. Letting N be the range limit and D ~= N * log10(N) the digit count of the largest value, computing one power by repeated multiplication costs O(N * D) digit-operations and summing its digits costs O(D), so the whole search is roughly O(N3 log N) in the worst case. For the canonical N = 99 this is trivially fast.

Solution Approach

Brute-force big-integer digit-sum search (Solution 0)

Both the Python and C files implement the same algorithm: iterate over every (base, exp) pair in the search range, compute base**exp exactly, sum its digits, and track the maximum. The search is parameterised by num_digits, with stop_at = 10**num_digits as the exclusive upper bound, so num_digits=2 reproduces the original Project Euler range 1..99.

A deliberate, pragmatic choice in both files is that they do not scan the full range. They restrict the base to the top 100 values and the exponent to the top 10 values below stop_at (for num_digits=2: bases 1..99, exponents 90..99). The reasoning is empirical. A digit sum tends to be largest when a number has many large digits, and the most promising candidates are large bases raised to high exponents - small bases or small exponents produce shorter numbers with fewer digits to sum. This pruning yields the correct answer for every test case here, but it is a heuristic shortcut rather than a proven bound; a fully rigorous solution would scan all pairs. The Python --show mode exists partly to sanity-check exactly this kind of assumption by visualising where the maximum actually lands in the grid.

In the Python solution, the built-in int is already arbitrary-precision, so base**exp directly produces the exact value with no library and no overflow. I extract the digit sum with the idiom sum(int(d) for d in str(base**exp)): convert the number to its decimal string and sum the integer value of each character. The entire search collapses into a single comprehension passed to max(). The optional --show branch builds NumPy matrices of digit sums, digit lengths, and average digit values across the grid and renders three Matplotlib heatmaps - a diagnostic aid entirely separate from the answer computation.

In the C solution, there is no native bignum, so I implement a minimal one. A BigInt is a heap-allocated struct holding a little-endian array of single decimal digits (index 0 = units place), a length, and a capacity. I chose base-10, one-digit-per-element storage on purpose: it makes the digit sum a trivial loop over the array with no extraction at all - exactly the operation this problem performs constantly. The only multiplication I need is bigint_mul_int (multiply by a scalar), a left-to-right carry-propagation loop. To compute the powers efficiently, for each base I build baseexp_start once via repeated scalar multiplies, then for each later exponent I multiply by the base just once. This incremental scheme means every exponent step after the first costs exactly one scalar multiply instead of recomputing the power from scratch. I pre-allocate the digit buffer to a conservative upper bound (max_exp * (num_digits+1) + 100) and keep a doubling-realloc fallback in the multiply routine as a safety net.

Programming Techniques

Python arbitrary-precision integers

Python's int grows automatically to any size, so base**exp produces an exact ~200-digit result with no library and no overflow risk. When numbers exceed 64 bits but raw speed is not the deciding factor, reaching for Python's native arbitrary-precision arithmetic is the right first instinct: it eliminates an entire class of overflow bugs and keeps the code short. Keep this in mind for any problem involving factorials, large exponentials, or combinatorial counts.

String-based digit extraction

The idiom sum(int(d) for d in str(n)) - convert to a decimal string, map each character back to an integer, and sum - is the cleanest way to compute a digit sum in Python. It uses a generator expression so no intermediate list is materialised. The alternative of repeated n % 10 / n //= 10 is more verbose and more error-prone. This pattern is worth memorising for any digit-manipulation task.

Little-endian base-10 big integer in C

Storing a big integer as an array of single decimal digits in little-endian order is a design choice tuned to this problem. Scalar multiplication becomes a simple forward loop - multiply, add carry, store, propagate - with no index reversal. More importantly, the digit sum is free: it is just a sum over the array, because we are already in base 10. When you build a custom bignum for a digit-sum problem, prefer base-10 storage for exactly this reason; for general arithmetic-heavy work a higher radix would be faster but would force you to extract digits afterward.

Incremental exponentiation (multiply-and-accumulate)

Rather than recomputing baseexp from scratch for each exponent, I compute baseexp_start once and then multiply by the base a single time per subsequent exponent. This is the incremental-update pattern: if you consume f(x), f(x+1), f(x+2), ... in order and f(x+1) is a cheap function of f(x), compute them in sequence. Here the cheap function is "multiply by base." It cuts the per-base multiplication count from O(exp_range * exp_max) down to O(exp_range + exp_start), and it generalises to running products, recurrences, and any sequence of powers.

Overflow-safe carry arithmetic in C

Inside bigint_mul_int I widen each digit to long long before forming digit * m + carry. A single base-10 digit times a multiplier up to ~999 plus a carry cannot overflow a 32-bit int in this problem, but widening first is the disciplined habit: it costs nothing and makes the routine correct for any multiplier the caller might pass. Integer overflow is silent and corrupting in C, so promote operands before the multiply, not after.

Capacity pre-allocation with realloc fallback in C

I compute a conservative upper bound on the digit count, allocate that much up front, and still keep a doubling-realloc guard inside the multiply routine. This is the canonical C dynamic array discipline: allocate generously to avoid reallocation in the hot loop, but keep a growth path so an underestimate can never corrupt memory.

Command-line parameterisation and timing harness

Both files accept a num_digits argument and run through a shared runner that times the solver and verifies the result. Separating this harness from the algorithmic core makes benchmarking and regression-checking effortless - a habit worth adopting in any performance-sensitive code.

Optional visualisation mode

The Python --show branch builds NumPy matrices and renders Matplotlib heatmaps of the digit sums, digit lengths, and average digit values across the (base, exponent) grid. Gating an expensive, dependency-heavy diagnostic behind a flag keeps the default path lean while giving an on-demand window into the structure of the search space - exactly what you want when validating a pruning heuristic like the top-100-bases / top-10-exponents shortcut used here.

Key Takeaways

The unlocking insight is that the whole problem reduces to "have exact big-integer arithmetic, then brute force a small grid." Python gives you that arithmetic for free; in C you write a minimal bignum tailored to the only two operations you need - multiply by a scalar and sum the digits.

The benchmarks reveal an instructive crossover. For the small and medium cases the C solution wins comfortably (0.0000 s vs 0.0001 s on dev, 0.0014 s vs 0.0069 s on main) because the work is tiny and CPython's interpreter overhead dominates. But on the large num_digits=3 case Python edges ahead (0.1941 s vs 0.2595 s): CPython's internal bignum uses well-optimised, high-radix C routines, whereas my hand-written single-digit base-10 representation pays a per-digit cost on every multiply. For large operands the quality of the underlying arithmetic outweighs the raw language advantage. This is a healthy reminder that "rewrite it in C" does not automatically mean "faster" - data representation and algorithm quality matter at least as much as the language.

  • When a problem involves large powers, factorials, or combinatorial numbers, reach for arbitrary-precision arithmetic before writing your own - native bignums are correct, overflow-free, and often fast.
  • If you need digit sums often, store your big integer in base 10, one digit per element; the digit sum then becomes a trivial loop with no extraction.
  • The incremental-update pattern - derive f(x+1) cheaply from f(x) instead of recomputing - is a powerful, broadly applicable optimisation whenever values are consumed in sequence.
  • In Python, sum(int(d) for d in str(n)) is the idiomatic, readable way to extract decimal digits; avoid hand-rolled division/modulo loops unless performance demands them.
  • Pre-allocate to a safe upper bound and keep a doubling-realloc fallback - the standard C dynamic-array discipline that avoids hot-loop allocation without risking overruns.
  • Widen operands before multiplying in C to keep integer overflow from silently corrupting results.
  • A faster language is not automatically a faster program: a naive single-digit C bignum can lose to a mature high-radix runtime on large inputs.
  • Gate expensive diagnostics and visualisations behind a flag so the default path stays lean while insight remains one command away.

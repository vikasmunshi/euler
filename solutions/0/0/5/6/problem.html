Problem 56: Powerful Digit Sum [Level 1]

Problem Statement

A googol ($10^{100}$) is a massive number: one followed by one-hundred zeros; $100^{100}$ is almost unimaginably large: one followed by two-hundred zeros. Despite their size, the sum of the digits in each number is only $1$.

Considering natural numbers of the form, $a^b$, where $a, b \lt 100$, what is the maximum digital sum?

Mathematical Complexity

The core mathematical requirement is computing a^b for bases and exponents up to 99, which produces numbers with roughly 200 digits - far beyond any native fixed-width integer type. The digit-sum function itself is O(d) where d is the number of digits, and d grows as O(b * log(a)). Iterating over all pairs in a range of size N gives an overall complexity of O(N^2 * b * log(a)). For the standard problem (N = 99) this is entirely manageable even without pruning, but the pruning heuristic I apply reduces the constant factor considerably.

The key observation that makes pruning viable is that digit sums are maximised when both the base and exponent are large. A large exponent produces a number with many digits, and a base close to (but not equal to) a power of ten keeps those digits spread across the full 0-9 range rather than collapsing to a single leading digit followed by zeros. This means the global maximum reliably lives in the upper-right corner of the (base, exponent) grid, so restricting the search to a narrow top-end window is justified empirically and confirmed by the optional heatmap visualisation I built.

Solution Approach

Brute-Force Enumeration with Top-Window Pruning (Python and C)

Both the Python and C implementations share the same algorithm, parameterised by num_digits so that stop_at = 10^num_digits is the exclusive upper bound on both base and exponent. Rather than iterating over the full N x N grid, I restrict bases to the top 100 values below stop_at and exponents to the top 10 values below stop_at. For the standard problem ( num_digits=2) this covers bases 1-99 and exponents 90-99, which is a substantial reduction from the full 99x99 grid while still capturing the region where the maximum digit sum lives.

In Python, big-integer exponentiation is handled natively, so the core loop computes base**exp, converts the result to a string, and sums the digit characters via sum(int(d) for d in str(n)). When invoked with --show, the Python solver switches to a full matrix computation over the top-100-by-100 window and produces three side-by-side Matplotlib heatmaps: one for the raw digit sum, one for the number of digits in the result, and one for the average digit value per cell. I built this primarily to confirm that the pruning window genuinely contains the global maximum and to develop intuition about where large digit sums cluster in the (base, exponent) plane.

The C implementation builds its own little-endian base-10 big-integer library with a bigint_mul_int function that multiplies in place. Rather than recomputing base^exp_start from scratch for each exponent, I accumulate the value incrementally by multiplying by the base once per exponent step, which avoids redundant work across the inner loop. Capacity is pre-allocated using a conservative upper bound of max_exp * (num_digits + 1) + 100 digits to avoid frequent reallocation. The C implementation ignores the --show flag and always runs the fast path.

Learnings

The central insight was realising that exhaustive enumeration over 99x99 = 9801 pairs - even without pruning - is trivially fast once you accept that Python handles big-integer arithmetic natively and that the digit-sum calculation is the only non-trivial per-pair cost. The pruning heuristic (top-100 bases, top-10 exponents) reduces the work further and is empirically justified by the heatmap: the maximum always appears in the upper-right corner of the grid. Building the visualisation first, before committing to a pruning strategy, turned a guess into a verified claim.

Looking at the benchmark results, the Python and C solutions behave in an interesting and instructive way. For the main case ( num_digits=2), C is faster at 0.0019 s versus 0.0091 s for Python. However, for the extra case ( num_digits=3), they are nearly identical at 0.2758 s (Python) versus 0.3325 s (C), with C actually being marginally slower on the larger input. This reflects the overhead of my hand-rolled big-integer multiply operating digit-by-digit in base 10, compared to Python's highly optimised arbitrary-precision integer library which uses a much larger internal word size and faster multiplication algorithms. It is a good reminder that reimplementing big-integer arithmetic from scratch in C does not automatically beat a mature language runtime's built-in implementation, especially for numbers in the few-hundred-digit range.

  • Python's arbitrary-precision integers remove the main algorithmic hurdle for problems involving large exponentials, and can be competitive with hand-rolled C for moderate number sizes.
  • The digit-sum idiom sum(int(d) for d in str(n)) is concise, correct for arbitrarily large n, and fast enough for numbers in the few-hundred-digit range.
  • Parameterising the problem generically via num_digits and validating against multiple test cases gives confidence that a pruning heuristic is not accidentally correct only for the primary case.
  • Interactive heatmap visualisation is a cheap and effective tool for building intuition about where extrema live in a 2-D search space before committing to a pruning strategy.
  • For big-integer work in C, the performance advantage over Python is not guaranteed - Python's runtime uses sophisticated algorithms internally that are hard to beat with a naive base-10 digit array.

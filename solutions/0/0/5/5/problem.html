Problem 55: Lychrel Numbers [Level 1]

Problem Statement

If we take $47$, reverse and add, $47 + 74 = 121$, which is palindromic.

Not all numbers produce palindromes so quickly. For example,

$$\begin{align} 349 + 943 &= 1292\\ 1292 + 2921 &= 4213\\ 4213 + 3124 &= 7337 \end{align}$$

That is, $349$ took three iterations to arrive at a palindrome.

Although no one has proved it yet, it is thought that some numbers, like $196$, never produce a palindrome. A number that never forms a palindrome through the reverse and add process is called a Lychrel number. Due to the theoretical nature of these numbers, and for the purpose of this problem, we shall assume that a number is Lychrel until proven otherwise. In addition you are given that for every number below ten-thousand, it will either (i) become a palindrome in less than fifty iterations, or, (ii) no one, with all the computing power that exists, has managed so far to map it to a palindrome. In fact, $10677$ is the first number to be shown to require over fifty iterations before producing a palindrome: $4668731596684224866951378664$ ($53$ iterations, $28$-digits).

Surprisingly, there are palindromic numbers that are themselves Lychrel numbers; the first example is $4994$.

How many Lychrel numbers are there below ten-thousand?

NOTE: Wording was modified slightly on 24 April 2007 to emphasise the theoretical nature of Lychrel numbers.

Mathematical Complexity

The reverse-and-add process is elementary to describe: repeatedly take a number, add it to its own digit-reversal, and check whether the result is a palindrome. The subtlety is that intermediate sums can grow well beyond 64-bit range - the problem notes that 10677 requires 53 iterations and produces a 28-digit result - so any implementation relying on fixed-width machine integers will silently produce wrong answers for the harder cases. The palindrome check itself is just a symmetry test on the digit sequence. A number is classified as Lychrel empirically: if no palindrome appears within the iteration budget (50 steps here), we treat it as Lychrel by assumption rather than by proof.

The overall complexity is O(N * I * D), where N is the upper limit (10,000), I is the iteration cap (50), and D is the number of digits in the largest intermediate value. D grows at most logarithmically with each addition, so it stays well below 128 digits for all inputs below 10,000 within 50 steps. Because N and I are both small fixed constants for this problem, the whole computation is effectively linear in N and finishes in well under a second.

Solution Approach

Iterative Reverse-and-Add with Early Exit (Python and C)

I implemented the same core algorithm in both Python and C. The predicate is_lychrel loops up to max_iterations times. On each iteration it forms the sum of the current value and its digit-reversal, then immediately tests whether the result is a palindrome. If a palindrome is found the function returns false and exits early. If the loop exhausts all iterations without finding one, the number is classified as Lychrel. The outer solve function counts how many integers from 1 to max_limit satisfy this predicate.

In Python, arbitrary-precision integers are built into the language, so I used string slicing with [::-1] for both digit reversal and the palindrome check. The palindrome test becomes str(number) == str(number)[::-1] and the reverse-and-add step is number += int(str(number)[::-1]). A generator expression inside sum avoids building an intermediate list. The code is very compact as a result.

The C solution required me to implement a small big-integer library, since intermediate sums can exceed 64-bit range. I represented numbers as little-endian base-10 digit arrays of up to 128 digits stored in a fixed-size BigNum struct on the stack. The function bignum_add_reverse adds the digit array to its own mirror in a single pass with carry propagation. The function bignum_is_palindrome checks symmetry by comparing digits from both ends toward the center. Advancing state between iterations is just a struct copy a = result. No dynamic memory allocation is needed anywhere in the hot path.

Learnings

The key insight that took a moment to appreciate is that palindromic numbers are not automatically non-Lychrel. The problem explicitly calls out 4994 as a palindromic Lychrel number. This means I must not short-circuit by testing whether the starting value is already a palindrome - the first reverse-and-add step must always be applied before any palindrome check. My loop structure handles this correctly because the palindrome test appears after the addition inside the loop body, so the original value is never tested directly.

Comparing the two implementations: the C version runs at roughly 0.0017 s versus 0.0183 s for Python - about an 11x speedup. The Python solution is already fast enough that it is imperceptible at this scale, but the C version demonstrates how a fixed-size stack-allocated big-integer representation avoids both dynamic allocation overhead and Python interpreter overhead. The trade-off is that I had to write the add-and-reverse routine manually in C, whereas Python handled arbitrary precision transparently.

Two implementation details from the C big-integer representation are worth remembering for future problems. First, little-endian digit storage (index 0 holds the least significant digit) makes carry propagation natural because it flows from low index to high index. Second, the palindrome check and the reverse-add both index from opposite ends of the same array, so the little-endian layout requires no index adjustment - they are symmetric about the center regardless of endianness. For any future problem involving digit-based iteration with an empirical termination criterion, the pattern of a bounded loop with early exit on success and a fall-through default return is clean and directly reusable.

Problem 59: XOR Decryption [Level 1]

Problem Statement

Each character on a computer is assigned a unique code and the preferred standard is ASCII (American Standard Code for Information Interchange). For example, uppercase A = 65, asterisk (*) = 42, and lowercase k = 107.

A modern encryption method is to take a text file, convert the bytes to ASCII, then XOR each byte with a given value, taken from a secret key. The advantage with the XOR function is that using the same encryption key on the cipher text, restores the plain text; for example, 65 XOR 42 = 107, then 107 XOR 42 = 65.

For unbreakable encryption, the key is the same length as the plain text message, and the key is made up of random bytes. The user would keep the encrypted message and the encryption key in different locations, and without both "halves", it is impossible to decrypt the message.

Unfortunately, this method is impractical for most users, so the modified method is to use a password as a key. If the password is shorter than the message, which is likely, the key is repeated cyclically throughout the message. The balance for this method is using a sufficiently long password key for security, but short enough to be memorable.

Your task has been made easy, as the encryption key consists of three lower case characters. Using 0059_cipher.txt (right click and 'Save Link/Target As...'), a file containing the encrypted ASCII codes, and the knowledge that the plain text must contain common English words, decrypt the message and find the sum of the ASCII values in the original text.

Mathematical Complexity

This problem is a statistical attack on a repeating-key XOR cipher, a classical form of polyalphabetic substitution. The structural property that makes the attack tractable is that because the key repeats with a fixed period, every ciphertext byte at position i mod key_length was XORed with exactly the same single key byte. The problem therefore decomposes into independent single-byte XOR subproblems - one per key position - rather than requiring an exhaustive search over all possible multi-character keys. Each independent slice can be cracked by scoring candidate decryptions against the known frequency of common English letters. The total search space shrinks to 26 * key_length candidates, and the scoring pass over each slice is O(n / key_length) per candidate, giving overall complexity of O(26 * n) - linear in the ciphertext length n.

XOR also has a convenient self-inverse property: applying the same key byte twice recovers the original value, so encryption and decryption are identical operations. This means no separate decryption step is needed - I XOR each candidate key byte against the slice and score the result directly. It also means that any multiple of the true key period works as the key_length parameter without breaking correctness, because over-partitioning preserves the independence property of each slice and the final summation still covers every ciphertext byte exactly once.

Solution Approach

Frequency-Analysis Slice Attack (s0 - Python and C)

Both the Python and C implementations follow the same algorithm. I partition the encrypted integer sequence into key_length interleaved slices, where slice i contains every element at index i mod key_length in the original sequence. In Python I use extended slice notation ( encrypted[i::key_length]) to build each slice in a single expression. In C I make two passes over the data - one to count the size of each slice and one to populate the slice arrays - reusing the slice_sizes array as a fill index during the second pass to avoid a separate allocation.

For each of the 26 lowercase ASCII values (97-122) I score the candidate decryption of each slice by counting occurrences of a configurable set of common English letters, passed in as the most_common_letters argument (I used eEaAnNtT for the main test). The candidate key byte producing the highest count is accepted as the true key byte for that slice position. The final answer is the sum of all ASCII values across all decrypted slices, which together cover the entire plaintext exactly once.

The Python implementation decodes each candidate slice into a string using chr and str.count, and supports a --show flag that prints the recovered plaintext by interleaving the decrypted slices with zip. The C implementation avoids string allocation entirely, scoring character by character against the common-letter set in a tight inner loop. Both implementations accept key_length and most_common_letters as configurable parameters rather than hardcoded constants, which made experimentation straightforward. Both produce the correct answer of 129448.

Learnings

The key insight was recognising that a repeating-key XOR cipher is trivially partitioned into independent single-byte streams, one per key position. Once I saw that structure, the brute-force space collapsed from exponential to linear and the rest was straightforward frequency scoring. The self-inverse property of XOR meant I could score candidate decryptions and accumulate the final ASCII sum in the same loop with no separate decryption pass.

The benchmark shows a clear language gap: the C solution runs at 0.0004 s while the Python solution takes 0.0031 s, roughly an 8x difference on this input. Both are comfortably under 10 ms, so either is practical; the C version is faster mainly because it avoids string object creation and Python interpreter overhead in the inner scoring loop.

A few implementation details worth remembering:

  • Making most_common_letters and key_length configurable parameters rather than hardcoded constants let me experiment with different scoring alphabets and key periods without touching the solver logic.
  • Any multiple of the true key period works as the key_length argument, because over-partitioning does not break the independence of each slice - slices 0, 3, 6 all contain characters encrypted with the same first key byte, so they score identically to the single slice that would result from using key_length=3.
  • The --show flag for printing the recovered plaintext was invaluable during development for visually verifying the key before trusting the numeric answer.
  • In the C version, reusing the slice_sizes array as a fill index during slice construction avoids an extra allocation and keeps the code compact.
  • For future problems involving repeating-key ciphers, the general recipe is: determine the key period, slice the ciphertext by that period, score each slice with a statistical measure, and combine the best-scoring key bytes. This scales to longer keys and different alphabets with no change to the algorithm structure.

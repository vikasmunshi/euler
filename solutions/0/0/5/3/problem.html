Problem 53: Combinatoric Selections [Level 1]

Problem Statement

There are exactly ten ways of selecting three from five, 12345:

123, 124, 125, 134, 135, 145, 234, 235, 245, and 345

In combinatorics, we use the notation, $\displaystyle \binom 5 3 = 10$.

In general, $\displaystyle \binom n r = \dfrac{n!}{r!(n-r)!}$, where $r \le n$, $n! = n \times (n-1) \times ... \times 3 \times 2 \times 1$, and $0! = 1$.

It is not until $n = 23$, that a value exceeds one-million: $\displaystyle \binom {23} {10} = 1144066$.

How many, not necessarily distinct, values of $\displaystyle \binom n r$ for $1 \le n \le 100$, are greater than one-million?

Test Cases

  • main: solve(max_n=100, threshold=1000000) → 4075

Mathematical Complexity

The problem rests on three properties of binomial coefficients that, taken together, make a brute-force factorial approach entirely unnecessary. First, the multiplicative row recurrence C(n, r+1) = C(n, r) * (n - r) / (r + 1) lets me walk along any row of Pascal's triangle using only integer arithmetic, since the division always comes out exact. Second, the symmetry property C(n, r) = C(n, n-r) means each row is a palindrome, so I only need to inspect values up to r = n // 2. Third, the unimodal property means values in each row rise monotonically from r = 0 to the central peak and then fall again. Combining all three, the moment I find the first r in the ascending half where C(n, r) exceeds the threshold, I know every remaining value in that row also exceeds it, and I can count the entire remaining block in O(1) using the formula n - 2*r + 1, which accounts for both the unvisited ascending positions and their symmetric partners simultaneously.

The brute-force upper bound is O(n^2) pairs for 1 <= n <= 100, giving at most 5050 evaluations. Because of the early-exit strategy the inner loop terminates as soon as the threshold is crossed, so in practice far fewer evaluations occur - particularly for large n and a high threshold where the crossing point is reached very early in the row. Worst-case complexity remains O(n^2), but the average-case behaviour is substantially sub-quadratic.

Solution Approach

Row-wise Incremental Traversal with Symmetry Early Exit (Python and C)

I implemented the same algorithm in both Python ( p0053_s0.py) and C ( p0053_s0_c). For each n from 1 to max_n, I initialise c = 1 (representing C(n, 0)) and walk r from 0 up to n // 2, updating via c = c * (n - r) // (r + 1) at each step. While c <= threshold I continue walking. The moment c exceeds the threshold, I add n - 2*r + 1 to the running count and break immediately to the next row, avoiding any evaluation of the remaining large central values. This approach requires no factorial storage and no floating-point arithmetic.

The Python version uses keyword arguments ( max_n, threshold) and Python's arbitrarily large integers, which means overflow is never a concern. The C version reads positional command-line arguments using atoi and atoll, and uses long long throughout. For this problem size the intermediate values of c stay well within long long range before the early exit fires, so no special overflow handling is needed. Both versions pass both test cases, returning 4724 for the dev case (threshold 100) and 4075 for the main case (threshold 1 000 000).

Learnings

The key insight was that I did not need factorials at all. The multiplicative recurrence walks any row of Pascal's triangle using only small integer operations, and the combination of unimodality and symmetry transforms what could be a full inner-loop count into a single arithmetic expression the moment the threshold is crossed. The formula n - 2*r + 1 was the satisfying "aha" step: it simultaneously accounts for all unvisited positions in the ascending half and their mirrored counterparts in the descending half.

Only one algorithmic approach exists here, so there is no meaningful performance comparison to make. Both the Python and C versions complete in effectively zero measurable time for this problem size, which reflects how little work the early-exit strategy actually does in practice. The dev test case with threshold 100 is a genuinely useful complement to the main case: because the threshold is much lower, the crossing point is reached later in each row, forcing the counting formula to fire across a wider range of r values and exercising more of the logic.

  • The recurrence C(n, r+1) = C(n, r) * (n - r) / (r + 1) always divides exactly and is far cheaper than computing or storing factorials.
  • Combining the unimodal and symmetry properties of Pascal's triangle turns a linear inner loop into an O(1) bulk count the moment a threshold is crossed - a pattern worth keeping as a mental template for any similar threshold-on-rows problem.
  • Walking a row of Pascal's triangle incrementally via the multiplicative recurrence is a general primitive that reappears in many combinatorics problems; it is worth internalising.
  • Pairing a high-threshold main test case with a low-threshold dev case is a good testing discipline: the low threshold forces later exits and exercises more of the counting code path, while the high threshold checks that the early-exit optimisation fires correctly under realistic conditions.

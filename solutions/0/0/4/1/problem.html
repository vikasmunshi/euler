Problem 41: Pandigital Prime [Level 1]

Problem Statement

We shall say that an $n$-digit number is pandigital if it makes use of all the digits $1$ to $n$ exactly once. For example, $2143$ is a $4$-digit pandigital and is also prime.

What is the largest $n$-digit pandigital prime that exists?

Mathematical Complexity

The key insight is a divisibility pre-filter based on digit sums. For an n-digit pandigital number, the digit sum is always exactly n(n+1)/2, regardless of how the digits are arranged. If that sum is divisible by 3, then every permutation of those digits is divisible by 3 and therefore composite. Checking each length: n=9 gives sum 45, n=8 gives 36, n=7 gives 28, n=6 gives 21, n=5 gives 15, n=4 gives 10, n=3 gives 6, n=2 gives 3, n=1 gives 1. Only n=4 (sum 10) and n=7 (sum 28) are not multiples of 3. The single-digit n=1 case yields only the number 1, which is not prime. This pruning eliminates lengths 2, 3, 5, 6, 8, and 9 entirely before generating a single candidate.

After pruning, the worst-case search space is 7! + 4! = 5040 + 24 = 5064 permutations, though in practice the search terminates much earlier because I iterate in descending order and stop at the first prime found. Each primality test uses trial division costing O(sqrt(N)), where N is at most a 7-digit number (sqrt(9999999) is roughly 3162 divisions). Total work is therefore a few thousand trial-division checks at most, placing the entire solution comfortably in the sub-millisecond range.

Solution Approach

Descending Permutation Search with Digit-Sum Pruning (s0 - Python and C)

Both the Python and C implementations share the same core algorithm: generate permutations of the valid digit sets in strictly descending order and return the first candidate that passes a trial-division primality test. Iterating largest-first means the very first prime encountered is guaranteed to be the answer, giving a clean early-exit search with no need to collect and sort results afterward.

In the Python version, I use itertools.permutations on a reversed digit tuple inside gen_n_digit_pandigital_numbers. Because itertools.permutations yields in lexicographic order relative to its input, reversing the input tuple causes the output to arrive in descending numeric order. I wrap the search in a lazy generator expression over lengths (7, 4) and call next() to stop at the first prime found. I also defined a fast_is_prime alias that simply delegates to is_prime, leaving a hook open for a future drop-in replacement such as Miller-Rabin if the candidate space ever grew.

The C version implements the same descent explicitly using a prev_permutation function, which is the mirror image of the standard next-permutation algorithm: find the rightmost index where the element is greater than its right neighbour, find the largest element to its right that is smaller, swap them, then reverse the suffix. The digit array is initialised in descending order (e.g. 7, 6, 5, 4, 3, 2, 1) and prev_permutation is called in a do-while loop until it signals that all permutations have been visited. A shared helper digits_to_number converts the digit array to an integer at each step. The C version is entirely stack-based with no dynamic allocation, contributing to its near-zero measured runtime.

Learnings

The central insight was recognising that the divisibility-by-3 rule applied to the digit sum n(n+1)/2 collapses the problem from a broad search over hundreds of thousands of permutations down to just two candidate lengths. Without this observation the problem looks expensive; with it the answer falls out almost instantly. Checking digit sums for divisibility before generating any permutations is a broadly useful pattern whenever combinatorial enumeration intersects with divisibility constraints.

Generating permutations in reverse lexicographic order combined with lazy evaluation via next() in Python or an early return inside the loop in C gives a clean early-exit maximum search. This avoids materialising the full result set and sorting it afterward, which matters when the search space is even moderately large.

Only one algorithm is implemented (identical in both languages), so there is no algorithmic trade-off to compare. The benchmark results confirm the approach is essentially free: the Python solution measures at 0.0001 s and the C solution at 0.0000 s. The C version is faster purely due to the absence of interpreter overhead, but Python is already well within any practical time budget. Trial division is entirely sufficient for 7-digit candidates; there is no motivation to introduce Miller-Rabin here.

  • Digit-sum divisibility by 3 is a powerful pre-filter - compute n(n+1)/2 and check divisibility before generating any permutations.
  • Reversing the input to itertools.permutations produces descending-order output, enabling a simple early-exit maximum search via next().
  • The prev_permutation pattern (find rightmost descent, find floor element to its right, swap, reverse suffix) is the mirror image of next_permutation and equally useful for descending enumeration in C.
  • For primes up to 7 digits, trial division is fast enough; Miller-Rabin would only matter if candidate sizes or search spaces were significantly larger.
  • Combining a mathematical pre-filter with an ordered lazy search is a broadly reusable pattern for combinatorial prime problems.

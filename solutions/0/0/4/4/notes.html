Problem Analysis

A pentagonal number is defined by the closed form P(n) = n(3n-1)/2, generating the sequence 1, 5, 12, 22, 35, 51, ... for positive integers n. The problem asks me to find two pentagonal numbers whose sum and difference are both themselves pentagonal, and among all such pairs to minimise the difference D. The crux is being able to answer one question very fast: "is this integer pentagonal?" If that test is constant time, then a straightforward double loop over candidate pairs becomes practical.

The test follows from inverting the defining quadratic. Setting n(3n-1)/2 = m and applying the quadratic formula gives n = (1 + sqrt(1 + 24m)) / 6. The integer m is pentagonal precisely when this expression is a positive whole number. That is a single square root plus a few arithmetic operations - O(1), with no table to build or search. The second decisive observation is about search order. If I fix the larger index i and scan the smaller index j downward, the difference P(i)-P(j) grows monotonically as j shrinks, and the smallest possible difference at outer index i is P(i)-P(i-1), which itself grows with i. So the first valid pair I encounter, with i ascending and j descending, is provably the global minimum. The work is O(i_answer^2) constant-time tests, where i_answer is the outer index at which the answer surfaces - a few thousand in practice.

Solution Approach

Ascending-Outer, Descending-Inner Pair Search with an Algebraic Membership Test

Both the Python (p0044_s0.py) and C (p0044_s0.c) files implement the same algorithm. The outer loop drives i upward from 1 with no fixed bound. For each i I compute p_i once via the closed-form helper, then run an inner loop with j descending from i-1 to 1. For each pair I form the difference p_i - p_j and the sum p_i + p_j and test both with the inverse-formula membership check. The moment both pass, I return the difference.

The descending inner loop is the design choice that justifies the early return. Scanning j from large to small means the first valid pair found for a given i has the smallest difference achievable at that i. Combined with the ascending outer loop and the monotonicity of P(i)-P(i-1) in i, the very first valid pair found anywhere is the global minimum, so there is no reason to keep searching.

The membership test differs subtly between the languages. In Python I compute n = (1 + sqrt(1 + 24*m)) / 6 and call float.is_integer() on it, relying on the float landing exactly on a whole value - acceptable for these magnitudes. In C I take the defensive floating-point route: I compute the same candidate, snap it to the nearest integer with round(), then recompute nth_pentagonal(ni) in exact integer arithmetic and compare against the original m, additionally guarding ni > 0. This round-then-verify pattern is immune to a value reading as 4.9999999999 instead of 5.0.

Programming Techniques

Inverting a Quadratic for O(1) Set Membership

Rather than maintaining a hash set or sorted list of pentagonal numbers and querying it, I invert the defining formula algebraically with the quadratic formula. Whenever a sequence has an easily invertible closed form, this is worth doing: it turns a membership query into one arithmetic expression with no memory allocation and no data structure that grows as the numbers do. Reach for this any time you are tempted to precompute a table that you could instead recognise by inverting a formula.

Walrus Operator for Loop-and-Increment in Python

The Python outer loop is written as while i := (i + 1), using the assignment expression (the "walrus operator") from Python 3.8. It increments i and tests it as the loop condition in one expression; since i is always positive after the first increment, the condition is always truthy, giving an effectively infinite loop exited only by the return. It is a compact idiom for "loop forever while updating a counter", though itertools.count(1) would be equally idiomatic and arguably clearer.

Hoisting the Loop-Invariant Outer Term

Both implementations compute p_i once before the inner loop instead of recomputing it on every inner iteration. This is classic loop-invariant code motion: any expression whose value does not change across iterations belongs outside the loop. p_j must stay inside because j changes, but p_i is constant across the inner sweep. A C compiler may hoist this automatically; the Python interpreter will not, so doing it explicitly matters most there.

Round-and-Verify for Floating-Point Integer Tests in C

When checking whether a square root lands on an integer, a naive floor(x) == x comparison can fail to floating-point rounding error. The C solution instead uses round(n) to get the nearest integer candidate, recomputes the pentagonal value in exact integer arithmetic, and compares it to m. This eliminates both false positives and false negatives at integer boundaries. Adopt this pattern whenever a floating-point approximation must satisfy an exact integer constraint.

Early Return on a Provably Optimal First Hit

Because the inner loop descends and the outer loop ascends, the first pair satisfying both conditions is provably minimal. I exploit this by returning at once rather than continuing to search. Recognising when an ordered enumeration makes the first solution optimal - and stopping there - converts what looks like an exhaustive search into an early-terminating one.

Key Takeaways

The unlocking insight is that pentagonality is testable in constant time by inverting the quadratic, removing any need to store values, and that an enumeration order in which differences grow monotonically lets me return the first valid pair as the guaranteed global minimum.

Both files implement the same algorithm, so the benchmark gap is pure language overhead: the C build runs in about 0.0133 s while the Python version takes about 0.3718 s, roughly a 28x difference. That ratio is typical for tight numerical loops with no heavy data structures - per-operation interpreter overhead dominates. When such a loop is genuinely the bottleneck, a compiled rewrite is the lever to pull.

  • When a sequence is defined by an invertible closed-form formula, invert it to get an O(1) membership test instead of building a lookup table.
  • Choose your enumeration order so that the first valid result is provably optimal, then return immediately.
  • Hoist loop-invariant subexpressions out of inner loops, especially in interpreted languages where the runtime will not do it for you.
  • Convert floating-point results to integers with round-then-verify in exact arithmetic, never with floating-point equality.
  • For pure arithmetic hot loops, expect a large constant-factor speedup from a compiled language - reserve that effort for when the loop is the bottleneck.

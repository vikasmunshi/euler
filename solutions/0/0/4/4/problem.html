Problem 44: Pentagon Numbers [Level 2]

Problem Statement

Pentagonal numbers are generated by the formula, $P_n=n(3n-1)/2$. The first ten pentagonal numbers are: $$1, 5, 12, 22, 35, 51, 70, 92, 117, 145, \dots$$

It can be seen that $P_4 + P_7 = 22 + 70 = 92 = P_8$. However, their difference, $70 - 22 = 48$, is not pentagonal.

Find the pair of pentagonal numbers, $P_j$ and $P_k$, for which their sum and difference are pentagonal and $D = |P_k - P_j|$ is minimised; what is the value of $D$?

Mathematical Complexity

The central mathematical tool is inverting the pentagonal number formula. Given P_n = n(3n-1)/2, testing whether an arbitrary integer m is pentagonal means solving 3n^2 - n - 2m = 0 for a positive integer root. The quadratic formula gives n = (1 + sqrt(1 + 24m)) / 6, so m is pentagonal if and only if this expression evaluates to a positive integer. That yields an O(1) membership test, which is the key to keeping the overall search tractable.

The search itself is a double loop over pairs of pentagonal indices, giving a worst-case O(k^2) iteration count where k is the outer index at the winning pair. The two simultaneous constraints - sum pentagonal AND difference pentagonal - are so restrictive that valid pairs are extremely sparse, and the answer appears at a moderate outer index. An important structural observation is that the gap between consecutive pentagonal numbers, P_n - P_{n-1} = 3n - 2, is strictly increasing with n. This means the smallest difference achievable for a pair whose larger element is P_i grows with i, which mathematically justifies returning immediately on the first valid pair found rather than continuing the search.

Solution Approach

Nested Pentagonal Pair Search with Inversion Test (s0 - Python and C)

I implemented the same algorithm in both Python and C. An outer loop advances index i from 1 upward, computing P_i. An inner loop walks index j downward from i-1 to 1, computing P_j and checking whether both P_i - P_j and P_i + P_j pass the pentagonal inversion test. Iterating j downward means differences are examined in ascending order for a fixed i, so the first hit for any outer index is the smallest possible difference anchored at that i. Because smaller-difference candidates within a given i are examined first, and because the minimum achievable difference strictly increases with i, the very first globally valid pair found is guaranteed to have the minimum D, and the function returns immediately.

The Python version uses math.sqrt and float.is_integer() for the membership test, which is concise and reliable for numbers of this magnitude within double-precision range. The C version uses sqrt from math.h followed by round and then a back-verification step - it rounds the floating-point result to the nearest integer and confirms that reapplying the forward formula recovers m exactly. This extra verification guards against floating-point rounding errors that could cause a false positive, and is a more defensive pattern than the Python is_integer() call alone. Both versions compute P_n with integer arithmetic via n*(3n-1)/2, using 64-bit integers in C and Python's arbitrary-precision integers natively.

Learnings

The key insight was recognising that inverting the pentagonal formula into a closed-form O(1) check via the quadratic formula makes the entire double loop tractable. Without that, every candidate sum and difference would need either a binary search over a precomputed list or a linear membership scan, both of which add a logarithmic or linear factor per iteration and would also require maintaining a growing set in memory.

The second important realisation was that the loop ordering is not arbitrary - it is mathematically load-bearing. Because P_n - P_{n-1} = 3n - 2 grows strictly with n, the minimum difference achievable at outer index i is larger than any difference achievable at outer index i-1. Early termination on the first valid pair is therefore not a heuristic shortcut but a provably correct stop condition.

Since only one algorithm is used here, the comparison between the Python and C files is purely a language performance question. The benchmark results show the C solution completes in about 0.016 s versus roughly 0.54 s for Python - approximately a 33x speedup. This is entirely expected given the tight inner loop over floating-point arithmetic and integer operations, which C executes with much lower per-iteration overhead than the CPython interpreter.

A technique worth carrying forward to future problems: whenever a problem asks about membership in a quadratically generated sequence, derive the closed-form inverse via the quadratic formula rather than maintaining a lookup set. This keeps memory O(1) and the test O(1) at the cost of a single floating-point square root, which is acceptable for numbers that fit comfortably in double precision. For much larger numbers, an integer Newton-Raphson square root with a back-verification step - as used in the C version here - would be the more robust approach.

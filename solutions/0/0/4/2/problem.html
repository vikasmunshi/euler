Problem 42: Coded Triangle Numbers [Level 0]

Problem Statement

The $n$ th term of the sequence of triangle numbers is given by, $t_n = \frac12n(n+1)$; so the first ten triangle numbers are: $$1, 3, 6, 10, 15, 21, 28, 36, 45, 55, \dots$$

By converting each letter in a word to a number corresponding to its alphabetical position and adding these values we form a word value. For example, the word value for SKY is $19 + 11 + 25 = 55 = t_{10}$. If the word value is a triangle number then we shall call the word a triangle word.

Using words.txt (right click and 'Save Link/Target As...'), a 16K text file containing nearly two-thousand common English words, how many are triangle words?

Mathematical Complexity

The central mathematical idea is deriving a closed-form membership test for triangle numbers. Triangle numbers follow t_n = n(n+1)/2. Treating this as a quadratic in n and applying the quadratic formula yields n = (-1 + sqrt(1 + 8m)) / 2, so a positive integer m is triangular if and only if 8m + 1 is a perfect square. This gives an O(1) check per word score and avoids pre-generating any lookup table, regardless of how large scores could theoretically grow.

The overall algorithm is O(W * L) where W is the number of words (roughly two thousand) and L is the average word length, since every character must be scored and summed. Both quantities are small and fixed by the input file, so the problem is effectively O(1) in practice - the runtime is dominated by file I/O and string splitting rather than any arithmetically interesting computation.

Solution Approach

Closed-Form Triangle-Number Test with Alphabetic Scoring (Python and C)

I implemented the same algorithm in both Python and C. The word file is read in full, split on commas, and each token has its surrounding quotes stripped. Every letter is mapped to its alphabetical position - in Python via ord(c) - 64, in C by checking whether the character falls in the upper- or lower-case ASCII range and subtracting the appropriate base. The resulting integer score is then tested for triangle-number membership using the discriminant condition described above.

In the Python version I used ((8 * n + 1) ** 0.5).is_integer() as the membership test. This works reliably here because all word scores are small enough that the floating-point square root is exact. I decorated word_to_num with @functools.lru_cache as a lightweight guard against duplicate words, though the file contains no meaningful repetition in practice. The final count is produced with a generator expression passed directly to sum, keeping memory usage minimal.

In the C version I used sqrt from math.h combined with round to obtain an integer candidate, then verified with an exact integer squaring check: sq_int * sq_int == 8 * n + 1. This avoids relying on floating-point integrality and is the safer pattern in C where there is no equivalent of Python's .is_integer(). File loading uses fread into a heap buffer and strtok_r for comma splitting. Path resolution reads the executable path via /proc/self/exe and derives a sibling resources/ directory from it, keeping the binary portable without requiring a specific working directory.

Learnings

The key insight was the algebraic trick for O(1) triangle-number membership testing. Rather than pre-generating all triangle numbers up to the maximum possible word score, I derived the inverse condition from the quadratic formula: a number m is triangular if and only if 8m + 1 is a perfect square. This pattern is worth memorising because it extends directly to other figurate numbers - for pentagonal numbers the analogous condition is that 24m + 1 is a perfect square, and hexagonal numbers have a similar closed-form test, both of which appear in later Project Euler problems.

The benchmark results show the C solution running at roughly 0.0001 s versus 0.0007 s for Python - about seven times faster in this run - though both are so fast in absolute terms that the difference is irrelevant in practice. The Python version is slightly more concise thanks to the .is_integer() call and the generator expression. The C version demonstrates the more conservative approach to floating-point square roots that I prefer in lower-level code: compute a rounded integer candidate and verify it with exact integer arithmetic rather than trusting a floating-point integrality check directly. I would carry forward both the general habit of deriving a closed-form integrality condition from the inverse of a sequence formula, and the round-and-verify pattern whenever correctness depends on exact results in C.

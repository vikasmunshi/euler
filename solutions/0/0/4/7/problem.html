Problem 47: Distinct Primes Factors [Level 1]

Problem Statement

The first two consecutive numbers to have two distinct prime factors are:

$$\begin{align} 14 &= 2 \times 7\\ 15 &= 3 \times 5. \end{align}$$

The first three consecutive numbers to have three distinct prime factors are:

$$\begin{align} 644 &= 2^2 \times 7 \times 23\\ 645 &= 3 \times 5 \times 43\\ 646 &= 2 \times 17 \times 19. \end{align}$$

Find the first four consecutive integers to have four distinct prime factors each. What is the first of these numbers?

Mathematical Complexity

The central concept is counting the number of distinct prime factors of an integer, ignoring multiplicity. For any integer n, trial division from 2 up to sqrt(n) identifies every prime divisor, each time dividing out the factor completely before moving on, so each factorization costs O(sqrt(n)). The search then scans consecutive integers looking for a run of exactly k numbers that each have exactly k distinct prime factors. Scanning up to the answer value N gives a total complexity of roughly O(N * sqrt(N)). The inner window check of length k short-circuits on the first failure, so in practice most candidates are rejected after only one or two factorizations.

The problem sits at the intersection of multiplicative number theory and consecutive-run search. Integers with exactly four distinct prime factors are moderately rare - the required run of four does not appear until 134043 - but the answer is well within reach of naive per-number trial division with no sieve precomputation. A smallest-prime-factor sieve would reduce each factorization from O(sqrt(n)) to O(log n), but that optimisation is unnecessary here since the answer is in the low hundreds of thousands.

Solution Approach

Trial Division with Sequential Window Scan (s0 - Python and C)

Both the Python and C solutions implement the same algorithm. I wrote a prime_factor_count helper based on trial division: starting from divisor 2, I divide out each factor completely to strip multiplicity and increment a counter each time a new prime divisor is found, then add 1 to the count after the loop if any remainder greater than 1 remains. This correctly counts distinct prime factors in O(sqrt(n)) time.

The main search loop iterates candidate starting numbers from 2 upward and checks whether all k consecutive integers beginning at that candidate each have exactly k distinct prime factors, breaking immediately on the first mismatch. In C I inlined the factor count directly into a single prime_factor_count function with no heap allocation. In Python I built on a richer prime_factorization function that returns a tuple of (prime, exponent) pairs, with prime_factor_count delegating to it via len. The Python window check uses an any expression inside a generator passed to next, which short-circuits cleanly without materialising any intermediate list.

Both versions parameterise solve on a single value k that simultaneously controls the required run length and the required distinct factor count, so exactly the same logic handles all three test cases without modification.

Learnings

The key insight was recognising that the run length and the required factor count are the same value k, making the problem fully parameterisable on one integer. Writing a generic solve(n=k) and validating it against all three examples - k=2 giving 14, k=3 giving 644, k=4 giving 134043 - gave me high confidence before submitting.

The benchmark results illustrate the performance gap between Python and C on tight numerical loops. Both implement the same algorithm, but C finishes the main case in roughly 0.029 s while Python takes about 0.749 s - approximately a 26x speedup - purely from the language overhead of interpreted execution and object allocation. For scan-heavy problems where the answer is in the hundreds of thousands, the C version pays off significantly even without any algorithmic change.

A few techniques worth noting for future problems: the any(...) short-circuit idiom inside a generator expression is an elegant Python pattern for sliding-window predicates without list allocation. Also, stripping multiplicity inside the factorization loop by dividing out a prime completely before incrementing the counter is the cleanest way to count distinct factors without a separate deduplication step. Finally, if this problem were extended to k=5 or beyond with answers in the millions, switching to a smallest-prime-factor sieve would be the natural next optimisation, reducing each factorization to O(log n) and cutting overall runtime proportionally.

Problem Analysis

This problem rests on prime factorization - decomposing an integer into the primes whose product equals it. The decisive subtlety is the word distinct: we count only how many different primes divide a number, ignoring multiplicity. Thus 644 = 22 x 7 x 23 has three distinct prime factors (2, 7, 23) even though the prime 2 appears squared. The task is to find the first integer that begins a run of four consecutive integers, each having exactly four distinct prime factors.

The workhorse here is trial division: to factor n, I test candidate divisors d from 2 upward, and each time d divides n I record it and strip out every copy. The first efficiency observation is that I only need to test divisors up to the square root of n - if n had a factor larger than its square root, the cofactor would be smaller than the square root and would already have been found, so whatever survives the loop above 1 is itself a prime factor. The second, equally important observation is that no sieve or precomputed table is needed: the answer (134,043) is small, so a plain linear scan suffices. Counting distinct prime factors of n costs O(sqrt(n)), and scanning candidates up to the answer A while testing windows of length n gives an overall cost of about O(A * sqrt(A)) - a few million operations, which is trivial at this scale.

Solution Approach

Linear Scan with Trial Division (Python and C)

Both files implement the same algorithm, so I describe them together. There are two parts: a function that counts the distinct prime factors of one integer, and a search loop that finds the first window of n consecutive qualifying integers.

The counter uses trial division with in-place reduction. I begin at d = 2 and loop while d*d <= n. When d divides n, I increment the count once, then drain out every copy of that factor with an inner loop (while n % d == 0: n //= d). Draining all copies is precisely what makes a separate primality test unnecessary: by the time d reaches a composite value such as 4, the factor 2 has already been fully stripped, so 4 can never divide the reduced n. Every d that successfully divides is therefore guaranteed prime. After the loop, if n still exceeds 1, that remainder is a prime larger than the square root of the original, so I count it as well. The result is exactly the number of distinct prime factors.

The search loop starts at number = 2 and increments by 1. For each starting position it checks whether number, number+1, ..., number+(n-1) all have exactly n distinct prime factors, using early exit: the instant one element fails, the window is abandoned. In C this is a flag (all_match) plus a break; in Python it is not any(prime_factor_count(number + i) != n for i in range(0, n)), where any short-circuits identically.

Both programs take n as a command-line argument, so a single executable solves all three test cases (n=2, n=3, n=4) without modification. The Python version adds one layer of structure: prime_factorization returns a tuple of (prime, exponent) pairs and prime_factor_count simply returns its length. That is slightly more than this problem needs, but it leaves a reusable building block. The C version counts inline and omits the tuple machinery that C does not express naturally, which is marginally simpler and faster.

Programming Techniques

Trial Division with In-Place Reduction

The inner loop that exhaustively divides out each factor is the load-bearing technique of this solution. By shrinking n as factors are removed, I guarantee that subsequent successful divisions occur only for primes - composites are skipped automatically because their prime constituents were already stripped. This eliminates any need to keep a prime list or run a primality check, keeping the function entirely self-contained. Reach for this whenever you need a factorization or a factor count for numbers in a modest range.

Short-Circuit Window Evaluation

Most candidate windows fail on their very first element, so testing all n elements every time would waste work. The C code sets a flag and breaks; the Python code uses any over a generator, which stops at the first truthy value. Short-circuiting reduces the expected per-candidate cost from n factor counts to roughly one. This bail-out-as-soon-as-the-answer-is-known pattern applies any time a test is a conjunction or disjunction of sub-tests.

Declarative Search with itertools.count and next

The Python solve expresses the search as next(number for number in itertools.count(2) if ...). itertools.count yields an unbounded stream of integers, and next pulls the first that satisfies the predicate. This reads as "the first number such that the condition holds," directly mirroring the problem statement and avoiding manual counters and termination logic - a common source of off-by-one bugs. Use it whenever you are searching for the first element of a possibly infinite sequence meeting a condition.

Separation of Factorization and Counting

The Python code splits prime_factorization (which yields primes with exponents) from prime_factor_count (which discards the exponents). This honors the single-responsibility principle and leaves a reusable component for future problems needing multiplicities. The C code collapses the two because C lacks convenient variable-length tuples and the generality is not required here. Both choices are deliberate: favor reusability where it is cheap, favor simplicity where the abstraction would only add boilerplate.

Parameterization via Command-Line Arguments

Passing n at runtime rather than hard-coding 4 lets one executable cover every test case. This is good engineering hygiene: it supports automated regression testing across inputs and lets me validate correctness on small, hand-checkable cases (n=2 yields 14, n=3 yields 644) before trusting the program on the large one.

Key Takeaways

The unlocking insight is that counting distinct prime factors by trial division with in-place reduction is both simple and entirely sufficient: no sieve is needed because the search range is small and contiguous and the per-number cost of O(sqrt(n)) is tiny. The two implementations share an algorithm; the difference is purely language overhead. On the main case the C binary runs in about 0.0210 s while Python takes about 0.4726 s, roughly a 22x gap that is typical of compiled versus interpreted tight numeric loops. At this scale Python is perfectly acceptable; for a far larger search range, C or a sieve-based factor-count table would become the better choice.

  • Trial division with exhaustive in-place factor removal counts distinct primes correctly without any separate primality test or precomputed prime list.
  • Only test divisors up to the square root; whatever remains above 1 afterward is itself a prime factor.
  • Short-circuit a conjunction of checks - bail out on the first failure to slash the average cost when most cases fail early.
  • Separate "compute the full structure" from "extract the summary" when the language makes reuse cheap; collapse them when the abstraction only adds noise.
  • Express "find the first element satisfying a predicate" with an unbounded stream plus a first-match pull to avoid manual loop bookkeeping.
  • Parameterize on input instead of hard-coding constants so one program serves all test cases and supports automated testing.
  • Before building a complex data structure, estimate answer size times per-step cost - if it fits your time budget, a plain linear scan is the simplest correct choice.

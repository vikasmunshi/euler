Problem Analysis

The task is to find the last ten decimal digits of the self-power sum 11 + 22 + ... + 10001000. The pivotal observation is that "the last ten digits" of an integer is precisely that integer reduced modulo 1010. Modular arithmetic is the arithmetic of remainders: rather than tracking a full value, I track only its remainder after division by a fixed modulus m. The crucial property is that the modulus distributes over both addition and multiplication, so (a + b) mod m = ((a mod m) + (b mod m)) mod m, and the same holds for products. This means I never need the true sum at all - the largest term, 10001000, has roughly 3000 decimal digits - because I can keep every intermediate value below 1010.

The problem then reduces to computing ii mod 1010 for each i from 1 to 1000. Multiplying i by itself i times would cost O(i) multiplications per term. The standard speedup is modular exponentiation via exponentiation by squaring, which decomposes the exponent into its binary digits and computes the power in O(log i) multiplications by repeatedly squaring the base. Across all 1000 terms the total work is O(N log N) - effectively instantaneous - and every reduced value fits comfortably in a machine integer.

Solution Approach

Modular Exponentiation with a Reduced Running Sum

Both the Python and C files implement the same algorithm. I iterate i from 1 to the limit inclusive, compute the term ii mod 1010 by fast modular exponentiation, add it to a running total, and reduce that total modulo 1010 after every addition. The final running total is the answer.

In Python I rely on the built-in three-argument pow(i, i, modulo). This is not the same as pow(i, i) % modulo; the three-argument form performs the modular reduction inside the squaring loop, so the full multi-thousand-digit power is never materialised. Python integers are arbitrary precision, so there is no overflow to consider, and the routine runs in optimised C under the hood.

In C I wrote mod_pow by hand as a classic binary exponentiation by squaring loop: while the exponent is positive, I test its lowest bit with exp & 1 to decide whether to fold the current base into the result, then square the base and right-shift the exponent. The subtlety is overflow. With a modulus of 1010, two reduced operands can each approach 1010, so their product can reach about 1020 - well beyond the roughly 9.2x1018 ceiling of a signed 64-bit long long. I avoid this by casting one operand to __int128 before each multiply, computing the product at 128-bit precision, and immediately reducing modulo mod back into 64-bit range. The 128-bit value is purely transient and never stored across iterations.

Both versions accept the limit as a runtime argument rather than hard-coding 1000, so the same binary or script handles the small development case (limit 10, answer 405071317) and the full case (limit 1000, answer 9110846700) through the identical code path.

Programming Techniques

Three-argument pow() in Python

Python's built-in pow(base, exp, mod) is a single call that performs modular exponentiation directly and never builds the full power. I always prefer it over pow(base, exp) % mod when the exponent is large: the former runs in O(log exp) multiplications on already-small numbers, whereas the latter first constructs an enormous integer. This is the canonical Python idiom for any number-theory task involving large modular powers.

Using __int128 as a transient widening cast in C

When two values near 1010 are multiplied, the exact product needs more than 64 bits, and a plain long long multiply wraps silently to a wrong answer. The GCC/Clang __int128 extension supplies a 128-bit integer type large enough to hold the product exactly. The pattern (__int128)a * b % mod widens one operand so the multiply happens at 128 bits, then the modulo reduces the result straight back into 64-bit range. No 128-bit value persists - it exists only for the duration of one expression. I reach for this whenever I do modular arithmetic in C with a modulus above roughly 3x109; it is far simpler than pulling in a big-integer library.

Modular accumulation with a reduced invariant

Rather than summing every term and reducing once at the end, I reduce after each addition with result = (result + term) % modulo. Since both term and result are always below the modulus, their sum is at most 2 * modulo - 2, which fits easily in a 64-bit integer. Maintaining the invariant "every accumulator is already fully reduced" is a general safeguard that guarantees no intermediate value can overflow.

Binary exponentiation loop

The exponentiation by squaring loop is a pattern worth memorising. Its invariant is that the product of the accumulated result and base raised to the remaining exp always equals the original power. Each iteration optionally folds in the base when the low exponent bit is set, then squares the base and halves the exponent, terminating after O(log exp) steps. Because it relies only on an associative operation, the same skeleton extends to matrix exponentiation, polynomial powers, and other domains.

Parameterising the problem size

Both solutions accept the limit as a command-line argument. This lets me validate the code on the small, hand-checkable case (limit 10) using the exact same code path as the full run, and it lets the benchmark harness exercise both cases without editing source. Parameterising sizes instead of hard-coding constants is a low-cost habit that pays off in testability.

Key Takeaways

The unlocking insight is that any "last N digits" question is really a request for a value modulo 10N, which immediately bounds every intermediate quantity and removes the need for big-integer arithmetic. The only remaining work is computing large powers efficiently, and the answer is always modular exponentiation.

The Python and C versions implement the identical algorithm. On the main case the C build registers 0.0000 s against roughly 0.0007 s for Python - the C version is faster thanks to compiled arithmetic and the native __int128 multiply, while Python's three-argument pow keeps it close and far more concise. Both are effectively instant at this scale, so the choice here is about clarity; the C technique remains valuable practice for larger moduli where it genuinely matters.

  • Restate "last N digits" as "modulo 10N" immediately - it bounds every intermediate value.
  • Use Python's three-argument pow(base, exp, mod) for modular powers; never raise to the full power and reduce afterward.
  • In C, apply (__int128) as a transient widening cast before any multiply whose product could exceed 263-1.
  • Keep every accumulator already reduced modulo m so intermediate sums never overflow.
  • Memorise the binary exponentiation loop - it turns O(N) multiplications into O(log N) and generalises to any associative operation.
  • Parameterise problem sizes so small, verifiable cases run through the exact same code path as the full problem.

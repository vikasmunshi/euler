Problem 48: Self Powers [Level 0]

Problem Statement

The series, $1^1 + 2^2 + 3^3 + \cdots + 10^{10} = 10405071317$.

Find the last ten digits of the series, $1^1 + 2^2 + 3^3 + \cdots + 1000^{1000}$.

Test Cases

  • dev: solve(limit=10) → 405071317
  • main: solve(limit=1000) → 9110846700

Mathematical Complexity

The only mathematical idea required here is modular arithmetic. Because the problem asks for the last ten decimal digits of the sum, I can reduce every term and the running total modulo 10 10 at each step without affecting correctness - contributions from higher decimal positions are discarded immediately, keeping all intermediate values small. The potentially expensive operation n n is tamed by modular exponentiation, which computes a b mod m in O(log b) multiplications via repeated squaring rather than the naive O(b) multiplications.

The overall asymptotic complexity is O(N log N), where N is the upper limit of the series. Each of the N terms requires O(log n) work for the modular exponentiation, and the loop runs N times. In practice the constant factors are tiny, and for N=1000 both implementations finish in well under a millisecond.

Solution Approach

Modular Exponentiation with Repeated Squaring (Index 0)

Both the Python and C solutions implement the same algorithm: iterate i from 1 to the limit, compute i i mod 10 10 using fast modular exponentiation, add the result to a running total, and reduce the total modulo 10 10 after each addition. The modulus 10 10 was chosen because it isolates exactly the last ten decimal digits - any contribution from higher positions is discarded immediately, keeping all intermediate values bounded throughout.

In Python I delegate the inner computation to the built-in three-argument pow(base, exp, mod), which performs repeated squaring internally at C speed, so no extra library is needed. In the C version I wrote a small mod_pow helper that implements the same binary exponentiation loop by hand. Because intermediate products of two 64-bit values can overflow a long long before the modulo reduction is applied, I cast each multiplication to __int128 before taking the modulus - a straightforward way to avoid overflow without resorting to a multi-word arithmetic library. The modulus constant is expressed as 10000000000LL to ensure 64-bit literal arithmetic throughout.

Learnings

The key insight was recognising immediately that the problem is entirely about modular arithmetic. Once I saw that only the last ten digits matter, the modulus 10 10 followed naturally, and the rest was standard modular exponentiation. The dev test case produces 405071317 rather than 10405071317 because the true sum for N=10 is exactly 10405071317, so taking it mod 10 10 drops the leading 1 - the code is correct.

Both solutions implement the same algorithm, so the benchmark comparison is purely a language overhead question:

The C version is roughly ten times faster on the main case, which is expected given it avoids Python interpreter overhead entirely. That said, both finish in well under a millisecond, so the Python version is perfectly adequate for this problem size.

  • pow(a, b, m) in Python uses the fast repeated-squaring algorithm internally and keeps all intermediate values bounded by m, making it far more efficient than computing a**b % m naively.
  • Working modulo a power of ten is a clean, idiomatic way to extract the last k digits of any integer expression, and it composes naturally with both addition and multiplication.
  • When implementing modular multiplication in C with a 64-bit modulus, casting intermediate products to __int128 before applying the modulus is the simplest way to prevent overflow without needing a big-integer library.
  • For future problems involving "last k digits of a large sum or product", the pattern is always the same: choose modulus 10 k , apply it term by term, and use modular exponentiation for any power operations.

Problem Analysis

This problem asks me to find a number that appears simultaneously in three figurate number sequences - triangular, pentagonal, and hexagonal. A figurate number is one that can be arranged into a regular geometric shape, and each of the three sequences here has a closed-form quadratic generator: T(n) = n(n+1)/2, P(n) = n(3n-1)/2, and H(n) = n(2n-1). I am told that 40755 is the smallest such common value (after the trivial 1) and must find the next one. A brute-force scan that generates all three sequences and intersects them would work, but there is a much sharper structural observation to be had first.

The decisive insight is an algebraic identity: every hexagonal number is also triangular. Substitute m = 2n-1 into the triangular formula and you get T(2n-1) = (2n-1)(2n)/2 = n(2n-1) = H(n). In other words, the hexagonal numbers are exactly the triangular numbers with odd index, so any number that is hexagonal is automatically triangular. That collapses a three-way intersection into a two-way one: I only need values that are simultaneously triangular (or equivalently hexagonal) and pentagonal. To test pentagonal membership efficiently I invert the formula. Solving P(k) = x with the quadratic formula gives k = (1 + sqrt(1 + 24x))/6, so x is pentagonal exactly when the discriminant 1 + 24x is a perfect square whose root is congruent to 5 mod 6 (making k a positive integer). Each test is constant-time arithmetic plus one square root, so scanning candidate triangular numbers and testing each is O(k) where k is the number of candidates examined before a hit - a tiny count in practice.

Solution Approach

Analytical Inverse-Formula Check (Python and C)

Both files implement the same algorithm. I start from a supplied index n - 285, which is the index of the known solution T(285) = 40755 - and increment n by one each iteration. For each n I compute the triangular number T(n) = n(n+1)/2 and apply two membership filters.

The first filter tests whether T(n) is pentagonal. From the inverse k = (1 + sqrt(1 + 24x))/6, k is a positive integer precisely when 1 + sqrt(1+24x) is divisible by 6, which means sqrt(1+24x) must be congruent to 5 mod 6. The code checks this directly: (1 + 24 * t) ** 0.5 % 6 == 5.0 in Python, and fmod(sqrt(1 + 24*t), 6.0) != 5.0 in C. This one comparison implicitly verifies that the discriminant is a perfect square - if it were not, the floating-point remainder would not land exactly on 5.0.

The second filter tests triangularity using k = (-1 + sqrt(1 + 8x))/2, which requires sqrt(1+8x) to be congruent to 3 mod 4: (1 + 8 * t) ** 0.5 % 4 == 3.0. Strictly this second filter is redundant - I am already iterating over triangular numbers, so T(n) is triangular by construction - but it keeps the formulation symmetric, self-contained, and trivially correct, and it costs almost nothing. When both filters pass, the value is returned. The answer is 1533776805.

The language-specific differences are purely mechanical. C uses fmod from <math.h> for the floating-point remainder and structures the filters with explicit continue statements; Python uses % on floats and nests two if blocks. Both rely on IEEE 754 double precision being exact for the integers involved (the answer is about 1.5 billion, well below 2^53), so the floating-point square roots are reliable.

Programming Techniques

Inverse-Formula Membership Testing

When a sequence is defined by a closed-form function f(n) = x, I can test whether a given x is a member by algebraically inverting f and checking that the result is a positive integer. For quadratic formulas this reduces, via the quadratic formula, to a "the discriminant must be a perfect square" condition, expressible as a modular residue check on the square root. This pattern eliminates lookup tables entirely and makes each test O(1). It recurs across many polygonal-number problems: derive the inverse once, identify the integrality condition, and reach for it whenever you need a fast "is this value in the sequence" test.

Floating-Point Square Root for Integer Checks

Both solutions use a floating-point square root and test its remainder against a small modulus. This is safe because the candidate values stay around 1.5 * 10^9, well below 2^53 - the largest integer a double-precision float can represent exactly. A perfect-square discriminant therefore produces an exact integer square root, so the residue comparison is reliable. For larger inputs I would instead compute an integer square root and verify by squaring, to remove any rounding risk.

Early-Continue Filtering

The C version applies the more selective pentagonal filter first and uses continue to abandon the iteration immediately on failure, before the second test runs; the Python version achieves the same short-circuit through nested if blocks. When two independent conditions must both hold, ordering them most-selective-first and exiting early minimises wasted work per iteration - a habit worth keeping in any tight loop with multiple guards.

Walrus Operator for Loop Control (Python)

The Python loop header while n := (n + 1): uses the assignment expression (walrus operator) to increment n and use its new value as the condition in a single step. Since n is always a positive integer the condition is always truthy, making this an effectively infinite loop exited only by return. It is compact, though an explicit n += 1 in the body reads more plainly where clarity outranks brevity.

Command-Line Parameterisation of Starting State

Both solutions read the starting index from the command line rather than hard-coding it, letting the harness resume the search from the known good point (285) instead of rescanning from 1. Making the search state an external parameter is good practice whenever the worked example sits close to the answer: it avoids redundant work and keeps the solver restartable.

Key Takeaways

The unlocking insight is the structural identity H(n) = T(2n-1): every hexagonal number is triangular, which turns a three-way intersection into a two-way membership test. Recognising relationships like this before writing any code is what separates a clean solution from a brute-force scan. Combined with inverting the pentagonal formula into a discriminant check, the whole problem reduces to a short loop with two arithmetic guards.

Both solutions implement the identical algorithm, so the benchmark gap is pure interpreter overhead: the C build runs in about 0.0004 s versus 0.0086 s for Python, roughly a 20x difference. Since both finish in well under a hundredth of a second, the language choice has no practical bearing here - the C version simply shows how much per-iteration overhead a lower-level implementation strips away.

  • Look for algebraic identities between sequences before coding; one substitution can shrink the search space dramatically.
  • Invert a closed-form formula to get a free O(1) membership test - for quadratics this becomes a perfect-square discriminant check.
  • Floating-point sqrt is safe for integer checks while values stay below 2^53; use an integer square root and verify by squaring above that.
  • When multiple guards must all pass, order them most-selective-first and exit early to avoid wasted work.
  • Parameterise the starting state of a search so it can resume from a known point rather than restarting from scratch.

Problem 40: Champernowne's Constant [Level 1]

Problem Statement

An irrational decimal fraction is created by concatenating the positive integers: $$0.12345678910{\color{red}\mathbf 1}112131415161718192021\cdots$$

It can be seen that the $12$ th digit of the fractional part is $1$.

If $d_n$ represents the $n$ th digit of the fractional part, find the value of the following expression. $$d_1 \times d_{10} \times d_{100} \times d_{1000} \times d_{10000} \times d_{100000} \times d_{1000000}$$

Mathematical Complexity

Champernowne's constant has a beautifully regular structure when the concatenated integers are grouped by their digit length. All 1-digit numbers contribute 9 digits, all 2-digit numbers contribute 2 * 90 = 180 digits, and in general the band of k-digit numbers contributes exactly k * 9 * 10^(k-1) digits. These band totals grow exponentially, so the cumulative total up through band k is a geometric-like sum that can be computed incrementally. Given any target position n, I can walk through bands in order - adding each band's contribution to a running total - and stop as soon as the running total first meets or exceeds n. Since position 10^6 falls in the 6-digit band, the loop terminates after at most six iterations for every lookup in the main problem. Once the containing band is identified, a single integer division locates the number within that band and a single modulo operation identifies which digit within that number is wanted, both O(1). The overall solve involves a fixed number of such lookups (one per power of ten from 10^0 to 10^i), giving a total complexity of O(i * log(10^i)) = O(i^2), which is negligible at any scale this problem demands.

No string needs to be built or stored in memory beyond a short representation of a single number at lookup time. The band boundaries grow exponentially, so the positional arithmetic stays well within 64-bit integer ranges throughout, and all the required arithmetic is exact integer arithmetic with no floating-point concerns.

Solution Approach

Analytical Band Decomposition (s0 - Python and C)

I chose to solve this purely analytically rather than by constructing any part of the concatenated string. Both p0040_s0.py and p0040_s0.c implement the same algorithm, with only language-level differences in how certain sub-expressions are written.

The helper function get_nth_digit_champernowne_s_constant (Python) and get_nth_digit_champernowne (C) maintain three state variables: the cumulative digit count before the current band, the cumulative digit count including the current band, and the current digit length. The loop increments the digit length and updates both accumulators - computing each band's size as num_digits * 9 * 10^(num_digits-1) - until the running total reaches or exceeds n. After the loop I compute a 0-indexed offset of position n within its band, divide by num_digits to find which number in the band contains position n (anchored at 10^(num_digits-1) to get the actual number), and take the remainder to find which digit within that number is wanted. The digit is then extracted by formatting the number as a decimal string and indexing it.

In Python, the outer solve function uses functools.reduce with a multiply lambda over a generator expression that yields get_nth_digit_champernowne_s_constant(10**k) for k from 0 to i inclusive. The function is parameterised by the exponent i rather than hardcoding seven terms, which made it straightforward to verify intermediate products across the full test suite. In C, I replaced the generator with a plain for-loop that multiplies a running product and advances pos by a factor of ten at each step, avoiding any floating-point exponentiation. The band-size computation in C uses an explicit integer multiplication loop rather than the ** operator, keeping the computation self-contained within 64-bit integer arithmetic without needing the math library for the core logic.

Learnings

The key insight was recognising that the concatenated digit sequence has perfectly predictable structure when partitioned by digit length. Once I saw that each band contributes exactly k * 9 * 10^(k-1) digits, the problem reduced entirely to small integer arithmetic and I never needed to materialise any string beyond a single number per lookup.

The most important detail to get right is the 0-indexing of the offset within the band. I compute offset_of_number = n - length_till_num_digits - 1 so that integer division and modulo both operate on a 0-based index. An off-by-one here shifts every digit lookup by one position and is the most natural source of error in this kind of positional decomposition.

Since I implemented only one algorithm for this problem, there is no performance comparison to make. Both the Python and C versions complete every test case in under a microsecond, so there was no practical motivation to look for a faster approach. The C version avoids Python's arbitrary-precision arithmetic overhead and the functools.reduce call, but neither has any measurable effect at this scale.

  • Partitioning a concatenated integer sequence by digit length is a reusable pattern applicable to any Champernowne-like positional lookup problem.
  • The product collapsing to zero when a zero digit is sampled (as in the i=11 case) is a natural consequence of multiplication and requires no special casing.
  • Using functools.reduce with a multiply lambda over a generator is a clean way to compute a running product without materialising an intermediate list.
  • Parameterising solve by the exponent i rather than hardcoding the number of terms made the extra test cases easy to add and the intermediate products easy to inspect.
  • In C, computing powers of 10 with an explicit integer loop rather than pow() avoids any floating-point rounding concerns and keeps the core logic self-contained.

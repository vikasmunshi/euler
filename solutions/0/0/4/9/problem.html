Problem 49: Prime Permutations [Level 1]

Problem Statement

The arithmetic sequence, $1487, 4817, 8147$, in which each of the terms increases by $3330$, is unusual in two ways: (i) each of the three terms are prime, and, (ii) each of the $4$-digit numbers are permutations of one another.

There are no arithmetic sequences made up of three $1$-, $2$-, or $3$-digit primes, exhibiting this property, but there is one other $4$-digit increasing sequence.

What $12$-digit number do you form by concatenating the three terms in this sequence?

Mathematical Complexity

This problem sits at the intersection of prime number theory, combinatorics, and arithmetic progressions. The core task is to find sets of n-digit primes that are mutual digit permutations of one another and simultaneously form a three-term arithmetic progression. Generating all n-digit primes via a Sundaram sieve costs roughly O(N log log N) where N = 10^n. For each prime p in the range, there are at most n! digit permutations to enumerate, and each permutation lookup into the prime set costs O(1) for a hash set or O(log P) for a binary-searched array, where P is the count of n-digit primes. The sibling group assembly therefore costs O(P * n! * log P) overall.

Once a sibling group of size k is assembled, the arithmetic progression check uses a difference-bucketing pass over O(k^2) pairs rather than iterating over O(k^3) triples explicitly. For n = 4 the entire search finishes in single-digit milliseconds; for n = 5 it takes hundreds of milliseconds because both P and n! grow substantially. The approach generalises cleanly to arbitrary digit length n, as confirmed by the extra test case at n = 5.

Solution Approach

Permutation Grouping with Difference Bucketing (Python and C)

Both p0049_s0.py and p0049_s0.c implement the same algorithm in their respective languages. I first build the full set of n-digit primes using a Sundaram sieve. For every n-digit prime I enumerate all digit permutations, keeping only those that are themselves n-digit primes, to form the "sibling group" of prime permutations sharing the same digit multiset. In Python I delegate to itertools.permutations and filter using a walrus operator inside a generator expression. In C I implement Heap's algorithm writing into a pre-allocated flat character buffer of stride 16, avoiding any dynamic allocation inside the inner loop.

When a sibling group contains at least three members I apply a difference-bucketing pass: I iterate over all pairs, compute the absolute numeric difference for each pair, and accumulate both primes into a bucket keyed by that difference. A bucket that ends up containing exactly three distinct primes is the signature of a valid three-term arithmetic progression, because if A, B, C all share the same pairwise difference d then B - A = C - B = d. In Python I use a defaultdict(set) for the buckets and a set of result strings for natural deduplication. In C I use flat parallel arrays for bucket keys and contents, plus a linear scan to deduplicate discovered sequences before appending them to the output list.

The main language-specific differences are: Python uses a hash set for O(1) prime lookup while C performs binary search over a sorted array of prime strings; Python's result deduplication is free via the set data structure while C requires an explicit linear scan; and permutation generation in Python allocates tuples via the CPython iterator machinery while Heap's algorithm in C operates entirely in-place on a stack buffer.

Learnings

The key insight was that "permutation of digits" is most naturally handled by directly enumerating all digit permutations for each prime and intersecting with the prime set, rather than pre-grouping primes by a sorted-digit canonical key. Both strategies work, but permutation enumeration makes the subsequent AP search fall out naturally from the sibling set already in hand.

The second insight was the difference-bucketing trick: instead of generating all C(k, 3) triples and testing each for the AP property, I accumulate C(k, 2) pairs by their difference value. A bucket containing exactly three primes is the unambiguous signature of an AP triple, which avoids an explicit triple iteration altogether. One caveat worth noting: the bucket-size-equals-three check can be brittle if a sibling group has more than three members that happen to include multiple AP triples sharing the same common difference - a safer guard is to verify the sorted triple satisfies A + C = 2B explicitly.

Comparing the two implementations using the benchmark results confirms a consistent roughly 3x speedup for C over Python: approximately 3.4 ms vs 12.0 ms for n = 4, and 204 ms vs 558 ms for n = 5. The speedup comes primarily from avoiding Python object allocation overhead in the permutation loop and from compiled tight loops in the difference-bucketing pass. For a Level 1 problem the Python version is perfectly acceptable, but the C translation was straightforward and the speedup at n = 5 is appreciable.

  • The Sundaram sieve is a clean, self-contained alternative to the Sieve of Eratosthenes and worth keeping in the toolkit for generating bounded prime sets.
  • Storing primes as strings rather than integers avoids repeated int-to-string conversions during permutation generation and primality lookup, at the cost of slightly more memory.
  • Using a set of result strings in Python naturally deduplicates sequences discovered from multiple starting primes in the same sibling group, eliminating the need for an explicit seen-check.
  • Heap's algorithm in C is a compact, cache-friendly way to enumerate all permutations in-place into a pre-allocated buffer, avoiding any dynamic allocation inside the inner loop.
  • The generalisation to arbitrary digit count n is straightforward and the n = 5 test case exercises the code well beyond the original problem constraint, providing useful coverage of edge cases in both the sieve and the permutation generator.

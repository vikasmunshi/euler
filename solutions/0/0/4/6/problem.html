Problem 46: Goldbach's Other Conjecture [Level 1]

Problem Statement

It was proposed by Christian Goldbach that every odd composite number can be written as the sum of a prime and twice a square.

$$\begin{align} 9 = 7 + 2 \times 1^2\\ 15 = 7 + 2 \times 2^2\\ 21 = 3 + 2 \times 3^2\\ 25 = 7 + 2 \times 3^2\\ 27 = 19 + 2 \times 2^2\\ 33 = 31 + 2 \times 1^2 \end{align}$$

It turns out that the conjecture was false.

What is the smallest odd composite that cannot be written as the sum of a prime and twice a square?

Mathematical Complexity

The conjecture claims every odd composite n can be written as n = p + 2k² for some prime p and positive integer k. Disproving it means finding the smallest odd composite for which no such decomposition exists. To test a given candidate n I fix n and iterate over all known primes p less than n, then ask whether (n - p) / 2 is a perfect square - a direct integer arithmetic question. Because p = 2 can be immediately excluded (n is odd, 2k² is always even, so 2 + 2k² is always even and can never equal an odd composite), only odd primes need to be considered. The inner loop for each candidate runs at most O(π(n)) steps, where π(n) grows as n / ln n by the prime number theorem. Since the answer (5777) is small, the entire search terminates almost instantly.

Two flavours of prime generation appear across the implementations. The Python solution uses an incremental dictionary-based sieve with memory proportional to O(π(n)), seeding each new prime's square into the dictionary on discovery and forwarding factors forward as composites are consumed. The C solution instead tests each new candidate by trial division against all previously accumulated primes up to its square root. Both are correct and more than adequate for the tiny search range required here.

Solution Approach

Incremental Prime Generation with Perfect-Square Residual Test (index 0, Python and C)

Both files share the same high-level strategy: maintain a growing collection of primes, walk odd composite numbers upward in order, and for each one check whether any known odd prime p satisfies the condition that (n - p) / 2 is a perfect square. The first candidate that fails every such check is returned as the answer.

In the Python version I used an incremental dictionary sieve. A dictionary known_composites maps future composite numbers to the list of primes whose multiples land there. When the main counter reaches a number absent from the dictionary it is prime; its square is seeded immediately. When it reaches a number present in the dictionary it is composite; its prime factors are forwarded to the next unseen multiples and the current entry is deleted to keep memory compact. The perfect-square check is expressed inline as ((current_number - p) / 2) ** 0.5 % 1 == 0 inside an any() call, which short-circuits as soon as a valid decomposition is found.

In the C version I chose a simpler approach: I maintain a dynamically grown integer array of known primes and determine primality of each new number by trial division against that array up to the square root of the candidate. For the conjecture check I compute half = (current - p) / 2, take an integer square root with (int)sqrt(...), and verify that root * root == half. This avoids the floating-point rounding concern present in the Python version entirely, since all arithmetic stays in integer domain. The C build runs in roughly 0.0004 s versus 0.0217 s for Python - approximately a 54x speedup - which reflects interpreter overhead rather than any algorithmic difference, as both solutions are effectively instant from a user perspective.

Learnings

The key insight was inverting the conjecture check: instead of enumerating all (p, k) pairs and testing their sums, I fix the candidate n, iterate over known primes, and ask whether the residual (n - p) / 2 is a perfect square. This turns the inner loop into a simple scan of length O(π(n)) with a cheap arithmetic test at each step, keeping the total work tiny even though the loop is written naively.

The floating-point modulo trick x ** 0.5 % 1 == 0 used in the Python version is concise and works reliably for values well below 10,000, but for significantly larger inputs floating-point rounding can produce false positives or negatives. The C version sidesteps this entirely with an integer square root and an exact equality check - a pattern worth preferring whenever the input range is uncertain. Python 3.8+ also provides math.isqrt as a clean, rounding-safe alternative.

Only one algorithmic index exists here so there is no algorithmic comparison to draw; the speed difference between Python and C is purely a matter of interpreter overhead. Both solutions are correct and solve the problem in well under a tenth of a second.

  • Inverting the conjecture check - fix n, iterate over known primes, test the residual - is far more efficient than forward (p, k) enumeration.
  • Excluding p = 2 from the conjecture check is mandatory: 2 + 2k² is always even and cannot equal an odd composite.
  • The incremental dictionary sieve is a useful pattern when no upper bound is known in advance, as it avoids pre-allocating a large boolean array and integrates naturally into a linear scan.
  • The floating-point square-root modulo test is elegant for small inputs but should be replaced with an integer isqrt check for larger ones to eliminate rounding hazards.
  • Short-circuit evaluation via any() avoids redundant prime checks once a valid decomposition is found, which is a small but clean optimisation worth applying generally.

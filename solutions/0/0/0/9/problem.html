Problem 9: Special Pythagorean Triplet [Level 0]

Problem Statement

A Pythagorean triplet is a set of three natural numbers, $a \lt b \lt c$, for which, $$a^2 + b^2 = c^2.$$

For example, $3^2 + 4^2 = 9 + 16 = 25 = 5^2$.

There exists exactly one Pythagorean triplet for which $a + b + c = 1000$.
Find the product $abc$.

Test Cases

  • main: solve(sum_sides=1000) → 31875000

Mathematical Complexity

The problem couples two simultaneous constraints over the natural numbers: the Pythagorean condition a² + b² = c² and the fixed sum a + b + c = N. Because the sum constraint makes c fully determined once a and b are chosen - it is simply the residual c = N - a - b - the search space collapses from three nested loops to two, giving a worst-case complexity of O(N²). In practice the ordering invariant a < b < c allows tight upper bounds to be derived that cut the constant factor substantially.

The key bounds follow directly from the ordering constraints. Because a is the smallest of three distinct positive integers summing to N, combining a < b and b < c with the fixed sum gives 4a < N, so a < N/4. This is tighter than the naive bound of a < N/3 and saves a meaningful fraction of outer iterations. For b, the condition b < c with b + c = N - a gives b < (N - a) / 2, which I conservatively bound as b < N/2. A closed-form parametric approach using Euclid's formula for Pythagorean triples would reduce the whole problem to O(1), but the bounded brute-force search is simple and fast enough for the given input.

Solution Approach

Bounded Brute-Force with Algebraic Elimination (Python and C)

I implemented the same algorithm in both Python and C. I iterate a from 1 up to N // 4 + 1 and b from a up to N // 2, then compute c directly as N - a - b using a single-element tuple trick inside the generator rather than adding a third loop. Only the Pythagorean condition a² + b² == c² needs to be checked per candidate pair. This eliminates an entire dimension of iteration compared to a naive triple-nested loop.

In Python I expressed the search as a generator expression fed into next(), which provides clean short-circuit semantics: the loops stop as soon as the first valid triple is found, and a StopIteration is caught and re-raised as a descriptive ValueError if no solution exists. In C I used nested for loops with an immediate return on success, achieving the same early-exit behaviour without any generator machinery. The C version casts to long long when computing the product to guard against overflow for larger inputs, even though the result for the main input (31,875,000) fits comfortably in 32 bits. The logic is otherwise identical between the two implementations.

Learnings

The central insight was that a sum constraint always makes one variable redundant - once a and b are fixed, c is determined algebraically. I should apply this pattern immediately whenever a fixed-sum condition appears: eliminate one variable before writing any loop, saving a full dimension of iteration.

The second insight was deriving the tight outer bound of a < N/4 from both ordering conditions simultaneously rather than just one. Using the looser bound a < N/3 would increase outer iterations by 33% for no benefit. It is worth spending a moment to combine all available constraints before fixing loop limits.

The benchmark makes the language difference concrete. The Python solution takes around 10 ms on the main input while the C solution takes around 0.1 ms - roughly a 100x speedup. Both produce the correct answer, and Python is entirely adequate for a problem at this scale, but the gap is a clear reminder that for brute-force integer arithmetic loops C is dramatically faster.

  • Use algebraic elimination to reduce loop dimensionality before searching - a sum constraint always makes one variable redundant.
  • Derive loop bounds from all available ordering invariants simultaneously; the a < N/4 bound is tighter than the naive a < N/3 and requires combining two conditions.
  • Python generator expressions with next() give clean short-circuit semantics without explicit break logic or mutable state.
  • For problems that admit a closed-form parametric solution (Euclid's formula for Pythagorean triples), the brute-force approach is a good starting point but can be replaced for large-scale inputs.
  • Always cast intermediate products to a wider integer type in C when the result might overflow, even when the immediate test case fits safely in 32 bits.

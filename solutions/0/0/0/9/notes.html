Problem Analysis

A Pythagorean triple is a set of three positive integers a < b < c satisfying a2 + b2 = c2. Finding such integers is a Diophantine equation - an equation for which we admit only integer solutions. Here I add a second constraint, the linear equation a + b + c = S, and must find the unique Pythagorean triple satisfying both, then report the product abc.

The crucial observation is that the linear constraint lets me eliminate one variable. Once a and b are chosen, c is completely determined: c = S - a - b. This collapses what would naively be a three-dimensional search into a two-dimensional one. Two further observations bound the loops tightly. First, because a < b < c are all positive and sum to S, the smallest value a cannot exceed roughly S/4 - if a were any larger we could not fit two strictly greater values b and c that still sum to S. Second, b is bounded above by S/2 to keep c positive. Each candidate (a, b) pair is checked in constant time, so the overall complexity is O(S2). For S = 1000 that is on the order of 105 iterations - trivially fast in any language.

Solution Approach

Bounded two-variable search (index 0, Python and C)

Both the Python and C solutions implement the same algorithm. The outer loop iterates a from 1 up to S/4 inclusive - the tightest upper bound derivable from the ordering a < b < c. The inner loop iterates b from a (which enforces a ≤ b) up to S/2 (which keeps c positive). For each pair, c is computed directly as S - a - b rather than searched for. This is the variable-elimination step that removes an entire loop level. I then test whether a2 + b2 == c2 and, because the problem guarantees exactly one solution, return a*b*c immediately on the first match. The two nested loops give the O(S2) bound established above.

I deliberately compute c as an integer expression rather than taking a floating-point square root of a2 + b2. Integer arithmetic is exact, so the equality check is reliable; a floating-point approach would demand a tolerance comparison and risk rounding errors. The closed-form c = S - a - b sidesteps all of that.

The Python version wraps the nested loops in a generator expression and calls next() to retrieve the first result, converting the resulting StopIteration into a descriptive ValueError when no triple exists. The C version uses plain nested for loops with an early return, writing a diagnostic to stderr and returning -1 on failure. In C I cast to long long before the multiplication: although the answer 31875000 fits comfortably in a 32-bit int, the product of three loop variables could overflow for larger inputs, and promoting first is a cheap, defensive habit. Both versions handle the dev case (S = 12, the classic 3-4-5 triple, product 60) and the main case correctly.

Programming Techniques

Variable elimination via a linear constraint

When two or more unknowns are tied together by a linear relation, you can always solve for one of them and substitute. Writing c = S - a - b turns a three-variable search into a two-variable one for free. This is the first transformation to look for whenever a linear equation connects your variables: it removes a loop level outright and shrinks the search space dramatically.

Tight loop bounds from ordering constraints

The constraints a < b < c with a + b + c = S imply 3a < S, and the ordering tightens this to a ≤ S/4. Choosing the smallest provable upper bound for each loop variable directly controls the iteration count. Whenever you write a loop, ask whether a mathematical argument lets you start later or stop earlier - each tighter bound multiplicatively reduces work and, here, keeps the O(S2) constant small.

Generator expression with next() for first-match search

A Python generator expression is lazy: it yields values on demand rather than building a full list. Wrapping the nested loops in a generator expression and calling next() stops the iteration the instant the first match is found, with no wasted computation. Catching StopIteration and re-raising it as a domain-meaningful ValueError turns a low-level iteration signal into a clear error. This is the idiomatic Python pattern for "find the first element satisfying a predicate," preferable to a manual flag or a break.

Single-element tuple as an inline binding

Inside the generator I write for c in (sum_sides - a - b,). Iterating over a one-element tuple is a lightweight way to give a derived value a name without an extra statement or function. The name c can then be reused in the if clause and the result expression without recomputing it. Reach for this idiom whenever you want a let-binding inside a comprehension.

Defensive integer promotion in C

In C, arithmetic on int operands yields an int, so a product can silently overflow past INT_MAX. Writing (long long)a * b * c promotes the whole computation to 64-bit arithmetic before any overflow can occur, because the cast on the first operand propagates through the chain. Apply this routinely to products of loop variables, not only when you have already proven overflow will happen.

Key Takeaways

The unlocking insight is that a linear sum constraint is a gift: it eliminates one variable at no cost, reducing a three-variable search to a two-variable one with easily computed bounds. Combined with the ordering constraint a < b < c, which caps a at S/4, the search space stays tiny enough that a direct nested loop is essentially instantaneous.

The benchmarks bear this out. On the main input (S = 1000) the C solution completes in about 0.00004 s, while the equivalent Python solution takes about 0.006 s. Both are effectively instant; the roughly 150x gap is the usual interpreter overhead, and for a problem this small either choice is perfectly acceptable. The C version is the faster, the Python version the more concise.

  • When variables are linked by a linear constraint, substitute to eliminate one before writing any loop - it is always free and always shrinks the search.
  • Derive the tightest possible bound for every loop variable from ordering and positivity constraints; each tighter bound multiplies into fewer iterations.
  • Prefer exact integer arithmetic over floating-point whenever you need an equality check - compute c = S - a - b rather than a square root.
  • In Python, a generator expression with next() is the idiomatic, lazy way to find the first element satisfying a condition.
  • In C, cast to a wider integer type before multiplying values whose product might overflow, as a default habit rather than an afterthought.
  • Estimate the iteration count before coding: O(S2) at S = 1000 is about 105 steps, well within reach of brute force.

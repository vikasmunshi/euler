Problem 1: Multiples of 3 or 5 [Level 0]

Problem Statement

If we list all the natural numbers below $10$ that are multiples of $3$ or $5$, we get $3, 5, 6$ and $9$. The sum of these multiples is $23$.

Find the sum of all the multiples of $3$ or $5$ below $1000$.

Mathematical Complexity

The core idea is the inclusion-exclusion principle applied to arithmetic progressions. The set of multiples of 3 or 5 below a limit is the union of two sets, and those sets overlap at every multiple of lcm(3, 5) = 15. Naive addition double-counts the overlap, so the correct sum is sum(multiples of 3) + sum(multiples of 5) - sum(multiples of 15). Each of those three sub-sums is itself an arithmetic progression starting at 0, and the sum of all multiples of d below n has a closed form: compute k = (n - 1) // d (the count of qualifying terms) and return d * k * (k + 1) // 2, which is just the triangular-number formula scaled by d.

A brute-force scan of every integer is O(n). Iterating only over multiples via a stepped loop is still O(n) but visits roughly (1/3 + 1/5 - 1/15) * n ~ 47% of integers. The closed-form arithmetic formula reduces the whole computation to O(1): three integer divisions, three multiplications, and two additions, entirely independent of the limit. The 10^9 extra test case makes the difference between these complexity classes very concrete.

Solution Approaches

Closed-Form Arithmetic Series (s0)

I derived the triangular-number formula and implemented it as sum_arithmetic_series in both Python and C. Given a common difference d and exclusive upper limit n, I compute k = (n - 1) // d and return d * k * (k + 1) // 2. I call this three times - for d = 3, 5, 15 - and combine via inclusion-exclusion. Both the Python and C versions are O(1) and finish in effectively zero measured time on all test cases, including the 10^9 limit. This is the approach I consider correct and final; all other solutions exist purely to illustrate the cost of skipping the closed form.

Brute-Force Modulo Scan (s1)

I loop over every integer from 0 to max_limit - 1 and accumulate those where term % 3 == 0 or term % 5 == 0. This is the most literal translation of the problem statement. In Python it takes about 55.7 s on the 10^9 case - far too slow for large inputs. The C version applies identical logic but runs roughly 78 times faster at 0.71 s, demonstrating how much of the Python overhead is pure interpreter cost rather than algorithmic cost. Even in C, performing two modulo operations on every integer makes this the slowest C variant by a factor of about 2.2 compared to the next-slowest C solution.

Range-Sum with Inclusion-Exclusion (s2)

The Python version calls sum(range(0, n, 3)), sum(range(0, n, 5)), and subtracts sum(range(0, n, 15)). Python's built-in sum over a range object is implemented at the C level and avoids per-iteration interpreter overhead, which is why it runs about 7x faster than the pure-Python s1.py (7.9 s vs 55.7 s) despite being algorithmically O(n) in terms of the range it covers. The C version takes a different angle: sum_range computes the result analytically using count * (start + last) / 2, making it O(1). This means s2_c (0.0038 s on the extra case) is nearly as fast as s0_c, though the formula is slightly more general since it handles an arbitrary start value rather than assuming the series begins at zero.

Range-Based Generator with Inclusion-Exclusion (s3)

In Python I wrapped a stepped range iteration inside an explicit generator function using for term in range(...): yield term. The logic is identical to s2.py but the generator boundary forces Python to invoke the interpreter callback for each yielded value, losing the C-level bulk-sum optimisation. This roughly doubles the cost relative to s2.py (17.4 s vs 7.9 s on the 10^9 case). In C, sum_arithmetic_series_range is a plain for loop that the compiler handles efficiently; s3_c runs in 0.33 s. The Python-to-C ratio for this variant is about 53x, versus effectively infinite for the O(1) solutions where both are below measurement noise.

While-Loop Generator with Inclusion-Exclusion (s4)

I replaced the for -over- range generator with one that manually increments a counter in a while loop. In Python this removes even the small optimisation that range provides for stepping, so every iteration requires an explicit Python addition and comparison. It is the slowest Python variant at 25.1 s - about 44% slower than s3.py and more than 3x slower than s2.py. In C, sum_arithmetic_series_loop is a while loop, and the compiler generates nearly identical code to a for loop; s4_c (0.34 s) is essentially the same speed as s3_c (0.33 s). The contrast between Python and C here is instructive: Python's interpreter overhead makes the difference between a for -range and a manual while meaningful, while in C both are irrelevant noise.

Learnings

The key insight was recognising that summing an arithmetic progression has a closed form. Once I reframed the problem as "sum of A union B where A and B are arithmetic progressions", inclusion-exclusion plus the triangular-sum formula gave an O(1) solution that handles any limit - including 10^9 - in unmeasurably small time. This pattern of replacing a loop over an arithmetic progression with a formula reappears constantly in number theory problems and is worth internalising as a first reflex.

The five solutions together form a useful benchmark ladder. For the 10^9 extra test case:

A few concrete takeaways from these numbers. First, the Python s2 vs s3/s4 gap (roughly 2-3x) is entirely due to generator overhead: yielding one value at a time forces a Python callback per term, while sum(range(...)) stays inside CPython's C layer. When a built-in can do the heavy lifting in C, I should keep the logic at that level rather than wrapping it in a Python generator. Second, in C the distinction between a for loop and a while loop disappears completely - s3_c and s4_c differ by less than 3% - confirming that the compiler eliminates micro-differences that are meaningful in Python. Third, the C s2 implementation is a useful reminder that "looks like iteration" does not mean "is iteration": because I happened to implement sum_range using the count-and-arithmetic-mean formula, it is genuinely O(1) and matches s0_c in practice. The algorithmic complexity is in the formula chosen, not the language construct used to express it.

Problem 5: Smallest Multiple [Level 0]

Problem Statement

$2520$ is the smallest number that can be divided by each of the numbers from $1$ to $10$ without any remainder.

What is the smallest positive number that is evenly divisible divisible with no remainder by all of the numbers from $1$ to $20$?

Mathematical Complexity

The problem reduces immediately to computing the least common multiple (LCM) of every integer from 1 to n. The essential identity is lcm(a, b) = a * b / gcd(a, b), which follows directly from the fundamental theorem of arithmetic: the LCM takes the maximum prime-power exponent across both operands while the GCD takes the minimum, so their product recovers a * b. Because LCM is associative and commutative, I can fold this pairwise formula left-to-right across the range 2 to n, maintaining a running accumulator rather than factorising every number in the range explicitly.

The asymptotic cost is O(n * log M) where M is the magnitude of the running LCM at each step, since each GCD call uses the Euclidean algorithm in O(log M) time. In practice the running LCM grows rapidly - for n=100 the answer is a 41-digit number - so native 64-bit integers become insufficient for larger inputs and arbitrary-precision arithmetic is required. The problem sits firmly in the "one identity to know" category once the LCM-GCD connection is recognised.

Solution Approach

Iterative LCM Reduction via GCD (Python and C, solution index 0)

Both implementations share the same algorithm: fold lcm(acc, y) = acc * y / gcd(acc, y) over integers 2 through n, starting from an accumulator of 1. The language-specific details differ considerably because Python provides arbitrary-precision integers and a fast built-in GCD, while C does not.

In Python I expressed the whole computation as a single functional pipeline using functools.reduce(lambda x, y: x * y // math.gcd(x, y), range(2, n + 1), 1). I used integer floor division ( //) rather than true division to keep every intermediate value exact. Python's arbitrary-precision integers handle the n=50 and n=100 cases transparently, and math.gcd delegates to a fast C implementation internally, so the entire computation finishes in negligible time.

In C, native 64-bit types are insufficient even for n=50 (the answer has 22 digits), so I implemented a minimal BigInt type: a little-endian array of 64 unsigned 32-bit limbs in base 2^32. Only two primitive operations are needed. bi_div_small divides a BigInt by a long long via a single right-to-left pass through the limbs, returning the remainder. bi_mul_small multiplies a BigInt by a long long in a single left-to-right pass propagating carry. The GCD itself stays in native long long arithmetic: because the small operand y fits in a long long, I reduce the BigInt modulo y with a single bi_div_small call and then apply the standard Euclidean algorithm on two machine-word values. The combined bi_lcm_small function divides the accumulator by the GCD and multiplies by y, matching the Python logic step for step.

Learnings

The central insight was recognising that "evenly divisible by all integers 1 to n" is precisely the definition of their LCM, and that LCM can be accumulated incrementally using only cheap GCD calls rather than requiring an explicit prime factorisation of every number in the range. Once I had the lcm(a, b) = a * b / gcd(a, b) identity in mind, the one-liner with reduce followed immediately.

Both the Python and C solutions record 0.0000 s across all test cases, so there is no meaningful runtime gap to discuss. The Python version is far more concise and readable. The C version required building a lightweight BigInt library from scratch - good practice in multi-limb arithmetic, but it illustrates that Python's built-in arbitrary-precision integers are a genuine productivity advantage for any problem whose intermediate or final values exceed 64 bits.

  • The identity lcm(a, b) = a * b / gcd(a, b) is worth memorising - it converts a multiplicative structure problem into inexpensive Euclidean divisions.
  • Using integer floor division ( //) instead of floating-point division ( /) is critical: floating-point silently loses precision for large intermediate products, whereas integer division remains exact throughout.
  • The reduce pattern generalises cleanly to any associative binary operation over a sequence, making intent explicit and keeping the code concise.
  • When working in C on problems whose answers exceed 64 bits, a minimal BigInt backed by a limb array with just two primitive operations (divide-by-small and multiply-by-small) is sufficient for many number-theory folding patterns - no general-purpose big-integer library is necessary.
  • For the GCD of a BigInt and a small integer, reducing the BigInt modulo the small value first brings the computation back into native-integer territory cheaply and avoids implementing a full multi-precision GCD.

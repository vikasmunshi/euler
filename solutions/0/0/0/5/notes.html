Problem Analysis

A number "evenly divisible by all of the numbers from 1 to N" is exactly the least common multiple of the set {1, 2, ..., N}: the smallest positive integer that has every member of the set as a divisor. So the smallest positive number divisible by all of 1..N is precisely the least common multiple of 1..N, and by definition nothing smaller can work.

The tool that makes this tractable is the identity lcm(a, b) = a × b / gcd(a, b), in which the greatest common divisor is the largest integer dividing both a and b. This identity lets me build the least common multiple of a whole set as a sequence of pairwise combinations, threading a running accumulator through the range. Each pairwise step needs only one greatest common divisor, which the Euclidean algorithm computes in O(log(min(a, b))) steps, so the overall cost is O(N log N) bit operations - negligible here.

One subtlety matters for scale. By the prime factorisation view, the least common multiple of 1..N is the product, over every prime p up to N, of the highest prime power of p not exceeding N, so the result grows rapidly. For N = 20 it fits in 32 bits, but for the extra cases (N up to 100) it reaches 40+ decimal digits, far beyond any native integer type - so I need arbitrary-precision arithmetic.

Solution Approach

Iterative least common multiple via the greatest common divisor (Python and C, index 0)

Both files implement the same algorithm: start with a running result of 1 (the identity element for the least common multiple), then for each y from 2 to N replace result with the least common multiple of result and y. After the loop, result equals the least common multiple of 1..N.

In the Python solution the entire computation is a single functools.reduce over range(2, n + 1) with initial accumulator 1 and the lambda x * y // math.gcd(x, y) - a direct transcription of the identity. I rely on Python's built-in arbitrary-precision integers, so multiplying and floor-dividing huge numbers just works with no special handling. I chose math.gcd over a hand-written loop because the standard-library greatest common divisor is implemented in C and is faster. I also write x * y // gcd (multiply first, then divide) rather than x // gcd * y; both are exact because the greatest common divisor divides x, but the chosen form mirrors the textbook identity directly.

In the C solution I have no native big integer, so I implemented a minimal BigInt: an array of 32-bit limbs in little-endian order (least-significant limb first), base 2^32. Three primitives drive everything:

  • bi_div_small divides the BigInt by a small value and returns the remainder, walking limbs from most significant to least and carrying the remainder forward in a 64-bit intermediate ((rem << 32) | limb).
  • bi_mul_small multiplies the BigInt by a small value, walking limbs from least to most significant and propagating carry - which is why little-endian order is convenient.
  • bi_lcm_small computes the least common multiple of result and y in place: it copies result, reduces the copy modulo y with bi_div_small to get a small remainder, finds the greatest common divisor of that remainder and y with the native gcd_ll, then divides result by that greatest common divisor and multiplies by y.

The crucial design choice in C is that I never implement a big-integer greatest common divisor. Because gcd(a, b) = gcd(a mod b, b) and y is small, one modular reduction step turns the big operand into a small one, after which the whole greatest common divisor runs in native long long arithmetic. Finally bi_to_str converts the result to decimal by repeated division by 10, collecting digits least-significant-first into a stack buffer and reversing them into a heap-allocated string.

Programming Techniques

Fold over a sequence of pairwise operations

When the answer is a single value built by repeatedly combining elements with one associative binary operation, a fold is the natural expression. In Python I express the fold as functools.reduce(op, iterable, identity), supplying 1 as the identity for the least common multiple. This makes the intent obvious: a running least common multiple threaded through the range. Whenever you find yourself writing "combine all elements of a list with some operator", reach for a fold in Python or an explicit accumulator loop in C, as the C solution does.

Reducing a hard big-number operation to a small-number one

The most reusable trick here is using gcd(BigInt, small) = gcd(BigInt mod small, small). Because the modular reduction yields a value smaller than the small operand, the rest of the greatest common divisor collapses to native arithmetic. This pattern - reduce modulo a small value first, then work in machine integers - recurs throughout number-theory code and saves writing a second, harder big-integer routine.

Little-endian limb-array big integers in C

For arbitrary-precision arithmetic without a library, I store the number as an array of word-sized limbs, index 0 being least significant. Little-endian order means carry-propagating loops iterate i = 0, 1, 2, ... in their natural direction. I chose 32-bit limbs so that a limb-times-limb product plus carry always fits in a 64-bit unsigned long long with no overflow, and a 64-limb array covers numbers up to 2^2048 - far more than this problem needs.

Stack buffer for statically bounded string conversion

Decimal conversion divides repeatedly by 10, emitting digits in reverse. I gather them in a local char buf[256] and copy them in the correct order into a heap string only at the end. Because the maximum digit count is statically bounded (well under 256 even for the largest BigInt the array can hold), a fixed stack buffer is safe and avoids heap churn inside the loop.

Separating solve() from a shared benchmarking runner

Both files implement only a solve() function and delegate everything else to a shared runner - @runner.main in Python (from solver.runners) and #include "runner.h" in C. The runner parses the input arguments, calls solve() repeatedly under --runs=N, times the repetitions, verifies the result is identical across runs, and prints the single benchmark line. Keeping the algorithm free of all timing and I/O concerns lets one harness drive every solution uniformly and keeps the core logic independently testable - a good default structure for any code you intend to benchmark.

Key Takeaways

The unlocking insight is recognising that "divisible by all of 1..N" is the definition of the least common multiple of 1..N, and that the least common multiple of a set is a left fold of pairwise steps using lcm(a, b) = a × b / gcd(a, b). The Euclidean algorithm makes each greatest common divisor cheap, and one modular reduction step lets a big-integer greatest common divisor be done in native arithmetic.

Both solutions implement the identical algorithm and both finish in effectively zero time across every test case up to N = 100. The recorded timings bear this out: at the main case (N = 20) the C solution averages 6.5e-07 s against the Python solution's 2.6e-06 s, and at the largest extra case (N = 100) 4.9e-06 s against 1.2e-05 s - the C build is a few times faster in absolute terms, but both are far below any threshold that matters, since the work is O(N log N) and N is tiny. The Python solution is the more elegant and pragmatic choice: arbitrary-precision integers and a single fold reduce the whole program to one line of logic. The C solution is more instructive, showing how to build just enough big-integer machinery to handle output that overflows 64 bits.

  • "Divisible by every element of a set" is the least common multiple of that set.
  • Compute the least common multiple of many numbers as a left fold of pairwise steps with identity 1.
  • lcm(a, b) = a × b / gcd(a, b); compute the greatest common divisor with the Euclidean algorithm.
  • gcd(a, b) = gcd(a mod b, b) - reduce modulo a small operand to drop into native arithmetic.
  • Estimate output magnitude before coding: if it can exceed 64 bits, use arbitrary-precision arithmetic (free in Python, hand-rolled in C).
  • Represent C big integers as little-endian limb arrays with 64-bit intermediates to avoid overflow.
  • Use a stack buffer for conversions whose output size is statically bounded.
  • Separate the compute function from the timing/IO harness to enable clean benchmarking and testing.

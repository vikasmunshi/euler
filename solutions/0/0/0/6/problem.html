Problem 6: Sum Square Difference [Level 0]

Problem Statement

The sum of the squares of the first ten natural numbers is,

$$1^2 + 2^2 + ... + 10^2 = 385.$$

The square of the sum of the first ten natural numbers is,

$$(1 + 2 + ... + 10)^2 = 55^2 = 3025.$$

Hence the difference between the sum of the squares of the first ten natural numbers and the square of the sum is $3025 - 385 = 2640$.

Find the difference between the sum of the squares of the first one hundred natural numbers and the square of the sum.

Mathematical Complexity

This problem reduces entirely to two classical closed-form summation identities. The sum of the first n natural numbers is given by n(n+1)/2, and squaring that result yields the square of the sum. The sum of the first n perfect squares is given by n(n+1)(2n+1)/6. Both identities are provable by induction or by telescoping finite differences and have been known for centuries. Because each identity involves only a fixed number of arithmetic operations on n, the asymptotic complexity of the entire computation is O(1) - constant time regardless of how large n grows.

The difference between the two quantities also has a tidy closed-form polynomial in n, but I chose to keep the two identities as separate named steps rather than pre-simplifying them into a single expression. That decision preserves the mathematical intent: a reader can immediately see which term is the square of the sum and which is the sum of squares, matching the structure of the problem statement directly.

Solution Approach

Closed-Form Arithmetic Formula (Python and C)

Both the Python and C implementations use exactly the same algorithm: compute the square of the sum via (n*(n+1)/2)**2 and the sum of squares via n*(n+1)*(2n+1)/6, then subtract the latter from the former. Neither version uses any loop or accumulator - the computation is a fixed sequence of multiplications and one subtraction.

In Python I used the // floor-division operator throughout to keep all intermediate results as exact integers and avoid any floating-point rounding. In C I relied on the standard integer division of long long operands, which is exact here because both formulas are guaranteed to produce integer results for any positive integer n. I also promoted n to long long immediately from the command-line argument so that intermediate products such as (2*n+1)*(n+1)*n do not overflow for large inputs. Both solutions are effectively instantaneous, as confirmed by the benchmark results showing 0.0000 seconds elapsed for every test case.

Learnings

The key insight was recognising that both quantities the problem names - the square of the sum and the sum of squares - have well-known closed-form identities, so no iteration over 1..n is needed at all. Once I recalled those two identities the solution collapsed to a one-liner in each language. I only implemented one algorithm for this problem; a naive accumulation loop would also be trivially correct for n = 100, but the closed-form version is cleaner, more general, and makes the underlying mathematics explicit.

Key points worth remembering for future problems:

  • Sum of the first n integers: n(n+1)/2.
  • Sum of the first n squares: n(n+1)(2n+1)/6.
  • In Python, using // integer division when a formula guarantees an integer result keeps the arithmetic exact and avoids any floating-point concerns.
  • In C, declaring intermediate values as long long early prevents silent integer overflow in product chains - even for inputs far beyond the n = 100 case here.
  • Whenever a problem involves a sum over 1..n, it is worth checking for a closed-form identity before reaching for iteration - the payoff is both speed and clarity.

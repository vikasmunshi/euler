Problem Analysis

The task is to compute the 10,001st prime number - an integer greater than 1 whose only positive divisors are 1 and itself. Naively, I could test each candidate integer for primality by trial division, but that scales poorly. The efficient route is to generate primes in bulk with a sieve. The most familiar is the Sieve of Eratosthenes, but here I use its odd-only cousin, the Sieve of Sundaram, which enumerates the odd primes directly by marking, for every pair i, j, the index i + j + 2ij. The surviving indices k then decode to the odd primes 2k+1. Its marking cost is O(M log M) over a range of size M - far better than per-candidate trial division.

A sieve needs to know how large M must be to contain the n-th prime. This is where the prime number theorem helps: it states that the n-th prime p(n) is asymptotically n * ln(n), where ln is the natural logarithm. For n = 10001, that estimate is about 92,000, and the true answer is 104,743 - so the raw n * ln(n) heuristic is a little tight, but in this solution it works because the Sieve of Sundaram indexes only odd numbers (each array index k stands for the much larger odd number 2k+1), so the effective covered range is roughly doubled. The key observation that makes this fast is that we never need to consider even numbers beyond 2: every other even number is composite, so we sieve only odd numbers and halve both memory and work.

Solution Approach

Sieve of Sundaram (Solution 0 - Python and C)

Both the Python and C files implement the same Sieve of Sundaram, so I describe them together and note language differences as they arise.

Step 1 - Special-case the only even prime. 2 is the sole even prime. If n == 1 I return 2 immediately; from then on I work only with odd numbers.

Step 2 - Size the sieve. I set max_expected_value = int(n * log(n)), the prime number theorem estimate, and allocate an array of that length. Because each index k represents the odd number 2k+1, the actual numeric range covered is about 2 * max_expected_value, which comfortably contains the 10,001st prime.

Step 3 - Sieve over odd numbers. I map odd number 2k+1 to index k, initialising numbers[k] = k so a non-zero value means "still a candidate." The crucial algebra behind the Sieve of Sundaram: multiplying two odd numbers gives (2i+1)(2j+1) = 2(i + j + 2ij) + 1, so their product corresponds to index i + j + 2*i*j. For each surviving index i, I set numbers[i + j + 2*i*j] = 0 for j running from i upward, marking those composites. Starting j at i rather than 1 avoids re-marking products whose smaller factor was already processed.

Step 4 - Extract the answer. The surviving (non-zero) entries are the odd primes in order. I take the (n-2)-th surviving entry: the offset of -2 accounts for index 0 decoding to the non-prime 1, and for 2 already being counted as the 1st prime. I decode it back with 2*k + 1.

Language differences. In Python the sieve is a list, and the inner loop relies on try/except IndexError to detect when the computed index runs past the end of the list, breaking out. In C, out-of-bounds writes are undefined behaviour, so I instead compute the index as a long long and explicitly compare idx > max_expected_value before writing; the C version also skips already-marked entries and frees the heap array at the end. Neither solve() parses arguments or times itself - that is the runner's job (see below).

Programming Techniques

Coordinate mapping to compact an implicit structure

Rather than storing every integer, I store only odd integers via the bijection odd = 2k+1 <=> index = k - the essence of the Sieve of Sundaram. This halves memory and halves the number of cells the sieve touches. Whenever your candidate set has known structure that lets you discard a fixed fraction of the space, baking that structure into the index arithmetic is a strong lever. The cost is that the "mark composite" formula becomes i + j + 2*i*j instead of a plain product - a small algebraic price for a real performance gain.

Using a closed-form bound to pre-size a fixed array

Instead of growing a dynamic list or re-running the sieve when it falls short, I pre-allocate exactly using n * ln(n), the bound supplied by the prime number theorem. Replacing a dynamic data structure with a fixed-size array (justified by a mathematical bound) makes allocation and access faster and the code simpler. The engineering skill is recognising when a reliable bound exists and being willing to lean on it - here, the odd-number indexing provides the extra headroom that keeps the tight estimate safe.

Exception-based termination in Python vs explicit bounds checks in C

Python's "easier to ask forgiveness than permission" style lets me catch IndexError to detect overrun and break. C has no such safety net - an out-of-bounds write is undefined behaviour - so I compute the index in a wider type and check it before writing. The same logical guard manifests differently depending on each language's error-handling philosophy.

Widening to long long before multiplication in C

I compute 2LL * i * j as a long long even though the final index fits in an int. With both factors near the sieve limit, their product can exceed 2^31 and silently overflow a 32-bit int. Casting the first operand to a wider type propagates that width through the expression. Make this a habit in any C code that multiplies two loop variables.

Delegating the benchmark harness to a shared runner

Each file implements only a pure solve(); argument parsing, the repeat-timing loop (--runs=N), the result-consistency check, and the single output line all live in a shared runner - @runner.main in Python and #include "runner.h" in C. Concentrating the harness in one place keeps every solution free of boilerplate, guarantees identical timing and output semantics across problems, and leaves the algorithm trivial to read and test in isolation. Factoring cross-cutting scaffolding into one reusable component is good structure for any program that must be both correct and uniformly measured.

Key Takeaways

The solution combines two ideas: the prime number theorem to fix the array size in advance, and the odd-number coordinate mapping of the Sieve of Sundaram, which halves memory and work while doubling the effective range covered. Together they turn prime enumeration into an about-36-ms computation even in interpreted Python.

Comparing the two implementations on the main case (n = 10001): the C build runs in about 0.33 ms while the Python version takes about 36 ms - roughly a 110x speedup for the identical algorithm. This is the expected outcome when the work is a tight inner loop over a large array: the Python interpreter's per-operation overhead dominates, whereas C compiles each iteration to a handful of native instructions. The algorithm is the same; only the constant factor differs.

  • When you need a batch of primes up to a limit, reach for a sieve - the Sieve of Sundaram is a compact, odd-only option that costs O(M log M).
  • If a number-theoretic bound exists for your target value (like n * ln(n) for the n-th prime), use it to pre-size arrays instead of growing them dynamically.
  • Encode known structure (e.g. "all candidates are odd") into your index mapping to cut memory and iterations.
  • In C, widen operands to a larger integer type at the first multiplication that could overflow.
  • Match your bounds-checking idiom to the language: exceptions in Python, explicit comparisons in C.
  • Push argument parsing, timing, and output into a shared runner so each solution is just a pure function - easy to read, test, and benchmark uniformly.
  • The same algorithm ported from Python to C typically yields a large constant-factor speedup with no change in asymptotic behaviour.

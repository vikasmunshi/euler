Problem 7: 10 001st Prime [Level 0]

Problem Statement

By listing the first six prime numbers: $2, 3, 5, 7, 11$, and $13$, we can see that the $6$th prime is $13$.

What is the $10\,001$st prime number?

  • main: solve(n=10001) → 104743

Mathematical Complexity

Finding the n-th prime is a classic application of the Sieve of Eratosthenes. To avoid dynamic resizing I relied on the Prime Number Theorem, which states that the n-th prime p(n) is asymptotically n ln(n). This gives a comfortable upper bound for the sieve size before any computation begins. With the sieve sized to N = n ln(n), the overall time complexity is O(N log log N), which is extremely fast in practice for n = 10 001.

I further restricted the sieve to odd numbers only, halving memory use and eliminating all even-composite crossings from the inner loop. Every odd number can be written as 2k+1 for a non-negative integer k. The product of two such numbers (2i+1)(2j+1) = 2(i+j+2ij)+1 maps back to index i+j+2ij in the same odd-indexed array. This algebraic identity is the key mechanical trick that makes the compact sieve work.

Solution Approach

Odd-Only Sieve of Eratosthenes with PNT Upper Bound (s0 - Python and C)

Both implementations share the same algorithm. I pre-allocate an integer array of size int(n * log(n)) + 1 where each position k implicitly represents the odd number 2k+1. The outer loop walks over every candidate index i that has not been zeroed out. The inner loop uses the closed-form composite-index formula i + j + 2*i*j to zero out positions corresponding to odd composites, stopping when the index exceeds the array boundary. After sieving, I scan the surviving non-zero indices and pick the (n-2)-th one (zero-indexed, because index 0 maps to the non-prime odd number 1), then convert it back with 2*k + 1. The edge case n=1 (the prime 2) is handled explicitly before the sieve runs, since 2 is the only even prime and the odd-indexed array cannot represent it.

There are two language-specific differences worth noting. The Python version uses a try/except IndexError to terminate the inner loop when the composite index runs off the end of the list, which is concise but incurs exception-handling overhead on each boundary hit. The C version avoids this by computing the index as a long long and comparing it against max_expected_value with an explicit conditional break, which is both safer and faster. The C version also skips already-zeroed outer indices with an early continue, a small optimisation that the Python version omits.

Learnings

The first key insight was recognising that the standard textbook sieve wastes roughly half its work on even numbers. By remapping odd numbers to a compact array and deriving the composite cross-off formula algebraically - (2i+1)(2j+1) lands at index i+j+2ij - I got a sieve that is both memory-efficient and avoids any explicit divisibility checks. The second insight was using the PNT to size the sieve in advance rather than guessing or growing the array dynamically; n * log(n) is a sufficient upper bound for n = 10 001 and comfortably overestimates the correct answer of 104 743.

The benchmark results show a large gap between the two language implementations of the same algorithm: the Python version takes about 0.059 s on the main case while the C version completes in roughly 0.0005 s, a speedup of around 120x. This difference is entirely down to interpreter overhead in Python's inner loop and the exception-based termination strategy, not any algorithmic difference.

  • The odd-sieve index formula i + j + 2*i*j is worth memorising for any future sieve variant that needs to skip even numbers.
  • The PNT asymptote n ln(n) is a reliable pre-sizing tool; for tighter bounds, n(ln(n) + ln(ln(n))) is a known improvement but was unnecessary here.
  • Using try/except IndexError as a loop-termination mechanism is concise in Python but trades a cheap comparison for exception overhead; the explicit bounds check used in the C version is the right habit for performance-sensitive inner loops.
  • Handling the n=1 edge case by returning 2 directly before the sieve runs is cleaner than trying to encode the only even prime inside an odd-only structure.

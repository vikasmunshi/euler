Problem 4: Largest Palindrome Product [Level 0]

Problem Statement

A palindromic number reads the same both ways. The largest palindrome made from the product of two $2$-digit numbers is $9009 = 91 \times 99$.

Find the largest palindrome made from the product of two $3$-digit numbers.

Test Cases

  • main: solve(n=3) → 906609
  • extra: solve(n=4) → 99000099
  • extra: solve(n=5) → 9966006699

Mathematical Complexity

The central insight is that any six-digit palindrome of the form abccba expands in place-value notation as 100001a + 10010b + 1100c, which factors as 11 * (9091a + 910b + 100c). This generalises: every even-length palindrome is divisible by 11. Because the product of two three-digit numbers that forms such a palindrome must therefore be divisible by 11, at least one of the two factors must itself be a multiple of 11. That observation prunes the inner search loop by a factor of 11 without missing any valid candidate.

A naive brute-force search over all ordered pairs of n-digit numbers is O(N 2 ) where N = 9 * 10 n-1 . Restricting one factor to multiples of 11 (when the other is not already one) reduces the inner loop to roughly O(N 2 / 11). Iterating both loops in descending order and breaking the inner loop as soon as the running product falls at or below the current best further cuts the practical work well below that ceiling. For n = 5 the C implementation finishes in under 1 ms and the Python version in under 20 ms.

Solution Approach

Divisibility-by-11 Pruned Descending Search (s0 - Python and C)

I implemented the same algorithm in both Python ( p0004_s0.py) and C ( p0004_s0_c), parameterised by n so that all four test cases (n = 2 through 5) can be validated with a single implementation. The outer loop iterates a downward from the largest n-digit number. For each a, the inner loop iterates b downward starting from the largest n-digit multiple of 11, stepping by 11 - unless a is itself a multiple of 11, in which case b starts at the full maximum and steps by 1. This guarantees at least one factor is always a multiple of 11. I restrict b >= a to avoid counting symmetric pairs twice. The inner loop breaks immediately when a * b falls at or below the current best, since all further products with this a can only be smaller.

The palindrome check differs slightly between the two languages. In Python I compare the string representation of the number to its reverse using str_number == "".join(reversed(str_number)). In C I use snprintf into a stack buffer and walk inward from both ends with two indices, avoiding any heap allocation. There is also one small behavioural difference in the inner loop: the Python version does not break after finding a palindrome for a given a, whereas the C version does. Both are correct because once a palindrome is recorded as the new best, any smaller b in the same inner loop produces a smaller product that cannot improve the result further - the C version simply makes that reasoning explicit with an early break.

Learnings

The key insight was recognising that even-length palindromes are always divisible by 11. It is not immediately obvious, but follows directly from expanding the place-value representation of any symmetric digit sequence with an even digit count. Once I had that factorisation I could step the inner loop by 11 instead of 1, cutting inner work by roughly an order of magnitude without sacrificing correctness.

The second important observation was the early-exit condition. Iterating downward means products are monotonically decreasing for a fixed a. As soon as the product drops at or below the running best, breaking immediately is both correct and essential for speed. Combining descending iteration with a monotone break condition is a powerful general pattern for "find the largest X" searches.

Since I wrote only one algorithmic approach (shared between Python and C), the benchmark comparison is purely a language-performance comparison. The C solution is roughly 25-30x faster than the Python version on the larger test cases: approximately 0.0007 s versus 0.0188 s at n = 5. Both are fast enough to be trivial here, but the translation demonstrates the kind of performance gap that becomes decisive on higher-difficulty problems.

  • Even-length palindromes are always divisible by 11 - a factorisation property worth memorising for any problem involving digit-symmetric structures.
  • Descending iteration combined with a monotone break condition is a clean pattern for bounded "find the maximum" searches without sorting or priority queues.
  • Restricting b >= a halves the search space by eliminating symmetric duplicates - a standard trick for unordered pair enumeration.
  • Parameterising by n rather than hard-coding three digits enables validation across multiple test cases and makes generalisation to larger inputs trivial.
  • In C, a two-pointer palindrome check on a stack-allocated buffer avoids heap allocation and is as readable as the Python string-reversal approach.

Problem 3: Largest Prime Factor [Level 0]

Problem Statement

The prime factors of $13195$ are $5, 7, 13$ and $29$.

What is the largest prime factor of the number $600851475143$?

Mathematical Complexity

The fundamental theorem of arithmetic guarantees that every integer greater than 1 has a unique prime factorization, so the problem reduces to identifying the largest prime in that factorization. The classical approach is trial division: test candidate divisors from 2 up to the square root of the number. The square root boundary is the key insight - if a composite number n has a factor larger than its square root, the paired cofactor must be smaller than the square root and would have been found first. In the worst case, when n is itself prime, this requires O(sqrt(n)) iterations, which for n = 600851475143 is roughly 775,000 steps and for the extra test case 79376318561249 is around 8.9 million steps.

A critical practical refinement is to fully divide out each confirmed factor before advancing the candidate pointer, and then recompute the search ceiling as the integer square root of the reduced remainder. Because small primes are peeled off first, the remaining number collapses quickly and the search limit shrinks far below its initial value, making the algorithm considerably faster in practice than the O(sqrt(n)) bound implies.

Solution Approach

Trial Division with Dynamic Search-Limit Reduction (s0 - Python and C)

I chose a straightforward trial division strategy with two focused optimizations. First, I handle the factor 2 separately at the start, then step through only odd candidates (3, 5, 7, ...), which immediately halves the number of divisions tested. Second - and more impactful - every time a factor is confirmed I call reduce(), which fully divides that factor out of the remaining number before the loop advances. After each such reduction I recompute search_limit as the integer square root of the now-smaller remainder. This causes the upper bound to collapse aggressively as factors are peeled off, so the algorithm self-accelerates through the factorization. When the loop exits, if remaining_number is still greater than 1 it must itself be prime and is returned as the answer; otherwise the last confirmed factor is returned.

The Python and C implementations use identical logic. The Python version computes the square root with int(remaining_number ** 0.5), while the C version uses (long long)sqrt((double)remaining_number). Both behave correctly for all inputs tested. The C version is noticeably faster in absolute terms: on the extra test case ( 79376318561249, largest factor 9999991) the C solution completes in about 0.0084 seconds while the Python solution takes about 0.25 seconds - roughly a 30x speedup - consistent with the overhead of the Python interpreter on a tight arithmetic loop. For the main problem input both are essentially instantaneous at well under a millisecond.

Learnings

The key insight for me was that fully dividing out each confirmed factor before advancing the candidate pointer implicitly guarantees that only prime factors are ever confirmed. By the time the loop reaches a composite candidate such as 9, all factors of 3 have already been stripped from the remaining number, so 9 will never divide it evenly. This means stepping through odd integers yields exactly the same result as stepping through an explicit list of primes, with no sieve or prime-generation overhead at all.

The second important realization was the dynamic search-limit update. Without it the loop would continue testing candidates up to sqrt(600851475143) even after the remaining number had already collapsed to a small cofactor. Recomputing the limit after each reduction keeps the bound tight and is a cheap operation that pays for itself immediately on any input with small prime factors.

Since I only wrote one algorithm for this problem there is no alternative approach to compare, just Python and C ports of the same logic. The roughly 30x speed difference on the extra test case is a useful data point: for tight trial-division loops where the iteration count runs into the millions, moving to C is a straightforward win. I decided that skipping only even numbers was preferable to wheel factorization (which would also skip multiples of 3, eliminating another third of candidates) because the current approach is already fast enough for all test cases and the code stays easy to read.

  • Fully dividing out each factor before advancing the candidate implicitly restricts confirmed factors to primes - no explicit primality test is needed.
  • Recomputing search_limit as isqrt(remaining_number) after each reduction is cheap but highly effective at shrinking the loop bound.
  • When the loop exits with remaining_number > 1, that remainder is always prime and always the largest factor - no additional check is required.
  • Wheel factorization beyond skipping evens (for example also skipping multiples of 3) is worth considering for inputs in the range of 10^15 and above where the raw loop count becomes large enough to matter.
  • For tight arithmetic loops the Python-to-C speedup is consistently around 30x, which matches the benchmark results here and serves as a useful rule of thumb for future problems.

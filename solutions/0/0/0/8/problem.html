Mathematical Complexity

The problem reduces to a sliding-window maximum over a finite digit sequence. With a string of length N = 1000 and a window of size k = 13, there are at most N - k + 1 = 988 candidate windows. Evaluating each window naively requires O(k) multiplications, giving an overall complexity of O(N * k), which for any fixed k is just O(N). No number theory is required - this is entirely a combinatorial scan over a known, bounded sequence, and the answer is guaranteed to fit in a 64-bit integer since the maximum possible product is 9^13, which is well within range.

The structurally important observation is that any window containing a zero digit has a product of zero and can never be the maximum, as long as at least one zero-free window exists (which it does for this input). Filtering those windows out before doing any multiplication is the key mathematical insight. Because N and k are both small, even a fully naive O(N * k) pass completes in microseconds, but recognising zero digits as hard product-reset boundaries is the conceptual move that matters regardless of scale.

Solution Approach

Sliding Window with Zero-Skip (Python and C, index 0)

I embedded the 1000-digit number as a string literal directly in both source files rather than reading from an external file, keeping each solution self-contained and avoiding I/O overhead during benchmarking. Both versions accept the window length as a runtime parameter, which lets me verify the dev test case (length 4, expected 5832) before running the real case (length 13).

The algorithm is identical in both languages: iterate over every consecutive substring of the specified length, skip any window that contains a zero digit, and for surviving windows compute the product of individual digits, tracking the running maximum. In Python I expressed this as a list comprehension with an if "0" not in sequence guard, converting each surviving character to an integer and passing the resulting generator to math.prod. I chose math.prod (available since Python 3.8) over a manual functools.reduce loop because it is concise and internally optimised. In C I used a two-pass inner loop: a short-circuit scan that sets a flag on the first zero encountered, followed by the multiplication loop only if no zero was detected. The logic is functionally identical; the C version simply avoids Python object allocation and interpreter dispatch entirely, which is why the benchmark shows it completing in effectively zero measurable time compared to the Python version's already fast sub-millisecond results.

Learnings

The key insight was treating zero digits as hard reset boundaries. Any window touching a zero is automatically disqualified, which immediately eliminates a large fraction of the search space without any arithmetic. For this input and window size the observation is almost trivially obvious, but the same pattern scales: for much larger N or k, partitioning the string around zeros first and only evaluating windows within each zero-free segment is the natural path toward a truly linear-time solution with minimal constant factors.

Since there is only one algorithmic approach (index 0) expressed in two languages, the benchmark comparison is simply a language overhead comparison. The Python solution completes in roughly 0.0004-0.0005 seconds and the C solution in effectively 0.0000 seconds. The difference is entirely interpreter startup, import costs, and object allocation overhead rather than anything intrinsic to the algorithm, since the actual computation is trivial in both cases. There was no compelling reason to pursue a more elaborate strategy such as maintaining a running product with lazy division or a log-sum transformation, given that even the slower Python version is comfortably fast.

Concepts worth remembering from this problem:

  • Zero digits in a product series act as hard reset boundaries - filter or partition around them before computing anything.
  • math.prod is a clean, readable way to compute the product of an iterable in modern Python (3.8+).
  • Parameterising the window length and providing a smaller dev test case is a useful sanity-check pattern when the full input is difficult to inspect by hand.
  • Embedding fixed input data as a string literal rather than reading from a file keeps solutions self-contained and avoids I/O overhead in benchmarks.
  • For sliding-window maximum-product problems at scale, a log-sum approach (converting products to sums of logarithms) or a deque-based running product with divide-and-conquer updates can reduce per-window cost from O(k) to O(1) amortised.

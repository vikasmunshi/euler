Problem 98: Anagramic Squares [Level 8]

Problem Statement

By replacing each of the letters in the word CARE with $1$, $2$, $9$, and $6$ respectively, we form a square number: $1296 = 36^2$. What is remarkable is that, by using the same digital substitutions, the anagram, RACE, also forms a square number: $9216 = 96^2$. We shall call CARE (and RACE) a square anagram word pair and specify further that leading zeroes are not permitted, neither may a different letter have the same digital value as another letter.

Using words.txt (right click and 'Save Link/Target As...'), a 16K text file containing nearly two-thousand common English words, find all the square anagram word pairs (a palindromic word is NOT considered to be an anagram of itself).

What is the largest square number formed by any member of such a pair?

NOTE: All anagrams formed must be contained in the given text file.

Mathematical Complexity

The core challenge is finding a bijective character-to-digit mapping that simultaneously converts two anagram words into perfect squares. The bijection is strict: each distinct letter must map to a unique digit, no two letters may share a digit, and no leading zero is permitted. A critical structural observation is that any valid mapping must preserve the repetition pattern of the source word - if a letter appears twice, the corresponding digit positions in the mapped square must carry identical digits. I encode this as a frequency-signature: I count per-symbol frequencies, sort the counts in descending order, and concatenate them into a short string. If the frequency-signature of a word differs from that of a candidate square, no valid bijection can exist, and I skip the pair entirely without any mapping work.

The asymptotic complexity is manageable at this problem size. The word file contains under two thousand words, and for each word length n the number of n-digit perfect squares is O(10^(n/2)). Grouping words into anagram buckets by their sorted-character canonical form reduces the set of candidate pairs to only genuine anagrams, so the expensive bijection checks are attempted only for promising (word, square) pairs. Processing word lengths from largest to smallest and stopping as soon as any valid pair is found at the longest matching length exploits the problem structure - the largest square must come from the longest word pair - turning a potentially full exhaustive scan into one that terminates very quickly in practice.

Solution Approach

Anagram-Bucketed Bijection Search with Frequency-Signature Pruning (s0, Python and C)

I implemented the same algorithm in both Python and C under index s0. Both follow identical logic; the C version is a manual translation that replaces Python's built-in dictionaries and sets with fixed-size arrays and linear scans.

I first parse the word list and assign every word to a bucket keyed by its sorted-letter canonical form. Buckets containing only a single word are discarded immediately, since a lone word cannot form an anagram pair. For each surviving bucket I precompute all perfect squares of the matching digit length as strings. Before attempting a full character-to-digit mapping for a (word, square) pair, I apply the cheap str_hash pre-filter, which counts per-symbol frequencies, sorts them descending, and concatenates the counts into a string. Only pairs whose str_hash values agree can possibly yield a valid bijection.

For pairs that pass the filter, I build an explicit char_map by zipping the word's characters against the square's digits. I then verify two things: consistency (no character mapped to two different digits) and injectivity (no two distinct characters share a digit). If the mapping is valid, I apply it to every other word in the same anagram bucket and check whether the resulting digit string also appears in the precomputed set of squares. When a match is found I record the maximum of the two square values and return after finishing the current word length.

The Python version uses a frozen=True dataclass whose field holds a mutable set. The frozen flag prevents reassigning the field reference but does not stop mutation of the set itself via .add(); a plain dictionary keyed by the canonical form would have been simpler and clearer. The C version avoids this entirely with fixed-size character arrays and flat arrays of Bucket structs. The C version also resolves the resource file path by reading the executable path via /proc/self/exe and constructing a sibling resources/ path, making the binary portable across different working-directory conventions without hard-coded absolute paths.

Learnings

The key insight that unlocked the solution was recognising that a valid character-to-digit mapping must preserve the repetition structure of the word. If the word has two identical letters, the corresponding digit positions in the mapped square must be identical, and globally the multiset of character frequencies must equal the multiset of digit frequencies. Encoding this as a sorted tuple of counts gives a very cheap pre-filter that eliminates the vast majority of (word, square) candidates before any mapping work is done.

A second important realisation was that once I derive a mapping from one specific (word, square) pairing, I can reuse that same mapping to test every other word in the bucket at negligible extra cost, rather than independently re-deriving a mapping for each word-pair combination. Because an anagram bucket groups all words sharing the same letter set, one derived mapping already encodes all letter-to-digit assignments for every word in the group.

The benchmark results show the C version completing in about 0.061 seconds versus 0.359 seconds for Python - roughly a six-fold speedup. The C version benefits from eliminating Python's dictionary and set overhead, replacing them with linear scans over small fixed-size arrays that are cache-friendly for the word lengths involved in this file.

  • Frequency-signature hashing is a broadly reusable technique for filtering any problem that requires bijective symbol substitutions; it prunes invalid candidates in O(n log n) before doing any mapping work.
  • Grouping candidates by a canonical key before doing expensive pairwise checks is a pattern worth applying whenever the raw pairwise count is much larger than the grouped count.
  • Processing candidates in decreasing order of a metric and returning early at the first successful length turns a full exhaustive search into one that terminates quickly when the answer lies at the largest candidate size.
  • Applying a derived mapping to all other members of a group - rather than re-deriving mappings independently for every pair - avoids redundant work and is worth recognising as a general optimisation whenever a shared structure links multiple candidates.
  • Using frozen=True on a Python dataclass that contains a mutable container field such as a set is misleading: the field reference is frozen but the object itself remains mutable. A plain dictionary keyed by the canonical form is simpler and clearer.
  • In C, resolving resource file paths relative to the executable via /proc/self/exe is a practical pattern for making binaries portable across different working-directory conventions without hard-coded absolute paths.

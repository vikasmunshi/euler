Problem 92: Square Digit Chains [Level 1]

Problem Statement

A number chain is created by continuously adding the square of the digits in a number to form a new number until it has been seen before.

For example, $$\begin{align} &44 \to 32 \to 13 \to 10 \to \mathbf 1 \to \mathbf 1\\ &85 \to \mathbf{89} \to 145 \to 42 \to 20 \to 4 \to 16 \to 37 \to 58 \to \mathbf{89} \end{align}$$

Therefore any chain that arrives at $1$ or $89$ will become stuck in an endless loop. What is most amazing is that EVERY starting number will eventually arrive at $1$ or $89$.

How many starting numbers below ten million will arrive at $89$?

Mathematical Complexity

The core observation is that every positive integer eventually converges to either 1 or 89 under repeated application of the "sum of squares of digits" function. The crucial insight enabling an efficient solution is that this function maps any n-digit number to a value at most n * 81, since each digit contributes at most 9^2 = 81. For numbers below 10^7, the maximum digit-square-sum is 7 * 81 = 567, which is tiny compared to the 10 million numbers in the problem. This compresses the problem into two tractable sub-problems: (1) determine which values in [1, 567] eventually reach 89 under the chain rule, and (2) count how many numbers in [1, 10^n] map to each such value.

The counting step is handled with a generating-function argument. Each digit d contributes d^2 to the digit-square-sum, so the contribution of a single digit position is represented by the polynomial 1 + x + x^4 + x^9 + x^16 + x^25 + x^36 + x^49 + x^64 + x^81, where the constant term accounts for digit 0. The coefficient of x^s in the n-th power of this polynomial gives exactly the count of n-digit sequences (with leading zeros permitted) whose digit-square-sum equals s. Multiplying this polynomial n times and accumulating into an array is equivalent to dynamic programming over the digit-square-sum value. The state space has size O(n * 81), so the total work is O(n^2 * 81^2), which is effectively constant for the problem sizes involved.

Solution Approach

Generating-function convolution (s0 - Python and C)

I maintain an array a where a[i] counts how many digit sequences considered so far produce a digit-square-sum of i, and a boolean array is89 that records whether index i eventually reaches 89. A helper function terminates_in_89 performs direct simulation for any individual value.

The iteration proceeds one digit position at a time. On each step I copy the current array into b, extend a by 81 zeroed slots, populate is89 for the new indices, and then for every index i with count v = b[i], I add v to a[i + s] for each of the nine non-zero digit squares s in {1, 4, 9, 16, 25, 36, 49, 64, 81}. The existing values a[0..prev_len-1] are left in place from the copy, which implicitly handles digit 0 (whose square is 0 and leaves the sum unchanged). This correctly implements multiplication by the per-digit polynomial factor. After n iterations, the answer is the sum of a[i] over all i with is89[i] true.

The Python version uses list slicing, enumerate, and map for conciseness, and supports a --show flag to print intermediate results for all powers of 10. The C version replicates the same logic using malloc, memcpy, and memset, allocating the full eventual maximum array size up front and freeing the temporary b buffer after each step. The C version computes all test cases in time that rounds to zero microseconds, while the Python version stays under 2.2 ms even for n = 9. Both are correct across all test cases.

Learnings

The key insight was recognising that I did not need to iterate over every number from 1 to 10^7 at all. Instead of asking "does number n reach 89?", I reframed it as "how many n-digit numbers have digit-square-sum s?" and answered that with a polynomial convolution running in well under 2 ms even in Python. The bound n * 81 is tight enough that the reachable state space is tiny no matter how large n grows, making the approach scale to n = 9 (answer around 854 million) with essentially no additional cost.

A subtlety I had to be careful about was the treatment of digit 0. Because 0^2 = 0, including digit 0 means the polynomial factor has a constant term of 1, and the coefficient array must not be zeroed before each distribution step - the old values must survive to represent the "add a zero digit" contribution. Implementing this by preserving existing array entries rather than re-zeroing them is a clean and correct way to express multiplication by the constant term.

Since only a single algorithm (s0) is implemented here, there is no algorithmic comparison to make. The benchmark does confirm, however, that the C translation offers a roughly three-orders-of-magnitude speedup over the Python version for larger inputs, consistent with the overhead difference between interpreted and compiled loops. For future problems involving large input sets that map into a small bounded state space, this generating-function counting trick - iterate over the small state space rather than over the large input set, and use a polynomial or DP to count paths - is well worth keeping in mind.

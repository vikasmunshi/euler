Problem 99: Largest Exponential [Level 2]

Problem Statement

Comparing two numbers written in index form like $2^{11}$ and $3^7$ is not difficult, as any calculator would confirm that $2^{11} = 2048 \lt 3^7 = 2187$.

However, confirming that $632382^{518061} \gt 519432^{525806}$ would be much more difficult, as both numbers contain over three million digits.

Using base_exp.txt (right click and 'Save Link/Target As...'), a 22K text file containing one thousand lines with a base/exponent pair on each line, determine which line number has the greatest numerical value.

NOTE: The first two lines in the file represent the numbers in the example given above.

Mathematical Complexity

The core challenge is comparing numbers of the form a^b where direct evaluation is impossible - the winning number contains over three million digits. The key insight is that the natural logarithm is strictly monotonically increasing over the positive reals, so a^b > c^d if and only if b*ln(a) > d*ln(c). This follows immediately from the identity log(a^b) = b*log(a). Applying this transformation converts an otherwise intractable comparison of astronomically large integers into a trivial comparison of ordinary floating-point numbers, each requiring exactly one logarithm evaluation and one multiplication.

The asymptotic complexity is O(n) in the number of base/exponent pairs. Each pair requires a constant amount of work - one logarithm call, one multiplication, and one comparison against a running maximum. There is no sorting, no recursion, and no combinatorial structure to exploit. A single linear scan over the 1000 lines is both necessary and sufficient.

Solution Approach

Logarithmic Linear Scan (s0 - Python and C)

Both implementations follow identical logic: parse the comma-separated file into (base, exponent) integer pairs, then perform a single pass maintaining a running maximum of exponent * log(base). Whichever line produces the largest value of this product is returned as the 1-based line index matching the problem convention. Both versions guard against non-positive values to avoid domain errors in the logarithm, and both handle malformed lines by printing a warning and continuing rather than aborting.

The Python version uses math.log inside a straightforward enumerate loop. Under the --show flag it also surfaces a human-readable summary including the approximate digit count of the winning number, computed as floor(max_val / log(10)) + 1. The C version uses strtok_r for line tokenisation and atoll for integer parsing, applying log from math.h. The file-loading helper in C resolves the resource path relative to the executable using readlink("/proc/self/exe"), mirroring the Python convention of looking in a sibling resources/ directory.

Learnings

The key unlock was recognising that comparing large exponentials is equivalent to comparing their logarithms. Once I applied log(a^b) = b*log(a), the problem collapsed from an impossible direct computation into a trivial floating-point scan. This trick appears frequently in Project Euler whenever numbers are too large to evaluate directly but their relative ordering is all that is required - I should reach for it immediately whenever large exponentials appear.

The benchmark confirms that both implementations are effectively instantaneous in practice. The C version completes in roughly 0.0002 s versus 0.0006 s for Python - about three times faster here. Both perform identical O(n) work, so the difference is driven entirely by interpreter overhead rather than any algorithmic distinction. The Python version is marginally more expressive, particularly with the --show digit-count summary, while the C version demonstrates how little overhead the algorithm itself carries when stripped of the runtime.

  • Strictly monotone transformations preserve ordering - applying ln to both sides of an inequality is always valid for positive reals, and this should be the first instinct whenever exponentials are too large to evaluate directly.
  • The approximate base-10 digit count of a^b is floor(b * log10(a)) + 1, a useful sanity check worth surfacing in verbose output.
  • Robust file parsing - skipping malformed lines with a warning rather than aborting - costs almost nothing and makes solutions far more resilient to unexpected input variations.
  • For future problems involving comparisons or orderings of very large exponentials, products, or ratios, taking logarithms immediately rather than attempting direct computation is almost always the correct first move.

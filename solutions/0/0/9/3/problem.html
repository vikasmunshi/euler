Problem 93: Arithmetic Expressions [Level 9]

Problem Statement

By using each of the digits from the set, $\{1, 2, 3, 4\}$, exactly once, and making use of the four arithmetic operations ($+, -, \times, /$) and brackets/parentheses, it is possible to form different positive integer targets.

For example,

$$\begin{align} 8 &= (4 \times (1 + 3)) / 2\\ 14 &= 4 \times (3 + 1 / 2)\\ 19 &= 4 \times (2 + 3) - 1\\ 36 &= 3 \times 4 \times (2 + 1) \end{align}$$

Note that concatenations of the digits, like $12 + 34$, are not allowed.

Using the set, $\{1, 2, 3, 4\}$, it is possible to obtain thirty-one different target numbers of which $36$ is the maximum, and each of the numbers $1$ to $28$ can be obtained before encountering the first non-expressible number.

Find the set of four distinct digits, $a \lt b \lt c \lt d$, for which the longest set of consecutive positive integers, $1$ to $n$, can be obtained, giving your answer as a string: abcd .

Mathematical Complexity

The core challenge is enumerating every value producible from four operands combined with three binary operators (+, -, x, /) under all possible parenthesizations and operand orderings. For n operands the number of distinct binary expression tree shapes is the (n-1)th Catalan number C(n-1), which for n=4 gives 5 tree shapes. Combined with 4! = 24 operand orderings and 4^3 = 64 operator assignments, the naive upper bound is 5 x 24 x 64 = 7,680 expressions per digit set. The outer loop over all four-element subsets of {1,...,9} adds C(9,4) = 126, keeping the total search entirely tractable. Consecutive-run scoring then requires only a linear scan per subset, so the overall work is dominated by expression enumeration.

Rather than enumerating expression trees explicitly by shape, I recognized that every binary expression tree over a multiset of values can be described recursively as: pick any two values currently in the pool, combine them with one operation to produce a single new value, and recurse on the reduced pool. This set-reduction framing covers every parenthesization and operand ordering simultaneously, because every distinct tree topology corresponds to some sequence of such pair-picks. When the pool is represented as a sorted tuple, equivalent intermediate states reached by different pair-pick paths collapse to the same cache key, enabling effective memoization.

Solution Approach

Recursive Pair-Reduction with Memoization (Python and C)

Both solutions share the same algorithmic idea: a recursive function accepts the current pool of values and returns the set of all reachable scalar results. The base case returns the single remaining value. In the recursive case I iterate over every unordered pair (i, j) from the pool, apply up to five operations, and recurse on the reduced pool with the newly combined value appended.

The five operations are: addition (a+b), absolute difference (|a-b|), multiplication (a*b), and both division orders (a/b if b is nonzero, b/a if a is nonzero). Using an unordered pair avoids duplicating commutative results for addition and multiplication, while computing both division orders and the absolute difference covers the non-commutative cases. Throughout, values are stored as floating-point numbers and integrality is tested at the very end - with x.is_integer() in Python and an epsilon comparison against floor(v + 0.5) in C - which cleanly handles rational intermediates arising from chained divisions without requiring exact rational arithmetic.

The Python version represents the pool as a sorted tuple so that lru_cache identifies equivalent intermediate states regardless of how they were reached. For example, the pool {3.0, 5.0} reached via two different initial pair-picks maps to the same cache key (3.0, 5.0), collapsing redundant sub-computations. The C version omits memoization entirely - the problem is small enough (126 subsets, each with a bounded recursion depth of 3) that the raw recursion still completes quickly. The C version uses a small dynamic DoubleSet struct with exact-equality deduplication, which is sound here because all intermediate values arise from the same finite sequence of arithmetic on integer starting points. Both solutions scan the produced result set for the longest unbroken consecutive integer run starting at 1 and track the best digit combination found across all 126 subsets.

Learnings

The key insight was reframing expression-tree enumeration as repeated operand-pair reduction. Once I recognized that "pick any two values, combine them, recurse on the reduced pool" is equivalent to evaluating every possible parenthesization and operand ordering, the recursive structure became obvious and the memoization fell out naturally. Representing the pool as a sorted tuple is the essential trick that makes caching effective - without sorting, permutations of the same multiset of intermediate values would appear as distinct cache keys, negating most of the memoization benefit.

A subtlety worth remembering: using |a-b| as the single subtraction result merges both signed differences into one non-negative value. This is sufficient for this problem because no negative intermediate is required to reach an otherwise-unreachable positive integer target, but in a more general setting both a-b and b-a would need to be tracked separately.

The benchmark results reveal an interesting reversal of the usual performance pattern:

Solution Average time (s)
p0093_s0.py 0.0184
p0093_s0_c 0.0238

The Python version is actually faster than the C version. I attribute this to the lru_cache memoization in the Python implementation eliminating enough redundant recursive work to more than compensate for interpreter overhead, while the C version performs the full recursion without any caching. This is a useful reminder that memoization can dominate raw execution speed even for problems that appear small enough to brute-force, and that a cached interpreted solution can outperform an uncached compiled one.

Two broadly reusable techniques emerged from this problem: the sorted-tuple memoization over multisets of intermediate values for any combine-and-reduce combinatorial search, and the pattern of accumulating floating-point intermediates throughout a computation and deferring integrality checks to the very end to handle chains of division without symbolic fractions.

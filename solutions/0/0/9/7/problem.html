Problem 97: Large Non-Mersenne Prime [Level 1]

Problem Statement

The first known prime found to exceed one million digits was discovered in 1999, and is a Mersenne prime of the form $2^{6972593} - 1$; it contains exactly $2\,098\,960$ digits. Subsequently other Mersenne primes, of the form $2^p - 1$, have been found which contain more digits.

However, in 2004 there was found a massive non-Mersenne prime which contains $2\,357\,207$ digits: $28433 \times 2^{7830457} + 1$.

Find the last ten digits of this prime number.

Mathematical Complexity

The central insight is that "last ten digits" is exactly the same as computing a value modulo 10 10 . Because multiplication distributes over modular reduction - (a * b) mod m = ((a mod m) * (b mod m)) mod m - I can reduce intermediate results at every step without affecting the final answer. This means I never have to materialise the astronomically large number 28433 * 2 7830457 + 1; the working value is always bounded to at most ten digits. The problem reduces entirely to modular exponentiation: compute (28433 * pow(2, 7830457, 10 10 ) + 1) mod 10 10 .

A naive loop that multiplies by 2 and reduces modulo 10 10 on each iteration runs in O(e) time where e = 7830457 is the exponent - nearly eight million steps. Fast modular exponentiation using the square-and-multiply algorithm reduces this to O(log e) multiplications, roughly 23 squarings, because the binary decomposition of the exponent drives the number of steps. For this particular exponent the linear approach still finishes in under a second, but the logarithmic method is the correct general tool whenever exponents are large.

Solution Approach

Iterative Modular Multiplication (Python and C, index 0)

My Python solution in large_non_mersenne_prime parses the prime expression string by splitting on spaces, extracting the coefficient (28433) from the first token and the exponent (7830457) by stripping the 2^ prefix from the third token. It then enters a plain loop that runs exactly exponent times, multiplying an accumulator by 2 and reducing modulo 10**num_digits on every iteration, then adds 1 and takes one final modulus. I chose to parse from a formatted string argument so the function remains general and avoids hard-coding any constants. The working integer is always bounded to at most ten digits, so each multiplication is O(1) in practice, but the loop still executes nearly eight million iterations, taking about 0.70 seconds.

My C solution implements a standard binary square-and-multiply algorithm in mod_pow. At each step of the binary decomposition of the exponent, I square the base and conditionally multiply into the accumulator, then right-shift the exponent by one bit. Because the intermediate product of two values each up to 10 10 can exceed 64 bits, I cast both operands to __int128 before multiplying and then reduce modulo the divisor before storing back into a long long. The main solve function concatenates command-line tokens into an expression string, extracts the coefficient and exponent using atoll and strchr, calls mod_pow(2, exponent, divisor), multiplies by the coefficient modulo the divisor, and adds 1. The algorithm is O(log e) rather than O(e), and the result arrives in effectively zero measurable time. Both the Python and C files implement conceptually the same overall formula, but differ in the exponentiation strategy: the Python file uses a linear loop while the C file uses binary exponentiation.

Learnings

The key insight is recognising that extracting the last k digits of any integer is identical to computing that integer modulo 10 k , and that this reduction can be applied eagerly at every multiplication step so the working value never grows large. Once I saw that, the implementation became almost mechanical.

The benchmark comparison makes the algorithmic difference vivid: the Python iterative loop takes about 0.70 seconds while the C fast-exponentiation solution is effectively instant (under 1 microsecond). Two compounding factors create this gap - the O(e) versus O(log e) step count, and C versus Python execution overhead. Either factor alone would give a large speedup; together they produce a difference of roughly four orders of magnitude. If I were writing the Python solution again, I would replace the manual loop with Python's built-in three-argument pow(2, 7830457, 10**10), which is implemented in C and uses fast exponentiation internally, reducing the entire computation to a single call running in microseconds and making the Python version competitive with the C version in elapsed time.

  • Last k digits of n equal n mod 10 k ; apply this reduction eagerly at every multiplication to keep intermediate values bounded.
  • Fast modular exponentiation (square-and-multiply) runs in O(log e) steps versus O(e) for a naive loop - always prefer it for large exponents.
  • In C, intermediate products of two mod-m values can exceed 64 bits when m is around 10 10 ; casting to __int128 before multiplying and reducing is a clean, portable fix.
  • Python's built-in pow(b, e, m) is implemented in C and uses fast exponentiation internally - it should be the first-reach tool for this pattern in Python.
  • The "last k digits" pattern recurs across several Project Euler problems and is worth keeping as an immediately recognised technique.

Problem 90: Cube Digit Pairs [Level 9]

Problem Statement

Each of the six faces on a cube has a different digit ($0$ to $9$) written on it; the same is done to a second cube. By placing the two cubes side-by-side in different positions we can form a variety of $2$-digit numbers.

For example, the square number $64$ could be formed:


In fact, by carefully choosing the digits on both cubes it is possible to display all of the square numbers below one-hundred: $01$, $04$, $09$, $16$, $25$, $36$, $49$, $64$, and $81$.

For example, one way this can be achieved is by placing $\{0, 5, 6, 7, 8, 9\}$ on one cube and $\{1, 2, 3, 4, 8, 9\}$ on the other cube.

However, for this problem we shall allow the $6$ or $9$ to be turned upside-down so that an arrangement like $\{0, 5, 6, 7, 8, 9\}$ and $\{1, 2, 3, 4, 6, 7\}$ allows for all nine square numbers to be displayed; otherwise it would be impossible to obtain $09$.

In determining a distinct arrangement we are interested in the digits on each cube, not the order.

  • $\{1, 2, 3, 4, 5, 6\}$ is equivalent to $\{3, 6, 4, 1, 2, 5\}$
  • $\{1, 2, 3, 4, 5, 6\}$ is distinct from $\{1, 2, 3, 4, 5, 9\}$

But because we are allowing $6$ and $9$ to be reversed, the two distinct sets in the last example both represent the extended set $\{1, 2, 3, 4, 5, 6, 9\}$ for the purpose of forming $2$-digit numbers.

How many distinct arrangements of the two cubes allow for all of the square numbers to be displayed?

Mathematical Complexity

The problem reduces to combinatorial enumeration over subsets of the digits 0 through 9. Each cube holds exactly six distinct digits, so there are C(10,6) = 210 possible configurations per cube. Because only the digit sets matter - not which physical cube is which - the number of unordered pairs I need to examine is C(210,2) + 210 = 22,155. For each candidate pair I must verify that all nine two-digit perfect squares (01, 04, 09, 16, 25, 36, 49, 64, 81) can be displayed, where for each square at least one assignment of its tens digit to one cube and its units digit to the other must be valid.

The non-trivial twist is that 6 and 9 are physically interchangeable when a face is rotated 180 degrees, so any cube carrying either digit effectively carries both for coverage purposes. Beyond this single rule the problem is entirely mechanical subset enumeration. Every bound is a fixed constant, making the total work O(1) in a strict sense - at most 22,155 pairs each checked against 9 target squares, each check requiring at most a small linear scan over six elements, giving roughly 200,000 elementary operations in total.

Solution Approach

Brute-Force Combination Enumeration with 6/9 Equivalence (Python and C)

I implemented the same algorithm in both Python and C. The approach has three layers. First, I generated all C(10,6) = 210 six-element combinations of digits 0-9. In Python I used itertools.combinations(range(10), 6); in C I used six nested loops with strictly increasing index bounds, which naturally produces the same 210 combinations stored in a fixed 210x6 array.

Second, I wrote a can_display predicate that takes a cube and a requested digit and returns true if the cube can show that digit, applying the 6/9 rule at query time: if the requested digit is 6 or 9, the cube satisfies the requirement as long as it contains either 6 or 9. I kept cube representations as plain tuples or integer arrays of exactly six digits without any preprocessing - the equivalence is enforced only at the point of checking, which keeps the data structures clean and avoids any risk of inconsistently applying the rule.

Third, can_pair_display_all iterates over the nine target squares and for each one checks both orientations (tens digit on cube1 with units digit on cube2, or vice versa) using an OR condition, returning false immediately on the first square that cannot be satisfied. The outer double loop runs over all unordered pairs with j >= i, which avoids double-counting symmetric arrangements without any explicit deduplication step. Both versions produce the correct answer of 1217. The C build runs in approximately 1.3 ms versus 15.3 ms for Python - roughly a 12x speedup - which is expected because the inner loops are entirely numeric with no memory allocation in the C version.

Learnings

The key insight that unlocked the problem was recognising the 6/9 symmetry as a set-membership broadening encoded in a single predicate rather than a special case scattered through the enumeration. By writing can_display to treat 6 and 9 as interchangeable at query time, the cube representations stayed clean and the rule was applied consistently everywhere without duplicating or modifying any combination data.

A second important realisation was that each square only needs to be satisfiable in at least one orientation - tens digit on cube1 or tens digit on cube2 - not in a fixed orientation. The OR inside can_pair_display_all is essential; omitting it would drastically reduce the count of valid arrangements and produce a wrong answer. Concepts and techniques worth remembering for future problems:

  • When two symbols are physically interchangeable, encode that symmetry as a predicate at query time rather than expanding or duplicating data in every combination structure.
  • Unordered pair enumeration with a j >= i guard is a clean, explicit way to avoid double-counting without a separate deduplication pass.
  • For small, fixed-size search spaces - here roughly 22k pairs - brute-force enumeration is entirely sufficient and far easier to verify for correctness than any clever pruning strategy.
  • Early exit on the first failing constraint provides a meaningful practical speedup even though it does not change the asymptotic bound, because most invalid pairs fail on the first or second square checked.

Problem 96: Su Doku [Level 4]

Problem Statement

Su Doku (Japanese meaning number place ) is the name given to a popular puzzle concept. Its origin is unclear, but credit must be attributed to Leonhard Euler who invented a similar, and much more difficult, puzzle idea called Latin Squares. The objective of Su Doku puzzles, however, is to replace the blanks (or zeros) in a 9 by 9 grid in such that each row, column, and 3 by 3 box contains each of the digits 1 to 9. A well constructed Su Doku puzzle has a unique solution and can be solved by logic, although it may be necessary to employ "guess and test" methods in order to eliminate options (there is much contested opinion over this). The complexity of the search determines the difficulty of the puzzle; the example above is considered easy because it can be solved by straight forward direct deduction.

By solving all fifty puzzles find the sum of the 3-digit numbers found in the top left corner of each solution grid; for example, 483 is the 3-digit number found in the top left corner of the solution grid above.

Mathematical Complexity

Sudoku is a constraint satisfaction problem (CSP). The puzzle imposes 27 all-different constraints - one for each of the 9 rows, 9 columns, and 9 three-by-three boxes - over 81 cells, each taking a value from 1 to 9. In the general n-by-n case the problem is NP-complete via reduction to graph colouring, but the fixed 9x9 size caps the theoretical worst case at O(9^k) where k is the number of empty cells. In practice, well-formed puzzles with unique solutions are solved far faster than that bound because constraint propagation prunes enormous branches of the search tree before they are ever explored.

The central algorithmic insight is the Minimum Remaining Values (MRV) heuristic: rather than filling empty cells in reading order, I always attempt the cell whose current candidate set is smallest. This fail-first strategy reduces the effective branching factor at each recursive step. When a cell has only one legal candidate its placement is forced, generating no branching at all. Combined with forward-checking - removing a newly placed digit from the candidate sets of all peer cells immediately after placement - the solver maintains tight domain sizes throughout the search and keeps the recursion shallow even for the harder puzzles in the input file.

Solution Approach

Constraint-Propagating Backtracking with MRV (Python and C)

I implemented the same algorithm in both Python ( p0096_s0.py) and C ( p0096_s0_c). The logic has three phases. First, I scan the grid once to build a list of empty cells, each annotated with its current set of legal candidates computed by checking the cell's row, column, and box against placed digits - this is get_all_empty_cells_with_possibilities in Python and get_all_empty_cells in C. Second, at the start of each recursive call I sort that list by candidate-set size ascending so the most-constrained cell is always attempted first - the MRV ordering. Third, when I place a digit I call update_possibilities to remove that digit from the candidate sets of every peer cell (same row, column, or box), propagating the constraint forward. If any peer's candidate set becomes empty the branch is immediately abandoned - this is the forward-checking pruning step.

There is one structural difference between the two implementations. In Python I sort the mutable cell list in-place at each recursive level and use pop(0) and insert(0, ...) to restore it on backtrack, keeping a single list reference throughout the call stack. Candidate sets are Python set objects, so propagation via discard is O(1) per peer. In C, because arrays are value-typed and the solver is recursive, I allocate a fresh copy of the remaining-cells array on each recursive call with malloc and memcpy, which trades a small heap-allocation cost for simpler ownership semantics with no undo logic needed. I also added an explicit feasibility pre-check in the C version: before recursing I scan the updated cell array and bail out immediately if any candidate count has reached zero, avoiding one extra recursive call per dead end. Both versions produce the correct answer of 24702.

Learnings

The key insight that made the solver fast was recognising that cell selection order matters enormously. Without MRV the solver still terminates, but the search tree can be orders of magnitude larger. With MRV, many cells are effectively forced - their candidate set shrinks to exactly one digit after neighbours are placed - so the solver traverses long chains of forced assignments with no branching at all. Every puzzle in the file, including the hardest, is handled comfortably within the total runtime budget.

The C version runs in 0.0086 s versus 0.0613 s for Python across all 50 puzzles, a roughly 7x speedup. Both are well inside any practical time budget, so the Python solution is entirely adequate for this problem. The speedup in C comes mainly from the absence of interpreter overhead and faster memory operations, not from any algorithmic difference - the solver logic is identical in both languages. I chose not to implement more advanced human-style techniques such as naked pairs, hidden singles, or X-wings because the MRV backtracker already solves all 50 puzzles comfortably; the added code complexity would not be justified.

Concepts worth remembering for future CSP-style problems:

  • MRV (fail-first) heuristic - always branch on the variable with the smallest remaining domain to minimise the effective branching factor.
  • Forward-checking - after each assignment propagate the constraint to peer domains immediately so dead ends are detected as early as possible.
  • Representing candidate sets as Python set objects makes propagation via discard O(1) per peer and keeps the backtrack code clean.
  • Sorting a flat list before each recursive call is simpler to implement correctly than maintaining a priority queue, and for at most 81 cells the sort overhead is negligible.
  • In C, allocating a fresh snapshot of the remaining-cell array per recursive frame is a clean ownership model that avoids explicit undo logic at the cost of small heap allocations; the malloc and free overhead is still fast enough to beat interpreted Python by a wide margin.
  • Remember to raise sys.setrecursionlimit in Python whenever a recursive solver may reach depths of several hundred frames.

Problem 95: Amicable Chains [Level 5]

Problem Statement

The proper divisors of a number are all the divisors excluding the number itself. For example, the proper divisors of $28$ are $1$, $2$, $4$, $7$, and $14$. As the sum of these divisors is equal to $28$, we call it a perfect number.

Interestingly the sum of the proper divisors of $220$ is $284$ and the sum of the proper divisors of $284$ is $220$, forming a chain of two numbers. For this reason, $220$ and $284$ are called an amicable pair.

Perhaps less well known are longer chains. For example, starting with $12496$, we form a chain of five numbers: $$12496 \to 14288 \to 15472 \to 14536 \to 14264 (\to 12496 \to \cdots)$$

Since this chain returns to its starting point, it is called an amicable chain.

Find the smallest member of the longest amicable chain with no element exceeding one million.

Mathematical Complexity

The problem sits at the intersection of number theory and graph theory. The central function is the sum of proper divisors, traditionally written s(n). Iterating s from any starting value produces an aliquot sequence; when such a sequence enters a cycle it forms an amicable chain - a generalisation of perfect numbers (fixed points, cycle length 1) and amicable pairs (cycle length 2). I recognised that s(n) defines a functional graph over the integers: every node n has exactly one outgoing edge pointing to s(n). The task then reduces to finding the longest cycle in that graph whose every node lies in [1, 10^6], and returning the minimum element of that cycle.

Precomputing s(n) for all n up to the limit using an additive divisor sieve costs O(N log N) time and O(N) space - the inner loop increments divisor_sum[j] for every multiple j of each divisor i, analogous in structure to the Sieve of Eratosthenes but accumulating sums rather than setting boolean flags. Walking all chains afterwards costs at most O(N) total steps provided already-visited nodes are memoised so that tails are never retraced. The dominant cost is therefore the sieve, giving an overall complexity of O(N log N).

Solution Approach

Additive Sieve with Path-Based Cycle Detection (Python and C)

Both implementations share the same two-phase algorithm. In the first phase I build a divisor_sum array by iterating every integer i from 1 to N/2 and adding i to every multiple of i in [2i, N] - a single nested loop with inner body divisor_sum[j] += i. This fills the entire table without factoring individual numbers.

In the second phase I walk a chain from each unvisited starting node, collecting the traversed nodes and their positions, then examine what the walk collides with. The Python version maintains a set ch for O(1) membership tests and a list path for ordering. It follows the chain until the next value leaves the valid range, hits zero, or already appears in ch. If the collision value equals the original start i then the entire path is a closed amicable chain; I check whether its length beats the current record and, if so, record i as the smallest member directly - no extra minimum scan needed because the ascending outer loop guarantees that i is already the smallest node in any chain it opens. All path nodes are stored in a global seen dictionary to short-circuit future traversals from interior nodes.

The C version uses the same logic but replaces the hash set with an in_path integer array that stores the 1-indexed position of each node in the current path, giving O(1) lookup without hashing overhead. Because the chain walk in C can close on any interior node rather than only the start, I handle the rho-shaped case explicitly: if in_path[c] is set but c != i, I compute the cycle length as path_len - (in_path[c] - 1), then scan the cycle slice of the path array for the minimum member. Tail nodes (those before the cycle entry point) are marked in seen with value -1 to prevent reprocessing without being counted as chain members. After each starting node the in_path entries for all path nodes are cleared to zero so the array can be reused without reinitialisation.

One key correctness distinction between the two versions: the Python implementation relies on the ascending iteration order to guarantee that when a cycle is first encountered it is entered from its smallest element, so the starting node i is already the answer and no minimum scan is required. The C version makes no such assumption and always scans the cycle slice explicitly, which is the correct general approach when entry order cannot be guaranteed.

Learnings

The central insight was recognising that s(n) defines a functional graph where every node has out-degree exactly one. Framing the problem that way made the algorithm obvious: build the edge table with a sieve, then walk edges while tracking each node's position in the current path so that any cycle closure is immediately detectable and measurable.

The most important correctness subtlety is checking c == i in the Python version rather than simply c in visited. If the walk re-enters the path at some interior node rather than at the start, then i is only a tail leading into someone else's chain and should not be counted as a chain member. The C version handles this more robustly by recording the cycle entry position and extracting only the cycle slice, which is the right approach for the general case.

The benchmark makes the performance gap concrete: the C solution completes in about 0.064 seconds while the Python solution takes roughly 1.93 seconds - a speedup of around 30x. Both are algorithmically identical, so the difference is entirely the cost of Python's interpreted inner sieve loop over one million iterations of i with up to half a million inner steps each. For this scale the C implementation is clearly the practical choice, while the Python version remains the more readable reference implementation.

  • Additive sieves - incrementing all multiples rather than marking composites - are the natural tool for computing divisor sums in bulk and run in O(N log N).
  • Functional graphs admit simple linear-time cycle detection by following the unique outgoing edge and tracking visited nodes with a position array or hash set.
  • Memoising already-processed nodes with a global seen table is essential for keeping total walk time O(N) rather than O(N^2) - without it every starting node on a long tail would retrace the entire tail.
  • When iterating in ascending order the first node to open a cycle is always its smallest member, which can eliminate the explicit minimum scan. The C approach of storing path positions and scanning the cycle slice is more general and worth keeping as a reusable pattern for functional-graph cycle problems.
  • Marking tail nodes differently from cycle nodes (value -1 versus a positive length in the C seen array) is a clean way to distinguish "already processed, not a cycle member" from "already processed, is a cycle member" without a second data structure.

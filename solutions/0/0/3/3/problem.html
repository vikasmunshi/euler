Problem 33: Digit Cancelling Fractions [Level 0]

Problem Statement

The fraction $49/98$ is a curious fraction, as an inexperienced mathematician in attempting to simplify it may incorrectly believe that $49/98 = 4/8$, which is correct, is obtained by cancelling the $9$s.

We shall consider fractions like, $30/50 = 3/5$, to be trivial examples.

There are exactly four non-trivial examples of this type of fraction, less than one in value, and containing two digits in the numerator and denominator.

If the product of these four fractions is given in its lowest common terms, find the value of the denominator.

Mathematical Complexity

The core question is when naive digit cancellation accidentally produces the correct result. For a two-digit numerator and denominator sharing one digit x, the interesting case is x appearing in the units place of the numerator (10n + x) and the tens place of the denominator (10x + d), such that (10n + x) / (10x + d) happens to equal n / d. Cross-multiplying that equality gives the exact integer condition (10n + x) * d = (10x + d) * n, which rearranges to 9nd = x(n - d). This collapses the entire search to single-digit loops over n, d, and x, capping the candidate space at 9 * 9 * 9 = 729 triples. The problem is effectively O(1) - a bounded exhaustive search with a closed algebraic filter.

Once the four valid fractions are found, their product must be reduced to lowest terms. Given the small values involved this is a trivial GCD computation. No asymptotic analysis beyond O(1) is meaningful here.

Solution Approach

Algebraic Constraint Enumeration with Fraction Accumulation (Python and C)

Both solutions share the same algorithm. I enumerate all triples (numerator, denominator, shared digit x) with 1 <= numerator < denominator <= 9 and 1 <= x <= 9, skipping cases where x equals either digit to exclude trivial cancellations such as 33/33. For each triple I test the cross-multiplication condition (10 * numerator + x) * denominator == (10 * x + denominator) * numerator, which is the exact algebraic statement that cancelling x from (10n+x)/(10x+d) yields the correct simplified fraction n/d. The constraint numerator < denominator ensures the fraction is less than one.

In the Python solution I accumulate matching fractions directly as fractions.Fraction objects using functools.reduce with a lambda multiplier, starting from an identity of Fraction(1, 1). Python's Fraction type maintains lowest terms automatically at every multiplication step, so reading the final .denominator attribute gives the answer with no extra GCD work. The entire solve function is a single expression, which I find clean and expressive for a search space this small.

In the C solution I instead accumulate the raw numerator and denominator products as long long integers, then apply one explicit GCD reduction at the end using the standard Euclidean algorithm. The logic is otherwise identical to the Python version. The C solution is more explicit about the GCD mechanics while the Python version is more concise due to the standard library's Fraction type doing the bookkeeping automatically.

Learnings

  • The key insight was to immediately write out the cross-multiplication condition (10n + x) * d = (10x + d) * n rather than reasoning informally about digit positions. Once I had that equation the search space collapsed to a handful of single-digit loops and the problem became straightforward.
  • Excluding x == n and x == d is essential to filter out trivial cancellations. The problem statement flags the 30/50 family as trivial; my loop starting x at 1 already excludes x = 0, and the equality guards handle the remaining degenerate cases naturally.
  • Python's fractions.Fraction combined with functools.reduce lets me express accumulation and automatic reduction as a single expression, with no risk of forgetting a final GCD step. For problems involving products of small rational numbers it is an excellent tool.
  • Both solutions report elapsed time of 0.0000 seconds, so there is no meaningful performance distinction to draw. The C version is marginally more explicit about mechanics while the Python version is more concise. I chose to implement both as the algorithm is short enough that the translation adds no real complexity.
  • For future problems involving accidental arithmetic identities it is worth remembering to translate the coincidence into an exact integer equation immediately. Cross-multiplication is almost always the right first step, and it typically reduces an open-ended digit-search to a constant-bounded loop.

Problem Analysis

The puzzle asks for two-digit fractions n/d below one in value where a naive "cancellation" of a digit shared between numerator and denominator happens, by coincidence, to give the correct reduced fraction. The canonical example is 49/98 = 4/8: cancel the shared 9 and the equation still holds. To pin this down algebraically, I write the numerator with digits a and x (value 10a + x) and the denominator with digits x and b (value 10x + b), so the shared digit x sits at the end of the numerator and the start of the denominator. Cancelling x leaves a/b, and I want exactly those triples where (10a + x) / (10x + b) = a / b. The problem excludes "trivial" cases such as 30/50 = 3/5, where x = 0 merely strips a factor of ten from both numbers; I enforce this by requiring x to be a genuine nonzero digit distinct from both a and b.

The decisive observation is that I must never test this equality in floating point. Comparing two fractions p/q = r/s is equivalent to comparing the integers p*s and r*q - a technique called cross-multiplication. Applied here it yields the exact integer test (10a + x) * b = (10x + b) * a, which sidesteps rounding error entirely. The search space is tiny: a from 1 to 9, b strictly greater than a (so the fraction stays below one), and the shared digit x from 1 to 9 distinct from both. That is at most a few hundred triples, so a brute-force search finishes instantly. Once the four qualifying fractions are found I multiply them and reduce the product to lowest terms; the answer is the resulting denominator. The whole computation is O(1) - a bounded, constant-size enumeration.

Solution Approach

Algebraic Digit Enumeration (Python and C)

Both files implement one algorithm. I iterate the single-digit denominator from 2 to 9, the single-digit numerator from 1 up to (but not including) the denominator - which guarantees the fraction is less than one - and the shared digit x from 1 to 9. I skip any x equal to the numerator or denominator, because those would either be the trivial cancellation or a degenerate match. For the surviving triples I apply the cross-multiplied test (10 * numerator + x) * denominator == (10 * x + denominator) * numerator. Each match contributes one fraction to a running product.

The Python version expresses the whole search as a single generator expression feeding functools.reduce. Each matching triple is materialised as a fractions.Fraction, and the fold multiplies them together starting from the identity Fraction(1, 1). Because Fraction keeps every intermediate result in lowest terms automatically, the final answer is simply the .denominator attribute of the product. I chose this representation because it removes any chance of an arithmetic bug in the reduction step - the standard library does the work.

The C version has no rational type, so I instead accumulate the product of all matching numerators in one variable and the product of all matching denominators in another, then reduce the pair at the end. The reduction uses Euclid's algorithm to compute the greatest common divisor of the two products, and the answer is den_product / gcd. The two implementations are mathematically identical; the C version just makes the accumulation and reduction explicit rather than hiding them behind a library type.

Programming Techniques

Cross-multiplication to avoid floating point

Rather than evaluate (10*n + x) / (10*x + d) == n / d as floating-point division, I rearrange it into an all-integer equality. This is the single most important technique in the solution: floating-point comparison of equality is unreliable because of rounding, whereas integer comparison is exact. Reach for cross-multiplication whenever you need to compare or order fractions - it works in any language and needs no special types.

Generator expression as a lazy search pipeline

The Python search is a generator expression: it yields qualifying fractions one at a time on demand instead of building a list. The filtering guards sit inline next to the values they constrain, which keeps the intent readable. For tiny inputs the memory saving is irrelevant, but the habit matters: whenever you generate values and immediately consume them (folding, summing, taking the first), a generator is the natural choice.

Chained comparison and ordered guards

The guard denominator != x != numerator uses Python's chained comparison, which reads as denominator != x and x != numerator. Placing this cheap exclusion before the expensive cross-multiplication test means the multiplication is skipped on rejected triples. Ordering guard clauses from cheapest to most expensive is a simple, transferable micro-optimisation.

Folding with functools.reduce and an identity element

functools.reduce collapses a sequence into one value by applying a binary operator cumulatively. Here it multiplies all matching fractions, with Fraction(1, 1) as the multiplicative identity element so the result is well-defined even for an empty sequence. Use a fold with an explicit identity whenever you combine a sequence with an associative operator - products, sums, GCDs, logical ANDs.

fractions.Fraction for exact rational arithmetic

Python's fractions.Fraction stores a ratio of integers in fully reduced form and re-normalises on every operation. It removes the need to track numerator and denominator by hand or to write a reduction step. This is the right tool any time a problem demands exact rational results - floating point would introduce errors that could silently corrupt the answer.

Iterative Euclidean GCD in C

The C reduction implements Euclid's algorithm iteratively: repeatedly replace (a, b) with (b, a % b) until b is zero, leaving the GCD in a. This runs in O(log(min(a, b))) time. I prefer the iterative form to avoid call overhead, though at this scale it is immaterial. When no rational library exists, this two-product-and-reduce pattern is the standard substitute for a fraction type.

Key Takeaways

The insight that unlocks this problem is translating the "naively cancelled fraction equals the true fraction" condition into a single cross-multiplied integer equation, then enumerating the handful of digit combinations that satisfy it. The domain is so small that no number theory is required - a direct scan over a few hundred triples is both faster to write and easier to verify than any clever alternative.

Both solutions are correct and run in unmeasurably small time (each reported as 0.0000 s), so performance does not distinguish them. The Python version is the more elegant: fractions.Fraction and the generator-plus-fold pattern make it self-documenting and free of manual reduction logic. The C version is instructive as a template for environments without a rational type, since it shows the accumulation and Euclidean reduction explicitly.

  • When comparing fractions for equality or order, cross-multiply into integers - never trust floating-point equality.
  • For problems with tiny, well-bounded domains, prefer a direct brute-force enumeration; clarity and correctness outweigh cleverness when the search space is constant-size.
  • Use a rational type (such as fractions.Fraction) whenever exact fractional arithmetic is needed; it handles reduction automatically and eliminates a whole class of bugs.
  • Express multi-variable searches as generator expressions with inline guards, ordering cheap exclusion tests before expensive ones to short-circuit work.
  • Know Euclid's GCD by heart - it underpins fraction reduction, modular inverses, and many other number-theoretic primitives.
  • Collapse a filtered sequence with a fold and an explicit identity element when combining values under an associative operator.

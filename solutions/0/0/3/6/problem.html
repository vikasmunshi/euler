Problem 36: Double-base Palindromes [Level 0]

Problem Statement

The decimal number, $585 = 1001001001_2$ (binary), is palindromic in both bases.

Find the sum of all numbers, less than one million, which are palindromic in base $10$ and base $2$.

(Please note that the palindromic number, in either base, may not include leading zeros.)

Mathematical Complexity

A double-base palindrome must read identically forwards and backwards in both base 10 and base 2. The key structural observation is that decimal palindromes are entirely determined by their left half: a d-digit decimal palindrome is parameterised by a ceil(d/2)-digit integer. This means there are only O(10^(d/2)) decimal palindromes below 10^d - a square-root reduction compared to the full range. Generating palindromes directly from their left half is therefore far more efficient than scanning every integer and testing each one for decimal palindromicity. The binary palindrome check requires O(log N) work proportional to the bit-length of N. The overall asymptotic complexity is O(d * 10^(d/2)) where d is the maximum number of decimal digits, compared to O(N log N) for a naive scan over all integers.

The no-leading-zeros constraint mentioned in the problem is automatically satisfied in both bases. Any positive decimal palindrome constructed from a non-zero left half trivially begins with a non-zero digit. Any positive integer's binary representation never begins with a zero by definition, so no special handling is required for the binary side.

Solution Approach

Palindrome Generation with Binary Verification (s0 - Python and C)

I implemented the same algorithm in both Python and C. Rather than iterating over every integer up to the limit and testing for decimal palindromicity, I wrote a generator in Python and an equivalent loop in C that constructs every decimal palindrome directly from its left half. The construction proceeds in two stages:

  • Single-digit numbers 1 through 9 are handled directly, since they are trivially palindromic in base 10.
  • For each integer digits from 1 up to 10^(max_digits/2), I form the reversed string digits_rev and build two families: the even-length palindrome digits_str + digits_rev, and - when 2 * num_digits < max_digits - ten odd-length palindromes digits_str + mid + digits_rev for each middle digit 0-9.

Each candidate is then tested for binary palindromicity. In Python I use bin(number)[2:][::-1] to reverse the bit string and parse it back as a base-2 integer, comparing against the original. In C I extract bits LSB-first into a char array and compare the array forwards and backwards directly - this avoids any string allocation for the binary check and works purely with bit-shifted integers. The guard condition 2 * num_digits < max_digits prevents generating odd-length palindromes whose total length would exceed the allowed digit count; without it, when max_digits is even and the left half is already at maximum length, inserting a middle digit would produce a palindrome one digit too long. The Python and C versions are algorithmically identical - the C version replaces the generator pattern with a flat loop using snprintf for string construction and atoll for parsing.

Learnings

The central insight was recognising that generating palindromes from their left half is far cleaner than testing every candidate number. Given the string "123", the even palindrome is "123321" and the odd palindromes are "1230321" through "1239321". This construction completely eliminates the O(N) scan. The technique of constructing structured objects directly rather than filtering a raw range - palindromes here, primes via sieve elsewhere - is worth carrying forward whenever the objects of interest have a square-root-sized parameterisation.

Since I only implemented one algorithm for this problem, there is no comparison of distinct approaches. I did write it in both Python and C, and the benchmark results show a consistent speedup of roughly 5x from the C version: the main case (6 digits) runs in 0.0002 s in C versus 0.0011 s in Python, and the extended 9-digit case takes 0.0130 s in C versus 0.0614 s in Python. Both are extremely fast in absolute terms, so the Python solution is entirely practical at this problem size. The C speedup comes mainly from avoiding Python object allocation per iteration and from the direct bit-manipulation binary palindrome check, which avoids any string handling entirely.

One subtle point I had to get right was the guard on odd-length palindrome generation. When max_digits is even, the left half can have exactly max_digits/2 digits, and inserting a middle character would produce a palindrome of max_digits + 1 digits - one too many. The check 2 * num_digits < max_digits catches exactly this case. It is also worth noting that the max_digits parameterisation makes the solution easily testable at smaller scales, which is how the dev test cases from 1 through 4 digits were useful for verifying correctness before running the main 6-digit case.

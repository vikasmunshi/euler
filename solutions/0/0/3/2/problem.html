Problem 32: Pandigital Products [Level 1]

Problem Statement

We shall say that an $n$-digit number is pandigital if it makes use of all the digits $1$ to $n$ exactly once; for example, the $5$-digit number, $15234$, is $1$ through $5$ pandigital.

The product $7254$ is unusual, as the identity, $39 \times 186 = 7254$, containing multiplicand, multiplier, and product is $1$ through $9$ pandigital.

Find the sum of all products whose multiplicand/multiplier/product identity can be written as a $1$ through $9$ pandigital.

HINT: Some products can be obtained in more than one way so be sure to only include it once in your sum.

Mathematical Complexity

The central insight is a digit-counting argument. If the multiplicand has a digits and the multiplier has b digits, their product has roughly a + b - 1 to a + b digits. For the concatenated string of all three numbers to use exactly the digits 1 through 9 once each, the total digit count must equal 9, so a + b + len(product) = 9. Combining this with the magnitude constraint - the product of an a -digit and a b -digit number cannot exceed roughly a + b digits - forces a + b to equal exactly 5. This leaves only two viable split configurations: (1, 4) and (2, 3). Any other split either overshoots or undershoots the nine available digit positions and can be ruled out entirely before any arithmetic is done.

The enumeration is purely combinatorial over a fixed alphabet of nine symbols. For each split the number of ordered selections is at most P(9, 5) = 15120 factor pairs, so the total search space is tiny and effectively O(1) in practical terms. Deduplication of products is necessary because the problem explicitly warns that some products can be reached by more than one valid identity - for example, a product reachable by both a (1, 4) and a (2, 3) factorisation must be counted only once in the final sum.

Solution Approach

Constrained Permutation Enumeration (Python and C)

Both solutions implement the same algorithm: enumerate all ordered selections of digits for the multiplicand and multiplier under each of the two valid length splits, compute the product, and check whether the full concatenated string is 1-through-9 pandigital. Valid products are collected into a deduplicating container before summing.

In the Python solution I used itertools.permutations to generate candidates. For each split (a_len, b_len) in ((1, 4), (2, 3)), I first pick an ordered a_len -tuple from the nine digit characters, then pick an ordered b_len -tuple from only the digits not already used in a. This ensures no digit is reused before the product is even computed. The walrus operator ( :=) lets me bind the intermediate string values a_str, b_str, and the integer product c inline within a single generator expression, avoiding redundant recomputation. Valid products flow into a set comprehension for automatic deduplication, and the result is the sum of that set.

The C solution replicates the same logic using flat nested loops. For the (1, 4) split I use four nested for loops over the remaining digit indices, and for the (2, 3) split I use three nested loops. Pandigitality is checked in a small helper, is_nine_pandigital, that formats the three integers into a character buffer with snprintf, verifies the total character count is exactly 9, and confirms each character is a distinct digit from 1 to 9 using a small seen-array. Unique products are stored in a flat integer array with a linear scan for deduplication - acceptable given the small number of valid products (well under 100). The C version runs at roughly 3.3 ms compared to 15.1 ms for Python, about a 4.6x speedup, consistent with the lower overhead of integer arithmetic in C versus Python's object model.

Learnings

The key insight that unlocked this problem immediately is the digit-count arithmetic: reasoning about how many digits each part of the identity must occupy before writing any code collapses the search space from something open-ended into exactly two small cases. Without that, a naive loop over all factor pairs up to some bound would require careful bounding and would run more slowly.

  • Digit-count arithmetic is a powerful pre-filter. Before generating anything, count how many digits each component can have and use the fixed total as a hard constraint.
  • Feeding only the remaining unused digits into the second permutation ( d for d in nine_digits if d not in a) avoids generating candidates with repeated digits entirely, rather than filtering them out after the fact.
  • Collecting results in a set is the natural tool when the problem explicitly warns that some answers appear more than once - it handles deduplication without any extra bookkeeping.
  • The walrus operator ( :=) is genuinely useful inside generator expressions to bind intermediate values; here it lets me compute the factor strings and their integer product once each, keeping the entire solution as a single compact comprehension.
  • Only one algorithm was needed for this problem. The Python and C files share the same approach, with the C version achieving about a 4.6x speedup (3.3 ms vs 15.1 ms) purely from language-level overhead differences rather than any algorithmic distinction.

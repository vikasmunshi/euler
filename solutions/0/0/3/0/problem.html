Problem 30: Digit Fifth Powers [Level 0]

Problem Statement

Surprisingly there are only three numbers that can be written as the sum of fourth powers of their digits: $$\begin{align} 1634 &= 1^4 + 6^4 + 3^4 + 4^4\\ 8208 &= 8^4 + 2^4 + 0^4 + 8^4\\ 9474 &= 9^4 + 4^4 + 7^4 + 4^4 \end{align}$$

As $1 = 1^4$ is not a sum it is not included.

The sum of these numbers is $1634 + 8208 + 9474 = 19316$.

Find the sum of all the numbers that can be written as the sum of fifth powers of their digits.

  • main: solve(n=5) → 443839

Mathematical Complexity

The first task was establishing a finite search space. For a number with k digits, the maximum possible sum of nth powers of its digits is k * 9^n. Once k grows large enough that 10^(k-1) > k * 9^n, no k-digit number can equal the digit-power sum, so the search terminates. For n = 5 this ceiling falls at 6 digits: 6 * 9^5 = 354294 is still 6 digits, but any 7-digit candidate would need at least 10^6 = 1000000 while the maximum 7-digit sum is only 7 * 9^5 = 413343, so it is impossible. I expressed this bound compactly as ceil(log10(n * 9^n)), which gives the digit count of the largest conceivable power sum.

A secondary combinatorial insight drove the efficiency gain: the digit-power property is invariant under permutation of digits. Whether a number equals the sum of the nth powers of its digits depends only on the multiset of those digits, not their order. This means I can enumerate distinct non-decreasing sequences of length k drawn from digits 0-9 rather than scanning every integer up to the bound. The number of such sequences is C(9+k, k), which for k = 6 is C(15, 6) = 5005 - orders of magnitude fewer than a linear scan up to 354294. The overall complexity is effectively O(C(9+k, k)) where k is the digit-count upper bound, a tiny constant for any fixed power.

Solution Approach

Digit-Multiset Enumeration with Candidate Verification (s0 - Python and C)

Both the Python and C implementations share the same algorithm. I first compute upper_bound_num_digits as ceil(log10(n * 9^n)), which gives the number of digits in the largest conceivable digit-power sum. I then enumerate all non-decreasing sequences of that length drawn from digits 0-9, which corresponds exactly to enumerating all multisets of digits up to that size. For each multiset I compute the candidate num as the sum of the nth powers of the multiset elements, then apply a two-stage check: num > 9 to exclude trivial single-digit cases, and then verify that the digits of num itself, when raised to the nth power and summed, reproduce num. The second check is needed because the candidate sum may have fewer digits than the chosen multiset length - leading zeros collapse naturally, and rechecking num 's actual digit decomposition catches any mismatch cleanly without special-casing.

In Python I used itertools.combinations_with_replacement(range(10), upper_bound_num_digits) with a generator expression and a walrus operator to keep the logic concise. In C I implemented the same non-decreasing sequence enumeration manually: I maintain an integer array of length k initialised to zero and advance it by finding the rightmost position not yet at 9, incrementing it, and filling all subsequent positions with the same value. The C version uses a custom ipow function for integer exponentiation and a digit_power_sum helper for the verification step. Both versions accept the power n as a runtime argument, making them general for any digit-power variant - the dev test uses n = 4 and the main test uses n = 5.

Learnings

The key insight that unlocked the solution was recognising that the digit-power property is permutation-invariant. Once I asked whether the property depends on digit order or only on the multiset of digits, the jump from a linear integer scan to combinations_with_replacement was immediate, collapsing the candidate pool from hundreds of thousands down to roughly 5000 multisets. The upper-bound formula ceil(log10(n * 9^n)) is a compact one-liner worth memorising for any narcissistic-number class of problem.

Since I only implemented one algorithm for this problem, there is no algorithmic comparison to make. The benchmark results do show the expected language-level speedup: the C version runs in around 0.0001-0.0002 seconds while the Python version takes 0.0026-0.0073 seconds, a roughly 25-35x difference. Both are effectively instantaneous given the tiny candidate space, so the language choice is immaterial for correctness or practical runtime here.

Key concepts worth remembering for future problems:

  • Digit-power properties are permutation-invariant; always consider multiset enumeration over integer scanning when the property depends only on the set of digits.
  • The upper bound on digit count follows from ceil(log10(n * 9^n)), a reliable ceiling for digit-power problems at any fixed power.
  • itertools.combinations_with_replacement is the natural Python tool for multiset enumeration; in C the equivalent is a simple non-decreasing array increment loop.
  • A two-stage check - compute a candidate sum from the multiset, then verify against the candidate's own digit decomposition - handles leading-zero collapse and keeps correctness easy to reason about without any special-casing.

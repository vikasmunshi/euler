Problem 34: Digit Factorials [Level 0]

Problem Statement

$145$ is a curious number, as $1! + 4! + 5! = 1 + 24 + 120 = 145$.

Find the sum of all numbers which are equal to the sum of the factorial of their digits.

Note: As $1! = 1$ and $2! = 2$ are not sums they are not included.

Mathematical Complexity

The core mathematical question is how large a candidate number could possibly be. For a number with n digits, the maximum digit-factorial sum is n * 9! = n * 362880. This ceiling grows only linearly in n, while the smallest n -digit number grows exponentially. At n = 8 the ceiling is 2,903,040, which is itself only a 7-digit number - meaning no 8-digit number can ever equal the sum of its digit factorials. The search space is therefore provably finite and confined to numbers with at most 7 digits, giving a concrete upper bound of around 10 million candidates in the naive case.

I exploited a further structural property to shrink the search space dramatically: the digit-factorial sum depends only on the multiset of digits, not their arrangement. Enumerating sorted combinations with replacement of up to 7 digits drawn from {0..9} - rather than scanning every integer - reduces the candidate count to a few hundred multisets. The number of such multisets for digit length k is given by the stars-and-bars formula C(k + 9, k), and summing across lengths 2 through 7 yields only a few hundred combinations total. The overall complexity is effectively constant for this fixed-size problem.

Solution Approach

Combinatorial Multiset Enumeration with Digit-Factorial Verification (s0)

Both the Python and C implementations share the same algorithm. I iterate over all non-decreasing sequences of digits of each length from 2 to 7, which corresponds to enumerating all multisets of digits without redundancy. For each multiset I compute the digit-factorial sum exactly once and then apply three filters in order. First, the factorial sum must have exactly the right number of digits for the current length. Second, every digit in the generating multiset must appear at least as many times in the factorial sum's decimal representation. Third, as a definitive self-referential check, the factorial sum must equal the digit-factorial sum of its own digits. This last check is necessary because a sum can pass the first two filters while still not being a true factorion - the digit frequencies of the factorial sum can differ from those of the generating multiset in subtle ways.

I precomputed all ten single-digit factorials in a lookup table in both languages - a dictionary keyed by digit characters in Python and a static integer array in C - so each per-digit factorial lookup is O(1). In Python I used itertools.combinations_with_replacement to generate the multisets declaratively inside a generator expression, keeping the code compact and readable. In C I implemented the same enumeration manually using a k -element integer array maintained in non-decreasing order, advancing it with a carry-like increment from the rightmost position. The logic is identical across both languages. The C version runs in approximately 0.4 ms versus 14 ms for Python, roughly a 35x speedup, though both are negligible in absolute terms.

Learnings

  • The key insight was recognising that the digit-factorial sum is symmetric in the digits - it depends only on which digits appear and how many times, not on their arrangement. Switching from iterating over all integers to iterating over sorted digit multisets collapses the search space from millions of candidates to a few hundred multisets.
  • Establishing a provable upper bound before writing any code is essential here. The crossing-point argument - n * 9! grows linearly while the smallest n -digit number grows exponentially - gives a concrete ceiling at 7 digits and eliminates any need for an arbitrary cutoff chosen by inspection.
  • The three-condition filter handles all edge cases cleanly. The first condition (correct digit count) eliminates most candidates immediately. The second (subset containment) catches cases where the factorial sum uses digits absent from or insufficiently represented in the generating multiset. The third (self-referential equality) is the definitive test and handles cases where the first two conditions are satisfied but the sum is still not a factorion.
  • Precomputing small lookup tables for digit factorials is a cheap habit worth keeping. Even in Python the dictionary lookup avoids calling math.factorial inside a tight inner loop, and in C the static array makes each access a single array dereference.
  • Only one algorithm is implemented here, so there is no cross-approach comparison to make. The C translation of the Python algorithm runs about 35 times faster (0.4 ms vs 14 ms), which reflects typical interpreter overhead rather than any algorithmic distinction. Both are well within any reasonable time budget for this problem.

Problem 38: Pandigital Multiples [Level 1]

Problem Statement

Take the number $192$ and multiply it by each of $1$, $2$, and $3$:

$$\begin{align} 192 \times 1 &= 192\\ 192 \times 2 &= 384\\ 192 \times 3 &= 576 \end{align}$$

By concatenating each product we get the $1$ to $9$ pandigital, $192384576$. We will call $192384576$ the concatenated product of $192$ and $(1,2,3)$.

The same can be achieved by starting with $9$ and multiplying by $1$, $2$, $3$, $4$, and $5$, giving the pandigital, $918273645$, which is the concatenated product of $9$ and $(1,2,3,4,5)$.

What is the largest $1$ to $9$ pandigital $9$-digit number that can be formed as the concatenated product of an integer with $(1,2, \dots, n)$ where $n \gt 1$?

Mathematical Complexity

The core constraint is that concatenating x*1, x*2, ..., x*n must produce a 9-digit number using each of the digits 1 through 9 exactly once. Because the total digit count must equal exactly 9, the digit length of each product is tightly constrained, which in turn bounds the starting value x for each choice of n. For n=2 the split must be 4+5=9 digits, so x lies in roughly [1000, 9876]. For n=3 the only workable split is 3+3+3=9, forcing x at most 987 (and in practice 333, since 334*3 already exceeds three digits). For n=5 through n=9 only a single-digit starting value can work. These bounds collapse so rapidly that the total search space is at most a few thousand candidates across all values of n, making this effectively O(1) in practice.

The pandigitality check itself is O(d) where d=9 is the fixed digit count. A digit-frequency array approach avoids sorting or set construction. No number theory beyond basic digit manipulation is needed, though the problem rewards careful reasoning about digit-count arithmetic to prune the search before any pandigitality testing takes place.

Solution Approach

Bounded Descending Search with Digit-Frequency Validation (s0 - Python and C)

I encoded the upper bound on x for each n directly in the loop driver as a sequence of (n, max_x) pairs: ((2, 9876), (3, 987), (4, 98), (5, 9), ...). Each upper bound is the largest x whose concatenated product can possibly reach exactly 9 digits in total. For each pair I decrement x from that maximum down to 1, build the concatenated number by joining the string representations of x*1, x*2, ..., x*n, convert to an integer, and test with is_nine_pandigital. That function uses a 10-element frequency array to reject zeros and repeated digits in O(9) time, and also applies a fast range check (must lie between 100000000 and 999999999) as an early exit.

Crucially, the outer loop visits n=2 before n=3 before n=4 and so on. Any valid result for n=2 starts with a 4-digit prefix (at least 1000...) while any valid result for n=3 starts with a 3-digit prefix (at most 333...), so results from smaller n are numerically larger. Combined with the descending inner loop, the very first pandigital number found is guaranteed to be the global maximum, and I return immediately without collecting further candidates.

The Python and C implementations share the same algorithm exactly. The C version builds the concatenated string using snprintf and strcat into a fixed-size buffer, then converts with atoll, while the Python version uses a list comprehension with str.join. Both implementations pass the same (n, max_x) table. The C version runs in approximately 0.0001 s versus 0.0005 s for Python, a factor of roughly 5x, but both are far below any practical threshold.

Learnings

The key insight was that digit-count arithmetic completely determines the feasible range of x for each n before I test a single pandigitality condition. Once I worked out that n=2 requires a 4-digit x paired with a 5-digit 2x (4+5=9) and that n=3 requires three 3-digit products (3+3+3=9), the upper bounds practically wrote themselves. The "aha moment" was recognising that the ordering of valid results by n is strict - larger n means a shorter numeric prefix, so results from n=2 are numerically larger than those from n=3, which are larger than n=4, and so on. This let me structure the search to terminate at the very first hit rather than collecting all candidates and taking the maximum.

Only one algorithm was implemented here. Both language variants run well under 1 ms, so there is no practical motivation to explore a different strategy. The C version is about 5x faster in absolute terms, which reflects the interpreter overhead of Python rather than any algorithmic difference.

  • Digit-count constraints are a powerful and cheap way to prune brute-force searches before applying any pandigitality condition.
  • A frequency-array digit check is faster and simpler than converting to a sorted string or a frozenset, and avoids allocation beyond a small fixed-size array.
  • When searching for a maximum, structuring the iteration so that larger candidates are visited first enables an early-exit strategy and avoids collecting all valid solutions.
  • The strict ordering of results by n - larger n yields a numerically smaller concatenated product because the prefix is shorter - is a reusable pattern whenever concatenation length is fixed and prefix length varies with a parameter.

Problem Analysis

This is a coin-change counting problem, a concrete instance of the broader integer partition family: in how many distinct ways can a target amount be written as a sum of values drawn from a fixed set of denominations? The decisive subtlety is that order does not matter. Spending a 50p coin and then a 20p coin is the same combination as spending the 20p first and the 50p second. We count unordered multisets of coins, not ordered sequences. Formally, we want the number of non-negative integer solutions to a1*c1 + a2*c2 + ... + ak*ck = T, where each ci is a denomination and each ai is how many of that coin we use.

The clean way to count these is dynamic programming (DP) - a technique that builds the answer to a large problem from the cached answers to overlapping smaller subproblems, so each subproblem is solved only once. This particular variant is the unbounded knapsack pattern, "unbounded" because every coin may be reused without limit. I define dp[i] as the number of ways to form amount i using the coins considered so far, with the base case dp[0] = 1 - there is exactly one way to make zero, namely by taking no coins. The complexity is O(N * T) time and O(T) space, where N is the number of denominations and T is the target. The single most important observation - the one that makes the count correct - is the loop order: coins in the outer loop, amounts in the inner loop, which guarantees we count combinations rather than permutations.

Solution Approach

Bottom-up Unbounded Knapsack DP

Both the Python and C files implement the identical algorithm. I allocate a 1D array dp (called result in Python) of length target_amount + 1, fill it with zeros, and set dp[0] = 1. I then loop over each coin denomination. For each coin I sweep the amount index i upward from coin to target_amount, applying the recurrence dp[i] += dp[i - coin]. After every coin has been processed, dp[target_amount] holds the answer.

The recurrence reads as: "the number of ways to make amount i that use at least one of the current coin equals the number of ways to make i - coin, to which we then add one more of this coin." Because the inner loop runs upward, by the time I reach index i the cell dp[i - coin] has already absorbed multiple uses of the current coin within this same pass - this is precisely what makes the coin unbounded.

The reason the outer/inner ordering matters is a correctness issue, not a performance one. By fixing a coin in the outer loop, I ensure that every count accumulated in dp[i] was built using only the coins processed so far, in a fixed order. That enforces the invariant that I never count two different orderings of the same multiset. Were I to swap the loops - amounts outer, coins inner - I would instead count ordered sequences (permutations), a different and far larger number.

The two language versions diverge only in arithmetic. For target_amount = 100000 the answer is 10056050940818192726001, a 23-digit number near 10^22, which overflows a 64-bit integer (whose maximum is about 1.8 x 10^19). In Python this is a non-issue: its integers are arbitrary precision and grow automatically. In C I use the GCC built-in __int128 for each dp cell, since 128 bits comfortably holds a 23-digit value. Because printf has no format specifier for __int128, I wrote a small int128_to_str helper that extracts decimal digits by repeated division by ten.

Programming Techniques

1D DP Array with In-place Update

Rather than building a 2D table indexed by both coin and amount, I keep a single 1D array and update it in place. This is valid because the recurrence only reads a smaller index (i - coin) within the same pass. The upward inner loop means that by the time I write dp[i], the value at dp[i - coin] already includes the current coin's contributions - exactly the behaviour the unbounded case requires. This drops memory from O(N*T) to O(T). Reach for this whenever a dynamic programming recurrence looks backward only and you can order the sweep so the dependencies are already updated.

Loop Order as a Correctness Invariant

The most important lesson here is that loop order can encode an algorithmic constraint, not merely a traversal pattern. Putting coins outside and amounts inside enforces "combinations, not permutations." Internalise this: in dynamic programming, the order in which you iterate your dimensions determines which subproblems are "in scope" at each step. Getting it wrong yields output that looks plausible but counts the wrong thing entirely.

Wide Fixed-width Arithmetic with __int128

When a C result is known to exceed 64 bits but stays under 128 bits, GCC's __int128 is the lightest correct tool - it gives 128-bit integer arithmetic as a compiler built-in with no external library. I chose it over a big-integer library because the need here is modest and a full bignum library would be heavy machinery. The trade-off is portability: __int128 is a GCC/Clang extension, not standard C, and it has no printf support - hence the manual digit-extraction helper.

Python Arbitrary-precision Integers

Python's built-in int is arbitrary precision and grows as needed without any code change, so the same Python solver that handles target 200 also handles the 23-digit result for target 100000. The cost is speed: heap-allocated bignum arithmetic is slower than fixed-width math. This is a good prototyping pattern - prove correctness in Python first, then port to C with __int128 when you need throughput.

Generic Solver with Flexible Argument Parsing

Both solutions are written as general coin-change counters rather than hardcoding the UK denominations. Python parses the coin list with its runner's parse_list helper; the C side walks each argument character by character, skipping brackets and commas and reading integers with strtol. Writing parameterised solvers pays off immediately when the test suite varies the input - here targets of 0, 200, 1000, and 100000 all flow through the same code, and the target-of-zero case naturally yields the correct answer 1.

Key Takeaways

The insight that unlocks this problem is recognising it as an unbounded knapsack count and choosing the "coins outer, amounts inner" loop order to count unordered combinations correctly. Once that is understood, the algorithm is just a few lines.

Both implementations share one algorithm, so the comparison is purely about language overhead. On the original target of 200 both finish in well under a millisecond, so performance is irrelevant there. The difference appears only at scale: for target 100000 the C version runs in about 0.0008 s versus the Python version's 0.0345 s - roughly 40x faster - driven entirely by fixed-width __int128 arithmetic versus Python's arbitrary-precision integers. The C solution is the right pick when scaling the target; the Python solution is the more elegant and immediately correct prototype.

  • To count unordered combinations with repetition, use a 1D DP array updated coin-by-coin in the outer loop - the unbounded knapsack pattern.
  • Loop order in DP is a correctness decision: swapping inner and outer loops can change what you count from combinations to permutations.
  • An in-place 1D table cuts space from O(N*T) to O(T) whenever the recurrence reads only smaller indices in the same pass.
  • Estimate the magnitude of your answer before coding in a fixed-width language - a quick check prevents silent overflow.
  • When a C result needs more than 64 but fewer than 128 bits, __int128 is the lightest correct tool, at the cost of portability and manual printing.
  • Python's arbitrary-precision integers let you prove correctness first and worry about speed later.
  • Write parameterised solvers instead of hardcoding inputs - it makes multi-case testing trivial and surfaces edge cases like a target of zero.

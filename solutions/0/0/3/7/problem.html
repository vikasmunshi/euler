Problem 37: Truncatable Primes [Level 0]

Problem Statement

The number $3797$ has an interesting property. Being prime itself, it is possible to continuously remove digits from left to right, and remain prime at each stage: $3797$, $797$, $97$, and $7$. Similarly we can work from right to left: $3797$, $379$, $37$, and $3$.

Find the sum of the only eleven primes that are both truncatable from left to right and right to left.

NOTE: $2$, $3$, $5$, and $7$ are not considered to be truncatable primes.

Mathematical Complexity

A truncatable prime must satisfy two independent chains of primality constraints simultaneously: every prefix formed by removing digits from the right must be prime, and every suffix formed by removing digits from the left must also be prime. For a d-digit candidate this imposes 2*(d-1) sub-checks. A useful number-theoretic observation is that every interior digit, when isolated by truncation, must itself be a single-digit prime (2, 3, 5, or 7), which dramatically thins the candidate pool. The problem guarantees exactly eleven such primes exist across all natural numbers, the largest being 739397, so the search space is effectively fixed and small. The practical cost per candidate is O(d) truncation lookups where d is the digit count, and the dominant cost overall is simply generating primes in order up to roughly 739397.

Because the solution set is finite and tiny, there is no meaningful asymptotic regime to optimise for. The whole algorithm is bounded above by the effort to sieve numbers up to 739397 and perform a handful of string membership tests per candidate. The theoretical complexity is effectively constant with respect to the final answer.

Solution Approach

Incremental Sieve with Running Prime String Set (Python and C)

Both implementations share the same core algorithm: an incremental (online) sieve of Eratosthenes paired with a running set of prime string representations. In the Python version I implement the sieve as a generator backed by a dict[int, int] that maps each composite to its smallest prime factor. When a new number is absent from the composite map it is prime and is yielded; otherwise its stored factor is popped, stepped forward by twice itself until a free slot is found, and re-inserted there. This avoids allocating a fixed-size array and lets the generator halt as soon as the eleventh truncatable prime is found.

As each prime is yielded, I store its string representation in a set[str]. For every candidate with two or more digits I iterate over all split positions i from 1 to len-1, forming the right-truncation prime[:i] and the left-truncation prime[i:], and check that both strings are already in the set. Because every prime smaller than the current candidate has already been inserted, a missing entry conclusively proves the candidate fails. Once eleven passing primes are collected their sum is returned.

The C translation replicates the same logic using two manually implemented open-addressing hash tables. The first is an integer-keyed table storing the composite-to-factor mapping for the sieve; entries are removed during sieve advancement and the chain is repaired using Robin Hood re-insertion. The second is a string-keyed table storing the string representations of seen primes, using a djb2-style polynomial hash with linear probing and no deletion, since entries are only ever added. One practical difference in the C version is that the sieve iterates only over odd numbers, stepping n by 2 and handling 2 as a special case, which halves the number of main-loop iterations.

Learnings

The key insight that unlocked this problem was recognising that by the time the incremental generator yields a prime p, every prime smaller than p is already in the string set. This means every legitimate truncation of p that is prime is guaranteed to be present, so a single set lookup per truncation completely replaces any separate primality test. There is no need to know an upper bound in advance and no need to pre-allocate a fixed sieve array - the generator and the set grow together naturally.

The Python solution runs in about 0.220 seconds and the C solution in about 0.019 seconds - roughly an 11x speedup. The algorithm is identical in both; the gap comes from Python's interpreter overhead and object allocation costs in the tight inner loop. I find the Python version cleaner and easier to read thanks to the generator abstraction and the concise list-comprehension truncation check, while the C version wins on raw speed without adding algorithmic sophistication.

Techniques worth remembering for future problems:

  • A dict-based incremental sieve expressed as a generator is a flexible alternative to a fixed Sieve of Eratosthenes whenever the upper bound is unknown or early termination is desirable.
  • Storing primes as strings in a set turns digit-manipulation tasks - truncation, rotation, permutation - into simple string slicing plus an O(d) membership test, avoiding repeated integer arithmetic.
  • When a problem guarantees a fixed finite count of solutions, using that count as the sole stopping condition is cleaner and more correct than imposing an arbitrary numeric ceiling.
  • When translating a Python dict-based sieve to C, open-addressing hash tables with Robin Hood deletion are a natural fit, but the deletion step must be handled carefully to avoid corrupting the probe chain for subsequent lookups.

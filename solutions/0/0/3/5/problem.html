Problem 35: Circular Primes [Level 1]

Problem Statement

The number, $197$, is called a circular prime because all rotations of the digits: $197$, $971$, and $719$, are themselves prime.

There are thirteen such primes below $100$: $2, 3, 5, 7, 11, 13, 17, 31, 37, 71, 73, 79$, and $97$.

How many circular primes are there below one million?

Mathematical Complexity

The core idea combines prime generation with a structural property of digit rotations. A circular prime requires every cyclic permutation of its decimal digit string to be prime. The key number-theoretic observation is that any multi-digit circular prime can only contain the digits 1, 3, 7, and 9. If any digit from {0, 2, 4, 5, 6, 8} appears anywhere in the number, then some rotation will place that digit in the units position, making the rotated value divisible by 2 or 5 and therefore composite. This prunes the candidate set dramatically - instead of checking every prime below the limit, I only need to examine those built entirely from {1, 3, 7, 9}, treating the single-digit primes 2, 3, 5, 7 as special cases that bypass the filter.

Prime generation via a sieve costs O(n log log n) time and O(n) space. After the sieve is built I store all primes in a set for O(1) membership queries. The rotation check per surviving candidate is O(d) where d = O(log n) is the digit count. Because the digit filter prunes the candidate pool heavily, the work beyond sieve construction is modest in practice. The answer stabilises at 55 above one million - there are no additional circular primes in the range 1,000,001 to 10,000,000 - which is consistent with the digit-composition constraint becoming increasingly restrictive as numbers grow larger.

Solution Approach

Sundaram Sieve with Digit Pruning (s0)

My first solution implements the Sundaram sieve from scratch in Python to generate all primes up to the limit, immediately stores them in a set for O(1) membership tests, then iterates over every prime as a candidate. For multi-digit candidates I first apply the digit-filter shortcut: if any character from "024568" appears in the decimal representation I skip the number entirely. Only after passing that filter do I call get_rotated_numbers, which builds all cyclic rotations as a set comprehension, and verify that every element is present in the prime set. Single-digit primes below 10 bypass both filters since 2 and 5 are legitimately circular.

The companion C file replaces the Sundaram sieve with a standard sieve of Eratosthenes - the more conventional and cache-friendly approach in C - but uses exactly the same digit-filter and rotation logic. One notable difference in the C sieve convention is that sieve[i] = 1 means composite, so the primality check is a zero-test rather than a one-test, the reverse of the Python convention where set membership indicates primality. The C version allocates exactly as much sieve memory as needed for the requested limit.

pyprimesieve with Minimum-1M Sieve (s1)

My second Python solution keeps the identical filtering and rotation-checking logic but replaces the hand-rolled sieve with pyprimesieve, a thin Python wrapper around Kim Walisch's highly optimised C++ primesieve library. Because pyprimesieve.primes always generates at least up to 10 6 in my implementation, I use bisect_right to slice the result down to the requested limit, ensuring the candidate loop only considers primes within bounds. This introduces a fixed baseline overhead at small limits, which is visible in the dev benchmarks where s1.py costs around 4 ms even for tiny inputs.

The C counterpart mirrors this minimum-generation guarantee by computing gen_limit = max(max_limit, 1000000) before running its own Eratosthenes sieve, then only iterates candidates up to max_limit. This means s1_c always allocates at least a one-million-entry sieve regardless of the actual limit, faithfully matching the Python variant's behaviour. The digit-filter and rotation code is otherwise identical to s0_c. The practical effect is that s1_c shows roughly 2-3 ms of fixed overhead on dev-scale inputs compared to the near-zero times of s0_c.

Learnings

The key insight that unlocked a clean solution was the digit-parity argument: because every digit of a circular prime must be capable of appearing as the final digit of some rotation, and a prime greater than 5 cannot end in 0, 2, 4, 5, 6, or 8, the entire multi-digit candidate set collapses to numbers built exclusively from {1, 3, 7, 9}. Applying this cheap character scan before constructing any rotations avoids the bulk of the rotation work. A secondary observation is that get_rotated_numbers naturally includes the original number itself among its outputs - rotating by the full string length wraps back to the start - so the set comprehension gives a self-check for free without a special case.

The pyprimesieve Python variant is roughly 6x faster than my pure-Python Sundaram sieve at one million and about 7x faster at ten million. The gap widens with scale because sieve construction dominates at larger limits and the C++ backend is far more cache-friendly than my Python loop. The two C implementations are essentially tied: s0_c is marginally faster at both limits because it allocates only as much sieve as needed, whereas s1_c always allocates at least a million entries regardless of the requested limit. The difference is small enough to sit near measurement noise but is consistent across runs. For small dev inputs, s1_c and s1.py show a fixed baseline overhead of around 2-4 ms from the minimum sieve allocation, whereas s0_c and s0.py scale down cleanly to near-zero.

  • Store sieve output in a Python set immediately - O(1) membership tests pay off whenever repeated prime-membership queries are needed across a large collection of candidates.
  • Apply cheap structural filters (digit composition, parity) before expensive combinatorial checks (rotation generation); the order of guards matters for performance.
  • The Sundaram sieve is an interesting theoretical alternative to Eratosthenes but offers no practical speed advantage in pure Python; a standard sieve or an external C library is preferable for large limits.
  • When comparing a Python sieve against a C extension library, the speedup scales with the limit because sieve construction is the dominant cost term - the Python-level application logic costs roughly the same in both cases.
  • Mirroring a minimum generation guarantee between Python and C ports is a useful correctness discipline, even if it occasionally allocates slightly more than strictly necessary.
  • The answer stabilising at 55 above one million is a good sanity check: it confirms the digit-composition constraint leaves no new circular primes in the entire 1M-10M range.

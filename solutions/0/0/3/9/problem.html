Problem 39: Integer Right Triangles [Level 1]

Problem Statement

If $p$ is the perimeter of a right angle triangle with integral length sides, $\{a, b, c\}$, there are exactly three solutions for $p = 120$.

$\{20,48,52\}$, $\{24,45,51\}$, $\{30,40,50\}$

For which value of $p \le 1000$, is the number of solutions maximised?

Mathematical Complexity

The problem reduces to counting integer right triangles for each perimeter value up to a bound. The key mathematical tool is Euclid's parametrisation: every primitive Pythagorean triple - one where gcd(a, b, c) = 1 - can be written as a = m² - n², b = 2mn, c = m² + n² for integers m > n > 0 with gcd(m, n) = 1 and m, n of opposite parity. The perimeter of such a primitive triple simplifies neatly to 2m(m + n). Every non-primitive triple is a positive integer multiple k of some primitive triple, so its perimeter is k · 2m(m + n). Enumerating all qualifying (m, n) pairs and then scaling by k = 2, 3, ... covers the complete set of integer right triangles without repetition or omission.

The number of primitive Pythagorean triples with perimeter at most P grows as O(P), so including all multiples the total work is O(P log P) - analogous to a harmonic sieve. The loop bounds on m and n are derived analytically by setting the minimum primitive perimeter for a given parameter value equal to the maximum perimeter and solving, giving tight upper limits of roughly O(sqrt(P)) for n and a corresponding bound for m. For P = 10⁶ the algorithm completes in well under a second in C and in about 0.3 seconds in Python.

Solution Approach

Euclid Parametrisation with Harmonic Scaling (Python and C)

I implement the same algorithm in both Python and C. I iterate n from 1 up to a bound derived as (int(8 * sqrt(max_perimeter)) - 6) // 8, and for each n I iterate m starting at n + 1 in steps of 2. The step of 2 automatically enforces the opposite-parity requirement - because n and n + 1 always differ in parity, incrementing by 2 keeps m and n on opposite parities without any explicit branch. I then check gcd(m, n) == 1 to ensure primitivity, compute the primitive perimeter p0 = 2 * m * (m + n), and record p0 together with every multiple k * p0 for k = 2, 3, ... up to max_perimeter. The m bound is also derived analytically as (int(sqrt(4 + 8 * max_perimeter)) - 2 * n) // 4, giving tight termination without wasted iterations.

In Python I collect all perimeter values into a list and use collections.Counter.most_common()[0][0] to find the most frequent one. In C I allocate a fixed-size integer array counts of length max_perimeter + 1, increment counts[p] for each perimeter value, and scan the array for the maximum. The C version avoids dynamic list growth and hash-map overhead, which accounts for its large speed advantage. The loop bounds are identical in both languages - I translated the Python integer floor expressions directly into C integer arithmetic, keeping the two implementations as faithful mirrors of each other.

Learnings

The central insight was recognising that I did not need to search (a, b) pairs directly over a grid of side lengths. Reformulating in Euclid's (m, n) parameters collapses a two-dimensional search over sides into a compact two-dimensional search whose inner check is just a single gcd call and whose bounds are derived analytically. The second insight was that every integer right triangle is a scaled primitive triple, so a short inner scaling loop replaces a much larger redundant outer search - making the algorithm feel like a sieve rather than brute-force enumeration.

Since only one algorithm is implemented, there is no algorithmic comparison to make. The Python and C versions are faithful translations of each other. The benchmark results confirm the expected language overhead gap:

The C version is roughly 50 times faster at P = 10⁶, purely due to language overhead - the algorithm and loop bounds are identical. The C version uses a plain integer array rather than a hash map, which removes allocation and hashing cost and benefits from cache-friendly sequential access during the final scan.

  • Euclid's parametrisation is the right starting point for any problem involving Pythagorean triples.
  • Primitive-triple enumeration plus harmonic scaling is the Pythagorean analogue of a prime sieve - enumerate the primitives, then sieve their multiples.
  • Deriving tight analytical loop bounds avoids wasted iterations and makes the intent explicit in code.
  • The step-of-2 trick silently enforces the opposite-parity constraint without an extra branch, and is worth reusing any time Euclid's formula appears in a future problem.
  • Counter.most_common() is clean and idiomatic for frequency-maximisation problems in Python; a plain count array serves the same role more efficiently in C.

Problem 66: Diophantine Equation [Level 10]

Problem Statement

Consider quadratic Diophantine equations of the form: $$x^2 - Dy^2 = 1$$

For example, when $D=13$, the minimal solution in $x$ is $649^2 - 13 \times 180^2 = 1$.

It can be assumed that there are no solutions in positive integers when $D$ is square.

By finding minimal solutions in $x$ for $D = \{2, 3, 5, 6, 7\}$, we obtain the following:

$$\begin{align} 3^2 - 2 \times 2^2 &= 1\\ 2^2 - 3 \times 1^2 &= 1\\ {\color{red}{\mathbf 9}}^2 - 5 \times 4^2 &= 1\\ 5^2 - 6 \times 2^2 &= 1\\ 8^2 - 7 \times 3^2 &= 1 \end{align}$$

Hence, by considering minimal solutions in $x$ for $D \le 7$, the largest $x$ is obtained when $D=5$.

Find the value of $D \le 1000$ in minimal solutions of $x$ for which the largest value of $x$ is obtained.

  • main: solve(max_d=1000) → 661

Mathematical Complexity

This problem requires finding, for each non-square D up to a given bound, the minimal positive integer x satisfying Pell's equation x² - Dy² = 1. The key mathematical fact is that the fundamental solution is always encoded in the convergents of the continued fraction expansion of sqrt(D). Every irrational square root has a periodic continued fraction of the form [a0; a1, a2, ..., ar-1, 2*a0, a1, ...], and the period length r determines which convergent to use: if r is odd, the (r-1)th convergent (0-indexed from a0) gives x directly; if r is even, one must go to the (2r-1)th convergent instead. The end of the period is easy to detect because the last coefficient in each period is always 2*a0.

The asymptotic complexity is hard to state cleanly. For each of the O(N) non-square values of D up to N, the continued fraction period has O(sqrt(D)) terms on average, so generating all periods costs O(N^(3/2)) arithmetic operations on small integers. However, the convergents themselves are astronomically large rational numbers - for some D the minimal x has hundreds of digits - so arbitrary-precision arithmetic dominates and the per-D cost grows with the bit-length of the fundamental solution rather than with the period length alone. For the main case (D up to 1,000) this is nearly instantaneous; for D up to 10,000 the cost becomes measurable but still modest.

Solution Approach

Continued Fraction Convergent Method (Python and C)

Both the Python and C implementations follow the same algorithm. For each non-square D I build the continued fraction incrementally using the standard three-term recurrence: starting from m=0, n=1, a=floor(sqrt(D)), each step applies m = n*a - m, n = (D - m²)/n, a = floor((sqrt(D) + m)/n), and I append the new a until it equals 2*a0, which marks the end of the first full period. The total length of the coefficient tuple (including a0) determines the convergent index: if the length is even, the period is odd and I use convergent index 2*len - 3; if the length is odd, the period is even and I use index len - 2.

I compute the required convergent by walking backward through the periodic coefficients. Starting from the tail term, I repeatedly apply the identity frac = term + 1/frac toward the head, then prepend a0 at the end. In Python I used fractions.Fraction for exact rational arithmetic and read the numerator directly as x. In C I implemented a custom BigInt type stored as a little-endian array of up to 512 base-10^9 limbs, with operations bi_add, bi_mul_small, and bi_cmp. The C convergent recurrence uses unsigned __int128 in bi_mul_small to safely accumulate intermediate products without overflow. Both versions iterate over all valid D, track the D yielding the largest minimal x, and return it.

The only meaningful language-level difference is that Python's built-in arbitrary-precision integers and the Fraction class handle all bookkeeping automatically, while the C version requires an explicit limb-array struct with manual memory management. The algorithm is otherwise identical.

Learnings

The central insight is that Pell's equation has a complete, exact solution via continued fractions - there is no need for any brute-force search. Once I recognised that the fundamental solution always appears at a precisely determined convergent index driven solely by the parity of the continued fraction period, the algorithm fell into place. The parity rule is easy to get wrong by one position, so I checked the index arithmetic carefully against the small examples in the problem statement (D=2, period length 1, odd period; D=5, period length 1, odd period; D=3, period length 2, even period).

The benchmark results show the practical cost of big-integer arithmetic across languages. For D up to 1,000 the Python solution takes about 27 ms and the C solution about 0.6 ms, a factor of roughly 45x. For D up to 10,000 Python takes about 874 ms and C takes about 16 ms, a factor of roughly 55x. Both are algorithmically identical; the entire gap comes from the overhead of Python's Fraction objects and general-purpose big-integer allocations versus the tightly packed, cache-friendly limb array in C. For the main test case Python is already fast enough, but the C version makes even the extended case essentially instantaneous.

  • Pell's equation x² - Dy² = 1 is solved exactly by taking a specific convergent of the continued fraction expansion of sqrt(D) - no trial and error required.
  • The period ends precisely when a coefficient equals 2*a0; this provides a clean termination condition for the expansion.
  • The parity of the period length determines whether to use convergent index r-1 or 2r-1, and getting this right is essential.
  • Python's fractions.Fraction class and native big integers make the convergent computation trivial to implement correctly and are fast enough for D up to at least 10,000.
  • Implementing a custom limb-array big integer in C with unsigned __int128 intermediate products delivers a 50x speedup over Python, driven entirely by big-integer multiplication costs at large D.
  • The "expand as continued fraction, read off the convergent" pattern is broadly applicable to problems involving quadratic irrationals and should be a first instinct for any future Pell-type problem.

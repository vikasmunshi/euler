Problem 60: Prime Pair Sets [Level 3]

Problem Statement

The primes $3$, $7$, $109$, and $673$, are quite remarkable. By taking any two primes and concatenating them in any order the result will always be prime. For example, taking $7$ and $109$, both $7109$ and $1097$ are prime. The sum of these four primes, $792$, represents the lowest sum for a set of four primes with this property.

Find the lowest sum for a set of five primes for which any two primes concatenate to produce another prime.

Mathematical Complexity

The central number-theoretic operation is decimal concatenation primality: given two primes a and b, I form two new integers by appending their decimal representations - ab and ba - and test whether both are prime using trial division up to the square root of each concatenated value. Because the concatenated number can be as large as a * 10^(digits of b) + b, primality testing is O(sqrt(a * 10^d)) where d is the digit count of b, somewhat more expensive than testing either input alone. The structural heart of the problem is a graph clique: I treat each prime as a node and draw an undirected edge between two primes when their mutual concatenation is prime in both orders. Finding five primes where every pair shares an edge is finding a 5-clique in this graph.

General clique detection is NP-hard, but two properties keep this instance tractable. First, the concatenation-prime graph is extremely sparse - most prime pairs fail the mutual concatenation test, so very few edges exist. Second, I restrict the candidate prime pool to primes up to 10^(set_length-1), a heuristic bound motivated by the known 4-prime answer (maximum element 673, well under 10^3). With a strict ascending-order constraint that eliminates all permutations of the same set, the practical search space is tiny and the whole computation finishes in well under a second.

Solution Approach

Incremental Clique Search with Cached Pair Validation (Python and C)

Both implementations share the same algorithm: build candidate cliques incrementally by extending partial solutions in strictly ascending prime order, validating each new candidate prime against every member already in the partial solution before committing it. The ascending-order constraint is the key pruning device - it ensures each unordered set of five primes appears exactly once in the enumeration, eliminating the 5! = 120 redundant orderings that would otherwise be visited per solution.

In the Python version I expressed the search as a lazy generator pipeline. The function solution_pairs yields all valid pairs and extend_solution yields valid extensions of any partial solution. For set_length=5 these are chained into four nested generator expressions inside solve, and I call next() on the outermost generator to stop the moment the first complete clique is found. The concatenate_is_prime function is decorated with lru_cache(maxsize=None) so any pair tested once costs O(1) on all subsequent encounters - important because the same low-index prime pairs recur across many different partial solutions. Primes are generated by a Sundaram sieve, which I find both fast and memory-efficient.

The C version implements the same depth-first search with an explicit recursive function dfs and a hash-table cache that mirrors the Python LRU cache. I use a 1M-bucket chained hash table with a multiplicative hash over the canonical (min, max) prime pair. One deliberate choice was to canonicalise the cache key to (min(a,b), max(a,b)) since the predicate is symmetric, halving the number of distinct cache entries. The dfs function carries its partial solution on the C call stack rather than building heap objects, and a global found flag lets every recursion level short-circuit as soon as a solution is discovered.

The key language-level difference is that the Python generator pipeline expresses the search structure very cleanly through lazy evaluation, while the C version uses an explicit recursive DFS with an early-exit flag. The algorithm and pruning logic are otherwise identical.

Learnings

The most important insight was framing this as a graph clique problem before writing any code. Once I saw it that way, the canonical ascending-order enumeration trick - always extend with a prime strictly greater than the current last element - followed naturally as the standard technique for enumerating unordered sets without repetition.

Caching the pairwise predicate was the second key insight. Because many distinct partial solutions pass through the same low-index prime pairs, concatenate_is_prime(3, 7) and similar calls would be executed thousands of times without caching. With caching, each unique pair is evaluated once and all future lookups are O(1). This mattered enough that even the C version warranted a full hash-table implementation rather than recomputing on every call.

Looking at the benchmarks, the C implementation is roughly seven times faster than Python for the main case (0.0367 s versus 0.2626 s). For the smaller dev cases the C binary appears slower than Python (0.0026 s versus 0.0001-0.0003 s) because those measurements are dominated by process startup time and the actual search is trivially short. The Python generator approach is arguably more elegant for this problem - the nesting of extend_solution generators reads almost like a declarative specification of the clique conditions.

The sieve bound heuristic 10^(set_length-1) is not guaranteed correct for arbitrary set lengths - it is a guess that worked here. A safer strategy for unknown inputs would be to retry with an expanded bound if the search returns empty. Key techniques worth remembering for future clique-style search problems:

  • Enforce a canonical ascending order over candidates to visit each unordered set exactly once, eliminating factorial redundancy.
  • Cache any expensive pairwise predicate that is evaluated across many overlapping partial solutions.
  • Use lazy generators (Python) or a short-circuit flag (C) so the search stops at the first valid solution rather than enumerating all of them.
  • Model the constraint structure as a graph before writing code - recognising the clique framing makes the pruning strategy obvious.
  • In a C hash table, canonicalising a symmetric predicate's cache key to (min, max) halves the number of distinct entries at essentially zero cost.

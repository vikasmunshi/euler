Problem 62: Cubic Permutations [Level 2]

Problem Statement

The cube, $41063625$ ($345^3$), can be permuted to produce two other cubes: $56623104$ ($384^3$) and $66430125$ ($405^3$). In fact, $41063625$ is the smallest cube which has exactly three permutations of its digits which are also cube.

Find the smallest cube for which exactly five permutations of its digits are cube.

Mathematical Complexity

The central observation is that two integers are digit-permutations of each other if and only if sorting their decimal digits produces the same string. This turns an otherwise expensive combinatorial search into a simple equivalence-class problem: assign every cube a canonical key and count how many cubes share each key. A second observation tightens the search further - digit-permutations of a number must all have the same digit count, so permutation families can never straddle two different digit lengths. This means I can process one digit-length bucket at a time and stop as soon as a qualifying group appears, keeping each pass finite and closed.

Let N be the number of cubes examined before a solution is found and d the number of digits in those cubes, so d = O(log N). Generating each cube is O(1); sorting its digits costs O(d log d); inserting into a hash map costs O(d) amortised. Total work is O(N d log d), effectively O(N log N) in the digit count. In practice the search terminates quickly - even the hardest extra cases requiring nine permutations finish in well under 60 ms in Python and around 32 ms in C.

Solution Approach

Digit-Sorted Canonical Key with Digit-Length Bucketing (s0, Python and C)

I implemented the same algorithm in both Python ( p0062_s0.py) and C ( p0062_s0.c). The core idea is identical: iterate over digit-length buckets, group cubes by their sorted-digit string, then check whether any group has exactly the target number of members. The permutation count is passed as a command-line argument in both versions, making it straightforward to run every test case without modification.

In the Python version I wrote a helper n_digit_cubes that computes the integer range whose cubes have exactly digit_length_n digits by inverting the cube root at the boundary values using math.ceil, with a deliberate overshoot of +1 on the upper boundary to guard against floating-point rounding errors near exact integer cube roots. Inside the main loop I use a collections.defaultdict(list) keyed on "".join(sorted(str(cube_number))). After processing the full bucket a set comprehension collects the minimum cube from every qualifying group, and I return the overall minimum.

The C version mirrors this logic but requires a hand-rolled hash map because no standard equivalent of defaultdict exists in C. I built an open-chaining hash map with 65536 buckets using djb2 hashing. Each entry stores the sorted-digit key as a fixed-length char array and a dynamically resized long long array of cube values, grown with realloc as needed. The sorted-digit key is produced with snprintf followed by qsort on the character array. After filling the map for a given digit length I scan all buckets for groups matching the target count and track the minimum cube found. Memory is freed explicitly after each digit-length pass. The long long type is sufficient for all test cases including the nine-permutation case, where the answer is around 1.3 x 10^13.

Learnings

The key insight was recognising that digit-permutation equivalence has a trivially computable canonical form: sort the digits. Once I had that, the problem collapsed entirely into frequency counting with a dictionary, and no explicit permutation enumeration was needed. The digit-length bucketing was equally important - it converted an open-ended search into a series of finite closed sets and provided a clean early-termination condition.

A practical subtlety worth remembering is the cube-root boundary calculation. Using ceil(cbrt(10^(n-1))) for the lower bound and ceil(cbrt(10^n - 1)) + 1 for the upper bound avoids off-by-one errors caused by floating-point imprecision near exact integer cube roots. The extra +1 is a deliberate safety margin; any harmless duplicate at the boundary costs nothing.

Comparing the two implementations using the benchmark results, the C version is roughly 1.7-1.8x faster across all test categories. For the main case Python takes 10.7 ms versus 6.8 ms for C; for the hardest extra cases Python sits around 56-57 ms versus about 32-33 ms for C. The speedup is modest and consistent with Python's interpreter and object-allocation overhead rather than any algorithmic difference. I consider the Python version more elegant for its brevity - the entire grouping logic fits in five readable lines using only standard library constructs, with no manual memory management. The C version is worth having if sub-10 ms latency is required, but it offers nothing algorithmically new and the hash map boilerplate is considerable.

  • A sorted-digit string as a canonical permutation key is a reusable pattern for any "same digits" problem on Project Euler.
  • Bucketing by digit length turns an open-ended search into a series of finite closed sets, enabling clean early termination.
  • Python's defaultdict(list) paired with a set comprehension makes grouping concise and readable with no boilerplate.
  • Floating-point cube-root boundaries need ceiling plus a small overshoot on the upper end to be numerically safe.
  • For a straightforward grouping problem the Python solution is fast enough for all practical inputs and substantially easier to read than the equivalent C hash map implementation.

Problem Analysis

This problem asks me to find the smallest perfect cube whose digits can be rearranged into exactly five distinct cubes (counting the cube itself). The mathematical core is the notion of a permutation of digits: two numbers are digit-permutations of one another precisely when they contain the same digits in the same counts - that is, they share the same multiset of digits. The observation that makes this tractable is that I can convert this "equal up to reordering" relation into an exact equality test by computing a canonical form. If I sort the digits of a number into ascending order and join them back into a string, then two numbers are digit-permutations of each other if and only if their sorted-digit strings are identical. That sorted string becomes a shared key for every member of a permutation class, collapsing an exponential equivalence relation into a single hash table lookup.

The second key insight is that permutations of a number always have the same number of digits. Therefore all cubes that could possibly be permutations of one another live in the same digit-length band, and I can process cubes batch by batch - one digit-length at a time - grouping each batch by its canonical key. As soon as I find a digit-length where some group has exactly the target size, the smallest cube in that group is the answer; because I processed shorter numbers first, no smaller cube can qualify. The count of d-digit cubes is roughly proportional to 10^(d/3) (the cube root of a d-digit number has about d/3 digits), and each cube costs O(d log d) to sort its digits. The overall work is therefore O(10^(d/3) * d log d) up to the answer's length, which is small enough that every test case finishes in well under 100 ms.

Solution Approach

Digit-canonical grouping by digit-length band

Both the Python and C files implement the identical algorithm; they differ only in how the grouping map is realised.

Step 1 - Bound the range of d-digit cubes. For a target digit-length d, the smallest d-digit number is 10^(d-1) and the largest is 10^d - 1. I need the integers whose cubes fall in this band, so I take the cube root of each bound. I use ceil(cbrt(...)) on the lower bound as the inclusive start, and ceil(cbrt(...)) + 1 on the upper bound as the exclusive stop. Using the ceiling rather than a floor or a bare cast guards against a slightly-under floating-point representation of an exact root pulling in a cube from the wrong digit band.

Step 2 - Build the canonical-key map. For each integer i in the range I compute i^3, convert it to a decimal string, sort the characters, and use that sorted string as a key into a map whose values are lists of cubes sharing that key. In Python this is collections.defaultdict(list) with permuted_cubes[key].append(cube). In C it is a custom hash table with 65536 buckets, each chaining entries that hold a dynamically growing long long array.

Step 3 - Scan for groups of the target size. Once the batch is fully grouped, I examine every entry whose list has exactly num_permutations members. Each such group is a candidate, and its answer is the minimum cube in the group. I take the minimum over all qualifying groups at this digit-length.

Step 4 - Advance. If no group qualified, I increment the digit-length and repeat. The first digit-length that yields a result holds the globally smallest answer, since all shorter cubes were already exhausted.

Language-specific differences. Python leans entirely on the built-in dictionary, which handles hashing and resizing transparently. The C version replicates these semantics by hand: a power-of-two bucket array, a djb2 string hash, and amortised-doubling value arrays. The extra effort buys roughly a 2x speedup, but both stay comfortably fast.

Programming Techniques

Canonical sorted-digit keys for multiset equality

Sorting the digits of a number into a string produces a representative that is identical for all its permutations. This is the universal trick for testing "equal up to reordering": compute a canonical form and use it as a key. It applies to anagram grouping, fingerprinting collections by their element multiset, and many combinatorial bucketing problems. Sorting costs O(d log d) per item but reduces the equivalence test to an O(1) lookup.

Batching by a structural property to bound an unbounded search

Instead of searching all integers up to some arbitrary limit, I partition the space by digit-length - a property that permutations preserve. Each batch is finite, and the monotone ordering of digit-lengths gives a clean correctness argument for early termination: the first batch that produces a hit also contains the smallest valid answer. Reach for this whenever a problem has a natural partition that respects the ordering you care about.

defaultdict for accumulator maps (Python)

collections.defaultdict(list) lets me write permuted_cubes[key].append(cube) unconditionally - a missing key auto-creates an empty list. This removes the noisy "check then insert" boilerplate that grouping code otherwise needs and is the idiomatic Python choice for any bucketing task.

Generator expressions fed straight into a constructor (Python)

The line tuple((i**3 for i in range(start_range, stop_range))) passes a lazy generator into tuple(), which materialises it without building an intermediate list. The same idiom works with set(), list(), and min(), saving an allocation when the sequence is large. The qualifying-group scan reuses it: set((min(v) for ... if len(v) == n)).

djb2 hashing with a power-of-two table size (C)

The C hash uses djb2: seed 5381, then h = (h << 5) + h + c per character (that is, h*33 + c). It distributes short strings well with no library dependency. Because the table has 65536 buckets - a power of two - the final reduction is a bitwise & (HASH_SIZE - 1) instead of a costly modulo, which matters when the hash runs millions of times.

Amortised-doubling dynamic arrays (C)

Each entry's values array starts at capacity 4 and doubles when full via realloc. Geometric (doubling) expansion makes the total copying cost across n insertions O(n), giving O(1) amortised cost per append - the same strategy used internally by C++ std::vector and Python's list. It is worth implementing by hand in C whenever you need a variable-length list without a standard container.

Ceiling arithmetic for floating-point root boundaries

When converting a real-valued cube root back into an integer index, I use ceil on the lower bound rather than floor or a bare cast. If the exact root is an integer, a slightly-under floating-point representation would make floor return one too few and pull in cubes from the wrong digit band. Ceiling on the lower bound, plus a small margin on the upper bound, is the safe default for this kind of conversion.

Key Takeaways

The unlocking insight is reducing a permutation-equivalence test to a canonical-key comparison: sort the digits and the whole equivalence class collapses to a single hashable string. The second insight is the batch-by-digit-length strategy, which both bounds each iteration and supplies a sound termination proof.

Comparing the two implementations, the C solution is consistently about 1.7x to 2x faster than Python - for example 0.0040 s versus 0.0078 s on the main case (five permutations), and roughly 0.0217 s versus 0.0439 s on the heaviest extra case (seven permutations). The gap is modest because most of the Python work happens inside built-in, C-implemented operations: string sorting and dictionary insertion. For problems at this scale Python's expressiveness makes it the pragmatic first choice; the hand-written C hash map is worthwhile only when the constant factor genuinely matters or the routine runs repeatedly.

  • To test whether two collections are equal up to reordering, compute a canonical sorted form and compare those - O(d log d) to build, O(1) to look up.
  • When the search space is unbounded, find a structural property (digit count, bit length, magnitude band) that partitions it into finite batches with a monotone ordering, then stop at the first hit.
  • A defaultdict (or an equivalent auto-initialising map) removes the check-then-insert pattern and makes grouping code cleaner.
  • Amortised-doubling arrays give O(1) amortised append and are the right manual choice for variable-length lists in C.
  • Use ceiling, not floor, when turning a real-valued lower bound into an integer index; floating-point underestimation of roots is a classic source of silent off-by-one errors.
  • Size hash tables as powers of two so the index reduction becomes a single bitwise AND instead of a division.

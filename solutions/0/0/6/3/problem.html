Problem 63: Powerful Digit Counts [Level 2]

Problem Statement

The $5$-digit number, $16807=7^5$, is also a fifth power. Similarly, the $9$-digit number, $134217728=8^9$, is a ninth power.

How many $n$-digit positive integers exist which are also an $n$th power?

Mathematical Complexity

The central insight is that valid bases are provably bounded to the range 1 through 9. For b^n to be an n-digit number we need 10^(n-1) <= b^n < 10^n. If b >= 10 then b^n >= 10^n, which already exceeds the upper bound for an n-digit number. So every valid base must be a single digit - this is a hard mathematical boundary, not a convenient approximation. The digit count of b^n is floor(n * log10(b)) + 1, and because log10(9) < 1, this quantity grows strictly slower than n for all bases 1 through 9. Eventually even base 9 stops producing n-digit nth powers, and that crossover happens around n = 22, so the outer loop terminates naturally without any manually imposed limit.

The overall complexity is effectively O(1). The total number of (n, b) pairs examined is bounded by a small constant - at most 9 bases times roughly 22 values of n. This is a classic bounded enumeration problem where a seemingly open-ended search collapses to trivial constant work once the logarithmic growth constraint is applied.

Solution Approach

Bounded Enumeration with Exact Digit-Count Verification (Python and C)

Both solutions implement the same core algorithm: iterate over increasing values of n, identify which bases in the valid range produce an n-digit nth power, accumulate the count, and stop as soon as a value of n yields no solutions. I chose this structure because the mathematical bound on valid bases makes the search both complete and guaranteed to terminate without any manual cutoff.

In the Python version I use ceiling arithmetic - ceil((10^(n-1))^(1/n)) for the lower bound and ceil((10^n - 1)^(1/n)) + 1 for the upper bound - to bracket the candidate base range without missing any values due to off-by-one errors. Because floating-point inverse powers can land just below an integer boundary, I apply a final exact guard: I compute i^n using Python's arbitrary-precision integers and confirm the digit count with len(str(r)) == n. The helper n_digit_nth_powers returns a tuple of valid powers for a given n, and the walrus operator in the while condition makes the termination logic clean - the loop exits the moment that tuple is empty.

The C version takes a slightly different but equivalent approach. Rather than computing floating-point inverse powers to narrow the base range, it simply iterates unconditionally over all nine candidates b = 1..9 for every n. For each pair it uses log10 to compute an approximate digit count and checks whether that equals n, then verifies with exact integer arithmetic via a custom ipow function that detects long long overflow. Digit counting is done by repeated integer division in ll_digit_count rather than string conversion. The termination condition is identical in spirit: the outer loop breaks as soon as a full pass over all nine bases finds no solutions.

The only meaningful language-level differences are that Python uses arbitrary-precision integers so overflow is never a concern, while the C version must guard against long long overflow explicitly - though in practice this guard is never triggered for bases 1-9 at the relevant exponents. Both produce the correct answer of 49.

Learnings

The key insight was recognising that b < 10 is not merely a convenient observation but a provable hard boundary that makes the problem finite with no arbitrary cutoff needed. Once I established that all valid bases are single digits, the termination condition for the outer loop also emerged naturally from the mathematics - when no base in 1..9 produces an n-digit nth power, the loop stops.

A subtlety worth remembering for future problems: floating-point inverse power operations such as x ** (1/n) can produce values just below an integer boundary, causing a ceiling to land one unit off. The safe pattern I used in the Python version - compute a slightly generous candidate range via floating point, then verify each candidate with exact integer arithmetic - is a reliable guard against those edge cases. The C version sidesteps the issue entirely by iterating all nine bases unconditionally and relying on log10 only as a fast pre-filter before the exact check. Both strategies are sound; the C approach is arguably simpler because it avoids the floating-point bracketing logic altogether.

The walrus-operator termination pattern in the Python while loop - while solutions := n_digit_nth_powers(n) - is a stylistic technique worth reusing whenever a loop naturally exits on an empty result. It avoids the need for a separate sentinel variable and keeps the loop condition self-documenting. At the benchmark scale shown (0.0001 s for Python, 0.0000 s for C) no algorithmic optimisation is warranted; the C implementation exists mainly as a cross-language consistency check.

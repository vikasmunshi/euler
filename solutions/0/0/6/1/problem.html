Problem 61: Cyclical Figurate Numbers [Level 3]

Problem Statement

Triangle, square, pentagonal, hexagonal, heptagonal, and octagonal numbers are all figurate (polygonal) numbers and are generated by the following formulae:

Triangle $P_{3,n}=n(n+1)/2$ $1, 3, 6, 10, 15, \dots$
Square $P_{4,n}=n^2$ $1, 4, 9, 16, 25, \dots$
Pentagonal $P_{5,n}=n(3n-1)/2$ $1, 5, 12, 22, 35, \dots$
Hexagonal $P_{6,n}=n(2n-1)$ $1, 6, 15, 28, 45, \dots$
Heptagonal $P_{7,n}=n(5n-3)/2$ $1, 7, 18, 34, 55, \dots$
Octagonal $P_{8,n}=n(3n-2)$ $1, 8, 21, 40, 65, \dots$

The ordered set of three $4$-digit numbers: $8128$, $2882$, $8281$, has three interesting properties.

  1. The set is cyclic, in that the last two digits of each number is the first two digits of the next number (including the last number with the first).
  2. Each polygonal type: triangle ($P_{3,127}=8128$), square ($P_{4,91}=8281$), and pentagonal ($P_{5,44}=2882$), is represented by a different number in the set.
  3. This is the only set of $4$-digit numbers with this property.

Find the sum of the only ordered set of six cyclic $4$-digit numbers for which each polygonal type: triangle, square, pentagonal, hexagonal, heptagonal, and octagonal, is represented by a different number in the set.

Mathematical Complexity

Each of the six polygonal families - triangle, square, pentagonal, hexagonal, heptagonal, and octagonal - is defined by a closed-form quadratic formula. Within the four-digit range 1000 to 9999, each family contains roughly 40 to 90 members, so the total pool of candidate numbers is only a few hundred. The structural insight is that the "last two digits equal first two digits of the next" linking rule defines a directed edge between numbers, converting the problem into a constrained directed cycle search: find a directed cycle of exactly length k that uses exactly one representative from each of k polygon families. Because every hexagonal number is also triangular, some numbers belong to multiple families simultaneously, introducing a coverage assignment problem on top of the cycle search.

Finding a Hamiltonian cycle in a directed graph is NP-hard in general, but here k is at most 6 and the node count is at most a few hundred, so a depth-limited depth-first search with backtracking finishes in negligible time. Building the successor adjacency structure is O(V^2) in the number of distinct figurate numbers V, which dominates the overall runtime. The polygon-type coverage verification after each candidate cycle is found is O(k times max_family_size), also trivial at this scale.

Solution Approach

Graph Construction and Depth-First Cycle Search with Polygon-Type Verification (Python and C)

I implemented exactly the same algorithm in both Python and C; the C version is a direct translation included for benchmarking. My approach proceeds in four phases: generate, link, search, and verify.

In the generation phase I produce every four-digit figurate number for each polygon type up to the requested chain length. In Python I used a FigurateNumber enum as a type-safe key and stored results in a dict[FigurateNumber, set[int]]. I paired each generator with an inverse "closest" function that computes the continuous index for a given value, allowing tight bounding of iteration ranges without starting from n=1 every time. In C I used a static array of function pointers and two-dimensional static arrays, avoiding heap allocation for the number lists entirely.

In the linking phase I built a directed successor graph over all unique figurate numbers in the pool. For every pair of numbers A and B, I added an edge A to B when A % 100 == B // 100 and the shared two-digit prefix is at least 10, eliminating degenerate leading-zero cases. Python stored successors in a dict[int, list[int]]; C used a static two-dimensional adjacency array with per-node counts.

The search is a recursive DFS function - find_cyclic_paths in Python and dfs in C. Starting from each node that has at least one outgoing edge, I explore paths of exactly length nodes while tracking visited nodes to prevent revisits. At depth length I check the closure condition: the last two digits of the current tail must equal the first two digits of the starting head. I canonicalise each found cycle by sorting its elements and inserting it into a set, which deduplicates all rotations of the same cycle without explicitly enumerating all k cyclic shifts.

The final verification step, verify_polygon_types, checks two things: that every polygon family contributes at least one member to the cycle, and that no number belonging to multiple families creates an unresolvable conflict where it is simultaneously the sole representative of two different families. In Python this is implemented with dictionary comprehensions and set intersections; in C it is a pair of nested loops over static arrays. Both versions implement identical logic.

I parameterised the solver by length, which let me validate against smaller cases before trusting the length-6 answer. The dev test cases confirm that lengths 2, 3, 4, and 5 return 1, 1, 2, and 5 valid cycles respectively.

Learnings

The key insight that unlocked the solution was recognising the digit-link condition as a directed graph edge. Once I framed "last two digits equal first two digits" as an adjacency relation rather than an arithmetic property, the whole structure became clear: build the graph, run a depth-limited DFS, and post-filter for polygon-type coverage. Without that reframing, the problem appears to require bespoke case analysis for each chain position.

A subtlety I had to handle carefully was numbers belonging to multiple polygon families. Every hexagonal number is also triangular, so a naive "each number in the cycle is figurate" check is insufficient. The verify_polygon_types function handles this by checking coverage at the family level and additionally rejecting cycles where a multi-family number is the sole representative of all its families simultaneously, because in that case the family assignment is unresolvable regardless of how one tries to label it.

Canonicalising cycles by sorting their elements is a clean trick for deduplicating rotations. It works because all four-digit integers in any valid cycle are distinct, so the sorted tuple is a unique canonical form - there is no need to explicitly generate all k cyclic shifts and check each one for prior existence.

The C version runs at roughly 0.0002-0.0018 s against Python's 0.0013-0.0350 s across all test cases, a speedup of around 17x to 20x on the heavier inputs. Both are fast enough that the difference is practically irrelevant for a one-shot solve, but the C port was a useful exercise in translating Python's dynamic data structures - enums, dict comprehensions, set intersections - into static arrays and function pointers. The translation is almost purely mechanical: every Python container maps to a fixed-size static array with an accompanying count variable, and every generator expression maps to a bounded for loop.

  • Modelling digit-link constraints as directed graph edges is a broadly reusable technique for any chaining or word-ladder style problem.
  • Inverse formulae for figurate numbers allow tight bounding of generator ranges without iterating from n=1, which matters more when the range is large.
  • Parameterising the solver over the chain length and testing against smaller instances is a reliable strategy for gaining confidence before committing to the full-scale answer.
  • Sorting cycle members for deduplication is simpler and cheaper than generating all rotations explicitly, as long as all cycle elements are distinct.
  • Coverage constraints on multi-valued assignments - one number, multiple family memberships - require explicit conflict detection, not just a membership presence check.

Problem Statement

Consider the following "magic" 3-gon ring, filled with the numbers 1 to 6, and each line adding to nine.


Working clockwise , and starting from the group of three with the numerically lowest external node (4,3,2 in this example), each solution can be described uniquely. For example, the above solution can be described by the set: 4,3,2; 6,2,1; 5,1,3.

Mathematical Complexity

A magic n-gon ring places 2n distinct positive integers on n spokes, where each spoke is a triple consisting of one outer node and two consecutive inner nodes. Because every inner node appears in exactly two spokes while every outer node appears in exactly one, the sum of all n line totals equals the sum of all outer values plus twice the sum of all inner values. For the 5-gon problem with values 1-10, the 16-digit constraint immediately forces the number 10 onto the outer ring: if 10 were on the inner ring it would contribute to two spoke strings and the total digit count would reach 17. This restricts valid inner ring values to 1-9 only, collapsing the search space before any enumeration begins.

Once an inner ring permutation is fixed, the n pairwise adjacency sums of consecutive inner nodes must all be distinct, because each is uniquely complemented by exactly one outer value. This distinctness condition prunes the vast majority of candidate inner permutations early. The raw enumeration is over permutations of n items chosen from at most min(9, 2n) candidates, giving O(min(9, 2n)! / (min(9, 2n) - n)!) inner candidates before pruning. For each surviving inner permutation the entire outer ring is then determined in O(n) time, so no independent search over outer values is needed. In practice the inner-sum distinctness filter makes the effective search far smaller than the raw permutation count.

Solution Approach

Constrained Outer Reconstruction (Python and C)

I implemented the same core algorithm in both Python ( p0068_s0.py) and C ( p0068_s0.c). The central idea is to enumerate inner ring permutations and reconstruct the outer ring deterministically rather than searching over it independently.

In the Python version I use itertools.permutations over range(1, min(9, 2*n)+1) to generate inner candidates. For each candidate I compute the n pairwise adjacency sums for consecutive pairs in the cyclic inner ring. If these are not all distinct I skip the candidate immediately. I then build the set of remaining values as outer candidates, place the minimum of those at position zero to enforce the canonical clockwise-lowest-external-node starting convention, derive the magic line sum as that minimum plus the first inner adjacency sum, and compute every remaining outer value by subtracting the appropriate inner adjacency sum from the fixed line sum. If any required outer value is absent from the candidate set the inner permutation is rejected. When a valid ring is found I concatenate the spoke triples into a string and track the maximum numeric value. The Ring and Line namedtuples are used only for debug output when --show is passed and have no effect on performance.

The C version uses a recursive backtracking function solve_recursive that builds the inner ring one position at a time, marking values as used in a boolean array. This allows pruning to happen during construction rather than only after a full permutation is assembled. Once a complete inner permutation is assembled, the outer reconstruction logic is identical to the Python version: find the minimum available outer value, set the line sum, and derive the rest. String comparison for the maximum uses a custom cmp_magic function that first compares string lengths (longer means larger for positive integers) and falls back to lexicographic comparison for equal lengths, avoiding any big-integer arithmetic throughout.

Learnings

The central insight was recognising that the outer ring is not an independent degree of freedom once the inner ring permutation and the canonical starting constraint are applied. Fixing the minimum available outer value at position zero and then deriving every other outer value from the magic-sum constraint reduces what initially looks like a two-ring search into a single-ring enumeration with O(n) verification per candidate. The distinctness of inner adjacency sums is the key structural property that makes this reconstruction unambiguous.

The digit-length argument for restricting inner values was a second key insight. Recognising that a two-digit number on the inner ring contributes twice to the concatenated string length - and that 10 is the only two-digit value in 1-10 - lets the 16-digit constraint be encoded as a simple upper bound of 9 on inner values, so the permutation generator never considers invalid placements.

The benchmark results show the C version is roughly 25-35x faster than Python across all ring sizes tested. At ring size 5 (the main problem) Python takes about 31 ms while C takes under 1 ms. At ring size 9 Python finishes in about 491 ms and C in about 20 ms, so both stay within reasonable budgets. The structural difference between the two - Python generates complete permutations before checking distinctness whereas the C recursive backtracker can prune mid-construction - is largely masked by the raw speed advantage of C, though at larger ring sizes the backtracker's early pruning would become increasingly valuable. For future ring or star-polygon problems the pattern worth remembering is: identify which variables are fully determined by the magic-sum constraint and a canonical starting rule, resolve those deterministically, and enumerate only over the remaining free variables.

Problem 67: Maximum Path Sum II [Level 2]

Problem Statement

By starting at the top of the triangle below and moving to adjacent numbers on the row below, the maximum total from top to bottom is 23.

3
7 4
2 4 6
8 5 9 3

That is, 3 + 7 + 4 + 9 = 23.

Find the maximum total from top to bottom in triangle.txt (right click and 'Save Link/Target As...'), a 15K text file containing a triangle with one-hundred rows.

NOTE: This is a much more difficult version of Problem 18 . It is not possible to try every route to solve this problem, as there are $2^{99}$ altogether! If you could check one trillion ($10^{12}$) routes every second it would take over twenty billion years to check them all. There is an efficient algorithm to solve it. ;o)

Mathematical Complexity

The naive approach of enumerating every root-to-leaf path through the triangle is exponential: for a triangle with n rows there are 2^(n-1) distinct paths, which for n = 100 is completely infeasible. The key observation is that the problem has optimal substructure - the maximum path sum passing through any node equals that node's value plus the larger of the two maximum path sums reachable from its two children. Those children are shared across many paths, creating overlapping subproblems, and the combination makes this a textbook dynamic programming application. Resolving each node's best-downstream value exactly once collapses the complexity to O(n^2) time, matching the total number of nodes in the triangle (5050 for n = 100), with O(n) additional working space if the reduction is performed in place on a shrinking copy of the triangle.

The triangle is effectively a strictly layered directed acyclic graph where each node at depth d has exactly two children at depth d+1, and the goal is a longest-path query from root to any leaf. Because the graph is layered, a single bottom-up sweep over the rows is sufficient - no general longest-path algorithm such as Bellman-Ford is needed. Each row depends only on the single row immediately below it, so no memoisation dictionary or recursion stack is required; the entire computation reduces to one linear pass over the rows from the bottom upward.

Solution Approach

Bottom-Up Triangle Collapse (Python and C)

I implemented the same bottom-up reduction algorithm in both Python and C. The triangle is read from a text file and parsed into a list-of-lists (Python) or array-of-arrays (C) of integers. Starting from the second-to-last row and working upward toward the apex, each element is replaced by itself plus the larger of its two neighbours in the row immediately below, after which the now-consumed bottom row is discarded. After n-1 such passes the structure collapses to a single element holding the global maximum path sum.

In my Python version I apply copy.deepcopy to the parsed triangle before passing it to max_path_sum_triangle, keeping that helper function free of side effects so the original data remains usable. The inner update is a compact list comprehension - v + max(triangle[-1][i], triangle[-1][i+1]) for each index i and value v in the second-to-last row - followed by del triangle[-1] to drop the consumed row. In my C version I allocate a fresh copy of every row using memcpy at the start, achieving the same purity, then update rows in place and call free on the bottom row after each pass. The path-finding logic is otherwise identical: second[i] = second[i] + (left > right ? left : right). Both implementations produce the correct answer of 7273.

File loading differs between the two implementations. My Python version delegates to a utility that locates the resource directory relative to the source file using pathlib.Path. My C version reads the executable's own path via readlink("/proc/self/exe", ...), strips the filename with dirname, and constructs the resource path from the resulting directory. Both approaches make the binary self-locating regardless of the working directory from which it is invoked.

Learnings

The key insight was flipping the direction of the search. Instead of asking which branch to follow downward from the apex - which requires knowing the future - I ask what the best total reachable from each node is, working from the leaves upward. By the time the sweep reaches the apex, every downstream decision has already been resolved optimally without any branching or backtracking. This bottom-up sweep is cleaner to implement than top-down memoisation for this problem because each row depends only on the single row immediately below it, and discarding each processed row keeps memory proportional to the shrinking triangle.

The benchmark confirms the expected language gap: my C solution runs in roughly 0.1 ms while the Python solution takes around 1.5 ms, a difference of about 15x. For a 5050-node triangle the actual DP computation is trivial in both languages; the gap is dominated by interpreter overhead and Python's dynamic memory model rather than any algorithmic difference. Both are fast enough to be imperceptible in practice.

  • Bottom-up DP avoids recursion depth limits and memoisation overhead for any strictly layered graph problem.
  • Discarding each processed row after the pass is a clean invariant - it makes the shrinking size of the working structure immediately visible and keeps memory usage proportional to the current triangle depth.
  • Using deepcopy in Python and explicit memcpy copies in C keeps transformation functions pure, which simplifies testing and allows the original parsed data to be reused if needed.
  • For any path-sum problem on a triangle or grid, always consider starting the DP sweep from the boundary and working inward toward the root - it is almost always simpler than working outward from the root because each update sees only already-resolved values.
  • The technique of collapsing a layered DAG in place from the leaves upward generalises directly to rectangular grid path problems and longer dependency chains that appear in later Project Euler problems.

Problem Analysis

This problem asks for the maximum-sum path from the apex of a triangle of integers down to its base, where at each step I may step only to one of the two numbers directly below. With 100 rows there are 299 distinct top-to-bottom paths, so trying every route - brute force - is hopeless. The problem statement makes the point explicitly: even at a trillion paths per second, exhaustive search would take longer than the age of the universe. The exercise is really about recognising that a far smarter method exists.

The crucial observation is that the problem has optimal substructure: the best path passing through any given cell equals that cell's value plus the better of the two best paths through its two children below. Because the answer for the whole is built from optimal answers to smaller, overlapping sub-problems, this is exactly the setting where dynamic programming (DP) applies - solving each sub-problem once, storing its result, and reusing it instead of recomputing. The triangle can also be viewed as a directed acyclic graph - a network of nodes connected by one-way edges that never loop back - and finding a maximum-weight path through such a graph is a textbook dynamic programming problem. Since each of the roughly n(n+1)/2 cells is visited exactly once, the algorithm runs in O(n2) time - about 5,000 operations for n = 100, which is effectively instantaneous.

Solution Approach

Bottom-Up Triangle Fold (Python and C)

The Python and C solutions implement one and the same algorithm; they differ only in syntax and in how they manage memory. I chose a bottom-up formulation - starting from the base cases (the last row) and working upward to the answer - rather than recursing from the top down. This sidesteps recursion overhead and removes any need for an explicit cache, because the base cases are simply the cells themselves and the merged values accumulate naturally as I climb.

Step 1 - Parse the input. The triangle is a text file, one row per line, with space-separated integers; row i holds i+1 numbers. I read the file and split it into a list of lists of integers (Python's text2triangle) or a heap-allocated array of arrays (C's text2triangle), indexed as triangle[row][col]. The C parser walks the buffer twice in spirit - first counting rows, then allocating each row to its exact length and filling it with atoi.

Step 2 - Deep copy. The fold mutates the triangle, so I copy it first to leave the parsed data intact and keep max_path_sum_triangle free of side effects on its input. Python uses copy.deepcopy; C allocates fresh rows and copies them with memcpy.

Step 3 - Fold from the bottom. While more than one row remains, I update the second-to-last row by adding to each entry the larger of its two children in the last row, then discard the last row. In Python this is a single expressive line:
triangle[-2] = [v + max(triangle[-1][i], triangle[-1][i + 1]) for i, v in enumerate(triangle[-2])]
followed by del triangle[-1]. In C I write the explicit loop, choose the larger child with a ternary expression (left > right ? left : right), and free the discarded row immediately after it is merged.

Step 4 - Read off the answer. When the loop ends, exactly one cell remains: triangle[0][0] holds the maximum path sum, 7273.

Programming Techniques

In-Place Bottom-Up Fold

Rather than allocating a separate DP table, I overwrite the triangle from the bottom up, merging each row into its parent. The recurrence "combine two children into a parent" is naturally a fold: I reduce the structure one row at a time until a single value remains. This keeps memory use minimal and lets the answer fall out at the apex. Whenever a dynamic programming recurrence has this "merge children upward" shape, folding in place is a clean and compact option worth reaching for.

List Comprehension with Atomic Reassignment (Python)

The row-update step builds an entirely new list and assigns it back to triangle[-2] in one statement. Because every read of triangle[-1] happens before the assignment completes, there is no danger of reading half-updated data - a subtle hazard of element-by-element in-place mutation. List comprehensions are idiomatic and typically faster than the equivalent explicit loop, so I prefer them whenever transforming one sequence into another.

Negative Indexing for Readability (Python)

triangle[-1] and triangle[-2] name the last and second-to-last rows directly, with no index variable to track and decrement. This makes the intent - "always operate on the two trailing rows" - immediately obvious and eliminates a whole class of off-by-one errors. In C I express the same idea with cur_rows - 1 and cur_rows - 2, decrementing cur_rows each iteration.

Explicit Memory Management in C

Each triangle row in the C version is its own malloc-allocated block. The moment a row is merged upward I call free on it, rather than deferring all cleanup to the end. Even though the data here is tiny, freeing memory as soon as it becomes unreachable keeps resource lifetimes explicit and prevents leaks accumulating in longer-running programs. I also wrap the allocate-compute-free cycle inside max_path_sum_triangle and handle allocation failure defensively - if any row's malloc fails mid-copy, I free the rows already taken before returning - so the ownership contract stays local and auditable.

Separation of I/O, Parsing, and Algorithm

Both solutions split into a file reader (get_text_file), a parser (text2triangle), and the core algorithm (max_path_sum_triangle). This lets me test the algorithm on any triangle without touching I/O, swap the data source freely, and benchmark only the work that matters. Single-purpose functions are easier to reason about, reuse, and verify.

Key Takeaways

The unlocking insight is recognising optimal substructure: when the best whole-problem solution is composed of best sub-problem solutions, an exponential search space collapses to a polynomial dynamic programming computation. The instant a problem reads "at each step choose one of two children and maximise the total", DP should be the first idea you reach for.

Both implementations run the identical O(n2) algorithm. On the benchmark the C version (p0067_s0_c) completed in 0.0001 s and the Python version (p0067_s0.py) in 0.0010 s - roughly a tenfold difference attributable to interpreter overhead and managed versus manual memory. For a problem this small both are instantaneous and the Python version is perfectly adequate; the C version simply shows the same idea expressed at the systems level with explicit allocation.

  • When an optimal value is sought over an exponential search space, look for optimal substructure - if it exists, dynamic programming makes the problem polynomial.
  • Bottom-up DP avoids recursion and cache-lookup overhead and is often simpler to implement than top-down memoisation.
  • Folding a recurrence in place, overwriting the input as you reduce it, saves both memory and code when intermediate states need not be preserved.
  • Build new sequences with list comprehensions and assign them atomically to avoid read-during-mutation bugs.
  • Use negative or end-relative indexing to refer to the tail of a sequence; it reads clearly and removes off-by-one mistakes.
  • In C, free heap memory as soon as it is no longer needed, and free partial allocations on the error path so failures never leak.
  • Decompose programs into small, single-purpose functions so parsing, I/O, and the core algorithm can be tested and benchmarked independently.

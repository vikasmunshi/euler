Problem 65: Convergents of $e$ [Level 2]

Problem Statement

The square root of $2$ can be written as an infinite continued fraction.

$$\sqrt{2} = 1 + \dfrac{1}{2 + \dfrac{1}{2 + \dfrac{1}{2 + \dfrac{1}{2 + ...}}}}$$

The infinite continued fraction can be written, $\sqrt{2} = [1; (2)]$, $(2)$ indicates that $2$ repeats ad infinitum . In a similar way, $\sqrt{23} = [4; (1, 3, 1, 8)]$.

It turns out that the sequence of partial values of continued fractions for square roots provide the best rational approximations. Let us consider the convergents for $\sqrt{2}$.

$$\begin{align} &1 + \dfrac{1}{2} &= \dfrac{3}{2} \\ &1 + \dfrac{1}{2 + \dfrac{1}{2}} &= \dfrac{7}{5}\\ &1 + \dfrac{1}{2 + \dfrac{1}{2 + \dfrac{1}{2}}} &= \dfrac{17}{12}\\ &1 + \dfrac{1}{2 + \dfrac{1}{2 + \dfrac{1}{2 + \dfrac{1}{2}}}} &= \dfrac{41}{29} \end{align}$$

Hence the sequence of the first ten convergents for $\sqrt{2}$ are:

$$1, \dfrac{3}{2}, \dfrac{7}{5}, \dfrac{17}{12}, \dfrac{41}{29}, \dfrac{99}{70}, \dfrac{239}{169}, \dfrac{577}{408}, \dfrac{1393}{985}, \dfrac{3363}{2378}, ...$$

What is most surprising is that the important mathematical constant,

$$e = [2; 1, 2, 1, 1, 4, 1, 1, 6, 1, ... , 1, 2k, 1, ...]$$

The first ten terms in the sequence of convergents for $e$ are:

$$2, 3, \dfrac{8}{3}, \dfrac{11}{4}, \dfrac{19}{7}, \dfrac{87}{32}, \dfrac{106}{39}, \dfrac{193}{71}, \dfrac{1264}{465}, \dfrac{1457}{536}, ...$$

The sum of digits in the numerator of the $10$ th convergent is $1 + 4 + 5 + 7 = 17$.

Find the sum of digits in the numerator of the $100$ th convergent of the continued fraction for $e$.

Mathematical Complexity

The continued fraction expansion of e follows the pattern [2; 1, 2, 1, 1, 4, 1, 1, 6, 1, ..., 1, 2k, 1, ...], where every third coefficient at 1-indexed positions 3, 6, 9, ... equals 2k for k = 1, 2, 3, ..., and all other coefficients equal 1 (except the leading 2 at position 1). The n-th convergent numerator satisfies the forward recurrence h_k = a_k * h_{k-1} + h_{k-2}, and these numerators grow roughly exponentially with n - the 10000th convergent numerator has tens of thousands of decimal digits. Because each multiplication step involves a big integer of O(n) digits and there are O(n) steps, the overall work is O(n^2) in digit operations.

The benchmark results confirm this quadratic scaling. Going from n = 1000 to n = 10000 is a 10x increase in input. The Python solution goes from about 0.014 s to about 9.9 s - roughly a 710x slowdown, worse than pure quadratic, because Python's Fraction class also pays a GCD cost at every step. The C solution scales far more gracefully, finishing n = 10000 in about 0.020 s, because it implements the recurrence directly over base-10^9 limb arrays with no GCD overhead.

Solution Approach

Top-Down Recursive Evaluation with Python Fraction and Forward Recurrence in C (s0)

Both files share the same coefficient rule encoded in e_denominator(n): return 2 when n == 1, return 2 * (n // 3) when n % 3 == 0, and return 1 otherwise. This compresses the infinite irregular-looking sequence into a three-line function with no lookup table.

The Python solution computes the convergent top-down via nth_convergent_of_e(n, _n=1), which recurses to depth n (the innermost coefficient) and on the way back up builds a_k + 1/remainder using Python's built-in Fraction class for exact rational arithmetic. I chose top-down recursion because it maps directly onto the mathematical definition of a nested fraction and produces correct numerators trivially. I set sys.setrecursionlimit(10**6) to prevent stack overflow for large n. The main cost is that Fraction reduces to lowest terms at every return via a GCD computation, which adds significant overhead when numerators have thousands of digits.

The C solution takes the opposite approach: it uses the forward recurrence h_k = a_k * h_{k-1} + h_{k-2} iteratively, tracking only the numerator since the problem asks only for its digit sum. I represented big integers as arrays of base-10^9 limbs in a BigInt struct with a fixed maximum of 2000 limbs. The combined operation bigint_mul_add(dst, a, scalar, b) computes a * scalar + b in one pass with a carry loop, which is cache-friendly and avoids any GCD step entirely. Three BigInt pointers are rotated at each step to avoid copying. The digit sum is extracted by walking each limb's decimal digits. This approach is purely iterative and eliminates all fraction overhead.

Learnings

The key insight was recognising the index pattern for the coefficients of e 's continued fraction. Once I expressed it as "return 2*(n//3) when n % 3 == 0, else 1 (with 2 at position 1)", the rule fit in a three-line function with no precomputed table needed. After that, both the recursive and iterative approaches fell out naturally.

The second insight was that for a numerator-only recurrence there is no need to track the denominator or perform any GCD reduction. The forward recurrence keeps everything as pure integer addition and multiplication, which is exactly what the C implementation exploits. Comparing the two approaches using the benchmark results makes this concrete:

n Python (s) C (s) Speedup
100 0.0003 ~0.000 -
1000 0.0139 0.0001 ~139x
10000 9.9261 0.0195 ~509x

The widening gap at larger n reflects both interpreter overhead and the hidden GCD cost inside Fraction. The C solution avoids both by working directly with integer limbs and never reducing fractions. For quick prototyping or small n, Python's Fraction is perfectly fine and produces correct results with minimal code. For performance-critical paths involving large big-integer arithmetic, maintaining numerator and denominator as raw integers and using the forward recurrence is strongly preferable.

  • The continued fraction coefficients of e obey a clean arithmetic rule that eliminates any need for a precomputed table.
  • The forward recurrence h_k = a_k * h_{k-1} + h_{k-2} is both simpler and faster than top-down recursive evaluation with a rational type, because it avoids GCD computation entirely.
  • Python's Fraction class carries hidden GCD overhead at every arithmetic step; for large-n big-integer paths, raw integer recurrence is strongly preferable.
  • Setting sys.setrecursionlimit explicitly is necessary whenever recursion depth is proportional to a problem parameter that can exceed Python's default limit of 1000.
  • A base-10^9 limb representation in C keeps the number of limbs small and makes decimal digit-sum extraction straightforward, since each limb contributes at most 9 decimal digits.

Problem 77: Prime Summations [Level 4]

Problem Statement

It is possible to write ten as the sum of primes in exactly five different ways:

$$\begin{align} &7 + 3\\ &5 + 5\\ &5 + 3 + 2\\ &3 + 3 + 2 + 2\\ &2 + 2 + 2 + 2 + 2 \end{align}$$

What is the first value which can be written as the sum of primes in over five thousand different ways?

Test Cases

  • dev: solve(num_prime_partitions=50) → 25
  • dev: solve(num_prime_partitions=500) → 45
  • main: solve(num_prime_partitions=5000) → 71
  • extra: solve(num_prime_partitions=50000) → 104

Mathematical Complexity

This problem is a restricted integer partition problem: rather than partitioning a number into any positive integer parts, every part must be prime. The key combinatorial challenge is counting each unordered multiset of primes exactly once. The standard technique is to enforce a non-increasing ordering constraint through a second argument - a cap on the largest prime allowed in the next position. This transforms an otherwise exponential enumeration into a clean two-argument memoised recursion. Each unique state is a pair (remaining sum, largest prime allowed), so the total number of distinct recursive calls is at most O(n * pi(n)) where pi(n) is the count of primes up to n. For the main case (threshold 5000, answer 71) this state space is tiny, and the recursion terminates almost instantly. Generating primes up to a bound m uses the Sieve of Sundaram in O(m log m) time.

The enumeration variant - materialising every distinct partition as an explicit list of integers - has the same state space in terms of unique (number, slots) pairs, but the storage and CPU cost scales with the total number of partitions across all states, which grows exponentially with the target. For small inputs this is manageable; beyond roughly 50-70 the Python version hits a practical memory and time wall, while the C version can push further by relying on manual heap allocation with a generous safe limit.

Solution Approach

Memoised Counting Recursion (s0 - Python and C)

I implemented a pure counting function num_prime_partitions_simple_recursion(number, slots) in Python, memoised with lru_cache. The slots parameter caps the largest prime allowed in the next recursive step, enforcing non-increasing order and ensuring each multiset is counted exactly once. The base case returns 1 when number == 0, and 0 when slots < 2. For each prime p not exceeding min(number, slots) I recurse on (number - p, p) and accumulate the total. The outer loop in solve increments n from 1 upward and returns the first value whose count meets the threshold. Because all intermediate counts are cached, work for increasing n reuses earlier results very efficiently.

The C translation in p0077_s0.c reproduces the same algorithm with an open-addressing hash map of 2^18 buckets as the memo table, keyed by (number, slots) pairs and resolved by linear probing. Primes up to min(number, slots) are generated fresh via the Sieve of Sundaram on each uncached call and freed before returning. The memo is reset between benchmark runs via memo_init. The C version is materially faster than the Python one - roughly 0.0001 s versus 0.0006 s on the main case - but both are well within comfortable time limits.

Memoised Enumeration Recursion (s1 - Python and C)

I also wrote a version that materialises every prime partition as an explicit collection of integers. In Python, get_prime_partitions_simple_recursion(number, slots) returns a list[list[int]] and is memoised with lru_cache. A safe_limit guard (default 50) raises OverflowError when number exceeds it. This is why the Python s1 only passes the three dev cases (thresholds 5, 50, 500 yielding answers 10, 25, 45) and raises on the main and extra cases whose answers (71 and 104) both exceed that safe limit. Every returned partition is checked with an assertion that its parts sum to the target, making s1.py useful as a correctness validator for s0 on small inputs.

The C translation in p0077_s1.c uses the same enumeration strategy but sets g_safe_limit = 200 inside solve, which is generous enough to reach both answers 71 and 104. Each partition is stored as a heap-allocated int array inside a Partition struct, and a PartList holds a dynamic array of those. Sub-partitions retrieved from the memo are extended by prepending the chosen prime into a newly allocated array. The C enumeration version therefore succeeds on all five test cases, unlike its Python counterpart. However, materialising every partition list is expensive compared to pure counting: on the main case s1_c takes about 0.0056 s versus 0.0001 s for s0_c, roughly a 56-fold slowdown, and on the extra case the gap widens to about 300-fold (0.0608 s versus 0.0002 s).

Learnings

The key insight was recognising that the "largest part allowed" second argument is what prevents any multiset from being counted more than once. Without that cap, the same multiset would be counted once per distinct ordering, inflating counts by the multinomial coefficient of the partition's multiplicities. Once I understood slots as an ordering enforcer rather than just a bound, the recursion wrote itself naturally and was trivial to memoise.

Comparing the four solutions via benchmark results makes the trade-offs concrete:

The counting approach dominates in every dimension. s0_c is the fastest overall and completes all cases in under a quarter of a millisecond. s1_c is correct on all five cases thanks to the generous safe limit of 200, but it is 56-300x slower than s0_c on the larger targets because storing every partition list consumes exponentially more memory and CPU time than accumulating a single integer. The Python s1 collapses entirely beyond its default safe limit of 50, which is why it cannot handle the main or extra cases at all.

  • Use a "largest part allowed" parameter to enforce non-increasing order - this is the standard technique for counting multisets without double-counting permutations, and it applies to any restricted partition problem.
  • Count, never enumerate, when only the quantity is needed. Partition counts grow exponentially while the count itself is one integer, so enumeration is orders of magnitude more expensive for no gain when the answer is a single number.
  • The Sieve of Sundaram is a compact drop-in replacement for the Sieve of Eratosthenes, convenient to call per recursion state when primes up to a varying bound are needed.
  • lru_cache on a two-argument counting recursion is a transparent substitute for a 2-D DP table and is much easier to write and validate correctly.
  • Keeping an enumeration version alongside a counting version is a good debugging strategy - I used s1 to verify s0 was correct on the dev cases before trusting it on the main case.

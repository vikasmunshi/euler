Problem Analysis

This problem asks me to count, for each integer n, the number of ways it can be written as an unordered sum of primes (with repetition allowed), and to find the first n whose count exceeds a given threshold. This is a constrained form of the integer partition problem: a partition of n is any multiset of positive integers summing to n, and here the allowed parts are restricted to primes. The word "unordered" is the crux: 7 + 3 and 3 + 7 describe the same decomposition and must be counted once. The standard device for enforcing this is to impose a canonical ordering on the parts - I require parts to be non-increasing, so each multiset is reached through exactly one ordered representative.

The tractability insight is to parameterise a recursion by two quantities: the remaining sum and a ceiling on the largest prime part still allowed. I call this ceiling slots. When I select a prime p, every later part must be at most p, so the recursive call passes p as the new ceiling - this single parameter eliminates double-counting. Because the same (number, slots) pair recurs across many branches, I apply memoization to cache each subproblem's answer, turning an exponential recursion into a dynamic programming computation. The number of distinct subproblems is O(N * pi(N)), where pi(N) is the prime-counting function, and each subproblem does O(pi(N)) work. Since the answer for the largest test case lies near n = 104, this is effectively instantaneous.

Solution Approach

Count-only memoised recursion (p0077_s0.py and p0077_s0.c)

My primary approach computes only the count of prime partitions, never the partitions themselves. The recursive function num_prime_partitions(number, slots) returns the number of ways to partition number using prime parts each at most slots. The base cases are direct: if number == 0 I have completed a valid partition and return 1; if slots < 2 no prime fits so I return 0. For the recursive step I iterate over every prime p up to min(number, slots) and sum num_prime_partitions(number - p, p). Passing p (not slots) as the next ceiling is precisely what enforces the non-increasing constraint and prevents duplicate counting.

In Python I let functools.lru_cache handle the cache transparently. In C I hand-roll an open-addressing hash table: each (number, slots) key is hashed to a bucket, and linear probing resolves collisions. I sized the table at 2^18 (262144) buckets so the load factor stays low across the input range, and the modulo reduces to a bitwise AND because the size is a power of two. The driver iterates n from 1 upward, calling num_prime_partitions(n, n), and returns the first n whose count reaches the threshold. This is the fastest design because it allocates nothing beyond the memo table and stores a single integer per subproblem.

Enumeration-based recursion (p0077_s1.py and p0077_s1.c)

The second approach constructs and stores every individual partition as a list of integers, returning a list-of-lists; the answer is then just its length. In Python, get_prime_partitions_simple_recursion returns list[list[int]] and is memoised by the (number, slots) key. Each recursive call prepends the chosen prime p to every sub-partition returned for the remainder. An assertion verifies each partition sums correctly - a useful debugging aid - and a safe_limit parameter (default 50) guards against runaway growth. That guard is exactly why the Python enumeration version reports overflow for thresholds 5000 and 50000: the answers (71 and 104) exceed the limit, so the search aborts before reaching them.

The C version is far more involved because C offers no built-in dynamic lists or garbage collection. I define a Partition struct (an int array plus a length) and a PartList struct (a growable array of Partitions). The memo is a separate hash table of PartList values, sized at 2^14. For each uncached query I iterate over primes, malloc and memcpy a new array with p prepended to each sub-partition, and append it to the result. Memory ownership is delicate: memo entries own their arrays, and enum_memo_init frees them all before clearing the table. The C driver raises its own g_safe_limit to 200, large enough to find the answer for every test case. This approach allocates and copies O(count) int arrays per subproblem, so it is dramatically slower than the count-only version.

Programming Techniques

Memoisation with lru_cache

Both Python solutions decorate the recursive function with functools.lru_cache. The decorator intercepts each call, uses the arguments as a dictionary key, and returns a cached result on a hit; maxsize=None makes the cache unbounded. This is the right tool for any pure (side-effect-free) recursive function with overlapping subproblems - reach for it whenever the return value depends only on the arguments and the same argument combinations recur.

Open-addressing hash table in C

Both C solutions implement their own open-addressing hash table with linear probing. I preferred this over a chained (linked-list) table for cache locality: all entries live in one flat array, so probing adjacent buckets touches the same cache lines. The hash mixes the two integer keys with a multiplicative constant and XOR-shifts to spread keys uniformly, and a power-of-two size lets the index reduce to a bitwise AND. This is the go-to pattern for a fast key-value store in C when the key space is bounded.

Sieve of Sundaram for prime generation

All solutions generate primes with the Sieve of Sundaram, which marks integers of the form i + j + 2ij and reads off odd primes 2k+1 from the unmarked positions. It works only on odd numbers, so it uses roughly half the memory of the Sieve of Eratosthenes. Here it is called with small ceilings, so its overhead is negligible; I include it for a clean, correct prime source.

Canonical ordering to avoid duplicate counting

The slots parameter encodes a non-increasing constraint: after picking prime p, all later parts must be at most p. The recursion therefore explores exactly one representative of each multiset, so no explicit deduplication is needed. This shrinking-upper-bound pattern is a general technique for enumerating or counting unordered selections with repetition.

Dynamic arrays with doubling reallocation (C enumeration)

The PartList grows by doubling its capacity (starting at 8) whenever it fills. This geometric expansion amortises the cost of realloc to O(1) per append. Always grow by a multiplicative factor rather than a fixed increment, which would degrade total allocation work to O(n^2).

Defensive recursion guards

The enumeration solutions carry a safe-limit parameter that aborts when number exceeds a threshold, raising OverflowError in Python or returning NULL in C. This is a deliberate guard against an algorithm whose memory grows explosively. When an approach has a known scaling cliff, an explicit limit converts silent resource exhaustion into a clear, diagnosable failure - the Python overflow verdicts in the benchmark are exactly this guard doing its job.

Key Takeaways

The unlocking insight is that prime partitions can be counted efficiently without ever being enumerated, via a memoised recursion over (remaining sum, maximum allowed part). The slots parameter does double duty: it enforces canonical ordering and keeps the memo key compact.

The benchmarks make the cost of enumeration vivid. The count-only C solution finishes both the main case (threshold 5000) and the extra case (50000) in about 0.0001 s. The enumeration C solution is markedly slower at 0.0003 s and 0.0034 s respectively, because it allocates and copies a growing collection of int arrays per subproblem. The count-only Python solution is comparably fast (0.0001 s and 0.0003 s) and beats the enumeration C version on the larger input, while the Python enumeration version cannot reach the answer under its safe limit and reports overflow on both the main and extra cases. When you only need a count, never enumerate.

  • Before enumerating combinatorial objects, ask whether a pure count suffices - enumeration is exponentially more expensive in time and memory.
  • A shrinking "maximum allowed part" parameter is the standard device for avoiding duplicate multisets in any unordered selection with repetition.
  • Memoisation turns exponential recursion into polynomial dynamic programming whenever the number of distinct subproblem keys is polynomial.
  • In C, prefer a flat open-addressing hash table with a power-of-two size for fast, cache-friendly memoisation.
  • Grow dynamic arrays by doubling, never by a fixed increment, to guarantee amortised O(1) insertion.
  • Add explicit limits to algorithms with known scaling cliffs so resource exhaustion surfaces as a clear, diagnosable error.
  • For primes in a bounded range, reach for a sieve rather than trial division.

Problem 70: Totient Permutation [Level 3]

Problem Statement

Euler's totient function, $\phi(n)$ [sometimes called the phi function], is used to determine the number of positive numbers less than or equal to $n$ which are relatively prime to $n$. For example, as $1, 2, 4, 5, 7$, and $8$, are all less than nine and relatively prime to nine, $\phi(9)=6$.
The number $1$ is considered to be relatively prime to every positive number, so $\phi(1)=1$.

Interestingly, $\phi(87109)=79180$, and it can be seen that $87109$ is a permutation of $79180$.

Find the value of $n$, $1 \lt n \lt 10^7$, for which $\phi(n)$ is a permutation of $n$ and the ratio $n/\phi(n)$ produces a minimum.

Mathematical Complexity

Euler's totient function phi(n) counts integers up to n that are coprime to n. For a semi-prime n = p1 * p2 with two distinct primes, phi(n) = (p1 - 1)(p2 - 1), and the ratio n / phi(n) factors as (p1 * p2) / ((p1 - 1)(p2 - 1)). Each prime factor p contributes a multiplicative term p/(p-1) to this ratio, and that term grows large as p shrinks, so any factorisation involving small primes inflates the ratio badly. To minimise the ratio while keeping n below 10^7, I need both prime factors as large as possible - ideally near sqrt(10^7) ~ 3162. Prime n can be ruled out immediately: phi(p) = p - 1, and p and p - 1 differ by exactly one, so they virtually never share the same digit multiset. The search therefore collapses to a nested loop over semi-prime pairs (p1, p2) with p1 in (sqrt_n / 2, sqrt_n] and p2 in (p1 + 2, limit / p1], which involves only a few hundred primes and makes the effective runtime negligible.

A full Euler totient sieve over all n below 10^7 would run in O(N log log N) time and O(N) space. By restricting to the semi-prime structure described above, the search collapses to a tiny nested loop. The digit-permutation check for each candidate is O(d) where d is the number of decimal digits, which is a constant for numbers below 10^7. Prime generation up to the limit dominates the actual runtime in both implementations.

Solution Approach

Semi-prime Pair Search with Digit Permutation Check (s0 - Python and C)

Both implementations share the same algorithm. I set the outer loop over primes p1 in the range (sqrt_n / 2, sqrt_n] and the inner loop over primes p2 in (p1 + 2, limit / p1]. For each pair I compute number = p1 * p2 and totient = (p1 - 1) * (p2 - 1), check whether the two values are digit permutations of each other, and track the pair with the smallest ratio number / totient among all passing candidates.

The Python version uses an on-demand infinite prime generator backed by a dictionary of composites. This avoids allocating a full sieve array and is memory-efficient at the cost of some per-prime overhead. The permutation check uses sorted(str(x)) == sorted(str(y)), which is concise and fast enough given the small number of candidates to evaluate. The Python solution runs in about 0.14 seconds.

The C version uses a classical Sieve of Eratosthenes to precompute all primes up to the limit and then walks the prime array with index arithmetic rather than re-generating primes for each inner loop. The digit permutation check uses a digit frequency array of length 10: it increments counts for one number and decrements for the other, then verifies all entries are zero. This avoids any string allocation and is slightly faster than sorting. The C version runs in about 0.045 seconds, roughly 3.1 times faster than Python, a modest gap that reflects the fact that the inner loop is tiny in both cases and prime generation overhead dominates.

Learnings

The critical insight was recognising that minimising n / phi(n) and satisfying the digit-permutation constraint simultaneously almost forces n to be a semi-prime with both factors clustered near sqrt(limit). Once I accepted that restriction, the brute-force sieve became unnecessary and the problem reduced to a small nested loop over a handful of primes near 3162.

Both solutions implement the same algorithm, so the benchmark comparison reflects language and implementation overhead rather than algorithmic differences. The C version is about 3x faster, which is a modest gap compared to problems where the algorithm itself is more compute-heavy. The Python version is arguably more elegant because the on-demand prime generator never materialises a full sieve array, which is a useful pattern when only primes in a bounded range are needed.

  • For ratio minimisation problems involving phi(n), factor the ratio expression first - the factored form immediately reveals which prime structures minimise it and allows you to prune the search space dramatically before writing any code.
  • Prime n can be ruled out for the permutation condition because phi(p) = p - 1, and consecutive integers almost never share the same digit multiset; checking digit sums mod 9 is a fast pre-filter if needed.
  • The digit frequency array trick - count[digit]++ for one number, count[digit]-- for the other, then check all zero - is faster and allocation-free compared to sorting digit strings, and is worth reaching for in any C digit-permutation check.
  • An on-demand prime generator backed by a dictionary of composites is a clean Python pattern when only a bounded prefix of primes is needed and avoiding large memory allocation matters.

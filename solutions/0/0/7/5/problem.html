Problem 75: Singular Integer Right Triangles [Level 4]

Problem Statement

It turns out that $\pu{12 cm}$ is the smallest length of wire that can be bent to form an integer sided right angle triangle in exactly one way, but there are many more examples.

  • $\pu{\mathbf{12} \mathbf{cm}}$: $(3,4,5)$
  • $\pu{\mathbf{24} \mathbf{cm}}$: $(6,8,10)$
  • $\pu{\mathbf{30} \mathbf{cm}}$: $(5,12,13)$
  • $\pu{\mathbf{36} \mathbf{cm}}$: $(9,12,15)$
  • $\pu{\mathbf{40} \mathbf{cm}}$: $(8,15,17)$
  • $\pu{\mathbf{48} \mathbf{cm}}$: $(12,16,20)$

In contrast, some lengths of wire, like $\pu{20 cm}$, cannot be bent to form an integer sided right angle triangle, and other lengths allow more than one solution to be found; for example, using $\pu{120 cm}$ it is possible to form exactly three different integer sided right angle triangles.

  • $\pu{\mathbf{120} \mathbf{cm}}$: $(30,40,50)$, $(20,48,52)$, $(24,45,51)$

Given that $L$ is the length of the wire, for how many values of $L \le 1\,500\,000$ can exactly one integer sided right angle triangle be formed?

Mathematical Complexity

Every primitive Pythagorean triple can be generated exactly once using Euclid's parametric formula: for coprime integers m > n > 0 with m - n odd, the three sides are m² - n², 2mn, and m² + n². The perimeter of such a primitive triple is the sum of those three terms, which simplifies neatly to 2m(m + n) - no need to compute the individual sides at all. Every non-primitive triple is an integer multiple k of a primitive one, so its perimeter is k · 2m(m + n). The problem therefore reduces to populating a frequency table over perimeter values and counting how many entries equal exactly one.

The outer loop over m runs up to roughly sqrt(L / 2), because the smallest perimeter achievable for a given m occurs at n = 1, giving 2m(m + 1). For each m the inner loop over valid n is O(m), and the total number of coprime pairs (m, n) satisfying the parity constraint is O(L log L) in aggregate - the same asymptotic order as a Sieve of Eratosthenes. Space is O(L) for the perimeter frequency array. Overall complexity is therefore O(L log L) time and O(L) space.

Solution Approach

Euclid Parametric Enumeration with Multiplicity Counting (Python and C)

Both implementations follow an identical algorithm. I iterate m from 2 up to floor(sqrt(max_perimeter / 2)). For each m, I start n at m % 2 + 1 so that m - n is guaranteed odd from the outset, then step n by 2 to maintain that parity automatically. Any pair where gcd(m, n) != 1 is skipped. The primitive perimeter p = 2 * m * (m + n) is then extended by integer multiples k = 1, 2, 3, ... in a while-loop for as long as k * p stays within the limit. Every generated perimeter increments the corresponding counter. After the full enumeration I count how many perimeter values have a counter of exactly one.

In Python I wrapped the generation logic in a gen_pythagorean_triangle_perimeters generator that yields each perimeter (primitive and scaled). Counts accumulate in a plain dictionary using math.gcd for the coprimality filter. In C I use a heap-allocated int array indexed directly by perimeter value, which avoids all hash-table overhead and gives contiguous, cache-friendly memory access. Neither version ever computes the actual side lengths - only the perimeter is needed at any point.

Learnings

The first key insight was recognising that the perimeter of any Pythagorean triple generated by Euclid's formula collapses to the elegant form 2m(m + n), making the individual sides completely irrelevant. The second insight was that every non-primitive triple is a scaled copy of a primitive one, so I could extend each primitive perimeter with a compact inner while-loop over k rather than needing a separate enumeration pass. Together these two observations make the core loop extremely tight and easy to reason about.

A subtle but important detail is the correct upper bound for m. The minimum perimeter for a given m occurs at n = 1, giving 2m(m + 1). Setting this expression less than or equal to max_perimeter and solving gives m <= sqrt(max_perimeter / 2), so I use int((max_perimeter / 2) ** 0.5) as the exclusive ceiling. A slightly wrong bound here would silently miss primitive triples near the limit and produce a wrong answer with no obvious error signal.

The C version runs roughly 43x faster than Python on the main case (0.0102 s versus 0.4349 s) and about 29x faster on the large extra case. The speedup comes from three sources: replacing a hash-map dictionary with a flat integer array, eliminating Python interpreter overhead on the tight inner loops, and the better cache behaviour of contiguous memory. The O(L log L) scaling is visible in both languages - going from 1.5 M to 10 M (a 6.67x increase in L) yields roughly a 10x increase in Python runtime and about 15x in C, both consistent with the expected logarithmic factor on top of linear growth.

For future problems involving primitive pair enumeration or lattice-point counting, the combination of a parity step (n += 2) and a gcd filter is a reusable pattern that is far cheaper than full factorisation while still guaranteeing exact coverage of all primitive triples. And when the key space is dense and bounded, replacing a dictionary with a plain integer array is a reliable way to approach C-level memory efficiency even before leaving Python.

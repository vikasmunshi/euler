Problem 73: Counting Fractions in a Range [Level 2]

Problem Statement

Consider the fraction, $\dfrac n d$, where $n$ and $d$ are positive integers. If $n \lt d$ and $\operatorname{HCF}(n, d)=1$, it is called a reduced proper fraction.

If we list the set of reduced proper fractions for $d \le 8$ in ascending order of size, we get: $$\frac 1 8, \frac 1 7, \frac 1 6, \frac 1 5, \frac 1 4, \frac 2 7, \frac 1 3, \mathbf{\frac 3 8, \frac 2 5, \frac 3 7}, \frac 1 2, \frac 4 7, \frac 3 5, \frac 5 8, \frac 2 3, \frac 5 7, \frac 3 4, \frac 4 5, \frac 5 6, \frac 6 7, \frac 7 8$$

It can be seen that there are $3$ fractions between $\dfrac 1 3$ and $\dfrac 1 2$.

How many fractions lie between $\dfrac 1 3$ and $\dfrac 1 2$ in the sorted set of reduced proper fractions for $d \le 12\,000$?

Mathematical Complexity

This problem lives at the intersection of two classical number-theoretic structures: the Farey sequence and the Stern-Brocot tree. The Farey sequence F n is the sorted enumeration of all reduced proper fractions with denominator at most n, and consecutive terms a/b and c/d satisfy the determinant property |bc - ad| = 1. The Stern-Brocot tree organises the same fractions as a binary search tree where every node is the mediant (a+c)/(b+d) of its two nearest ancestors, making it natural for recursive interval-splitting between 1/3 and 1/2. Both structures support traversal algorithms that visit only the target fractions, but their cost is proportional to the answer itself, which grows as O(max_d 2 ) - around 500 million steps for max_d = 100 000.

A fundamentally faster approach replaces traversal with a rank function that counts how many reduced fractions lie at or below a given value. This uses Mobius-style inclusion-exclusion - the same sieve underlying Euler's totient function - and runs in O(max_d log max_d) time. The answer then becomes rank(1/2) - rank(1/3) - 1, turning the problem from a quadratic traversal into two near-linear sieve passes.

Solution Approaches

Iterative Farey Neighbour Traversal (s0)

I start from the denominator of the fraction just to the left of 1/2 in F max_d and walk leftward using the Farey recurrence new_d = max_d - (max_d + prev_d) % d, incrementing a counter at each step until I return to the denominator 2, which signals arrival back at 1/2. The crucial observation is that fractions in this interval are uniquely determined by their denominator alone - numerators are implicit - so I only need to track two consecutive denominators at any time. The starting denominator is computed as d = 2 + 3 * ((max_d - 2) // 3), which aligns to the largest valid denominator adjacent to 1/3 in the Farey sense. The Python and C implementations are structurally identical; the C version runs roughly 70x faster on the extra test case (1.72 s vs 34.31 s), but both scale as O(answer) = O(max_d 2 ) and are unsuitable for very large inputs.

Recursive Stern-Brocot Mediant Search (s1)

I exploit the Stern-Brocot property that every fraction strictly between a/b and c/d is reachable as the mediant (a+c)/(b+d). My recursive function takes a lower and upper denominator, computes their sum as the mediant denominator, and if the sum does not exceed max_d it counts that fraction and recurses into both sub-intervals. Seeding the search with recursion(3, 2) correctly covers the open interval (1/3, 1/2). This is a clean DFS over the relevant Stern-Brocot subtree, but it still visits every fraction in the answer set, so complexity remains O(answer). In Python it is slower than the Farey approach (0.91 s vs 0.53 s for max_d = 12 000) due to call-stack overhead; I also set sys.setrecursionlimit(10**6) to avoid hitting the default limit. The C version avoids stack depth concerns and is noticeably faster than s0_c at the same scale (1.01 s vs 1.72 s for max_d = 100 000), likely because the balanced divide-and-conquer structure fits CPU cache access patterns slightly better than purely sequential traversal. The Python s1 benchmark was not run for the extra test case, presumably due to recursion depth concerns.

Rank Function with Mobius Sieve (s2)

I define a rank(n, d) function that counts all reduced fractions a/b with b <= max_d and a/b <= n/d, then return rank(1, 2) - rank(1, 3) - 1. Inside rank, I initialise an array data[i] = floor(i * n / d), which counts all fractions (reduced or not) with denominator i that are at most n/d. I then apply a Mobius-style sieve: for each i from 1 upward, I subtract data[i] from every later multiple data[j] where j = 2i, 3i, and so on. After this pass, data[i] holds exactly the count of reduced fractions with denominator i at most n/d by Mobius inversion, and summing the whole array gives the rank. The sieve runs in O(max_d log max_d) time due to the harmonic series, and the constant factor is tiny because it is pure integer arithmetic on a flat array. The Python version handles the extra case in 0.175 s, roughly 196x faster than s0.py at 34.31 s. The C version is even more striking: 0.0003 s for the main case and 0.0032 s for the extra case - making it the clear winner by a wide margin.

Learnings

The key insight for the traversal approaches was that fractions in the Stern-Brocot tree between 1/3 and 1/2 are uniquely identified by their denominator alone - carrying numerators is unnecessary. This made both s0 and s1 surprisingly compact, but it did not change their fundamental O(answer) complexity.

The deeper insight for s2 was reframing counting as a rank computation. Once I recognised "count reduced fractions below a threshold" as a standard Mobius inversion - the same structure underlying the Euler totient sieve - the rest followed mechanically: initialise with floor values, sieve out multiples, sum. Two rank evaluations and a subtraction replace the entire traversal.

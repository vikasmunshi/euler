Problem 72: Counting Fractions [Level 3]

Problem Statement

Consider the fraction, $\dfrac n d$, where $n$ and $d$ are positive integers. If $n \lt d$ and $\operatorname{HCF}(n,d)=1$, it is called a reduced proper fraction.

If we list the set of reduced proper fractions for $d \le 8$ in ascending order of size, we get: $$\frac 1 8, \frac 1 7, \frac 1 6, \frac 1 5, \frac 1 4, \frac 2 7, \frac 1 3, \frac 3 8, \frac 2 5, \frac 3 7, \frac 1 2, \frac 4 7, \frac 3 5, \frac 5 8, \frac 2 3, \frac 5 7, \frac 3 4, \frac 4 5, \frac 5 6, \frac 6 7, \frac 7 8$$

It can be seen that there are $21$ elements in this set.

How many elements would be contained in the set of reduced proper fractions for $d \le 1\,000\,000$?

Mathematical Complexity

The central insight is that counting reduced proper fractions n/d with n < d and HCF(n, d) = 1 for a fixed denominator d is exactly what Euler's totient function φ(d) measures: it counts the integers from 1 to d that are coprime to d. Summing φ(d) for d from 2 to N therefore gives the total number of elements in the Farey-like set for all denominators up to N. This reframes the combinatorics problem about reduced fractions into a number-theory problem about efficiently computing and summing totient values.

Computing each totient individually via GCD checks would cost O(N²). A sieve-based approach propagates the multiplicative identity φ(n) = n · ∏(1 - 1/p) over all prime factors p of n using nested loops whose total work follows a harmonic series, yielding O(N log log N) overall - the same asymptotic class as the Sieve of Eratosthenes. For N = 1,000,000 this is comfortably fast even in Python, and for the extended input of N = 10,000,000 it finishes in roughly 7 seconds in Python and under 0.4 seconds in C.

Solution Approach

Euler Totient Sieve (s0 - Python and C)

Both p0072_s0.py and p0072_s0.c implement the same multiplicative totient sieve. I initialise an array euler_totients[i] = i for every index from 0 to max_d. I then iterate n from 2 to max_d. The sentinel condition euler_totients[n] == n detects that n has not yet been modified by any smaller prime factor, which means n is itself prime. For each such prime n I walk through all multiples j in steps of n and apply the update euler_totients[j] = euler_totients[j] / n * (n - 1), folding in the contribution of that prime factor via the multiplicative property of φ. After processing n, I add euler_totients[n] to a running total. The entire computation is a single forward pass with no separate prime list needed.

The Python version uses a plain list[int] with floor division ( //), while the C version uses a heap-allocated long long array with equivalent integer division. The logic is otherwise identical. The C version is roughly 50x faster at the smaller benchmark size and about 20x faster at the larger one: 0.0125 s versus 0.6167 s at N = 1,000,000, and 0.3608 s versus 7.3294 s at N = 10,000,000.

Learnings

The key "aha moment" was recognising that counting elements in the Farey sequence is equivalent to summing Euler totients. Once that equivalence is established, the problem reduces to a well-known algorithmic exercise: adapt a prime sieve to propagate a multiplicative arithmetic function instead of just marking composites. The outer loop structure is identical to the Sieve of Eratosthenes.

A subtle but important correctness detail is the order of operations in the sieve update. I write euler_totients[j] // n * (n - 1) rather than euler_totients[j] * (n - 1) // n. Because the sieve processes prime factors in order, p already divides euler_totients[j] at the point the update is applied - all earlier prime factors of j have already been folded in. The floor division is therefore exact and introduces no rounding error. Reversing the order would multiply first by a potentially non-integer factor and could silently produce a wrong result.

Since I only implemented one algorithm (the multiplicative totient sieve), there is no algorithmic comparison to make. The Python and C ports are direct translations of each other, and the benchmark confirms expected scaling. The roughly 20x to 50x speedup of the C version is typical for array-intensive sieve work where Python's interpreter overhead and list indexing costs accumulate over millions of iterations.

  • Summing φ(d) for d = 2 to N counts all reduced proper fractions with denominator at most N - Farey sequence cardinality equals a totient sum.
  • A multiplicative totient sieve runs in O(N log log N) and requires only a single integer array of size N+1.
  • The sentinel euler_totients[n] == n is a lightweight primality check that avoids storing a separate prime list.
  • Division order in the sieve update ( val // p * (p - 1)) must be respected to guarantee exact integer arithmetic at each step.
  • The same sieve skeleton used for marking primes can be adapted to propagate any multiplicative arithmetic function, making it a reusable pattern for totient, Mobius function, and related sieves.

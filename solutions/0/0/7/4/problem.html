Problem 74: Digit Factorial Chains [Level 2]

Problem Statement

The number $145$ is well known for the property that the sum of the factorial of its digits is equal to $145$: $$1! + 4! + 5! = 1 + 24 + 120 = 145.$$

Perhaps less well known is $169$, in that it produces the longest chain of numbers that link back to $169$; it turns out that there are only three such loops that exist:

$$\begin{align} &169 \to 363601 \to 1454 \to 169\\ &871 \to 45361 \to 871\\ &872 \to 45362 \to 872 \end{align}$$

It is not difficult to prove that EVERY starting number will eventually get stuck in a loop. For example,

$$\begin{align} &69 \to 363600 \to 1454 \to 169 \to 363601 (\to 1454)\\ &78 \to 45360 \to 871 \to 45361 (\to 871)\\ &540 \to 145 (\to 145) \end{align}$$

Starting with $69$ produces a chain of five non-repeating terms, but the longest non-repeating chain with a starting number below one million is sixty terms.

How many chains, with a starting number below one million, contain exactly sixty non-repeating terms?

Mathematical Complexity

Every positive integer has exactly one successor under the digit-factorial map: replace the number with the sum of the factorials of its digits. This defines a functional graph - a directed graph where every node has out-degree exactly one. Because any number with at most seven digits maps to a value no larger than 7 * 9! = 2,540,160, the reachable state space is finite and bounded regardless of the starting point, which guarantees that every walk eventually enters a cycle. The problem states there are exactly three such cycle attractors, seeded by 169, 871, and 872. The question reduces to counting how many starting numbers below one million produce a non-repeating chain of exactly sixty terms before revisiting a value.

The key structural insight is that memoization on this bounded functional graph converts a naive O(max_num * chain_length) traversal into an amortized O(max_num) algorithm: once a node's chain length is computed and cached, any subsequent walk that reaches it terminates immediately. The main correctness subtlety is distinguishing two different termination events during a forward walk - hitting a previously cached node (where the remaining length is read directly from cache) versus revisiting a node encountered in the current walk (a cycle event, where the length is inferred from the walk prefix alone). Conflating these two cases produces off-by-one errors in every cycle member's length.

Solution Approaches

s0 - Integer cache with graph dict

I chose to store only a scalar chain length per number in a cache dictionary, together with a separate graph mapping each number to its precomputed successor. For each starting number I walk forward, collecting visited nodes in a seen list, until I either find the current node already in the length cache or detect a cycle by finding it in seen. In the cache-hit case the total length is len(seen) + cached_length; in the cycle case it is len(seen), since the entire cycle is contained within the current walk. I then back-propagate through seen, assigning each node the correct length as total - i. I also track the running maximum inline to avoid a second pass. The C translation allocates flat arrays of size 3,000,000 to cover all reachable values, and uses a short linear scan over seen for cycle detection - acceptable because chains are short (at most around 60 elements). Cycle detection with a linear scan over a tiny array is faster in practice than maintaining a hash set for such small walks.

s1 - Full chain memoization with list splicing

My second approach stores for every number the full suffix chain starting at that number as a Python list. The add_chains function builds a current_chain list and a visited set as it walks forward. It stops either when a cached number is reached (appending that number's stored list via extend) or when a cycle is detected. Each prefix position in the walk is then assigned a slice of current_chain as its cached value. The final count is obtained by computing all stored list lengths and calling list.count on the maximum. The C translation reproduces the same logic with dynamic arrays and a boolean visited-flag array, freeing the per-walk visited list after each iteration. Storing full chain lists rather than scalars increases both memory allocation work and garbage-collection pressure, which is directly visible in the benchmarks.

s2 - List memoization with optional visualization

This variant shares the identical chain-building logic from s1. I added it primarily for interactive exploration: the Python print_chains function renders the chain graph using ANSI colour codes, highlighting cycle entry points in blue and starting numbers in green, and reports aggregate statistics such as the ratio of total chain entries to unique numbers. A --show flag and a guard restricting full rendering to max_num <= 1000 keep it usable at small scales without polluting benchmark runs. The C translation omits the visual output but is otherwise identical to s1_c. The benchmarks confirm that the visualization infrastructure adds no measurable overhead when the flag is absent.

Learnings

Key insights

  • Precomputing the ten digit factorials (0! through 9!) into a tuple and extracting digits with modulo and integer division is substantially faster than calling math.factorial inside the hot loop.
  • Cycle detection and cache lookup are two distinct termination conditions requiring different length formulas. A cache hit gives len(seen) + cached_length; a revisit gives len(seen). Mixing them up corrupts the chain lengths of every cycle member.
  • Because the digit-factorial map is strongly contractive for large numbers (any number above roughly 2.2 million maps to a smaller value), the effective graph explored is much smaller than max_num, which is why memoization pays off so heavily even in Python.
  • On a functional graph (out-degree 1 everywhere), iterative memoization with a seen list and a scalar cache is the standard O(n) pattern. Recursive DFS should be avoided to prevent stack overflows on chains longer than Python's default recursion limit.

s0 is the clear winner in every benchmark category. In Python the performance gap grows from roughly 2.9x at one million to about 4x at ten million, because list allocation and slicing scale poorly compared with simple integer assignment. In C the gap is smaller in absolute terms but still visible - s0_c is about 1.6x faster than s1_c and s2_c at ten million. The lesson is that when only aggregate statistics (counts, maximums) are needed rather than the full path structure, storing derived scalars rather than raw sequences is almost always the right trade-off: it reduces both memory usage and garbage-collection pressure. s1 and s2 are essentially equivalent in speed at scale, confirming that the visualization code in s2 adds no overhead when the flag is absent. The C implementations deliver roughly a 60x to 100x speedup over their Python counterparts, reaching the ten-million case in under 150 ms for s0_c.

Problem Analysis

This problem asks for the smallest n whose value under the partition function p(n) - the number of ways to write n as an unordered sum of positive integers - is divisible by one million. The phrase "separated into piles" is exactly the integer-partition notion: order does not matter, only the multiset of pile sizes. The crucial fact is how fast p(n) grows. By the Hardy-Ramanujan asymptotic, p(n) grows roughly like exp(pi*sqrt(2n/3)), so any approach that explicitly enumerates partitions is hopeless even for small n. The entire problem hinges on having a fast recurrence for p(n).

That recurrence is Euler's pentagonal number theorem: p(n) = sum over k != 0 of (-1)^(k+1) * p(n - g(k)), where g(k) = k(3k-1)/2 are the generalized pentagonal numbers 1, 2, 5, 7, 12, 15, 22, 26, ... generated by letting k = 1, -1, 2, -2, 3, -3, .... The signs come in pairs: +, +, -, -, +, +, .... Because g(k) grows like 3k^2/2, only about O(sqrt(n)) pentagonal numbers do not exceed n, so each p(n) costs O(sqrt(n)) given all smaller values; building the table up to the answer N is therefore O(N*sqrt(N)). Two further observations clean up the implementation. First, since we only test divisibility we can work entirely in modular arithmetic, reducing every value mod one million so nothing overflows or grows large. Second, the recurrence reads back many smaller values, making it a textbook case for dynamic programming in a flat array indexed by n.

Solution Approach

Euler Pentagonal Recurrence with Modular Reduction (Python and C)

The Python and C files implement the same algorithm; the C version exists purely for speed. Both are structured as a bottom-up dynamic programming build-up.

Base case. I seed partitions[0] = 1. There is exactly one way to partition zero - the empty sum - and this single seed is what makes every later term of the recurrence consistent. Without it the whole chain collapses to zeros.

Outer loop. I iterate n = 1, 2, 3, ... with no fixed upper bound, because I do not know the answer in advance. In Python this is itertools.count(1); in C it is a while(1) with an internal return.

Inner loop - applying the recurrence. For each n I accumulate pval by iterating k = 1, 2, 3, .... Each k contributes two generalized pentagonal numbers at once: the positive-index value g(k) = k(3k-1)/2 and the negative-index value g(-k) = k(3k+1)/2. Computing both in one iteration lets a single sign variable (-1)^(k-1) cover the pair. The loop terminates as soon as g(k) > n - that is the smaller of the two pentagonal numbers, so the test is conservative and never references a negative index. The second, larger term is guarded separately (g(-k) <= n) because it can exceed n while g(k) does not.

Reduce and test. After the inner loop I reduce pval with modular arithmetic and store it, then check for zero. The first n giving zero is the answer. A language difference surfaces here: Python's % always returns a non-negative result, so partition_value %= divisor suffices. In C the alternating subtractions can leave pval negative, so I use the idiom ((pval % divisor) + divisor) % divisor to land cleanly in [0, divisor).

Storage. The Python list simply appends each new value. The C version pre-allocates 100,000 entries and doubles capacity with realloc if it ever runs out. For the actual answer of 55,374 the resize never triggers, but the pattern keeps the code correct for larger divisors.

One stylistic note on the Python: it decorates the pentagonal helper with functools.lru_cache. Since pentagonal is a pure formula evaluated once per (n, k) pair, the cache provides no real benefit here - the algorithmic substance is identical to the C version.

Programming Techniques

Modular Arithmetic for Divisibility Testing

Because the question is purely about divisibility, I never store full partition values (which become enormous - p(200) already has 13 digits). Reducing with modular arithmetic at every step caps all stored numbers below the divisor, preventing overflow in C and keeping Python's big-integer arithmetic from ballooning. This is a general principle: when a recurrence is additive and you only need a divisibility or remainder condition, fold the modulo into every step. It does not change the result and bounds all intermediate magnitudes.

Bottom-Up Dynamic Programming with a Flat Array

The recurrence reads back O(sqrt(n)) earlier values, so all smaller terms must be available. A flat array indexed by n gives O(1) lookup and excellent cache locality in C. I prefer this dynamic programming form over top-down memoization here because it avoids function-call and stack overhead and makes the dependency order obvious - every value is computed before anything that needs it.

Paired Generalized Pentagonal Numbers

Instead of generating the interleaved sequence 1, 2, 5, 7, 12, ... and indexing it, I compute both members of each (k, -k) pair together in one loop iteration. This halves the iteration count and makes the sign bookkeeping trivial: a single (-1)^(k-1) applies to both members. The loop's single termination test on the smaller generalized pentagonal number keeps the logic compact and correct.

Amortized Dynamic Array Resizing in C

The C array starts at 100,000 and doubles on demand via dynamic array growth. Doubling keeps the total number of realloc calls at O(log N) and wasted memory under a factor of two - the same growth strategy Python lists use internally. It is the right default in C whenever the final output size is unknown ahead of time.

Unbounded Iteration Idioms

Both solutions express a "loop until a condition is met" pattern. Python uses itertools.count for a clean infinite ascending sequence with no arbitrary hard-coded ceiling; C uses while(1) with an internal return. Reaching for an explicit infinite loop, rather than guessing a large upper bound, makes the intent clear and removes a class of off-by-one and too-small-limit bugs.

Key Takeaways

The single insight that unlocks this problem is recognizing that p(n) is a well-studied object with a fast recurrence - Euler's pentagonal number theorem. Once you have the recurrence, the modular arithmetic trick turns a big-integer problem into a bounded one, and the rest is ordinary bottom-up dynamic programming with early termination.

Both files run the identical algorithm, so the benchmark gap is pure language overhead. On the main case (divisor = 1,000,000) the C solution finishes in about 0.011 s while the Python solution takes about 2.02 s - roughly a 180x difference. For a tight inner loop running tens of thousands of times with heavy array indexing and arithmetic, C's static typing and direct memory access dominate. Python remains perfectly adequate at around two seconds, but the port is worth it when benchmarks are tight.

  • When a problem reduces to "find the first n where f(n) satisfies a condition" and f(n) has a recurrence, bottom-up dynamic programming with early termination is almost always the right tool.
  • If you only care about divisibility or a remainder, apply the modulo at every step of an additive recurrence - never let intermediate values grow unbounded.
  • Recognize famous sequences: the partition function, generalized pentagonal numbers, and their identities are standard combinatorial tools worth keeping in your mental library.
  • Quantities that come in natural symmetric pairs (here k and -k) can often be processed together, halving loop work and simplifying sign handling.
  • In C, watch the sign of a remainder after subtractions: use ((x % m) + m) % m to normalize into [0, m).
  • Use an explicit infinite loop (itertools.count or while(1)) when the answer's location is unknown, rather than guessing a fixed bound.
  • A doubling dynamic array is the correct default in C when the output size is not known in advance, giving amortized O(1) appends.

Problem 71: Ordered Fractions [Level 2]

Problem Statement

Consider the fraction, $\dfrac n d$, where $n$ and $d$ are positive integers. If $n \lt d$ and $\operatorname{HCF}(n,d)=1$, it is called a reduced proper fraction.

If we list the set of reduced proper fractions for $d \le 8$ in ascending order of size, we get: $$\frac 1 8, \frac 1 7, \frac 1 6, \frac 1 5, \frac 1 4, \frac 2 7, \frac 1 3, \frac 3 8, \mathbf{\frac 2 5}, \frac 3 7, \frac 1 2, \frac 4 7, \frac 3 5, \frac 5 8, \frac 2 3, \frac 5 7, \frac 3 4, \frac 4 5, \frac 5 6, \frac 6 7, \frac 7 8$$

It can be seen that $\dfrac 2 5$ is the fraction immediately to the left of $\dfrac 3 7$.

By listing the set of reduced proper fractions for $d \le 1\,000\,000$ in ascending order of size, find the numerator of the fraction immediately to the left of $\dfrac 3 7$.

Mathematical Complexity

The problem is a direct application of Farey sequence theory. The Farey sequence F_n is the ordered list of all reduced proper fractions with denominator at most n, and a classical result states that any two consecutive fractions a/b and c/d in F_n satisfy the mediant identity bc - ad = 1. I fixed the right-hand neighbor as c/d = 3/7 and needed to find the largest valid b such that the left neighbor a/b satisfies 3b - 7a = 1. Rearranging gives 3b ≡ 1 (mod 7), so b must fall in a specific congruence class modulo 7. The largest b not exceeding max_d in that class is exactly 7 * floor(max_d / 7), which gives the numerator a = 3k - 1 where k = floor(max_d / 7).

This observation collapses the entire search over up to a million (or a trillion) denominators into a single integer division, making the algorithm O(1) in both time and space, completely independent of max_d. The answer grows proportionally to (3/7) * max_d, so for max_d = 10^12 the numerator exceeds 4 * 10^11, which means 64-bit integer arithmetic is required in any low-level implementation. The closed form holds exactly for all inputs because the Farey-neighbor identity is a pure number-theoretic result with no approximation involved.

Solution Approach

Direct Closed-Form Farey Neighbor (s0 - Python and C)

Both implementations encode the same single arithmetic formula derived from the Farey-neighbor identity. I compute k = floor(max_d / 7) and then form the fraction (3k - 1) / (7k), which is the immediate left neighbor of 3/7 in the Farey sequence F_max_d. In the Python version I expressed this as fractions.Fraction(3, 7) - fractions.Fraction(1, 7 * (max_d // 7)), letting Python's Fraction class perform the rational subtraction and automatic GCD reduction before reading off result.numerator. This formulation makes the algebraic intent transparent - I am literally stepping one unit to the left of 3/7 in the Farey sequence - and it eliminates any risk of a manual GCD error. The optional --show path prints the difference from 3/7 alongside the result, which helped me verify the derivation during development.

The C implementation mirrors the same algebra directly: it computes num = 3 * k - 1 and den = 7 * k, then divides both by their GCD via a small Euclidean helper before returning num. All variables are declared as long long to handle inputs up to 10^12 safely. There is no algorithmic difference between the two languages; the C version is a straightforward translation of the closed form with explicit GCD bookkeeping replacing Python's built-in rational arithmetic class.

Learnings

The key insight was recognising that "fraction immediately to the left of 3/7 in a bounded Farey sequence" is precisely the Farey-neighbor problem. Writing down the identity bc - ad = 1 with c/d = 3/7 fixed immediately converts the problem into a linear congruence: find the largest b not exceeding max_d with 3b ≡ 1 (mod 7). From there, rounding down to the nearest multiple of 7 rather than tracking the exact congruence class gives the same reduced numerator. The dev test case (max_d = 10, answer = 2, matching the example in the problem statement) confirmed the formula before running the main input.

Both Python and C solutions run in effectively zero measurable time across all test cases including max_d = 10^12, confirming the O(1) nature of the solution. Even interpreter startup dominates the Python elapsed time, so the C version has no practical speed advantage for this particular problem.

  • The Farey-neighbor identity bc - ad = 1 is the right tool whenever a problem asks for the fraction immediately adjacent to a known target within a bounded denominator range.
  • Fixing one neighbor and solving the resulting linear congruence converts an apparent search over millions or trillions of denominators into a single division.
  • Python's fractions.Fraction class is ideal for prototyping exact rational formulas - it eliminates GCD bookkeeping and makes the algebraic intent readable at a glance.
  • Always check whether intermediate products can overflow 32-bit integers when a formula scales with the input; here 3 * k with k up to roughly 1.4 * 10^11 requires long long in C.
  • Verifying a closed-form solution against a small hand-computable case before trusting it on the main input is a habit worth keeping - the dev test with max_d = 10 doubles as a proof of correctness for the formula.

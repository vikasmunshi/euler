Problem 10: Summation of Primes [Level 0]

Problem Statement

Mathematical Complexity

The problem reduces to enumerating all primes below a fixed upper bound and accumulating their sum. The key mathematical concept is the distribution of primes and the trade-off between per-candidate primality testing and batch sieve-based enumeration. Testing each candidate individually with trial division costs O(sqrt(n)) per candidate, giving O(n sqrt(n)) across all integers below the bound - acceptable for tiny bounds but painfully slow at two million and beyond. Sieve methods amortise the marking work: the Sieve of Eratosthenes strides through multiples of each prime and runs in O(n log log n) time with O(n) space. The Sieve of Sundaram marks indices of the form i + j + 2ij and recovers odd primes as 2k + 1 for each unmarked index k; it sits in the same asymptotic class. The incremental dictionary generator produces primes lazily with no fixed bound but pays per-candidate hash-map overhead, making it less competitive here.

Because the problem specifies a fixed upper bound of two million, any sieve approach dominates over iterative testing. Summing the primes is then a linear O(n) scan over the sieve output, which does not affect the dominant complexity term. The practical boundary between sieve methods is determined not by asymptotics but by memory access patterns and the overhead imposed by the host language's runtime.

Solution Approaches

s0 - Sieve of Eratosthenes

I implemented the classic Sieve of Eratosthenes in both Python and C. In Python I allocated a bytearray of length max_num + 1 initialised to ones, then used slice assignment sieve[i*i::i] = bytearray(...) to zero out all multiples of each prime in a single C-level operation, avoiding a Python-level loop over composites. The outer loop runs up to int(max_num ** 0.5) + 1. In C I used calloc followed by memset to fill the array with ones, then a nested loop marking multiples starting from each prime's square, with the outer limit computed via sqrt. Both implementations scan the final sieve once to accumulate the sum. The C version completes the two-million case in about 0.007 s; the Python version takes around 0.079 s - roughly eleven times slower but still very comfortable for the problem size.

s1 - Sieve of Sundaram

I implemented the Sieve of Sundaram as an academic exercise in both Python and C. The algorithm works on a reduced domain of size n = (max_num - 1) / 2: for each pair (i, j) with i >= 1 and j >= i it marks the index i + j + 2ij. Every unmarked index k then corresponds to the odd prime 2k + 1, and 2 is added separately. In Python I used a bytearray for the marked array and a nested loop to perform the marking. In C I used calloc and a comparable nested loop. The Python version is roughly six times slower than the Python Eratosthenes sieve (0.52 s versus 0.079 s) because the double-nested Python-level loop dominates. In C the picture reverses: the Sundaram sieve (0.005 s) edges out the Eratosthenes sieve (0.007 s), likely because the tighter inner loop has slightly better cache locality at these problem sizes when compiled with optimisation.

s2 - Incremental Dictionary-Based Generator

I implemented the "sieve in a dictionary" prime generator, commonly attributed to David Eppstein, in both Python and C. A dictionary maps each composite encountered so far to the prime that generated it. When a candidate is absent from the dictionary it is prime; its square is registered as the next composite to watch. When a candidate is present the corresponding entry is deleted and the prime is advanced to the next unmarked odd multiple. In Python I used a plain dict and yielded primes lazily, accumulating the sum until the current value exceeded the bound. In C I translated this using a custom open-addressing hash table with linear probing and cluster rehashing on deletion. The per-candidate hash-map overhead makes this approach significantly slower than the sieves: 0.37 s versus 0.079 s in Python, and 0.12 s versus 0.007 s in C. Its principal advantage is that it requires no fixed upper bound and streams primes lazily, which matters more for problems where the stopping condition is not a simple numeric ceiling.

s3 - Trial Division

I included a straightforward trial-division primality test in both Python and C. The is_prime predicate checks divisibility by 2 and then by odd integers up to the square root of the candidate; the solve function sums all primes in range(2, max_num). Both Python and C follow the same logic. This is the most readable implementation but by far the slowest: 4.98 s in Python and 0.17 s in C on the two-million case, rising to 50.7 s and 1.6 s respectively at ten million. The O(n sqrt(n)) cost is very tangible at scale. I include it only as a baseline and a sanity check for verifying the answer.

s4 - pyprimesieve Library (Python) / Eratosthenes Sieve Variant (C)

The Python solution wraps pyprimesieve.primes_sum, a third-party C++ segmented sieve. It completes in under a millisecond for the two-million case and around 3 ms at ten million, serving mainly as a performance ceiling and an answer verification tool rather than a handwritten solution. The corresponding C file is another Eratosthenes sieve, functionally equivalent to s0_c but with the loop termination condition written as (long long)i * i < max_num instead of using sqrt. This avoids the floating-point rounding concern entirely. The two C Eratosthenes implementations perform identically in the benchmarks (0.006 s versus 0.007 s for the main case), confirming that the termination condition difference has no measurable effect here.

Learnings

The key insight is that whenever a fixed upper bound is known, pre-computing a sieve reduces an O(n sqrt(n)) problem to O(n log log n). Once I recognised that the summation is just a post-processing step, the problem collapsed entirely to choosing the right enumeration strategy. Using bytearray slice assignment to mark composites in bulk was the single most impactful micro-optimisation inside the Python sieve loop, replacing an interpreted Python iteration with a C-level memset.

One result I found genuinely surprising is that the C Sundaram sieve is marginally faster than the C Eratosthenes sieve, reversing the Python ranking completely. My hypothesis is that the tighter inner loop of the Sundaram variant has slightly better cache behaviour at these problem sizes once the compiler applies optimisation, whereas in Python the overhead of the double-nested interpreted loop dominates and makes Sundaram the slowest sieve. The incremental dictionary generator sits in the middle tier in both languages - its main strength (no fixed bound, lazy streaming) is simply not relevant to this problem. For any future problem with a known prime bound, my default will be the bytearray -backed Eratosthenes sieve in Python or either C sieve when raw speed matters. Trial division remains useful only for spot-checking individual numbers or for very small bounds where code clarity outweighs performance.

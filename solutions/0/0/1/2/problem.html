Problem 12: Highly Divisible Triangular Number [Level 0]

Problem Statement

The sequence of triangle numbers is generated by adding the natural numbers. So the $7$ th triangle number would be $1 + 2 + 3 + 4 + 5 + 6 + 7 = 28$. The first ten terms would be: $$1, 3, 6, 10, 15, 21, 28, 36, 45, 55, \dots$$

Let us list the factors of the first seven triangle numbers:

$$\begin{align} \mathbf 1 &\colon 1\\ \mathbf 3 &\colon 1,3\\ \mathbf 6 &\colon 1,2,3,6\\ \mathbf{10} &\colon 1,2,5,10\\ \mathbf{15} &\colon 1,3,5,15\\ \mathbf{21} &\colon 1,3,7,21\\ \mathbf{28} &\colon 1,2,4,7,14,28 \end{align}$$

We can see that $28$ is the first triangle number to have over five divisors.

What is the value of the first triangle number to have over five hundred divisors?

Mathematical Complexity

The core mathematical tool here is the divisor-counting function d(n), sometimes written sigma-0. For a number n with prime factorisation p1^e1 * p2^e2 * ... * pk^ek, the number of divisors is (e1+1)(e2+1)...(ek+1), because each divisor is formed by independently choosing an exponent from 0 to ei for each prime. The property that made my optimised solution possible is that d is multiplicative: whenever gcd(a, b) = 1, d(a*b) = d(a)*d(b).

The i-th triangle number is T(i) = i*(i+1)/2. Because i and i+1 are consecutive integers their GCD is always 1, so the factor of 2 can be absorbed cleanly into whichever of i or i+1 is even. If i is even, T(i) = (i/2)*(i+1) and the two factors are coprime; if i is odd, T(i) = i*((i+1)/2) and again the two factors are coprime. Trial division runs in O(sqrt(n)), so factorising the full triangle number T(i) directly costs O(sqrt(i^2/2)) = O(i/sqrt(2)), whereas factorising the two coprime halves each of size roughly i/2 costs about 2*O(sqrt(i/2)) = O(sqrt(2*i)). The split approach therefore becomes increasingly cheaper compared with the naive approach as i grows.

Solution Approaches

Split Coprime Factorisation (s0)

I exploited the coprimality of consecutive integers to avoid ever factorising the full triangle number. At each step I check whether i is even or odd, split T(i) = i*(i+1)/2 into its two coprime halves, call my num_factors helper on each half, and multiply the results. The loop exits as soon as the divisor count strictly exceeds the threshold, returning the current triangle number. Both the Python version (p0012_s0.py) and the C version (p0012_s0_c) implement exactly the same parity check and two num_factors calls; the C version benefits from compiled arithmetic to reach 0.0021 s on the main case, roughly 22x faster than the Python equivalent at 0.0458 s. The Python and C code are direct translations of each other, which I find is a useful property to maintain when comparing language overhead in isolation.

Naive Direct Factorisation (s1)

My simpler baseline computes each triangle number T(i) = i*(i+1)//2 directly and passes the full value to num_factors. The loop in both Python (p0012_s1.py) and C (p0012_s1_c) advances while the divisor count is strictly less than the threshold and returns as soon as it meets or exceeds it. This is easier to read at a glance, but pays the full O(sqrt(T(i))) trial-division cost at every candidate. The Python version takes 0.2127 s on the main case and the C version takes 0.0083 s - both meaningfully slower than their s0 counterparts.

Learnings

The key insight was recognising that the formula T(i) = i*(i+1)/2 already hands you a factorisation into two coprime parts for free. Once I recalled that the divisor function is multiplicative, the rest followed immediately: never factorise the big product when you can factorise each coprime piece separately and multiply the counts.

Within the same algorithm, C is roughly 22x faster than Python for the split variant and about 26x faster for the naive variant - consistent with typical compiled-versus-interpreted overhead. The split coprime approach is the clear winner on both correctness grounds and performance grounds.

Key techniques worth remembering for future problems:

  • The divisor-count function d(n) is multiplicative: d(a*b) = d(a)*d(b) whenever gcd(a, b) = 1.
  • Consecutive integers are always coprime, so any product of the form n*(n+1) or n*(n+1)/2 can be split into coprime factors before applying multiplicative arithmetic functions.
  • Trial division is O(sqrt(n)), so halving the argument saves a factor of sqrt(2) per call - modest on its own but meaningful when the loop runs thousands of iterations.
  • When a product formula provides coprime factors for free, always prefer factorising the pieces over factorising the assembled product.

Problem 15: Lattice Paths [Level 1]

Problem Statement

Starting in the top left corner of a $2 \times 2$ grid, and only being able to move to the right and down, there are exactly $6$ routes to the bottom right corner.

How many such routes are there through a $20 \times 20$ grid?

Mathematical Complexity

Any path from the top-left to the bottom-right of an n x n grid, using only rightward and downward moves, consists of exactly 2n steps - n steps right and n steps down, in some order. Counting distinct orderings of those steps is identical to choosing which n of the 2n steps are rightward moves, which is the binomial coefficient C(2n, n) = (2n)! / (n! * n!). For the main problem n = 20 this is the central binomial coefficient C(40, 20). Recognising the geometric traversal as a sequence-selection problem reduces the whole question to a single closed-form identity, with no search, no recurrence, and no dynamic programming table required.

Computing the result requires evaluating factorials up to 2n. Using the multiplicative form C(2n, n) = product_{i=1}^{n} (n + i) / i, each factor is applied one at a time in O(n) multiplications and divisions, making the algorithm O(n) in time. For n = 200 the answer is a 120-digit integer; arbitrary-precision arithmetic is needed in any language that does not provide it natively. The number of digits grows as O(n), so storage is O(n) as well.

Solution Approach

Closed-Form Central Binomial Coefficient (Python and C)

Both the Python and C solutions implement the same combinatorial identity and differ only in how they handle big-integer arithmetic. The Python solution is a single expression: math.factorial(2 * lattice_size) // math.factorial(lattice_size) ** 2. Because Python integers are natively arbitrary-precision, this is exact for any n with no additional library or overflow concern.

The C solution cannot rely on built-in big integers, so I implemented a small arbitrary-precision library using base-10 9 limbs stored in a little-endian array of unsigned long long. Rather than computing two enormous factorials separately and then dividing, I used the multiplicative form: starting from 1, for each i from 1 to n I multiply the running product by (n + i) and then immediately divide by i. Because C(2n, n) is always an integer and the factors are applied in increasing order, each intermediate result is also an integer, so every division is exact with no remainder discarded. I capped the limb array at 64 limbs, giving 576 decimal digits of capacity - well beyond any reasonable n. The print routine emits the most-significant limb without leading-zero padding and all remaining limbs with nine-digit zero-padding, producing the correct decimal string.

Learnings

The key insight was reframing the grid traversal as a sequence-selection problem. Once I asked "how many ways can I assign n of the 2n steps to be rightward moves?", the formula C(2n, n) followed immediately without drawing any recurrence or DP table. This is a useful reminder that counting problems with a geometric flavour are frequently combinatorial identities in disguise, and checking for a closed form before reaching for dynamic programming is always worth a moment's thought.

Because both solutions implement exactly the same algorithm, there is no meaningful performance difference to compare - all six benchmark runs report 0.0000 s elapsed time, meaning the computation itself is negligible and any measured time would be dominated by process startup. The Python version is clearly more elegant: it is a one-liner that relies on the language's built-in big integers. The C version required writing a small big-integer library from scratch - specifically mul-by-scalar and div-by-scalar operations on a limb array - which was the only real engineering effort involved.

  • Grid path counting from corner to corner always yields C(rows + cols, rows) when only right and down moves are allowed; this generalises to non-square grids by replacing n with the appropriate pair of dimensions.
  • The central binomial coefficient C(2n, n) appears frequently in combinatorics and is worth recognising on sight.
  • The multiplicative form product_{i=1}^{n} (n + i) / i keeps all intermediate values as integers, avoiding the need to compute and then divide two enormous factorials separately.
  • Before reaching for dynamic programming, always check whether a closed-form combinatorial identity exists - it often eliminates both asymptotic complexity and code volume.
  • When implementing big-integer arithmetic in C for a specific use case, a limb-based structure with mul-by-scalar and div-by-scalar operations is sufficient and straightforward to verify for correctness.

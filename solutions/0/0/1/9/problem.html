Problem 19: Counting Sundays [Level 2]

Problem Statement

You are given the following information, but you may prefer to do some research for yourself.

  • 1 Jan 1900 was a Monday.
  • Thirty days has September,
    April, June and November.
    All the rest have thirty-one,
    Saving February alone,
    Which has twenty-eight, rain or shine.
    And on leap years, twenty-nine.
  • A leap year occurs on any year evenly divisible by 4, but not on a century unless it is divisible by 400.

How many Sundays fell on the first of the month during the twentieth century (1 Jan 1901 to 31 Dec 2000)?

Test Cases

  • main: solve(end_year=2000, start_year=1901) → 171

Mathematical Complexity

This problem reduces to calendar arithmetic under modular arithmetic. Days of the week repeat with period 7, so if the day of the week for any anchor date is known, the day of the week for any later date can be computed by accumulating elapsed days and reducing modulo 7. The only non-trivial calendar rule is the leap year definition - divisible by 4, except century years unless also divisible by 400 - which the problem states explicitly and which only affects February's day count.

The search space is fixed: 100 years times 12 months gives exactly 1200 first-of-month dates to examine. The asymptotic complexity is O(Y * M) where Y is years and M is months per year, which is effectively O(1) for this fixed input. There is no combinatorial explosion, no dynamic programming, and no number theory beyond modular arithmetic modulo 7.

Solution Approach

Standard Library Delegation (Python) and Hand-Rolled Day Counter (C)

Both the Python and C files implement solution index 0 but handle the calendar logic differently.

In the Python solution I chose to delegate all calendar arithmetic to datetime.date. For each (year, month) pair in the target range I construct a date object for the first of that month and call isoweekday(), comparing against 7 for Sunday. The entire body is a single generator expression fed to sum. I deliberately avoided reimplementing leap-year logic because Python's datetime module already encodes the proleptic Gregorian calendar correctly, including exactly the leap year rule the problem specifies. The outer loop iterates over months and the inner loop over years; this ordering is arbitrary since every (year, month) pair is visited exactly once either way.

In the C solution I hand-rolled the calendar arithmetic, since C has no equivalent standard library convenience. I wrote a small is_leap_year predicate and a days_in_month function encoding the standard month lengths with the leap year correction applied to February. The day-of-week state is tracked as a single integer dow using 0 = Monday through 6 = Sunday, anchored to the fact that 1 Jan 1900 was a Monday. I first advance dow from 1900 to the start year by accumulating full-year day counts modulo 7, then iterate month by month over the target range, testing whether dow == 6 before advancing by the current month's day count. The argument parsing mirrors the Python interface: the first positional argument is end_year and the second is start_year.

Learnings

The key insight was recognising that the leap year rules given in the problem are not an invitation to derive calendar arithmetic from scratch - they are precisely the rules Python's datetime module already implements. Reaching for the standard library in Python kept the solution to a single readable expression. In C, where no such library is available, the hand-rolled arithmetic is still straightforward because the rules are simple and the iteration space is tiny.

Comparing the two implementations using the benchmark results is instructive even though they share the same algorithmic index. The C solution clocks in at essentially 0.0000 seconds, while the Python solution takes 0.0003 seconds. Both are negligible in absolute terms, but the C version is faster by at least an order of magnitude because it avoids constructing 1200 datetime.date objects and the associated Python interpreter overhead. For a problem this small the Python solution is clearly the more elegant choice; the C version is useful as a performance baseline for the timing framework.

Two technique notes worth keeping for future calendar problems. First, isoweekday() returns values in the range 1-7 with Sunday as 7, while weekday() uses 0-6 with Sunday as 6 - I prefer isoweekday() == 7 because the intent reads more naturally. Second, when hand-rolling a day-of-week counter in C, anchoring to a known Monday and advancing with modulo 7 arithmetic is simpler than implementing a closed-form formula such as Zeller's congruence, especially for a fixed bounded range.

Mathematical Complexity

This problem is a pure combinatorial search over a fixed 20x20 grid. The only mathematical observation needed is that "four adjacent numbers in the same direction" partitions into exactly four direction families: horizontal, vertical, main diagonal, and anti-diagonal. For a grid of dimension N and a run length of k, a valid anchor cell (x, y) must satisfy x in [0, N-k] and y in [0, N-k] to keep every direction in-bounds simultaneously. That gives at most (N - k + 1)^2 anchor positions with four products computed per anchor, so the total work is O(N^2 * k). For N=20 and k=4 that is 4 * 17 * 17 = 1156 candidate products, each requiring four multiplications - a trivially small computation with no number theory whatsoever required.

The one genuine indexing subtlety is the anti-diagonal direction, where the row index must decrease as the column index increases. I express the anti-diagonal row as x + k - 1 - i for step i, which counts the row down from x + k - 1 to x while i counts the column up from y. Crucially, this stays within bounds automatically whenever the anchor already satisfies the shared constraint x in [0, N-k], y in [0, N-k], so no extra boundary check is needed for any of the four directions - all four can live under one unified anchor loop.

Solution Approach

Simultaneous Four-Direction Product Accumulation (Python and C)

I implemented the same core algorithm in both Python and C. For each anchor cell (x, y), I accumulate four running products simultaneously through a single inner loop over i in [0, k). At each step, the horizontal accumulator is multiplied by grid[x][y+i], the vertical accumulator by grid[x+i][y], the main-diagonal accumulator by grid[x+i][y+i], and the anti-diagonal accumulator by grid[x + num_adjacent - 1 - i][y+i]. After the inner loop, all four candidates for that anchor are compared against the running best.

In Python I expressed this using nested generator expressions inside a single max() call. The trick is seeding the four accumulators at (1, 1, 1, 1) via a single-element tuple in a generator for clause, then re-binding all four names on each iteration of the inner loop using a further single-element tuple unpack. This avoids explicit loop variables or auxiliary lists and lets the entire solve body be a pure expression. In C the same algorithm becomes a conventional nested for loop with four long long accumulators, which is considerably more readable and significantly faster.

Learnings

The key insight was that all four directions can share a single anchor constraint. As long as both x and y are bounded by dimension - num_adjacent (inclusive), no direction ever goes out of bounds - including the anti-diagonal. Recognising this meant I only needed one loop nest rather than four separate boundary-checked sweeps, which keeps the code compact and avoids a common source of off-by-one errors.

The anti-diagonal index x + num_adjacent - 1 - i is worth remembering as a reusable pattern for any grid direction problem. It pairs a downward row sweep with an upward column sweep inside a single unified loop body, and the bounds fall out naturally from the shared anchor constraint.

On the Python side, I found the idiom of carrying mutable accumulator state across generator iterations by repeatedly re-binding a tuple of names inside a nested generator clause to be a compact trick for small grid scans. It trades some readability for conciseness and is only really justified when the problem is this small. The C version, by contrast, is both clearer and dramatically faster: the benchmark shows the C solution completing in effectively 0.0000 s versus 0.0004 s for Python - roughly an order of magnitude difference even on this tiny problem. For anything involving tight inner loops over numeric data, the C translation is the right choice.

  • Brute-force enumeration is entirely sufficient for fixed small grids; O(N^2 * k) with N=20 and k=4 is only 1156 products.
  • Unifying all four directions under one anchor constraint eliminates redundant boundary checks and off-by-one risk.
  • Anti-diagonal indexing with start_row + (k - 1 - i) is a reusable pattern worth committing to memory.
  • Python generator expressions can carry accumulator state via nested single-element tuple unpacks, enabling concise one-expression solutions for simple grid scans - though readability suffers.
  • Even at this trivial scale the C solution is measurably faster, reinforcing that tight numeric loops benefit strongly from compiled code.

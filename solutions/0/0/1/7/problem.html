Problem 17: Number Letter Counts [Level 2]

Problem Statement

If the numbers $1$ to $5$ are written out in words: one, two, three, four, five, then there are $3 + 3 + 5 + 4 + 4 = 19$ letters used in total.

If all the numbers from $1$ to $1000$ (one thousand) inclusive were written out in words, how many letters would be used?


NOTE: Do not count spaces or hyphens. For example, $342$ (three hundred and forty-two) contains $23$ letters and $115$ (one hundred and fifteen) contains $20$ letters. The use of "and" when writing out numbers is in compliance with British usage.

Test Cases

  • main: solve(max_number=1000) → 21124

Mathematical Complexity

This problem is primarily a structured string-generation exercise rather than a number-theory puzzle. English number words decompose hierarchically: any positive integer splits into groups of three digits (triplets), each triplet splits into a hundreds part and a sub-100 remainder, and the sub-100 range splits further into a tens word and a units word. Because the word form of a triplet value depends only on that value and not on its position in the full number, there are at most 1000 distinct sub-problems (triplet values 0-999), regardless of the upper bound.

The asymptotic complexity is O(n) in the upper bound n: each integer from 1 to n is visited exactly once, and the per-integer work is O(1) once the triplet cache is warm. For n=1000 the cache is fully populated after a single pass, so virtually every call beyond the first occurrence of a triplet value is a cache hit. The constant factor is tiny because the lookup tables cover all irregular forms and the recursive decomposition has depth at most two (hundreds then sub-100).

Solution Approach

Recursive Triplet Decomposition with Memoization (Python and C)

I implemented the same algorithm in both Python and C, so I cover them together here. Both solutions share the same structural design: a lookup table handles every irregular or base-case word (the integers 0-19 and each multiple of ten up to 90), a cached triplet converter handles values 0-999 by recursion, and an outer function assembles full numbers from their triplets with appropriate scale words.

In the triplet converter, numbers whose word form is in the lookup table are returned immediately. Numbers between 20 and 99 that are not multiples of ten are formed by joining the tens word and units word with a hyphen, for example "twenty-one". Numbers from 100 to 999 prepend the hundreds word, and when a non-zero remainder exists, "and" is inserted before the recursive result for that remainder - this follows British English usage as required. The outer converter splits the integer into triplets (using Python's f"{n:,}" format string, or integer arithmetic in C), pairs each triplet with a scale suffix, and joins the pieces. A special case inserts "and" when a large number ends in a sub-100 tail.

In Python I used @functools.lru_cache on number_triplet_in_words to memoize results automatically. I also over-engineered the scale suffixes list to support numbers up to centillions, which costs nothing in correctness and makes the converter reusable. Letter counting is done by calling str.translate with a pre-built deletion table that strips spaces and hyphens. In C I pre-allocated a fixed array of 1000 character buffers ( triplet_cache[1000][64]) alongside a boolean flag array ( triplet_cached[1000]) to replicate the same caching pattern without dynamic allocation. Letter counting in C is a simple character walk that skips spaces and hyphens, avoiding any regex overhead.

Learnings

  • The key insight was recognising the purely hierarchical, recursive structure of English number words. Once I had a correct sub-100 builder and a hundreds layer on top of it, any arbitrarily large number follows automatically by splitting into triplets and appending scale words.
  • British English mandates "and" between the hundreds part and any non-zero remainder below 100 - for example "three hundred and forty-two". This rule also applies across the thousands boundary when the tail of the number is below 100 - for example "one thousand and five". Encoding this explicitly was essential to get the correct answer.
  • Using @lru_cache in Python and a manually managed buffer array in C are both clean ways to ensure each triplet value is converted at most once. For n=1000 the speedup over a naive approach is modest, but the pattern scales well to much larger ranges.
  • Stripping non-letter characters with str.translate and a pre-built deletion table is faster and cleaner than a character loop or a regex in Python, and is worth keeping in mind for any letter-counting task.
  • Since I only implemented one algorithm (triplet decomposition, in two languages), there is no algorithmic comparison to make. The benchmark results confirm the expected language gap: the C solution runs in roughly 0.0002 s versus 0.0024 s for Python - approximately a 12x speedup - driven almost entirely by avoiding interpreter overhead in the inner loop rather than any algorithmic difference. Both are effectively instantaneous at this scale.
  • The Python version's use of generator expressions and decorator-based caching made it the more enjoyable implementation to write, even though the C version is the faster one. For a problem this small, any correct structured approach would be fast enough.

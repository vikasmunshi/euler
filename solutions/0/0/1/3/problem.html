Mathematical Complexity

The core mathematical operation is multi-precision integer addition. Summing one hundred 50-digit numbers produces a result of at most 52 digits, so the work is O(n * d) where n = 100 addends and d = 50 digits each - a fixed and trivially small workload. The interesting mathematical observation is the carry-propagation bound: when summing n numbers in base 10, carries can ripple at most ceil(log10(n)) digit positions to the left of the addend width. With n = 100 that bound is exactly 2, meaning the result has at most 52 digits. A practical consequence is that summing only the first digits + 1 leading characters of each addend is sufficient to obtain the correct first digits characters of the total - no need to process all 50 digits when only 10 are asked for. This prefix-truncation idea is essential in environments without arbitrary-precision integers, as it keeps intermediate values inside native types.

The problem has no hidden depth in number theory or combinatorics; it is fundamentally an exercise in handling large integers correctly. The asymptotic class is O(n * d) which, for fixed n and d, is O(1) in any practical sense.

Solution Approach

Digit-prefix sum (Python and C, solution index 0)

Both the Python and C solutions implement the same conceptual algorithm: slice only the leading digits + 1 characters from each 50-digit string, sum those truncated values as integers, and return the first digits characters of the result. The extra leading character absorbs any carry that propagates out of the truncated sum. When the requested digit count is 50 or more, use_digits is capped at the full NUM_DIGITS width so no information is lost.

In Python I stored all one hundred numbers as a single triple-quoted newline-separated string constant. The solve function splits on newlines, filters blank entries, slices each line to digits + 1 characters, converts each slice to int, and passes the list to the built-in sum(). Python's arbitrary-precision integers handle all carry propagation invisibly, so the entire logic fits in a single expression. The function is parameterised on digits, which makes it straightforward to verify the result at different precisions using the extra test cases.

In C there are no arbitrary-precision integers, so I implemented the addition manually using a flat array of base-10 digits. Each of the 100 number strings is stored as a const char * in a static array. The solve function allocates an integer accumulator array of length use_digits + 4 (the extra four cells absorb all possible carries from 100 addends), iterates over every number adding its digits into the right-aligned accumulator, then makes a single right-to-left carry-propagation pass after all addends have been processed. Finally it scans past any leading zeros, copies digits characters into a heap-allocated result string, and returns it. Doing one carry sweep at the end rather than propagating after every addend is cleaner and equally correct.

Learnings

  • The key insight is that Python's arbitrary-precision integers reduce what looks like a daunting problem to a one-liner. No manual carry arithmetic is needed at all in Python; the language does the heavy lifting invisibly.
  • The carry-propagation bound - summing n numbers introduces at most ceil(log10(n)) extra digits of carry - is a useful mental model to keep. It means only a short prefix of each addend is needed when only a short prefix of the answer is required, which matters in constrained environments such as C without a big-integer library.
  • Both solutions produce identical results across all three test cases (10-digit, 20-digit, and 100-digit). The 100-digit case is particularly useful as a correctness check because it exercises the code path where use_digits is capped at NUM_DIGITS and no truncation occurs.
  • Writing the C version forced me to think explicitly about array sizing and carry propagation in a way the Python version hides. Allocating use_digits + 4 cells is a safe upper bound: even if every digit position accumulated a sum of 9 * 100 = 900, the carry chain cannot exceed 3 extra positions, so 4 spare cells is always sufficient.
  • Deferring carry propagation until after all addends are accumulated, rather than propagating after each one, is a cleaner pattern: it separates the accumulation phase from the normalisation phase and requires only a single right-to-left pass over the digit array.
  • Embedding static input data as a string constant and splitting at runtime is a clean pattern for self-contained Euler problems where the data never changes. In C the equivalent is a static array of string literals, which the compiler places in read-only memory with no runtime overhead.
  • Both solutions benchmark at effectively zero elapsed time, as expected for an O(100 * 50) workload. The Python version is more elegant given the language's built-in support for arbitrary-precision integers; the C version is useful mainly as an exercise in manual big-integer addition and explicit memory management, and as a demonstration that the same algorithm can be expressed without any library dependency.

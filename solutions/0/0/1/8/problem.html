Problem 18: Maximum Path Sum I [Level 1]

Mathematical Complexity

The problem asks for the maximum sum along any root-to-leaf path through a triangular grid. A naive approach enumerates every path by making a binary left-or-right choice at each row, producing O(2^n) distinct routes for an n-row triangle. For the 15-row triangle here that is only 16384 routes - still feasible - but the problem explicitly flags Problem 67 with 100 rows, where brute force yields roughly 2^99 paths and is completely impractical. The efficient strategy exploits optimal substructure: the maximum path sum passing through any node equals that node's value plus the larger of the two maximum sub-path sums rooted at its two children. This recurrence supports bottom-up dynamic programming, which resolves the entire triangle in O(n^2) time - linear in the total number of elements.

A critical observation is that greedy top-down selection - always stepping to the larger immediate neighbour - does not guarantee a global optimum. A locally smaller child can sit above a much larger subtree, so purely greedy descent can be misled. Bottom-up DP sidesteps this entirely: by the time a decision at level k is considered, the optimal totals for all subtrees below it are already fully folded in.

Solution Approach

Bottom-Up Dynamic Programming - In-Place Row Collapse (s0, Python and C)

I implemented the same algorithm in both Python and C. The triangle is stored as a list of rows - a list of lists in Python, and a jagged array of individually malloc'd int arrays in C. The reduction loop repeatedly targets the second-to-last row: for each element at position i, I replace its value with itself plus the maximum of positions i and i+1 in the final row. The final row is then discarded. After n-1 such iterations only the apex remains and its value is the answer.

In the Python version I use copy.deepcopy before the reduction loop to avoid mutating the caller's triangle data, which matters when the same parsed triangle might be reused by test harnesses. The collapse step itself is written as a list comprehension with enumerate, and the exhausted row is removed with del triangle[-1]. In the C version I operate directly on the int **triangle array without a defensive copy - the data is freshly allocated inside each call to solve, so mutation is safe - and I track a current_rows counter rather than physically removing array entries. The C parsing requires more ceremony: I count rows by scanning for newlines, then process each line twice using strdup and strtok - once to count tokens and allocate the row array, and once to populate it. All heap memory is freed explicitly before returning the result. Despite these language-level differences, the algorithmic core is identical in both.

Learnings

The key insight that unlocked this solution is recognising optimal substructure: once I understood that the best path through any node is determined purely by that node's value and the best paths available below it, the bottom-up DP formulation followed immediately. The in-place collapse pattern - progressively shrinking the triangle by one row at a time until only the apex scalar remains - is especially clean because it repurposes the input array as the DP table with no auxiliary storage.

Since I only implemented one algorithm (index s0), there is no algorithmic comparison to draw. The benchmark results do confirm the expected language-level gap: the C build records 0.0000 s on both test cases while Python records 0.0001 s on the main triangle, though both are effectively instantaneous at this scale. The real payoff of the DP approach appears at scale - the same code, unchanged, will handle Problem 67's 100-row triangle without modification or any meaningful increase in runtime.

  • Optimal substructure is the condition that makes DP applicable: max path through a node = node value + max of children's optimal values.
  • Bottom-up traversal (leaves to root) avoids the exponential branching of top-down enumeration.
  • Greedy descent is not equivalent to DP here - this is a useful concrete example to remember when evaluating whether greedy is safe for a new problem.
  • Defensive copying in Python ( deepcopy) is worth the minor overhead when an algorithm mutates its input and the same data structure may be referenced again by tests or callers.
  • The in-place row-collapse pattern - iteratively reducing a hierarchical structure until only the answer remains - is worth remembering for similar problems.
  • This solution scales directly to Problem 67 with 100 rows, making it the correct general approach to internalise rather than the brute-force alternative.

Problem 14: Longest Collatz Sequence [Level 0]

Problem Statement

The following iterative sequence is defined for the set of positive integers:

  • $n \to n/2$ ($n$ is even)
  • $n \to 3n + 1$ ($n$ is odd)

Using the rule above and starting with $13$, we generate the following sequence: $$13 \to 40 \to 20 \to 10 \to 5 \to 16 \to 8 \to 4 \to 2 \to 1.$$

It can be seen that this sequence (starting at $13$ and finishing at $1$) contains $10$ terms. Although it has not been proved yet (Collatz Problem), it is thought that all starting numbers finish at $1$.

Which starting number, under one million, produces the longest chain?

NOTE: Once the chain starts the terms are allowed to go above one million.

Test Cases

  • main: solve(max_number=1000000) → 837799
  • extra: solve(max_number=10000000) → 8400511

Mathematical Complexity

The Collatz map is defined by two rules: halve an even number, triple-and-add-one an odd number. The conjecture that every starting value eventually reaches 1 is unproven but holds for all values tested up to enormous bounds. The key structural observation that makes an efficient search feasible is that Collatz trajectories share common suffixes - once two chains merge at the same value, every subsequent step to 1 is identical. This overlapping-subproblems property is exactly what memoization exploits. Without caching, brute-force simulation for all n under 1,000,000 costs O(N * L) where L is the average chain length (typically several hundred steps), giving hundreds of millions of individual operations. With memoization, each distinct intermediate value is computed exactly once across all chains, bringing the practical cost much closer to O(N log N).

A secondary combinatorial insight cuts the search space roughly in half. For any starting number n where 2n also falls within the search bound, the chain from 2n is exactly one step longer than the chain from n, because 2n is even and maps directly to n. Therefore the globally longest chain cannot start in the lower half of the range - its double would produce a strictly longer chain. The largest power of two strictly below the upper bound defines a clean cutoff: every candidate below that threshold has an in-range double that dominates it, so the search only needs to cover the upper portion of the interval.

Solution Approach

Recursive Memoization with Upper-Half Search (Python and C)

I implemented one core algorithm in both Python and C. The chain-length function is recursive: return 1 at the base case n=1, otherwise recurse on n/2 (even) or 3n+1 (odd) and add 1. Because the 3n+1 step can temporarily push values above the original starting number, the cache must cover values that exceed the initial search bound.

In Python I decorated collatz_sequence_length with @functools.lru_cache(maxsize=None), which stores every result keyed by argument on first computation and returns it in O(1) on any repeat. Python's arbitrary-precision integers handle intermediate values that exceed the search bound without any special treatment. The outer loop runs from max_number down to one step above 2 ** (max_number.bit_length() - 1), which is the largest power of two strictly below max_number, applying the upper-half pruning exactly. I also included an optional --show path that plots chain lengths using matplotlib.

In C I replaced the implicit hash map of lru_cache with a flat long long array of length max_number + 1 allocated with calloc. Zero-initialisation serves as the sentinel for "not yet computed". Values that exceed cache_size during recursion are computed and returned normally but not stored - only values within the original range are cached. The same upper-half loop bounds apply. The cache is freed after each call to solve, which is important when the benchmark runner invokes solve multiple times in a single process, so that stale data from a previous run cannot affect subsequent runs.

Learnings

The central insight was recognising that the overlapping-subproblems structure of Collatz sequences maps perfectly onto memoization: any large starting value's trajectory inevitably passes through smaller values whose lengths are already cached, so the amortised cost per starting number drops dramatically compared to naive simulation. In Python, @lru_cache is a one-line way to achieve this without writing an explicit dictionary or a bottom-up table.

The upper-half pruning is a reusable pattern: whenever f(2n) is strictly greater than f(n) for all n in the valid range, the global maximum over [1, N] must lie in the interval (N/2, N]. The largest power of two below N gives a clean boundary because it is the highest value x such that 2x is still within range, so everything strictly below x is dominated. This halves the search with zero loss of correctness.

The C version is roughly 16x faster at the main input size and about 8x faster at the larger input. The C flat-array cache has lower per-access overhead than Python's hash-based lru_cache, which accounts for most of the gap. Both implementations produce identical answers at both input sizes, confirming correctness.

  • Use @lru_cache for any recursion with overlapping subproblems - it is faster to write than a manual dictionary and often equally fast at runtime.
  • The upper-half dominance argument - any n below N/2 is beaten by 2n - is a reusable search-space reduction for problems where doubling monotonically extends a chain or sequence.
  • In C, a plain array cache indexed by value is much faster than a hash map for integer keys in a known bounded range, and calloc gives zero-initialisation for free as a sentinel.
  • Remember to free and reset a static cache between benchmark runs in C, or results from a second run may silently reuse stale memory.
  • Raise sys.setrecursionlimit when memoised recursion may descend through many uncached levels before hitting a base case, to avoid unexpected stack overflow errors in Python.

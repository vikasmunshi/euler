Problem Analysis

The Collatz sequence follows a deceptively simple rule: halve an even number, otherwise triple it and add one. The Collatz conjecture - verified computationally for vast ranges but still unproven - holds that every positive integer eventually reaches 1. The task is to find the starting value below one million whose chain has the most terms. The naive approach iterates over every start and counts its steps independently, but this repeats enormous amounts of work, because distinct chains converge and then share a long common tail down to 1.

The structural observation that unlocks the problem is that the chain-length function has overlapping subproblems: once the length from some intermediate value n is known, it can be reused by every other chain that passes through n. That is precisely the condition under which memoization - a form of dynamic programming that caches the result of each subproblem the first time it is solved - pays off. A second, subtler observation prunes the search: any start x in the upper half of the range (above the largest power of two below the limit) is enough, because every value in the lower half is reached "for free" as the halved successor of 2x, and the chain from 2x is just the chain from x with one extra step prepended, hence never shorter. With memoization, each value contributes only a handful of new steps before landing on a cached entry, giving an overall cost of roughly O(N log N) for a limit of N.

Solution Approach

Memoised Recursive Collatz (Solution 0 - Python and C)

Both languages run one algorithm. Its heart is a function collatz_length(n) returning the number of terms from n down to 1: the base case n == 1 has length 1, and otherwise the length is one more than the length of the next value - n / 2 when even, 3n + 1 when odd. This is a direct transcription of the mathematical recurrence, made efficient purely by caching.

The Python version defines collatz_length as a nested function decorated with functools.lru_cache, so the cache is a fresh hash table created on every call to solve. This is deliberate: the benchmark harness calls solve repeatedly, and a cache built once at import (or shared across calls) would leave only the first call paying for it and make every later call almost free, understating the true cost. Building it inside solve means each timed run rebuilds the table from scratch and reports an honest figure. Python's arbitrary-precision integers mean 3n + 1 can never overflow, so value ranges are a non-issue.

The C version allocates a flat array of long long indexed by n with calloc, using the value 0 as a "not yet computed" sentinel value. Because 3n + 1 can exceed the limit, the recursion computes any out-of-range value without caching it; the result still bubbles up correctly, and the first in-range value on the way back up is cached. The array is allocated and freed inside solve, mirroring the Python cache's per-call lifetime so the two benchmarks are measured on the same footing. A contiguous array is chosen over a hash table deliberately: indexing a flat block benefits from CPU locality of reference and avoids per-lookup hashing overhead.

In both versions the search loop starts at max_number and walks down only to the largest power of two strictly below it (found with int.bit_length() in Python and a doubling loop in C). As argued above, any maximum-length chain must begin in this upper half, so the lower half is skipped entirely. A running pair of variables tracks the best length and its starting number, updated whenever a new record appears.

Programming Techniques

Per-call memoization with a nested lru_cache

functools.lru_cache is Python's built-in decorator for transparent memoization: it checks whether the argument tuple has been seen, returns the stored result if so, and otherwise computes and stores it. With maxsize=None the cache is unbounded, which suits a run where every computed length is potentially reusable. Defining the decorated function inside solve rather than at module scope is the key choice here: it ties the cache lifetime to a single call so a repeated-runs benchmark measures real work rather than a warm cache. Reach for memoization whenever a recursive function has overlapping subproblems and a manageable argument space - it converts exponential recomputation into amortised linear work in one line.

Flat array cache in C

Instead of a general-purpose hash table, the C solution allocates one contiguous block with calloc, which zero-initialises the memory and so supplies the sentinel 0 for free. Indexing by the number itself gives O(1) lookup and store with excellent locality of reference, since the values touched during one chain traversal tend to be numerically close. The trade-off - values above max_number cannot be cached - is acceptable because they appear only transiently.

Sentinel-value pattern

Both versions distinguish "not yet computed" from a real result with a sentinel value: the integer 0 in C (safe because no chain has length 0; the minimum is 1) and the absence of a key in Python's cache. Choosing a sentinel outside the valid result range lets the cache array double as its own "visited" marker, sparing a separate boolean array and improving memory density.

Half-range search pruning

The top-level loop runs from max_number down to power_of_two + 1 rather than to 1. In Python the boundary uses int.bit_length(), the number of bits in the binary representation, so 2 ** (bits - 1) is the largest power of two not exceeding the limit; in C the same value comes from a doubling loop. This is a domain-specific pruning rule: it does not change the asymptotic class but roughly halves the iteration count for a steady constant-factor gain.

Multiple assignment for the running maximum

When a new record is found in Python, both the length and the starting number are updated in a single multiple assignment: max_length, starting_number = (length, x). Besides being concise, this multiple assignment keeps the two related variables in lockstep, eliminating any window where one is updated and the other is not - a habit that prevents a class of bugs under later refactoring.

Pure solver behind the runner framework

Each solution implements only a pure solve - returning the answer as a string - while the shared runner framework supplies main, parses --runs=N, repeats the call, times it, and prints the result line. Keeping the algorithm free of timing and I/O code makes it shorter and lets the same harness benchmark every problem identically. It also makes the cache-lifetime point above matter: because the runner re-invokes solve, any state a fair benchmark should re-pay must live inside solve, not at module scope.

Key Takeaways

The unlocking insight is recognising chain length as a function with overlapping subproblems, which immediately suggests memoization and turns wasteful repeated traversal into amortised linear work. The secondary insight - that the maximum must start in the upper half of the range - converts a simple property of the halving rule into a concrete constant-factor speedup.

The two implementations run the identical algorithm, so the benchmark gap measures the language layer itself. With both caches rebuilt per call for a fair comparison, the C version solves the main case (limit 1,000,000) in about 0.018 s against 0.832 s for Python - roughly a 45x difference - and the 10,000,000 case in about 0.214 s against 6.93 s, around 32x. The gap comes from C's lower per-call overhead, the absence of Python's boxed-integer object model on every arithmetic step, and the flat-array cache's better locality of reference than a Python hash table. At this scale Python is perfectly adequate and far more concise; the C version shows the ceiling reachable with the same algorithm when low-level control matters.

  • When a recursive problem has overlapping subproblems, memoize before hand-rolling a bottom-up table - the speedup is large and the code change is minimal.
  • Build a cache inside the function being benchmarked, not at module scope, so each timed run pays the real cost and the measurement stays honest.
  • A flat array indexed by the problem value beats a hash map for bounded, dense integer keys, thanks to memory locality.
  • Pick a sentinel outside the valid result range so the cache itself records what has been computed, sparing a separate "visited" structure.
  • Before looping over a whole domain, look for a structural property that lets you prune it - here the halving step makes the upper half sufficient.
  • Keep tightly coupled state variables in a single assignment to avoid inconsistent intermediate states.

Problem 82: Path Sum: Three Ways [Level 3]

Problem Statement

NOTE: This problem is a more challenging version of Problem 81 .

The minimal path sum in the $5$ by $5$ matrix below, by starting in any cell in the left column and finishing in any cell in the right column, and only moving up, down, and right, is indicated in red and bold; the sum is equal to $994$.

Mathematical Complexity

This problem extends the two-direction grid path sum idea from Problem 81 by permitting vertical movement within each column before stepping right. The key observation is that optimal substructure still applies column by column: the minimum cost to travel from cell (r, c) to the right edge depends only on the already-reduced costs stored in column c+1. The complication is that within column c, a path may move freely up or down before exiting rightward, so every source row must be compared against every possible target row in the same column. The transition cost from source row r to target row t is the range sum of column c-1 values between rows r and t inclusive, plus the already-reduced value at row t in column c. This makes each column-reduction step O(n^2) rather than O(n), giving an overall complexity of O(n^3) for an n-by-n matrix.

For the 80-by-80 input that amounts to roughly 512,000 inner operations, which is entirely tractable. The working state is a single column vector of length n that is updated in place as the sweep moves from right to left. After all columns are reduced, the answer is the minimum of the n values remaining in column 0, found in a final O(n) scan. Auxiliary space beyond the matrix itself is O(n) for a temporary per-column buffer that prevents in-progress values from contaminating later rows within the same sweep.

Solution Approach

Right-to-Left Column Reduction with Vertical Range Sums (s0 - Python and C)

I implemented the same algorithm in both Python and C. The core routine is a reduce_column function that sweeps one column at a time from right to left. For each source row in the current column, it considers every possible target row in that same column. The transition cost is computed as the sum of all cells in the source column between the source and target rows - inclusive of both endpoints - plus the already-reduced value stored at the target row in the column to the right, which represents the minimum onward cost to the right edge. The minimum over all targets becomes the new reduced value for that source row. All new values are collected into a temporary buffer before being written back, so no partially updated entry is seen by subsequent rows in the same pass.

After reducing every column down to column 0, the answer is the minimum over all entries in that leftmost column. In Python I expressed the reduction as a nested list comprehension, passing a generator expression to min for compactness. Range sums use a generator over range(min(row, target), max(row, target) + 1). In C I used explicit nested loops with a local best-tracking variable and computed range sums with a third inner loop over the same index bounds. Both versions mutate the matrix in place. The C version parses the comma-separated input manually using pointer arithmetic, while Python uses str.split and a list comprehension. There are no algorithmic differences between the two languages - only syntax and I/O handling differ.

Learnings

The key insight that unlocked the solution was recognising that vertical freedom within a column means the column-reduction step cannot look only at adjacent rows. A path can shift to any row in the current column before stepping right, so every source-target row pair must be evaluated. The transition cost is a range sum over all cells between source and target in the column being left behind, including the source cell itself, because the cell you depart from contributes its value to the path total. Carrying over the simpler neighbour-only logic from Problem 81 would produce incorrect results.

The benchmark confirms correctness on both the sample (994) and the full 80-by-80 matrix (260324). The C version finishes in about 0.011 seconds while the Python version takes about 0.93 seconds - roughly 85 times slower - reflecting the cost of interpreter overhead and Python object arithmetic on the inner loops. A further optimisation would be to precompute column prefix sums so that each range sum becomes O(1) rather than O(n), reducing total complexity to O(n^2). That was unnecessary here given how fast even the Python solution completes, but it is worth keeping in mind for larger inputs. The general pattern - reducing a 2-D grid to a 1-D cost vector by sweeping columns and exploiting optimal substructure - is a useful template for any grid problem where movement is permitted in more than one direction along the same axis.

Problem 86: Cuboid Route [Level 7]

Problem Statement

A spider, S, sits in one corner of a cuboid room, measuring $6$ by $5$ by $3$, and a fly, F, sits in the opposite corner. By travelling on the surfaces of the room the shortest "straight line" distance from S to F is $10$ and the path is shown on the diagram.


However, there are up to three "shortest" path candidates for any given cuboid and the shortest route doesn't always have integer length.

It can be shown that there are exactly $2060$ distinct cuboids, ignoring rotations, with integer dimensions, up to a maximum size of $M$ by $M$ by $M$, for which the shortest route has integer length when $M = 100$. This is the least value of $M$ for which the number of solutions first exceeds two thousand; the number of solutions when $M = 99$ is $1975$.

Find the least value of $M$ such that the number of solutions first exceeds one million.

Mathematical Complexity

The core geometric insight is surface unfolding: the shortest path along the surface of a cuboid from one corner to the opposite can be found by cutting along an edge and flattening two adjacent faces into a single rectangle. The shortest surface path becomes a straight Euclidean line across that rectangle, whose length follows directly from the Pythagorean theorem. For a cuboid with integer dimensions a >= b >= c, the three candidate unfoldings yield distances sqrt(a^2 + (b+c)^2), sqrt(b^2 + (a+c)^2), and sqrt(c^2 + (a+b)^2). Under the canonical ordering a >= b >= c, the minimum is always the first expression, so I only need to check whether a^2 + (b+c)^2 is a perfect square. This collapses the problem from a three-variable search to a double loop over (a, s) where s = b+c.

Once a Pythagorean triple is confirmed for a given pair (a, s), counting the valid integer splits (b, c) with b+c = s, b >= c >= 1, and b <= a is a closed-form O(1) calculation using floor arithmetic: the count is s//2 when s <= a+1, and (2*a - s + 2)//2 when s > a+1. The outer loop iterates a from 1 upward and s from 1 to 2*a, giving O(M^2) total iterations where M is the answer. For M = 1818 this amounts to roughly three million (a, s) pairs, each requiring only a single integer square-root check and an O(1) count.

Solution Approach

Pythagorean Triple Scan with Closed-Form Pair Counting (s0)

I iterate a from 1 upward using an infinite loop, and for each a I scan every possible value of b_plus_c from 1 to 2*a. For each (a, b_plus_c) pair I test whether a^2 + b_plus_c^2 is a perfect square. In Python I use math.sqrt(...).is_integer() for this check, which is clean and readable. In C I compute the floating-point square root and use the round-and-verify pattern: long long isq = (long long)(sq + 0.5) followed by isq * isq == (long long)val, which avoids floating-point ambiguity in direct equality tests on doubles.

When the Pythagorean check passes, I add the closed-form count directly to a running total without any inner loop over individual (b, c) pairs. The moment the running total meets or exceeds the target I immediately return a. No solution list is stored and no redundant work is done beyond the crossing point. The Python and C implementations are algorithmically identical; the only differences are the square-root check idiom and the usual loop structure syntax.

Learnings

The key insight that unlocked the solution was recognising that fixing a as the largest dimension and treating b+c as a single variable s reduces a three-dimensional problem to a two-dimensional one. Instead of triple-nested loops over (a, b, c), I only need a double loop over (a, s), and within that the only non-trivial work is a single integer square-root check. Once that check passes, counting the valid (b, c) splits of s is a trivial closed-form calculation - no second inner loop is ever needed.

The benchmarks show a clear language difference: Python solves the main case in about 387 ms while C solves it in about 11 ms - roughly a 35x speedup, which is expected for arithmetic-heavy inner loops. Both are fully correct across all test cases.

Key techniques and concepts worth remembering for future problems:

  • Surface-path minimisation on a polyhedron reduces to Euclidean geometry by unfolding faces into a plane - the shortest path becomes a straight line across the unfolded net.
  • For a >= b >= c, only one of the three candidate unfoldings needs to be checked; the others are provably longer under the canonical ordering.
  • Treating a multi-variable sum as a single variable (s = b+c here) can reduce a higher-dimensional search to a lower-dimensional one with no loss of correctness.
  • Counting integer pairs with a fixed sum and ordering constraints is always O(1) via floor arithmetic - an inner loop over individual pairs is never necessary.
  • Maintaining a running total and terminating the moment a threshold is crossed avoids precomputing or storing the full solution set, saving both time and memory.
  • When using floating-point square roots for integer Pythagorean checks in C, the round-and-verify pattern isq = (long long)(sq + 0.5); isq * isq == val is safer than a direct equality test on the double.

Problem 83: Path Sum: Four Ways [Level 4]

Mathematical Complexity

Allowing movement in all four cardinal directions - up, down, left, and right - turns the grid into a general undirected graph rather than a directed acyclic graph. Problems 81 and 82 restricted movement to subsets that preserved a topological order, making dynamic programming with a fixed sweep direction correct. Here, any fixed processing order can fail: a later update to one cell may need to propagate backwards through cells already processed, so DP is no longer valid. The problem reduces to single-source shortest paths on a node-weighted graph with V = N squared nodes and at most 4N squared edges. Dijkstra's algorithm with a binary-heap priority queue solves this in O((V + E) log V), which is O(N squared log N) for a grid. My implementation uses a linear scan to find the minimum-distance unvisited node at each step, which costs O(V) per iteration and raises the overall complexity to O(V squared) = O(N to the fourth). For the 80-by-80 matrix (V = 6400) this is still tractable, completing in about 39 milliseconds in C and 2.6 seconds in Python.

Because all cell values are positive integers, all edge weights in the equivalent graph are positive, which satisfies the key precondition for Dijkstra's algorithm. The cost of moving into a cell equals that cell's value, making this a node-weighted rather than edge-weighted graph. That distinction matters for initialisation: the start cell's own value must be included in the initial distance, and neighbour relaxation adds the neighbour's cell weight rather than a fixed edge cost.

Solution Approach

Dijkstra's Algorithm with Linear Minimum Search (Python and C)

I modelled the grid as a mapping from coordinates to cell weights and ran Dijkstra's algorithm starting from the top-left corner (0, 0). The initial distance for the start node is set to its own cell value. All other distances are initialised to a sentinel equal to one more than the total sum of all cell weights - a data-driven substitute for infinity that is guaranteed to exceed any real reachable distance. At each iteration I select the unvisited node with the smallest tentative distance. Both implementations perform this as an O(V) linear scan: the Python version uses min(unvisited, key=lambda node: distances[node]) over a set of coordinate tuples, and the C version scans the flat distance array directly. For the selected node I relax its four in-bounds unvisited neighbours by adding each neighbour's cell weight to the current node's distance. The loop terminates as soon as the target node (size-1, size-1) is selected, exploiting Dijkstra's early-exit property: the first time a node is popped its tentative distance is already final.

The Python and C implementations follow exactly the same algorithm. The C version represents the grid as a flat integer array indexed by row * size + col and uses two malloc 'd arrays for distances and visited flags. The Python version uses a dictionary keyed by tuples for distances and a set for the unvisited collection. Both produce 2297 for the 5-by-5 example and 425185 for the 80-by-80 file. The benchmark shows a roughly 67-fold speedup from Python (2.63 s) to C (0.039 s) with no algorithmic difference at all - pure language overhead.

Learnings

The central insight was recognising that unrestricted four-directional movement breaks the DAG property that made earlier problems amenable to simple DP sweeps. As soon as left and up moves are allowed, any fixed propagation order can miss updates that flow backwards, so correctness requires a proper shortest-path algorithm. Dijkstra's algorithm resolves this immediately because its correctness depends only on non-negative edge weights, which is satisfied since all cell values are positive integers.

A secondary insight concerns node-weighted versus edge-weighted graphs. Here the cost of a move is the weight of the destination cell. The start cell's own value must therefore appear in the initial distance, and every neighbour relaxation adds the neighbour's cell value rather than a fixed edge cost. Getting this wrong would undercount the total path cost by exactly the start cell's weight.

The benchmark makes the Python-to-C speedup concrete without any algorithmic change. Replacing the O(V) linear scan with a heapq -based priority queue in Python would lower the complexity to O(N squared log N) and could bring the Python runtime well under a second, which would be worth doing if performance mattered. Key techniques worth remembering for future grid shortest-path problems:

  • When movement in all directions is allowed, use Dijkstra or BFS (for uniform weights) rather than adapting DP.
  • A priority queue gives O(log V) extraction instead of O(V) linear scans and should be preferred for larger grids.
  • Early termination once the target node is popped is valid in Dijkstra because the first pop gives the final distance.
  • A sentinel equal to the sum of all weights plus one is a clean, data-driven substitute for infinity in grid problems.
  • In node-weighted graphs the start node's value belongs in the initial distance, and neighbour relaxation adds the neighbour's weight, not an edge cost.

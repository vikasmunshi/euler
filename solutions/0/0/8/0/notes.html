Problem Analysis

I need to extract the first one hundred decimal digits of the square root of every non-perfect-square integer in a range, sum those digits, and total the sums. The mathematical foundation is that the square root of a natural number is either an integer or an irrational number: a quantity that cannot be written as a ratio of two integers and whose decimal expansion neither terminates nor settles into a repeating cycle. So for every number that is not a perfect square, I have an unending string of digits and must read off a fixed prefix from it without rounding error contaminating my answer.

The observation that makes this tractable is a scaling trick. If x = sqrt(n), then x * 10^d = sqrt(n * 10^(2d)). The integer part of sqrt(n * 10^(2d)) is therefore exactly the first d significant digits of sqrt(n) written without a decimal point. This converts a problem about real-number precision into a pure integer square root problem: compute floor(sqrt(n * 10^(2d))). For n up to 999 and d = 100, the scaled value has roughly 206 decimal digits - far beyond the 64-bit range - so I need arbitrary-precision arithmetic (big integers). The cost of each big-integer multiply or divide grows with the digit count d, so the overall complexity is roughly O(N * d^2), where N is the count of irrational roots: the d^2 comes from schoolbook multiplication of two d-digit numbers, and the choice of root-finding algorithm controls how many such operations I perform per number.

Solution Approach

Heron's (Newton's) Method for Integer Square Root

Both p0080_s0.py and p0080_s0_c compute the integer square root using Heron's method, which is Newton's method applied to the equation x^2 - S = 0. The iteration is x_new = (x + S / x) / 2. Each step roughly doubles the number of correct digits, so the method converges quadratically: about seven iterations suffice to reach 100 digits.

  • Scale the input to S = n * 10^(2d) using big integers.
  • Pick a starting guess at or above the true root, so the sequence decreases monotonically toward floor(sqrt(S)). In Python I seed with x = S itself, which is always a valid overestimate for S >= 1; the quadratic convergence quickly absorbs the loose start. In C I bootstrap with a double-precision estimate sqrt(n) * 1e15 (accurate to about 15 digits), add a margin of 100, then scale up by 10^(d-15) to land within a few iterations of the answer.
  • Iterate until x_new is no longer strictly less than x; that fixed point is floor(sqrt(S)).
  • Stringify the result, take the first d characters, and sum their digits.

The Python version is almost a direct transcription of the math, using the walrus operator to assign the next iterate and compare it to the previous value in one expression. The C version is far larger because Heron's method requires big-integer division, which I implemented with Knuth's Algorithm D (long division on limb arrays, with divisor normalization and a two-step quotient-digit correction). This is the most intricate routine in the file and the reason the C solution earns its speed.

Binary Search for Integer Square Root

Both p0080_s1.py and p0080_s1_c find floor(sqrt(S)) by binary search. The invariant is low^2 <= S < high^2. Each step computes mid = (low + high) / 2, squares it, and moves low or high according to whether mid^2 <= S. When high - low == 1, low is the answer.

Binary search is conceptually simpler than Heron's method - it needs only multiplication, never division - but it converges only linearly: it takes about log2(sqrt(S)) iterations, roughly 166 squarings of a 100-digit number for d = 100. That is fine in Python but markedly slower in C. The C version adds a helper bi_diff_is_one that tests the termination condition high - low == 1 directly, avoiding a full subtraction allocation each iteration. Both versions start with low = 0 and high = S, a maximally loose but always-correct bracket.

Programming Techniques

Scaling to Convert Decimal Digits to Integer Arithmetic

The central pattern in every solution is replacing "first d digits of sqrt(n)" with "floor(sqrt(n * 10^(2d)))". This turns a question about real-number precision into a pure integer square root computation, eliminating floating-point rounding error entirely. Reach for this trick whenever a problem demands a fixed number of digits of an irrational quantity: multiply by the appropriate power of ten, do exact integer work, then read the digits off the result.

Python Arbitrary-Precision Integers and the Walrus Operator

Python integers are arbitrary-precision by default, so the Heron and binary-search loops read exactly like their mathematical definitions with no limb management. The Heron loop also uses the walrus operator (:=) inside the while condition to assign the next iterate and compare it to the previous value in a single expression - a compact idiom for "iterate until a fixed point is reached".

Custom Big-Integer Library in C with Limb Representation

The C solutions implement big integers as base-10^9 limbs stored least-significant-first. Base 10^9 is deliberate: it fits in a 64-bit word with carry room, two limbs multiplied together fit in GCC's __uint128_t, and it maps cleanly to decimal output via %09llu. With MAXLIMBS = 64 I get 576 decimal digits of headroom, comfortably more than needed. Choosing a base that is a power of ten makes decimal I/O trivial at a small cost in space efficiency - a good trade whenever the primary output is itself decimal digits.

Knuth Algorithm D for Big-Integer Division

Heron's method requires dividing one big integer by another, so the C Heron file implements long division limb by limb via Knuth's Algorithm D. It normalizes the divisor so its top limb is at least BASE/2 (which tightens the estimate of each quotient digit), estimates each quotient digit from the top two dividend limbs, refines that estimate with at most two corrections, subtracts the partial product, and adds the divisor back if it overshot. This is the standard industrial-strength division routine and the correct starting point any time you need division in a hand-rolled big-integer library.

Double-Precision Bootstrap for the Initial Guess

The C Heron solution seeds its iterate from sqrt((double)n) * 1e15, capturing about 15 correct digits cheaply, then scales it up to full precision. This is the general pattern of "use a fast imprecise method to get close, then let the quadratically convergent method finish". The only requirement is that the guess sit at or above the true root so the monotone-decreasing convergence holds; the + 100.0 margin guarantees that.

Cheap Perfect-Square Detection

Before any expensive root computation, each solution skips perfect squares. C uses is_perfect_square, rounding sqrt((double)n) and checking the nearby integers exactly; Python uses the equally cheap test i**0.5 % 1 == 0. Excluding degenerate cases with an O(1) filter before invoking heavy machinery is a habit worth keeping.

Key Takeaways

The unlocking insight is to treat "decimal digits of an irrational square root" as "integer square root of a scaled number", which removes floating point from the picture entirely. Given that, the choice of root-finding algorithm dominates performance. Heron's quadratic convergence crushes binary search's linear convergence at high precision, and the gap widens with problem size. On the extra benchmark (digits 100, up to 999), the C Heron solution ran in 0.0052 s versus 0.2730 s for C binary search - about 52x faster - because Heron performs roughly seven iterations where binary search performs over a hundred and sixty. In Python the gap is smaller (0.1466 s vs 0.2170 s) since CPython's bignum operations are uniformly well-optimized in C, but Heron still wins. The C Heron solution is the fastest overall; the Python Heron solution is the most readable and is plenty fast for this problem size.

  • To extract d decimal digits of an irrational root, compute the integer square root of the value scaled by 10^(2d) - turning analysis into exact integer arithmetic.
  • Prefer quadratically convergent methods (Newton/Heron) over linear ones (binary search) whenever precision is high and each operation is expensive.
  • Bootstrap an iterative numerical method with the best cheap approximation available, then let the fast-converging method finish the remaining precision.
  • When iterating toward a fixed point, ensure the starting guess lies on the side that gives monotone convergence; a guaranteed overestimate or underestimate makes termination simple.
  • Lean on arbitrary-precision integers when prototyping; only drop to a hand-rolled limb library in C when the language's native bignum is the bottleneck.
  • For custom big integers, a power-of-ten base with 128-bit intermediate products gives clean decimal I/O and safe carries; Knuth's Algorithm D is the reference division routine.
  • Filter out degenerate inputs with an O(1) check before invoking expensive routines.

Problem 80: Square Root Digital Expansion [Level 3]

Problem Statement

It is well known that if the square root of a natural number is not an integer, then it is irrational. The decimal expansion of such square roots is infinite without any repeating pattern at all.

The square root of two is $1.41421356237309504880\cdots$, and the digital sum of the first one hundred decimal digits is $475$.

For the first one hundred natural numbers, find the total of the digital sums of the first one hundred decimal digits for all the irrational square roots.

Mathematical Complexity

The core challenge is arbitrary-precision arithmetic. Standard floating-point types offer only 15-17 significant decimal digits, so extracting 100 decimal digits of an irrational square root requires a different strategy. The key insight is that floor(sqrt(n * 10^(2*d))) gives exactly the first d decimal digits of sqrt(n): multiplying under the radical by 10^(2*d) is equivalent to multiplying the result by 10^d, shifting the decimal point d places to the right. This converts a floating-point precision problem into a straightforward exact integer computation.

For each of the 91 irrational square roots below 100 we compute an integer square root of a number with roughly 200 decimal digits. The asymptotic cost per root depends heavily on the algorithm. Newton-Raphson (Heron's method) converges in O(log d) iterations due to quadratic convergence, each requiring one big-integer division on an O(d) -digit number, giving roughly O(d^2 log d) total work per root with schoolbook arithmetic. Binary search requires O(d) iterations (the search space has O(d) bits), each involving a big-integer multiply, giving O(d^3) or worse - a much heavier cost that dominates at scale.

Solution Approach

Heron's Method / Newton-Raphson Integer Square Root (s0)

Both p0080_s0.py and p0080_s0.c implement the classic Heron iteration for integer square roots. In Python I scale n by 10^(2*digits), then iterate x = (x + scaled // x) // 2 until two consecutive iterates agree - that is, until the iterate stops decreasing. Because Newton-Raphson converges quadratically, doubling the number of correct digits per step, it reaches the correct 100-digit result in roughly log2(200) ~ 8 steps. The Python version uses the scaled number itself as a safe initial overestimate, which is harmless because the first few iterations rapidly collapse it to the right neighbourhood. The final result string is truncated to digits characters and its digits summed.

The C version implements the same algorithm but requires a full arbitrary-precision library written from scratch using base- 10^9 limbs, including addition, subtraction, multiplication, Knuth Algorithm D division, and base conversion. A key difference from the Python version is the initial guess: rather than starting from the scaled number (which would be enormous in C and waste the first iteration), I seed the iteration using a double approximation scaled to about 15 significant digits, then multiply up by 10^(digits-15) to produce a tight starting estimate. This means the C Heron loop typically needs only 2-3 iterations rather than starting cold.

Binary Search Integer Square Root (s1)

Both p0080_s1.py and p0080_s1.c replace Newton-Raphson with a binary search over the interval [0, scaled_number]. The invariant is that low^2 <= scaled_number < high^2; each step computes mid = (low + high) // 2, squares it, compares with the scaled number, and narrows the bracket. When high - low == 1, low is the floor square root. This is conceptually simpler than Newton-Raphson - there are no convergence subtleties and no need to detect a fixed point - but it requires roughly d * log2(10) ~ 660 iterations for d = 100, each involving a big-integer multiply.

The C binary search reuses the same BigInt library as the Heron solution. I added a dedicated bi_diff_is_one helper that checks whether high - low == 1 without allocating a full BigInt result, avoiding unnecessary overhead in the tight loop. Even so, the hundreds of schoolbook multiplications per root add up significantly.

Learnings

The key realisation was that the entire problem reduces to integer arithmetic. The scale-and-floor trick - isqrt(n * 10^(2*d)) gives the first d digits of sqrt(n) exactly - is a reusable pattern for any problem requiring high-precision irrational values. Once that transformation is in place, both algorithms are straightforward to formulate; the challenge shifts to implementation details of big-integer arithmetic in C.

The benchmark results tell a very clear story:

The C Heron implementation is by far the fastest - about 21x faster than its Python equivalent on the extra case and over 44x faster than the C binary search. The most striking result is that the C binary search is actually slower than the Python binary search on the main and extra cases. The explanation is that Python's arbitrary-precision integers are highly optimised internally (using Karatsuba multiplication and other techniques), while my handwritten C BigInt uses schoolbook multiplication. For Heron's method this difference is irrelevant because so few iterations are needed, but for binary search the hundreds of schoolbook multiplications per root accumulate painfully. This vividly demonstrates that algorithmic complexity dominates language choice when the iteration counts differ by two orders of magnitude.

  • Scale-and-floor trick: isqrt(n * 10^(2*d)) gives the first d digits of sqrt(n) exactly - a reusable pattern for any problem requiring high-precision irrational roots.
  • Newton-Raphson integer square root converges in O(log d) iterations versus O(d) for binary search; this gap dominates all other performance factors including language choice.
  • When implementing Heron's method in C without native big integers, seeding with a double-precision approximation scaled to ~15 digits saves the majority of iterations compared to starting from the raw scaled number.
  • Writing a minimal big-integer library for a fixed precision target is feasible and educational, but schoolbook multiplication becomes a bottleneck under heavy iteration counts; Karatsuba or higher-order algorithms would be needed to compete with Python's built-in integers on the binary search variant.
  • Checking for perfect squares via i**0.5 % 1 == 0 is reliable for small integers, but for larger inputs an exact check using math.isqrt followed by squaring and verification is safer.

Problem 88: Product-sum Numbers [Level 10]

Problem Statement

A natural number, $N$, that can be written as the sum and product of a given set of at least two natural numbers, $\{a_1, a_2, \dots, a_k\}$ is called a product-sum number: $N = a_1 + a_2 + \cdots + a_k = a_1 \times a_2 \times \cdots \times a_k$.

For example, $6 = 1 + 2 + 3 = 1 \times 2 \times 3$.

For a given set of size, $k$, we shall call the smallest $N$ with this property a minimal product-sum number. The minimal product-sum numbers for sets of size, $k = 2, 3, 4, 5$, and $6$ are as follows.

  • $k=2$: $4 = 2 \times 2 = 2 + 2$
  • $k=3$: $6 = 1 \times 2 \times 3 = 1 + 2 + 3$
  • $k=4$: $8 = 1 \times 1 \times 2 \times 4 = 1 + 1 + 2 + 4$
  • $k=5$: $8 = 1 \times 1 \times 2 \times 2 \times 2 = 1 + 1 + 2 + 2 + 2$
  • $k=6$: $12 = 1 \times 1 \times 1 \times 1 \times 2 \times 6 = 1 + 1 + 1 + 1 + 2 + 6$

Hence for $2 \le k \le 6$, the sum of all the minimal product-sum numbers is $4+6+8+12 = 30$; note that $8$ is only counted once in the sum.

In fact, as the complete set of minimal product-sum numbers for $2 \le k \le 12$ is $\{4, 6, 8, 12, 15, 16\}$, the sum is $61$.

What is the sum of all the minimal product-sum numbers for $2 \le k \le 12000$?

  • main: solve(max_k=12000, min_k=2) → 7587457

Mathematical Complexity

The central insight is a mapping from multiplicative factorizations to product-sum sets. Given any factorization of N into m factors f1, f2, ..., fm with running sum S, the product already equals N by construction. To make the sum equal N as well, I can pad the multiset with exactly (N - S) ones, since multiplying by ones leaves the product unchanged while raising the sum. The resulting set has size k = m + (N - S) = N - S + m. This means every factorization of every integer N directly nominates N as a candidate minimal product-sum number for exactly one k, with no additional search needed. A tight upper bound also exists: the set containing (k-2) ones, a 2, and a k always satisfies the product-sum property with value 2k. Therefore no minimal product-sum number for any k can exceed 2k, which bounds the entire search to factorizations of integers up to 2 * max_k.

The number of multiplicative factorizations of integers up to 2 * max_k grows slowly in practice because each factor must be at least 2, limiting the recursion depth to at most log2(2 * max_k). The effective work is well within O(max_k * log(max_k)) for the given constraints, which explains the sub-100ms runtimes observed for max_k = 12000. Deduplication at the end is a linear pass over the result array, negligible compared to the factorization work.

Solution Approach

Recursive Factorization with Ones-Padding (s0 - Python and C)

I implemented a single depth-first recursive enumeration of all multiplicative factorizations. The function find_product_sum(prod, total, count, start) builds a factorization incrementally: prod accumulates the running product, total the running sum, and count the number of factors chosen so far. At each call I compute k = prod - total + count, which is the set size after padding with ones, and update min_prod[k] if the current product is smaller than the stored candidate. The factor loop starts at start to enforce non-decreasing order and avoid duplicate factorizations, and is capped at max_k // prod * 2 + 1 to respect the 2 * max_k upper bound on any candidate value. The array min_prod is initialised to 2 * max_k for every index, exploiting the known upper bound as a sentinel that doubles as a valid upper-bound candidate. After the recursion completes, I collect unique values from min_prod[2:] and sum them, correctly handling cases where multiple k values share the same minimal N.

The C version follows exactly the same algorithm, using global state ( g_max_k and g_min_prod) to avoid passing extra pointers through the recursion. Deduplication in C uses a heap-allocated boolean array indexed by value, which is simpler and faster than a hash set at this scale - the array fits comfortably in cache for max_k = 12000. The Python version uses a set() literal for deduplication, which is idiomatic and equally correct. Both implementations take min_k as the starting value for the outermost factor loop, making the search range configurable and keeping the dev test case (max_k=6) fast.

Learnings

The key insight that unlocked the solution was the ones-padding trick: any factorization of N into m factors with sum S can be extended to a valid product-sum set of size k = N - S + m by appending (N - S) ones. This collapses the problem from a two-dimensional search (over all N and all multisets summing and multiplying to N) into a single pass over multiplicative factorizations, each directly recording which k it satisfies. The second critical insight is the tight upper bound of 2k, which prunes the recursion aggressively and makes the algorithm practical even for max_k = 12000.

Since I only implemented one algorithm, there is no algorithmic comparison to make. The benchmark results do show a clear language gap: the C version solves the main case in about 0.0014 seconds versus 0.0757 seconds for Python, roughly a 54x speedup. Both are entirely practical, but the C version demonstrates how much overhead Python carries for tight recursive loops, since each recursive call here is extremely cheap and the bottleneck is interpreter dispatch rather than algorithm complexity.

  • The formula k = product - sum + count is a reusable identity for any problem involving product-sum factorizations.
  • Padding with ones is the algebraic reason the minimal N for any k is always at most 2k, giving a free and tight upper bound.
  • Initialising the answer array to the known upper bound avoids needing a separate infinity sentinel and gives every k a valid candidate from the start.
  • Enforcing non-decreasing factors via a start parameter is the standard technique for enumerating unordered factorizations without duplicates.
  • Deduplicating with a flat boolean array indexed by value is faster and simpler than a hash set when the value range is bounded and small enough to fit in cache.
  • Summing the unique values of a result array (rather than summing all entries) is the correct pattern when multiple problem instances can share the same optimal answer.

Problem 81: Path Sum: Two Ways [Level 2]

Problem Statement

In the $5$ by $5$ matrix below, the minimal path sum from the top left to the bottom right, by only moving to the right and down , is indicated in bold red and is equal to $2427$.

$$ \begin{pmatrix} \color{red}{131} & 673 & 234 & 103 & 18\\ \color{red}{201} & \color{red}{96} & \color{red}{342} & 965 & 150\\ 630 & 803 & \color{red}{746} & \color{red}{422} & 111\\ 537 & 699 & 497 & \color{red}{121} & 956\\ 805 & 732 & 524 & \color{red}{37} & \color{red}{331} \end{pmatrix} $$

Find the minimal path sum from the top left to the bottom right by only moving right and down in matrix.txt (right click and "Save Link/Target As..."), a 31K text file containing an $80$ by $80$ matrix.

Mathematical Complexity

This is a classic dynamic programming problem on a grid. The key observation is that each cell's minimum-cost contribution to the overall path depends only on the two cells immediately reachable from it - the one to the right and the one below. This optimal substructure means each sub-problem needs to be solved only once, and a single linear scan over all n^2 cells of the grid suffices. The time complexity is O(n^2) and, when done in-place on the matrix itself, the additional space complexity is O(1), ignoring the traversal coordinate bookkeeping.

Without memoization, enumerating all right-or-down paths from the top-left to the bottom-right of an n-by-n grid would require examining C(2n-2, n-1) distinct routes, which grows combinatorially. For n=80 this is astronomically large. Dynamic programming collapses that explosion entirely by reusing sub-results, turning an otherwise intractable search into a straightforward grid scan.

Solution Approach

In-place reverse anti-diagonal DP (Python and C, index 0)

Both the Python and C solutions implement the same algorithm. I process cells in reverse anti-diagonal order, starting from the bottom-right corner and sweeping anti-diagonals toward the top-left. The move_diagonally generator in Python (and the equivalent move_diagonally_all function in C) produces cell coordinates such that by the time any cell (r, c) is visited, the already-updated values for (r+1, c) and (r, c+1) are guaranteed to be in place. At each cell I add the minimum of the available right and down neighbor values, using min(neighbors, default=0) in Python and a sentinel-based minimum in C to handle the bottom-right corner cleanly without special casing. When the sweep finishes, matrix[0][0] holds the answer.

The anti-diagonal traversal is what makes a single-pass, in-place update correct without an auxiliary DP table: because each anti-diagonal is processed in full before moving to the next, and dependencies only point right or down, no value is ever read after it has been overwritten. A simpler row-by-row sweep from bottom to top would also work for this specific movement model, but I chose the diagonal approach because it generalises more naturally to variants where neither axis alone respects the full dependency order.

The C implementation mirrors the Python logic exactly. The main language-specific differences are manual file I/O, comma-delimited string parsing with strtok, a static int matrix[MAX_SIZE][MAX_SIZE] buffer, and pre-generating all (row, col) coordinate pairs into two flat int arrays before the update loop. Python handles all of this implicitly through list comprehensions and a generator function.

Learnings

The key insight was recognising that a bottom-up DP starting from the destination and propagating backward is clean and natural for this movement model. Because all allowed moves point right or down, reversing direction and walking anti-diagonals from the bottom-right corner back toward the top-left ensures every dependency is satisfied by the time it is needed - with no auxiliary structure and no topological sorting required.

The benchmark results show the C solution runs in about 0.3 ms on the main input while the Python solution takes about 3.5 ms - a roughly 12x speedup. The difference is attributable almost entirely to interpreter overhead and list construction in Python rather than any algorithmic difference; both implement exactly the same O(n^2) pass over the grid.

  • In-place DP on grids avoids extra memory and keeps the code concise; modifying the input matrix directly is safe when the traversal order is chosen so that dependency order is strictly respected.
  • Processing cells in reverse dependency order - from destination back to source - is often the most straightforward way to set up a bottom-up DP without needing to track which cells are ready.
  • The anti-diagonal sweep generalises to situations where neither pure row-wise nor pure column-wise ordering satisfies the dependency constraints, making it a useful pattern to keep in mind.
  • This problem is the first of a trilogy (Problems 81, 82, 83) that progressively extend the allowed movement directions. The same DP mindset applies throughout, but the dependency structure and traversal order must be adapted carefully for each variant.

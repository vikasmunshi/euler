Problem 87: Prime Power Triples [Level 3]

Problem Statement

The smallest number expressible as the sum of a prime square, prime cube, and prime fourth power is $28$. In fact, there are exactly four numbers below fifty that can be expressed in such a way:

$$\begin{align} 28 &= 2^2 + 2^3 + 2^4\\ 33 &= 3^2 + 2^3 + 2^4\\ 49 &= 5^2 + 2^3 + 2^4\\ 47 &= 2^2 + 3^3 + 2^4 \end{align}$$

How many numbers below fifty million can be expressed as the sum of a prime square, prime cube, and prime fourth power?

Mathematical Complexity

This problem belongs to additive number theory: I need to count integers below a limit N that can be expressed as p^2 + q^3 + r^4 for primes p, q, r. The three exponents impose natural ceilings on the size each base prime can reach before the partial sum alone exceeds N. For N = 50,000,000 the fourth-power base is capped at roughly N^(1/4) ~ 84, the cube base at N^(1/3) ~ 368, and the square base at N^(1/2) ~ 7071, so the outer loops each visit only a small number of primes while the innermost loop carries the bulk of the iterations. A critical subtlety is that the same integer can be produced by more than one distinct prime triple, so a deduplication step is mandatory to avoid overcounting.

The asymptotic cost of the triple enumeration is O(pi(N^(1/4)) * pi(N^(1/3)) * pi(N^(1/2))), where pi is the prime counting function. Each factor is tiny relative to N, making the nested loop extremely fast in practice. The sieve used to produce candidate primes runs in O(N / 2) time and O(N / 2) space and dominates for large N, but it is still linear in N.

Solution Approach

Nested Prime-Power Enumeration with Deduplication (s0 - Python and C)

Both the Python and C implementations share the same core algorithm. I first generate all primes up to sqrt(N) using a Sieve of Sundaram, which operates on a half-sized boolean array and marks composite indices k of the form i + j + 2ij, then recovers odd primes as 2k + 1 and prepends 2. A single sieve run is sufficient to supply all three power families, since the largest base prime needed is bounded by sqrt(N).

I then iterate over the three families of prime powers in three nested loops ordered from the most constraining term outward: fourth powers in the outermost loop, cubes in the middle, squares in the innermost loop. Each loop breaks as soon as its running partial sum exceeds the remaining budget. I precompute two sentinel constants - max_quadruple (N minus the smallest possible cube and square, i.e. N - 8 - 4) and max_quadruple_cube (N minus the smallest possible square, i.e. N - 4) - so the break conditions involve no repeated arithmetic inside the loops.

The language-specific difference lies in deduplication. The Python version inserts every valid sum into a set and returns its length. The C version allocates a flat unsigned char array of length N: on the first visit to any value the byte is set to 1 and a counter is incremented; subsequent visits are silently ignored. The C approach avoids Python's hash-table overhead and replaces it with a 50 MB byte array that supports cache-friendly sequential writes. The benchmark confirms this: the C solution finishes in about 0.095 s versus roughly 0.405 s for Python - a speedup of approximately 4x on the main input.

Learnings

The key insight was recognising that the three exponents impose very different upper bounds on their respective prime bases. Ordering the loops from the most constraining exponent (fourth power) outward means each outer loop iterates over only a handful of values and each break fires as early as possible. Reversing the order - squares outermost, fourth powers innermost - would make the pruning far less effective because the square loop is the longest of the three.

Choosing the right deduplication structure was the other critical decision. In Python a set is the natural tool: it collapses multiple representations of the same integer at essentially no extra code cost. In C a flat boolean array indexed directly by value is faster and simpler than a hash set: for N = 50,000,000 the array is 50 MB, which fits comfortably in memory, and random writes into it are still fast enough that they do not become a bottleneck. The benchmark result of 0.095 s versus 0.405 s shows that Python set overhead and interpreted loop cost together account for roughly a 4x slowdown.

  • When enumerating multi-term sums, nest loops from the most restrictive term outward to enable the earliest possible pruning and minimise total iterations.
  • Precompute partial-sum budget constants outside inner loops to avoid redundant arithmetic on every iteration.
  • A set in Python and a flat boolean array in C are both clean deduplication tools; the array wins on speed when the value range is small enough to fit comfortably in memory.
  • The Sieve of Sundaram is a useful alternative to the Sieve of Eratosthenes when a compact half-sized array is preferable; the only extra cost is a 2k + 1 decode step when recovering primes.
  • For a problem where the answer must be a count of distinct values in a bounded integer range, a direct boolean array is often the simplest and fastest deduplication mechanism available in C.

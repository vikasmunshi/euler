Problem 89: Roman Numerals [Level 3]

Problem Statement

For a number written in Roman numerals to be considered valid there are basic rules which must be followed. Even though the rules allow some numbers to be expressed in more than one way there is always a "best" way of writing a particular number.

For example, it would appear that there are at least six ways of writing the number sixteen:

IIIIIIIIIIIIIIII
VIIIIIIIIIII
VVIIIIII
XIIIIII
VVVI
XVI

However, according to the rules only XIIIIII and XVI are valid, and the last example is considered to be the most efficient, as it uses the least number of numerals.

The 11K text file, roman.txt (right click and 'Save Link/Target As...'), contains one thousand numbers written in valid, but not necessarily minimal, Roman numerals; see About... Roman Numerals for the definitive rules for this problem.

Find the number of characters saved by writing each of these in their minimal form.

Note: You can assume that all the Roman numerals in the file contain no more than four consecutive identical units.

Mathematical Complexity

Roman numerals form an additive positional system with a subtractive notation twist: any symbol whose value is strictly less than the symbol immediately to its right contributes negatively to the total. The minimal representation of any positive integer is uniquely determined by a greedy canonical decomposition over 13 meaningful tokens - the seven base symbols (I, V, X, L, C, D, M) and the six two-character subtractive pairs (IV, IX, XL, XC, CD, CM). Converting a valid but non-minimal numeral to its minimal form therefore reduces to two independent sub-problems: parsing the string into a plain integer, then reconstructing that integer via a canonical greedy descent. Both operations are uniquely defined for every positive integer in the representable range, so there is no ambiguity to manage.

The asymptotic complexity is O(N * L), where N is the number of numerals in the file and L is the maximum string length per line. Because Roman numerals for numbers in the input range are bounded in length by a small constant, the per-numeral cost is effectively O(1), making the whole solution linear in the input size O(N). The greedy reconstruction loop iterates over a fixed lookup table of at most 28 entries a small constant number of times per number, so its per-number cost is also O(1) in practice.

Solution Approach

Greedy Parse-and-Reconstruct (s0 - Python and C)

I chose a two-phase strategy that is identical in logic across both the Python and C implementations, with only language-level differences in how the lookup table and file I/O are handled.

In the first phase, roman_to_number walks the numeral string in reverse, maintaining a running total and a last value. For each character, if its numeric value is less than last it is subtracted (subtractive notation), otherwise it is added. This is the standard O(L) reverse-scan algorithm for interpreting Roman numerals correctly without needing explicit look-ahead.

In the second phase, number_as_roman_numeral (Python) and number_as_roman_numeral_length (C) reconstruct the minimal form via a greedy descent through a static lookup table. I encoded every canonical partial value from 1 to 1000, including all subtractive pairs, as explicit entries. At each step the largest table entry not exceeding the remaining value is selected, its string is appended (or its length counted in C), and its value is subtracted. Because the table encodes all valid subtractive combinations, the output is guaranteed to be the shortest possible representation. The character saving per numeral is then just the difference between the original string length and the reconstructed length.

There are two language-specific differences worth noting. The Python version stores the lookup table as a dict[int, str] and uses a max() call with a generator expression to find the largest fitting entry on each iteration, which is idiomatic but involves allocating an intermediate generator on every loop step. The C version stores parallel arrays of values and strings sorted in descending order and uses a simple linear scan, which avoids any heap allocation during reconstruction - it also only counts the output length rather than building the string, since only the length difference matters for the final answer.

Learnings

The key insight that unlocked the problem quickly was recognising that I did not need to implement or understand the full Roman numeral validation ruleset. The problem statement guarantees all input is already valid, so I only needed two things: a reliable integer parser and a reliable minimal-form generator. Those two pieces are each simple and independently verifiable, which made the overall solution straightforward to get right on the first attempt.

A second useful realisation was that encoding every two-digit decimal position as its own table entry - rather than decomposing procedurally from raw digit values - keeps the reconstruction code trivially correct and eliminates any edge-case handling. This "canonical decomposition via fixed lookup table" pattern is worth remembering: when the output token set is finite and enumerable, a static greedy table is both simpler and faster than a procedural rule system. The same pattern appears in coin-change problems and unit-conversion problems where subsets of values have special combined forms.

The benchmark results show the C version completing in roughly 0.0001 s versus 0.0148 s for Python - approximately 148 times faster. The entire gap comes from Python's interpreter overhead and per-operation cost rather than any difference in the underlying algorithm. For a problem of this scale the Python version is already fast enough in absolute terms, but the C port illustrates how much overhead the Python runtime carries even for trivially small inputs.

Problem 85: Counting Rectangles [Level 2]

Problem Statement

By counting carefully it can be seen that a rectangular grid measuring $3$ by $2$ contains eighteen rectangles:

Although there exists no rectangular grid that contains exactly two million rectangles, find the area of the grid with the nearest solution.

Mathematical Complexity

The key combinatorial insight is that every rectangle in an m-by-n grid is uniquely determined by choosing two distinct horizontal grid lines and two distinct vertical grid lines. An m-row grid has (m+1) horizontal lines, giving C(m+1, 2) = m(m+1)/2 ways to pick a horizontal pair, and similarly n(n+1)/2 for the vertical pair. So the total rectangle count is T(m) * T(n), where T(k) = k(k+1)/2 is the k-th triangular number. This factorisation decouples the two dimensions entirely and is the structural property that makes the problem tractable.

The search space is bounded because T(k) grows as O(k^2), so T(m)*T(n) reaches two million well before either dimension exceeds roughly 100 units. A naive approach that re-derives the count by summing all sub-rectangle positions costs O(h*w) per pair and O(S^4) overall for a grid of side bound S. Applying the closed form brings this down to O(S^2) with O(1) per pair. The optimised approach precomputes the triangular sequence and uses binary search, reducing the height scan to O(log S) per width and giving O(S log S) overall.

Solution Approaches

Brute-Force Double Product Scan (s0)

In the Python version I used itertools.product to iterate over all (height, width) pairs and computed the rectangle count with a nested generator that explicitly sums (height - h + 1) * (width - w + 1) over all sub-rectangle positions - effectively re-deriving T(h)*T(w) term by term rather than applying the closed form. Results were accumulated in a dictionary keyed by rectangle count, and at the end I selected the entry closest to the target with min(..., key=delta). An early break fires when the running count falls within max_error of the target. The C version follows the same nested loop structure with explicit inner loops, tracking the best delta inline rather than using a dictionary, and uses goto done to break out of the double loop early. Both versions are correct but carry the O(h*w) cost of the inner sum for every pair: the Python implementation takes about 0.23 s on the main case and the C implementation takes about 0.0014 s.

Precomputed Triangular Numbers with Binary Search (s1)

I first built a sequence numbers where numbers[k-1] = T(k) = k(k+1)/2 for each side length from 1 to max_side - 1. Then for each width I retrieved its triangular value num_width and used bisect_left to find the insertion point for target // num_width in the sorted triangular-number sequence. Because the insertion point j marks the first index where the value is at least the quotient, the exact solution for the height lies at either index j or j+1 in zero-indexed terms - I checked both one-indexed heights j+1 and j+2, which straddles any exact solution. The Python version uses bisect.bisect_left from the standard library and stores candidates in a dictionary; the C version implements its own binary search with a straightforward lo/hi loop and tracks the running best delta inline. Both versions return the area of the closest grid found. On the main case the Python version runs in about 0.0004 s and the C version is effectively instant at under 0.0001 s - roughly 570x and 14000x faster than their s0 counterparts respectively.

Learnings

The key insight that unlocked the problem was recognising that the rectangle count factors completely along each axis as T(m) * T(n). Before seeing that, I was recomputing an O(h*w) double sum for every pair, which is both slow and hides the structure. Once I had the factorisation, the 2-D search collapsed into a 1-D nearest-value lookup for each width - exactly the scenario where binary search on a precomputed sorted list shines.

Comparing the approaches using the benchmark results: s0 in Python averages about 0.23 s on the main case while s1 in Python averages about 0.0004 s, a speedup of roughly 570x. In C, s0 averages 0.0014 s while s1 is under 0.0001 s, a speedup of over an order of magnitude. The gain comes from two compounding improvements: replacing the O(h*w) inner sum with an O(1) precomputed triangular-number lookup, and replacing an O(S) height scan with an O(log S) bisect. The s1 approach is both faster and more elegant because it makes the mathematical structure of the problem visible in the code itself.

  • Counting rectangles in a grid reduces to choosing pairs of grid lines: the result is always a product of two triangular numbers T(m) * T(n).
  • When one dimension of a 2-D search is monotone given the other, binary search converts an O(S^2) scan into O(S log S).
  • bisect_left returns the insertion point; checking the element just before and just after that point covers both neighbours of a target value in a sorted sequence.
  • Precomputing an intermediate sequence and binary-searching it is a broadly applicable pattern whenever a closed form exists but iteration is otherwise the natural approach.
  • The C implementations show the same algorithmic gains as Python but with much lower absolute times; the relative speedup between s0 and s1 is consistent across both languages.

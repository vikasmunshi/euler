Problem 25: $1000$-digit Fibonacci Number [Level 1]

Problem Statement

The Fibonacci sequence is defined by the recurrence relation:

$F_n = F_{n - 1} + F_{n - 2}$, where $F_1 = 1$ and $F_2 = 1$.

Hence the first $12$ terms will be:

$$\begin{align} F_1 &= 1\\ F_2 &= 1\\ F_3 &= 2\\ F_4 &= 3\\ F_5 &= 5\\ F_6 &= 8\\ F_7 &= 13\\ F_8 &= 21\\ F_9 &= 34\\ F_{10} &= 55\\ F_{11} &= 89\\ F_{12} &= 144 \end{align}$$

The $12$th term, $F_{12}$, is the first term to contain three digits.

What is the index of the first term in the Fibonacci sequence to contain $1000$ digits?

Test Cases

  • dev: solve(n=3) → 12
  • main: solve(n=1000) → 4782

Mathematical Complexity

The core idea is translating a digit-count requirement into an inequality on Fibonacci values. A positive integer k has exactly floor(log10(k)) + 1 decimal digits, so finding the first index where F_n reaches 1000 digits is equivalent to finding the smallest n such that F_n >= 10^999. Because the Fibonacci sequence grows as F_n ~ phi^n / sqrt(5) where phi = (1 + sqrt(5)) / 2 ~= 1.618, taking base-10 logarithms gives an analytic estimate n ~= 999 / log10(phi) ~= 4782, which the iterative solution confirms exactly.

Each addition step involves numbers that grow to roughly 1000 decimal digits by the end of the loop, so each individual step costs O(d) time where d is the current digit count. Because the digit count grows roughly linearly with the step index (approximately one new decimal digit every 1 / log10(phi) ~= 4.78 steps), the total work over all ~4782 iterations is O(n * d) = O(d^2) where n is proportional to d. For d = 1000 this is entirely feasible and completes in milliseconds.

Solution Approach

Iterative Big-Integer Fibonacci (s0 - Python and C)

I maintained just two rolling variables a and b representing consecutive Fibonacci numbers, advancing them with the standard recurrence and incrementing a counter on each step. The loop exits as soon as the current value reaches the digit threshold. I initialised both variables to 1 with the counter at 2 to match the problem's 1-indexed convention where F_1 = F_2 = 1, which the dev test case (first 3-digit Fibonacci at index 12) confirms is correct.

In the Python version I used b < 10 ** (n - 1) as the loop condition, which is a clean mathematical statement equivalent to "b has fewer than n digits". Python's built-in arbitrary-precision integers handle all big-number arithmetic without any special libraries, making the implementation extremely concise.

The C version required me to implement big-integer arithmetic manually. I chose a base-10^9 limb representation, storing each number as an array of unsigned long long limbs where each limb holds up to 9 decimal digits. Addition iterates over limbs with carry propagation in the standard way. I allocated 200 limbs, giving a ceiling of 1800 digits - comfortably above the 1000-digit target. Rather than precomputing 10^999 as a big integer for a threshold comparison, I used a digit-count function that inspects the most significant limb and adds 9 for each lower limb, which is simpler to express in C. Two extra BigInt structs act as ping-pong buffers with memcpy advancing the pair each step.

Learnings

The key insight was recognising that the digit-count threshold translates directly into a comparison against a power of ten, which terminates the loop without any string conversion. In Python this was almost trivial to write correctly because native integers are arbitrary-precision; I just wrote the recurrence and let the runtime handle the rest. The analytic estimate n ~= (d - 1) / log10(phi) is a useful sanity check and could even be used to compute the answer in a single floating-point expression, though care around off-by-one rounding would be needed.

Comparing benchmark results for the main case, the C solution ran in roughly 0.001 s versus 0.014 s for the Python solution - about a 14x speedup. The gap is modest in absolute terms because the problem is small, but it reflects the overhead of Python's interpreted big-integer arithmetic versus the tight limb-loop in C. The Python solution is considerably more concise and readable, so for a problem at this scale I would normally prefer it; the C version is a useful exercise in manual big-integer implementation.

  • Python's native bignum makes Fibonacci problems of this kind almost trivial - there is nothing special to configure.
  • Comparing against a power-of-ten threshold is cleaner than repeated string conversion in a tight loop.
  • When implementing big integers in C, a base-10^9 limb layout is convenient because digit counting only requires inspecting the top limb rather than converting between bases.
  • Keeping only two rolling variables rather than a full list is sufficient whenever only the current Fibonacci value is needed, saving memory and simplifying the code.
  • The closed-form growth rate of Fibonacci via the golden ratio provides a fast sanity check on any iterative result and can double as a near-exact analytic solution.

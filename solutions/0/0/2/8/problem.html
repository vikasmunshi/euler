Problem 28: Number Spiral Diagonals [Level 0]

Problem Statement

Starting with the number $1$ and moving to the right in a clockwise direction a $5$ by $5$ spiral is formed as follows:

21 22 23 24 25
20 7 8 9 10
19  6 1 2 11
18 5 4 3 12
17 16 15 14 13

It can be verified that the sum of the numbers on the diagonals is $101$.

What is the sum of the numbers on the diagonals in a $1001$ by $1001$ spiral formed in the same way?

Mathematical Complexity

The spiral grows outward from a central 1 in concentric rings. For ring k (k = 1 being the first ring around the centre), the side length of the bounding square is 2k+1, so the largest corner value is (2k+1) 2 . The other three corners of that ring are each separated by a step of 2k, giving (2k+1) 2 - 2k, (2k+1) 2 - 4k, and (2k+1) 2 - 6k. Summing all four corners for ring k yields 16k 2 + 4k + 4. Summing this over k from 1 to n where n = (s-1)/2, and adding the central 1, reduces to a closed-form cubic via the standard sum-of-squares identity (sum k 2 = n(n+1)(2n+1)/6) and the sum-of-integers identity. After substituting back in terms of the original side length s, the total diagonal sum is (s * (s * (4s + 3) + 8) - 9) / 6. This closed-form evaluation is O(1) - it performs a fixed number of arithmetic operations regardless of spiral size.

I also implemented a simulation path used only for small visualisation grids. It places s 2 values into a coordinate map one by one, each step scanning at most four neighbours, giving O(s 2 ) overall. For the puzzle input of side 1001 that would mean over one million operations and is only present as a development and correctness aid, not as a serious solver.

Solution Approach

Closed-form Polynomial Formula (Python and C)

Both p0028_s0.py and p0028_s0.c implement the same O(1) formula. I derived it by writing out the four corner values per ring symbolically, summing them, and applying textbook summation identities. The final expression is evaluated in Horner's method form as (s * (s * (4 * s + 3) + 8) - 9) // 6, which keeps intermediate values small and makes the cubic structure explicit. I verified that divisibility by 6 always holds for positive odd inputs. Both implementations pass all test cases and report elapsed times of 0.0000 seconds, reflecting the essentially zero computational cost of evaluating a single arithmetic expression.

The Python version also carries an optional simulation path that activates when the --show flag is passed and the requested size is 10 or smaller. In that mode I build the spiral by maintaining a dictionary mapping (x, y) coordinates to values. At each step I filter the four cardinal neighbours of the current position to those not yet occupied, then select the candidate with the smallest squared distance from the origin. This proximity-to-origin tie-breaker naturally traces the clockwise outward path without requiring any explicit direction state. After construction the grid is rendered to the terminal with ANSI colour highlighting on diagonal cells, and the summed result is cross-checked against the formula with a coloured tick or cross. The C implementation mirrors this simulation for the same small-size visualisation case, but uses a flat integer array instead of a hash map, and shares the same ANSI rendering logic.

Learnings

The key insight was recognising that I did not need to simulate the spiral at all. Writing out the four corner values for the first two or three rings immediately revealed a quadratic pattern in the per-ring contribution, and from there applying standard summation identities was mechanical. The productive move was grouping the four diagonal values of each ring together and treating the ring index k as the summation variable, rather than trying to reason about individual cell positions.

Only one algorithmic approach - the formula - is implemented, shared across Python and C. Both are effectively instantaneous on every test case, so there is no meaningful performance distinction to draw. The Python version carries extra machinery for the simulation and visualisation path, but that code is never reached for the actual puzzle input.

Key concepts worth keeping in mind for future problems:

  • When a 2-D structure grows by adding concentric rings, the corner or edge values almost always follow a polynomial pattern in the ring index - reach for algebraic summation before simulation.
  • The proximity-to-origin heuristic is a clean spiral tracer that requires no direction variable, but it is O(n 2 ) per step and is only practical for small visualisation grids.
  • Keeping a simulation path alongside the formula during development is worth the small extra coding effort: it acts as an independent correctness oracle and produces a readable grid rendering that makes the diagonal pattern visually obvious.
  • Writing polynomials in Horner's method form - (s * (s * (4 * s + 3) + 8) - 9) // 6 - is a good habit for clarity and avoids intermediate large values when working in fixed-width integer types such as C's long long.

Problem Analysis

We are asked to find the one-millionth arrangement of the digits 0 through 9 when all such arrangements are listed in lexicographic order - that is, dictionary order, where we compare element by element from left to right. A naive approach would generate all permutations in order and stop at the millionth, but the answer reveals a much more elegant structure. The total number of arrangements of n distinct symbols is n! (the factorial of n). For ten digits that is 3,628,800 permutations, so generating a million of them is feasible but wasteful.

The key observation is the factorial number system: among all permutations sorted in lexicographic order, exactly (n-1)! of them begin with each possible leading digit. Therefore, to find the permutation at rank k (counting from zero), I divide k by (n-1)!. The quotient tells me the index, into the sorted list of remaining digits, of the digit to place first; the remainder is the rank of the desired permutation among those sharing that leading digit. Removing the chosen digit leaves a strictly smaller, identically structured subproblem. This technique - converting a rank into its corresponding object directly - is called unranking. The complexity is O(n^2): we make n digit selections, and each selection rebuilds the shrinking digit list in linear time. For n = 10 this is trivially fast.

Solution Approach

Factorial-Decomposition (Recursive Unranking) - Algorithm 0

Both the Python and C implementations realise the same recursive algorithm, which mirrors the mathematical structure directly: at each level of recursion one digit is fixed and the remainder of the work is an identical, smaller problem. I chose recursion precisely because the subproblem after fixing a digit is the original problem in miniature.

Each call receives the sorted string of available digits and a 1-based target rank. I compute fact = (len - 1)!, then convert the rank to 0-based and divide: current = (permutation_number - 1) / fact gives the index of the digit to place here, and remaining = (permutation_number - 1) % fact gives the sub-rank. I remove digits[current] from the list and recurse on the smaller list with rank remaining + 1 (converting back to 1-based). The base case is a single remaining digit, which I return directly. The chosen digit is prepended to the result of the recursive call as the call stack unwinds.

In Python, removing the chosen digit is a single clean slice expression, digits[:current] + digits[current + 1:], and the factorial comes from math.factorial. In C, where strings are raw character buffers, I allocate a new array, copy every character except the chosen one, recurse, then build the result by writing the chosen character followed by a strcpy of the sub-result. The C version implements its own factorial loop returning long long, which comfortably holds 9! = 362880, and carefully frees both the intermediate digit buffer and the sub-result at each level to avoid leaks.

Programming Techniques

Divmod for Quotient and Remainder Together

Python's built-in divmod(a, b) returns the quotient and remainder of integer division as a tuple in one call, which I unpack as current, remaining = divmod(permutation_number - 1, math.factorial(len(digits) - 1)). This is both efficient and expressive: it makes plain that the same division produces both pieces of information the algorithm needs. In C I write the two operations separately with / and %. Reach for this idiom in any number-system conversion, pagination, or rank/unrank routine where you need both results.

Recursive Decomposition of Self-Similar Problems

Both solutions are expressed as direct recursion rather than an explicit loop with an accumulator, because the problem has clean optimal substructure - fixing one digit yields a smaller instance of the same problem. The recursion depth is bounded by the input length (at most 10), so stack overflow is not a concern even in languages without tail-call optimisation. When a problem decomposes naturally into a single smaller instance of itself, recursion produces code whose shape matches the mathematics.

1-Based to 0-Based Index Conversion at the Boundary

The permutation number is given as a 1-based count. I subtract 1 immediately, perform all the division and modulo arithmetic in 0-based space, and add 1 back only when handing the sub-rank to the next recursive call. Doing the conversion at the boundary of the computation keeps the core arithmetic free of off-by-one corrections. This convert-at-the-edges discipline applies whenever human counting (starting at 1) meets array indexing or modular arithmetic (starting at 0).

Heap Allocation for Computed-Length Strings in C

Because the result is assembled character by character as recursion unwinds, its length is known only at runtime. At each level I malloc exactly the bytes I need - len for the reduced digit buffer, len + 1 for the result including its null terminator - and free each buffer as soon as it is no longer referenced. Pairing every allocation with a matching free in the same logical scope makes ownership clear and prevents leaks. This is the standard pattern for C functions that must return strings of computed length.

Standard Library Over Reinvention

The Python solution uses math.factorial rather than a hand-written loop: it is exact for arbitrary-size integers, implemented in C, and signals intent clearly. The C standard library offers no factorial, so there I write a trivial loop - but only because no library equivalent exists. As a general rule, prefer well-tested standard library functions for standard mathematical operations unless precision, performance, or portability gives you a concrete reason to do otherwise.

Key Takeaways

The insight that unlocks this problem is that lexicographic rank can be decoded digit by digit using factorial arithmetic, sidestepping enumeration entirely. Recognising that (n-1)! permutations begin with each leading digit turns a search into a short sequence of divisions. Both implementations finish far below human-perceptible time: on the main case (the millionth permutation of all ten digits, averaged over 21 runs) the C version takes about 0.4 microseconds and the Python version about 6.9 microseconds. The C version is thus roughly eighteen times faster in absolute terms, but the gap is immaterial - the work itself is negligible, so it is the algorithmic choice, not the language, that makes the problem trivial at this scale. The Python version is the more elegant of the two, expressing the whole algorithm in a handful of lines with slicing and divmod, while the C version is instructive for showing the explicit memory management the same logic requires.

  • When objects are ordered in lexicographic order, the factorial number system lets you unrank any position directly in O(n) selections - never enumerate permutations one at a time.
  • Recursion shines when a problem reduces to a single smaller instance of itself and the depth is small and bounded; the code then mirrors the mathematical definition.
  • Convert between 1-based and 0-based representations explicitly and at the boundaries of your computation to keep the core arithmetic clean.
  • Use divmod in Python (or paired / and % in C) whenever you need both quotient and remainder - a recurring pattern in number-system and rank/unrank problems.
  • In C, allocate exactly what you need and free it as early as possible, pairing every malloc with a free in the same scope to make ownership unambiguous.
  • Before writing brute force, ask whether the problem's ordering structure admits a direct formula; combinatorial rank and unrank problems almost always do.

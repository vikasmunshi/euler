Problem 27: Quadratic Primes [Level 1]

Problem Statement

Euler discovered the remarkable quadratic formula:

$n^2 + n + 41$

It turns out that the formula will produce $40$ primes for the consecutive integer values $0 \le n \le 39$. However, when $n = 40, 40^2 + 40 + 41 = 40(40 + 1) + 41$ is divisible by $41$, and certainly when $n = 41, 41^2 + 41 + 41$ is clearly divisible by $41$.

The incredible formula $n^2 - 79n + 1601$ was discovered, which produces $80$ primes for the consecutive values $0 \le n \le 79$. The product of the coefficients, $-79$ and $1601$, is $-126479$.

Considering quadratics of the form:

$n^2 + an + b$, where $|a| < 1000$ and $|b| \le 1000$

where $|n|$ is the modulus/absolute value of $n$
e.g. $|11| = 11$ and $|-4| = 4$

Find the product of the coefficients, $a$ and $b$, for the quadratic expression that produces the maximum number of primes for consecutive values of $n$, starting with $n = 0$.

Mathematical Complexity

The central number theory here is primality testing applied across a parametric family of quadratic polynomials of the form f(n) = n² + an + b. The key structural observation is that evaluating at n = 0 gives f(0) = b, so b must itself be prime for any run of consecutive primes to begin at n = 0. This collapses the candidate set for b from the 2001 integers in [-1000, 1000] down to the roughly 168 primes up to 1000 - a 12x reduction before any polynomial evaluation takes place. A secondary parity argument applies once b is known to be an odd prime: f(1) = 1 + a + b must also be prime, and since 1 + b is even, a must be odd to keep the sum odd and thus a candidate prime. This halves the search space for a as well. The special case b = 2 is handled separately with an unrestricted step for a.

The asymptotic cost is O(P(L) * L * R * sqrt(V)), where P(L) ~ L / ln L is the count of primes up to the limit L (bounding b), L covers the range of a, R is the average prime run length per candidate pair, and sqrt(V) is the trial-division cost per primality test. In practice the runs are short and trial division exits early, so the total work is well under a second in both Python and C.

Solution Approach

Constrained Brute-Force with Sieve-Pruned Search Space (s0 - Python and C)

I used a Sundaram sieve to pre-generate all primes up to max_limit and restricted the candidate values for b exclusively to that list, exploiting the structural constraint that f(0) = b must be prime. For each prime b, I iterated a starting at 1 with a step of 2 to enforce the odd- a parity constraint for odd prime b, or starting at 0 with a step of 1 for the special case b = 2. Rather than running four separate outer loops over sign combinations, I evaluated all four sign variants - (a, b), (a, -b), (-a, -b), and (-a, b) - within each iteration of the inner loop. The helper prime_run(a, b) counts how many consecutive values of n starting from zero produce a prime under |n² + an + b|, using trial-division is_prime with early exits for even numbers.

In the Python version I folded the four sign combinations into a single max() call over tuples of the form (run_length, a*b_product). Python's lexicographic tuple comparison means the run length acts as the primary sort key and the product as the tiebreaker, so no explicit best-so-far variable is needed. The outer list comprehension produces one such winning tuple per (a, b) pair, and a final max() selects the global winner; indexing [1] extracts the desired product. The C version implements exactly the same algorithm but tracks best_run and best_product explicitly with conditional updates, replicating Python's tuple-comparison tie-breaking by keeping the larger product on equal run lengths. Both versions produce the answer -59231.

Learnings

The most valuable insight was recognising that b is structurally forced to be prime by the requirement that the consecutive run starts at n = 0. Generating candidate values for b with a Sundaram sieve and iterating only over that list eliminated roughly 92% of the naive search space before a single polynomial was evaluated. The follow-on parity argument - that for odd prime b, the coefficient a must be odd so that f(1) has a chance of being prime - then halved the remaining work for essentially no implementation cost.

Since there is only one algorithmic approach here, there is no algorithmic comparison to make. The Python implementation runs in about 0.476 s and the C implementation in about 0.018 s, a roughly 26x speedup, driven by the lower per-call overhead of trial division and loop dispatch in compiled code. The Python time is already comfortable for a Level 1 problem, so the C version is present purely for archival completeness.

  • Always evaluate a formula at its boundary inputs first - here n = 0 forces b to be prime, collapsing the search space dramatically before any search begins.
  • Parity arguments are cheap to derive and can halve a search space essentially for free; always check whether an output being odd constrains the parity of an input parameter.
  • Folding sign-combination variants into a single max() call over tuples is an elegant Python idiom for "find the X that maximises Y, return Z" problems without writing multiple near-identical loops.
  • Comparing tuples lexicographically with the target metric first and the desired output second avoids maintaining explicit best-so-far variables in Python.
  • The Sundaram sieve is a practical alternative to the Eratosthenes sieve for generating a prime list; it produces only odd primes naturally and requires appending 2 as a special case, which fits naturally here since b = 2 already requires separate handling of the step size for a.

Problem 26: Reciprocal Cycles [Level 2]

Problem Statement

A unit fraction contains $1$ in the numerator. The decimal representation of the unit fractions with denominators $2$ to $10$ are given:

$$\begin{align} 1/2 &= 0.5\\ 1/3 &=0.(3)\\ 1/4 &=0.25\\ 1/5 &= 0.2\\ 1/6 &= 0.1(6)\\ 1/7 &= 0.(142857)\\ 1/8 &= 0.125\\ 1/9 &= 0.(1)\\ 1/10 &= 0.1 \end{align}$$

Where $0.1(6)$ means $0.166666\cdots$, and has a $1$-digit recurring cycle. It can be seen that $1/7$ has a $6$-digit recurring cycle.

Find the value of $d \lt 1000$ for which $1/d$ contains the longest recurring cycle in its decimal fraction part.

Test Cases

  • main: solve(limit=1000) → 983

Mathematical Complexity

The recurring cycle length of 1/d in decimal is determined by the multiplicative order of 10 modulo d. When the long-division algorithm is applied to 1/d, the state at each step is entirely captured by the current remainder. Because remainders lie in the finite set {0, 1, ..., d-1}, they must eventually repeat, and the cycle begins precisely when a remainder recurs. When gcd(d, 10) = 1, the first remainder is 1 and the cycle closes when 1 reappears; the length of that cycle is the smallest positive integer k such that 10^k ≡ 1 (mod d), which is the multiplicative order ord_d(10). Denominators sharing a factor with 10 only introduce a finite non-repeating prefix; the repeating tail still originates in the coprime reduced part, so restricting attention to gcd(d, 10) = 1 is both correct and sufficient for finding the longest cycle.

Computing the multiplicative order of 10 modulo d naively requires up to d-1 iterations of r = r * 10 mod d, making it O(d) per candidate. Checking all candidates up to a limit L would therefore be O(L^2) in the worst case. The top-down heuristic I apply reduces the empirical search space to roughly L/10 candidates, bringing practical cost well below the theoretical ceiling for the problem's limit of 1000.

Solution Approach

Multiplicative-Order Search from the Top (s0 - Python and C)

I chose to exploit the observation that the maximum possible cycle length for 1/d is d-1, achieved when d is prime and 10 is a primitive root mod d. This means denominators near the top of the search range are the most likely candidates for the longest cycle. Rather than scanning all d from 2 up to the limit, I scan only the top 10% of the range - from limit down to limit - limit//10 - keeping only candidates where gcd(d, 10) = 1. For each such d I compute multiplicative_order(10, d), which iterates successive powers of 10 modulo d until the result equals 1, returning None (Python) or -1 (C) if no order is found within d-1 steps.

The Python version expresses the search as a single max() call over a generator of (order, d) tuples, relying on Python's lexicographic tuple comparison to handle ties in favour of the larger d naturally. The walrus operator ( d := limit - i) keeps the generator concise. The C version mirrors exactly the same loop logic with explicit comparisons and uses long long arithmetic inside multiplicative_order to prevent silent overflow when computing r * a % modulus, since even a modulus that fits in an int can produce an intermediate product that overflows one. Both implementations produce identical answers of 983 for the main case.

Learnings

The key insight was recognising that long division is modular arithmetic in disguise: tracking the remainder sequence under repeated multiplication by 10 modulo d is precisely how the decimal expansion is generated. Translating "find the longest recurring decimal cycle" into "find the largest multiplicative order of 10 over candidates up to the limit" turned a digit-simulation problem into a clean number-theory problem.

The practical optimisation insight was that the answer almost always lives near the top of the search range, because a prime p close to the limit can have a cycle as long as p-1, and no d smaller than that prime can beat it. Scanning only the top 10% is therefore a well-motivated heuristic, not an arbitrary shortcut - though it does assume the answer is not an outlier far below the limit, which is worth remembering if the approach were generalised to unusual limits.

Both solutions use the same algorithm, so the benchmark comparison is a pure language-overhead measurement. The C version runs in roughly 0.1 ms versus 0.7 ms for Python - about a 7x speedup - but both are comfortably under 1 ms in absolute terms, so the choice between them is a matter of language preference for this problem size.

  • Multiplicative order ord_n(a) is the foundation of cyclic decimal expansion theory and reappears in many number-theory problems involving periodicity.
  • Filtering early by gcd(d, 10) = 1 eliminates all denominators whose only effect is to shift the decimal point, reducing work without missing any candidate for the longest cycle.
  • Searching from the top of a bounded range is a cheap but powerful optimisation whenever the objective function is positively correlated with the magnitude of the parameter.
  • Python's lexicographic tuple comparison makes argmax with a tie-breaking key trivially expressible as max((value, key) for ...)[1], avoiding explicit conditional logic.
  • In C, intermediate products in modular multiplication loops should use long long even when the modulus fits in an int, to prevent silent overflow in the product before the modulo reduction.

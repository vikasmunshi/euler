Problem 22: Names Scores [Level 0]

Problem Statement

Using names.txt (right click and 'Save Link/Target As...'), a 46K text file containing over five-thousand first names, begin by sorting it into alphabetical order. Then working out the alphabetical value for each name, multiply this value by its alphabetical position in the list to obtain a name score.

For example, when the list is sorted into alphabetical order, COLIN, which is worth $3 + 15 + 12 + 9 + 14 = 53$, is the $938$th name in the list. So, COLIN would obtain a score of $938 \times 53 = 49714$.

What is the total of all the name scores in the file?

Mathematical Complexity

The arithmetic here is elementary. Every uppercase ASCII letter maps to its 1-based alphabet position via ord(c) - 64, because 'A' sits at code point 65. Summing those values across all characters in a name gives its alphabetical score; multiplying that score by the name's 1-based rank in the sorted list gives its contribution to the total. There is no number theory, combinatorics, or dynamic programming involved - this is a file-parsing and string-sorting exercise wrapped in straightforward arithmetic.

The dominant step is the initial lexicographic sort, which runs in O(N log N) where N is the number of names (roughly 5 000 in the supplied file). Scoring every name afterward costs O(N * L) where L is the average name length, a small constant in practice. Total complexity is therefore O(N log N + N * L), and because both N and L are modest the entire computation finishes well under a second in either language.

Solution Approach

Sort-then-enumerate (Python and C, index 0)

Both implementations share the same algorithm: read the raw comma-separated file into memory, split it into individual name tokens, sort them lexicographically, then iterate over the sorted list with a 1-based rank counter and accumulate rank-times-score for each name. The only differences are language-level implementation details.

In Python I resolve the resource file path relative to the script using Path(__file__).parent, so the solution runs correctly from any working directory. Splitting the file content on commas yields raw quoted tokens that are passed directly to sorted(). I then feed the sorted iterator into enumerate(..., start=1) to produce 1-based ranks without any manual counter variable, and wrap everything in a single sum() generator expression to avoid building intermediate lists. The helper word_to_num strips surrounding quote characters with .strip('"') before summing letter values, and I decorated it with @functools.lru_cache as a low-cost defensive habit for a pure string-to-integer function, even though no name repeats in practice.

In C I replicate the same steps manually. get_text_file derives a resource path relative to the executable directory using readlink("/proc/self/exe") and dirname, then reads the entire file into a heap-allocated buffer in one fread call. I estimate the name count by counting commas, allocate a char ** array of that size, and tokenise with strtok on commas. Standard library qsort with a strcmp comparator handles the sort. The inline word_to_num function skips quote and space characters and subtracts 64 from each remaining character, matching the Python logic exactly. I chose not to add a caching layer in C because the per-name scoring overhead is negligible for this input size, and a hash map would add complexity with no measurable benefit.

Learnings

The central insight is that ord(c) - 64 gives the alphabet position of any uppercase ASCII letter with zero overhead - no lookup table or dictionary needed. Keeping that expression inside a generator fed directly into sum() keeps the Python solution concise and avoids allocating any intermediate list.

Only one algorithm exists across both language implementations, so there is no algorithmic comparison to make. The benchmark does show a meaningful speed gap: the C build finishes in roughly 0.0011 s versus 0.0029 s for Python - a factor of about 2.6 on this run. For a problem of this scale the difference is almost entirely interpreter startup and import overhead rather than any meaningful algorithmic distinction, which confirms the Python solution is already as efficient as it needs to be.

A few techniques from this problem are worth carrying forward:

  • ord(c) - 64 - zero-overhead alphabet-position mapping for uppercase ASCII, no lookup table needed.
  • enumerate(iterable, start=1) - eliminates a manual counter variable for 1-based indexing and makes intent self-documenting.
  • Combining sorted() with a generator inside sum() keeps the Python solution to a single expression with minimal intermediate allocations.
  • Locating sibling resource files via Path(__file__).parent in Python and via readlink("/proc/self/exe") with dirname in C makes solutions runnable from any working directory without hardcoded absolute paths.
  • Applying @lru_cache to pure string-to-integer helper functions is a low-cost defensive habit even when the cache is unlikely to fire on a given input.

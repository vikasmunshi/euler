Problem 20: Factorial Digit Sum [Level 0]

Problem Statement

$n!$ means $n \times (n - 1) \times \cdots \times 3 \times 2 \times 1$.

For example, $10! = 10 \times 9 \times \cdots \times 3 \times 2 \times 1 = 3628800$,
and the sum of the digits in the number $10!$ is $3 + 6 + 2 + 8 + 8 + 0 + 0 = 27$.

Find the sum of the digits in the number $100!$.

Test Cases

  • dev: solve(n=10) → 27
  • main: solve(n=100) → 648

Mathematical Complexity

This problem sits in the domain of arbitrary-precision integer arithmetic. The mathematical content is minimal: computing a factorial is a straightforward multiplicative reduction over the integers from 2 to n, and summing decimal digits is a linear scan over the result. The only real challenge is that 100! has 158 digits, far beyond the range of any fixed-width integer type, so the interesting question is how to represent and multiply numbers of unbounded size. By Stirling's approximation, n! has roughly O(n log n) digits, so the naive algorithm requires O(n) multiplications each acting on an O(n log n)-digit number, giving an overall time complexity of O(n 2 log n) digit operations.

For n = 100 this is entirely negligible - the problem sits firmly in the trivial complexity class. The digit-sum step adds O(n log n) work, which is dominated by the factorial computation itself. There is no number-theoretic trick required; the only thing to get right is ensuring multiplication does not overflow the chosen representation.

Solution Approach

Big-Integer Factorial with Decimal Digit Array (Python and C, index 0)

Both solutions implement the same conceptual algorithm - compute n! exactly and sum its decimal digits - but they reach it through very different mechanisms. The Python solution I wrote exploits the standard library entirely. math.factorial returns an exact arbitrary-precision integer using Python's native int type, which silently promotes to unlimited precision with no effort on my part. I convert the result to a decimal string with str() and accumulate the digit characters using a generator expression passed to sum(). The entire algorithm collapses to a single return statement. I accepted n as a keyword parameter so the same function handles both the dev case (n=10, expected 27) and the main case (n=100, expected 648) without any hard-coding.

The C solution required me to implement the big-integer layer myself, since C has no built-in arbitrary-precision type. I stored the factorial as a little-endian array of single decimal digits - index 0 is the least significant digit - in a fixed buffer of MAX_DIGITS = 512 elements, which is more than enough for 100!, which needs only 158. The multiplication loop walks over the digit array, multiplies each element by the current factor i, adds any carry from the previous position, stores the new digit as prod % 10, and propagates carry = prod / 10. After the main digit positions are exhausted I expand the array by continuing to drain any remaining carry. Tracking num_digits as a running high-water mark avoids scanning unnecessary leading zeros on every iteration. The digit sum is then a trivial accumulation over the same array. Working directly in base 10 makes the digit extraction free - there is no final base-conversion step needed, unlike a base-2 32 big-integer representation.

Learnings

The core insight is recognising that the language runtime can do all the heavy lifting in Python. Trusting math.factorial and the arbitrary-precision int type reduces the solution to a one-liner. The key lesson was simply not reaching for anything more complicated when the standard library already provides exactly what is needed.

Writing the C solution was instructive for understanding what Python is doing behind the scenes. Choosing base 10 for the digit array was a deliberate design decision: it makes digit extraction O(1) per digit rather than requiring a base-conversion pass at the end, at the cost of slightly less efficient multiplication compared to a power-of-two base. For a problem this small the trade-off is irrelevant, but the pattern is worth remembering when implementing manual big-integer arithmetic in low-level languages in future problems.

Both solutions produce the correct answer and both benchmark at effectively 0.0000 seconds, so there is no meaningful performance distinction at n = 100. A few techniques worth carrying forward:

  • The sum(int(d) for d in str(value)) idiom in Python is directly reusable for any problem that asks for the digit sum of a large computed integer, and it recurs in several later Euler problems.
  • Python's int type never overflows - there is never a need to import a separate big-integer library for pure arithmetic tasks.
  • When implementing big-integer multiplication in C, storing digits in little-endian order (least significant first) simplifies carry propagation because extending the array only ever appends to the high end.
  • Parameterising the solver with n rather than hard-coding 100 and validating against the dev case (n=10, expected 27) is a cheap defensive check that confirms both the factorial and the digit-sum logic are correct before committing to the main case.

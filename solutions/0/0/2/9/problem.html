Problem 29: Distinct Powers [Level 1]

Problem Statement

Consider all integer combinations of $a^b$ for $2 \le a \le 5$ and $2 \le b \le 5$: $$\begin{array}{rrrr} 2^2=4, &2^3=8, &2^4=16, &2^5=32\\ 3^2=9, &3^3=27, &3^4=81, &3^5=243\\ 4^2=16, &4^3=64, &4^4=256, &4^5=1024\\ 5^2=25, &5^3=125, &5^4=625, &5^5=3125 \end{array}$$

If they are then placed in numerical order, with any repeats removed, we get the following sequence of $15$ distinct terms: $$4, 8, 9, 16, 25, 27, 32, 64, 81, 125, 243, 256, 625, 1024, 3125.$$

How many distinct terms are in the sequence generated by $a^b$ for $2 \le a \le 100$ and $2 \le b \le 100$?

Test Cases

  • dev: solve(a_min=2, a_max=5, b_min=2, b_max=5) → 15
  • main: solve(a_min=2, a_max=100, b_min=2, b_max=100) → 9183
  • extra: solve(a_min=2, a_max=1000, b_min=2, b_max=1000) → 977358

Mathematical Complexity

The core challenge is counting distinct values among all a^b without materialising enormous integers for comparison. Two expressions a1^b1 and a2^b2 are equal if and only if their prime factorisations are identical. If a = p1^e1 * p2^e2 * ..., then a^b = p1^(e1*b) * p2^(e2*b) * ..., so the prime signature of a^b is fully determined by scaling each exponent in the factorisation of a by b. This reduces equality testing to comparing short tuples of (prime, scaled-exponent) pairs rather than comparing numbers with hundreds or thousands of digits.

The naive approach iterates over all (a_max - a_min + 1) * (b_max - b_min + 1) pairs and inserts each computed value into a set - O(a_max * b_max) insertions where each element can be O(b_max * log(a_max)) digits long. The signature approach costs O(sqrt(a_max)) per factorisation via trial division and O(log(a_max)) to hash the resulting short tuple, giving an overall O(a_max * b_max * log(a_max)) runtime that stays tractable even for large ranges. I chose a two-branch design: for small ranges (min(a_max, b_max) at most 100) I compute a^b directly and deduplicate with a set, since the constant overhead of factorisation and tuple construction exceeds the cost of a few big-integer multiplications when numbers are short; for larger ranges I fall back to prime-signature hashing.

Solution Approaches

Index 0 - Two-Branch Strategy with Trial-Division Factorisation (s0.py and s0_c)

I implemented a hand-rolled trial-division factoriser that returns a tuple of (prime, exponent) pairs for any integer. In the small-range branch, Python's arbitrary-precision integers handle a^b directly and a set comprehension collects distinct values. In the C version I replicated this with a hand-rolled base-10^9 big-integer representation stored in a fixed-size array, converting each computed value to a decimal string and inserting it into a custom open-chained hash table with 4M buckets. In the large-range branch, both Python and C compute the canonical signature string "p1:e1*b,p2:e2*b,..." for each (a, b) pair and insert it into the same hash set. The threshold of 100 was chosen pragmatically after observing that factorisation overhead dominates only above that point.

Index 1 - Two-Branch Strategy with pyprimesieve Factorisation (s1.py and s1_c)

This approach is structurally identical to index 0 but replaces the hand-rolled trial-division factoriser in the Python version with pyprimesieve.factorize, a C-backed library call. I added this variant specifically to measure whether delegating factorisation to a compiled library would reduce wall time on the large extra test case. The C version (s1_c) mirrors s0_c almost exactly - since there is no direct C binding to pyprimesieve, I copied the same trial-division factoriser across and named it factorize to signal the intended correspondence. The same two-branch threshold logic, hash set, and big-integer path are present in both language variants, making the structural difference between index 0 and index 1 purely the factorisation call site in the Python layer.

Learnings

The key insight was recognising that equality of powers is a statement purely about prime-factorisation exponents. Once I saw that a^b and c^d are equal exactly when their (prime, exponent) tuples match after scaling each exponent by the respective power, I could avoid multi-thousand-digit integer arithmetic entirely for large inputs. Representing each computed value as a canonical signature and deduplicating with a set is a clean substitution for big-integer equality comparisons.

A second insight was that the direct-computation branch is worth keeping for small ranges even in the more general solution, because the constant overhead of factorisation and tuple construction is non-trivial relative to a few big-integer multiplications when numbers are short.

For dev and main all four solutions are essentially identical because the direct-computation branch dominates and factorisation never runs. For the extra case the two Python solutions are very close to each other - s0.py at 2.63 s is actually slightly faster than s1.py at 2.68 s, meaning the Python/C boundary overhead of pyprimesieve outweighs any gain from a compiled factoriser when bases are only up to 1000. The C solutions are roughly 3.4x faster than their Python counterparts on the extra case. The tiny margin of s1_c (0.76 s) over s0_c (0.79 s) is within noise since their factorisers are effectively identical. The C solutions appear slower than Python on dev and main because process startup and hash-table initialisation dominate at that scale while the Python solutions incur near-zero overhead. I would call s0.py the most elegant overall: it has no external dependency, the trial-division factoriser is self-contained and easy to reason about, and the performance difference is negligible at the problem's intended scale.

  • Prime-signature hashing is a reusable technique for comparing products of primes without materialising huge numbers.
  • A two-branch solve function - fast direct path for small inputs, general signature path for large ones - keeps code readable while staying efficient across test categories.
  • For factorisation of integers up to a few thousand, trial division is fast enough that a C extension library offers no benefit and may even add overhead through the Python/C call boundary.
  • Representing mathematical objects as canonical tuples or strings and using a set for deduplication is a broadly applicable pattern when direct equality comparison is expensive or impossible in fixed-size types.
  • When porting a Python solution to C, a custom string-keyed hash table with chaining is a straightforward substitute for Python's built-in set, and the speedup on large inputs can be significant due to reduced interpreter overhead.

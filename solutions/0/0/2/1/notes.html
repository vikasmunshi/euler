Problem Analysis

This problem revolves around the divisor function, specifically the sum of proper divisors of a number n - the positive integers strictly less than n that divide n evenly. We call this d(n). Two distinct numbers a and b form an amicable pair when d(a) = b and d(b) = a. The task is to sum every number under 10,000 that participates in such a pair. The crucial observation is that we do not need to search for pairs directly: for each candidate x we compute y = d(x), then check whether d(y) = x and y differs from x. If so, x is amicable. The condition y != x is what excludes perfect numbers, where d(n) = n - these are not amicable because the definition demands two distinct numbers.

The efficiency hinges on how we compute d(n). A naive approach tests every integer from 1 to n-1, costing O(n) per number. Instead I enumerate divisors in pairs: every divisor i below the square root of n has a matching co-divisor n/i above it. By iterating only from 2 to floor(sqrt(n)) and harvesting both i and n/i whenever i divides n, the cost per number drops to O(sqrt(n)). The single subtlety is a perfect square, where sqrt(n) pairs with itself and would be counted twice - so I subtract it once. Across all numbers up to a limit N, the total work is O(N * sqrt(N)), which for N = 10,000 is roughly one million operations - trivial for a modern machine.

Solution Approach

Trial Division with Divisor Pairing (Python and C)

The Python solution (p0021_s0.py) and the C solution (p0021_s0.c) implement the same algorithm in two languages. The heart of both is sum_factors(n), which computes d(n). I initialise the running total to 1, since 1 is a proper divisor of every n greater than 1. I then loop i from 2 up to n_sqrt = floor(sqrt(n)); whenever n % i == 0, I add both i and n/i to the total. Finally, if n_sqrt * n_sqrt == n, the number is a perfect square and its square root was added twice (once as i, once as n/i), so I subtract it once. This formulation is correct by construction and avoids any conditional inside the hot loop other than the divisibility test.

The main loop walks x from 2 to max_num inclusive. For each x, it evaluates y = sum_factors(x), then accepts x as amicable when both y != x and sum_factors(y) == x. I deliberately placed the inexpensive comparison y != x first so that the more expensive second divisor-sum call is short-circuited whenever x is a perfect number or self-mapping. I do not constrain y to lie within max_num, because an amicable partner can legitimately fall outside the search window while the number itself is inside it - the membership condition is on x, not on its partner.

The Python version memoises sum_factors with functools.lru_cache, but defines the helper inside solve() rather than at module scope, so the cache is rebuilt on every call. This is a deliberate benchmarking choice: the harness invokes solve() many times in one process to average its runtime, and a module-level cache would survive between those runs - the first run would pay the full cost and every later run would read cached answers, reporting a falsely low time. A per-call cache is discarded and rebuilt each run, so the recorded average reflects the true cost of one solve. The C version omits memoisation entirely because C offers no built-in equivalent and the problem is small enough that recomputation is cheap. The benchmark bears this out: C runs in about 1.6 ms while Python takes about 22 ms, a roughly 14x gap driven mainly by interpreter overhead rather than the caching difference.

Programming Techniques

Square-Root Divisor Enumeration

Rather than testing every integer below n, I iterate only to floor(sqrt(n)) and collect divisors in pairs (i, n/i). This is the canonical technique for enumerating or summing divisors and should be your default whenever you need the full divisor set of a number. The one pitfall is the perfect-square case, where the middle divisor is counted twice and must be corrected; recognising and handling that edge case explicitly is what makes the routine correct.

Memoisation with lru_cache

functools.lru_cache wraps a pure function with a dictionary-backed cache: the first call with a given argument runs the body and stores the result, and later calls with that argument return it instantly. I applied this memoisation because sum_factors is pure - the same input always yields the same output with no side effects - and may be invoked twice with the same argument when verifying a pair. Crucially, I define the cached helper inside solve() rather than at module scope: the benchmark harness calls solve() repeatedly in one process, so a module-level cache would stay warm across runs and understate the real cost, whereas a per-call cache is rebuilt every run and measures it honestly. Memoisation is the right tool whenever a function is pure, has a bounded argument space, and might be called repeatedly with the same input - but always build the cache inside the timed function so the benchmark stays truthful.

Walrus Operator for Inline Assignment

The Python filter uses the walrus operator (:=) to bind y inside the condition: (y := sum_factors(x)) != x and sum_factors(y) == x. This computes sum_factors(x) once and reuses the result both for the comparison and for the second call. The walrus operator (Python 3.8+) is especially valuable inside comprehensions and generator expressions, where ordinary assignment statements are not permitted. Reach for it whenever you need a value both as part of a test and for subsequent computation in the same expression.

Generator Expression in sum()

The solve function passes a generator expression directly to sum() instead of building a list. A generator produces values lazily, one at a time, so no intermediate list is allocated. For 10,000 numbers the memory saving is negligible, but the habit is worth keeping: whenever you only need an aggregate such as sum, min, max, any, or all, prefer the generator expression form sum(... for x in ...) over the list form sum([... for x in ...]).

Short-Circuit Ordering of Conditions

I placed the cheap comparison y != x before the expensive sum_factors(y) == x call. Because both and in Python and && in C short-circuit, the second divisor-sum is skipped whenever the first test fails. Ordering boolean tests from cheapest and most-discriminating to most expensive is a small but consistently rewarding habit in any hot loop.

Key Takeaways

The unlocking insight is that amicability can be checked with just two evaluations of d per candidate: compute y = d(x), then verify d(y) = x. There is no need to enumerate pairs, and combined with O(sqrt(n)) divisor summing the whole search is fast in either language. The C implementation finishes in about 1.6 ms versus Python's 22 ms - roughly a 14x speedup - which is typical when per-iteration work is small and interpreter overhead dominates. At this scale either language is perfectly comfortable; the gap only matters once N grows large.

  • Enumerate divisors in pairs up to the square root of n to cut per-number work from O(n) to O(sqrt(n)) - the standard approach for any divisor-sum computation.
  • Handle the perfect-square edge case explicitly: when the integer square root pairs with itself, subtract it once to avoid double-counting.
  • Reframe pair-finding problems as a one-directional check followed by a verification, rather than enumerating all pairs.
  • Apply memoisation to any pure function with a bounded argument space that may be called repeatedly with the same input - and build the cache inside the timed function so benchmarks stay honest.
  • Use the walrus operator inside generator expressions to bind intermediate values needed for both filtering and further computation.
  • Prefer generator expressions over list comprehensions when only an aggregate is required.
  • Order boolean conditions cheapest-first so short-circuit evaluation skips expensive tests when possible.

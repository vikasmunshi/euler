Problem 21: Amicable Numbers [Level 1]

Problem Statement

Let $d(n)$ be defined as the sum of proper divisors of $n$ (numbers less than $n$ which divide evenly into $n$).
If $d(a) = b$ and $d(b) = a$, where $a \ne b$, then $a$ and $b$ are an amicable pair and each of $a$ and $b$ are called amicable numbers.

For example, the proper divisors of $220$ are $1, 2, 4, 5, 10, 11, 20, 22, 44, 55$ and $110$; therefore $d(220) = 284$. The proper divisors of $284$ are $1, 2, 4, 71$ and $142$; so $d(284) = 220$.

Evaluate the sum of all the amicable numbers under $10000$.

Mathematical Complexity

The core number theory here is the sum-of-proper-divisors function d(n). The key efficiency insight is that divisors come in pairs: if i divides n then n/i also divides n, so I can collect both in a single loop up to the square root of n. This reduces computing d(n) from O(n) naive trial division down to O(sqrt(n)). One subtle correction is needed for perfect squares: the integer square root would otherwise be counted twice as both i and n/i, so I subtract it once in that case. Checking all candidates below a limit L therefore costs O(L * sqrt(L)), which for L = 10000 is trivially fast.

Identifying amicable pairs requires two evaluations of d per candidate: compute b = d(a), then verify d(b) == a with the constraint a != b to exclude perfect numbers. Many values of b coincide with values of a already computed, so memoising the function converts repeated lookups into O(1) cache hits and keeps total work close to O(L * sqrt(L)) unique evaluations rather than up to 2 * L separate computations. An alternative worth knowing for much larger limits is a sieve-style accumulation: for each divisor d from 1 to L, add d to every multiple of d up to L. That runs in O(L log L) and would be preferable if L grew into the millions.

Solution Approach

Memoised Square-Root Trial Division (s0 - Python and C)

Both the Python and C solutions implement the same core algorithm. I wrote a helper sum_factors(n) that initialises a running total to 1 (the trivial divisor 1 is always proper), then iterates i from 2 up to floor(sqrt(n)). Whenever i divides n evenly, both i and n/i are added to the total. After the loop, if n is a perfect square the integer square root is subtracted once to correct the double-counting. The main loop iterates x from 2 to max_num, computes y = sum_factors(x), skips the case y == x (which would be a perfect number rather than an amicable pair), and checks the reciprocal condition sum_factors(y) == x. Any x satisfying both conditions is accumulated into the result.

In the Python version I decorated sum_factors with @functools.lru_cache() so repeated calls with the same argument are free. I also used the walrus operator := inside a generator expression to assign y = sum_factors(x) and immediately test both conditions in one compact expression passed to sum(). The C version follows the identical logic with a plain local variable inside the for loop, since C has no equivalent syntactic shorthand. The C version has no memoisation and recomputes sum_factors for each y from scratch, but the problem is small enough that this makes no measurable difference in correctness or practical runtime at this scale.

Learnings

The central insight was recognising that divisor pairs let me compute d(n) in O(sqrt(n)) rather than O(n), and that the lru_cache in the Python version converts what would otherwise be up to 2 * L evaluations into at most L unique evaluations plus L cache hits. Getting the perfect-square guard right - - (n_sqrt if n_sqrt ** 2 == n else 0) - is a subtle but important correctness point; omitting it silently inflates every perfect-square divisor sum and produces wrong answers without any obvious error signal.

There is only one algorithmic approach here (index s0), implemented in both Python and C. The benchmarks show the Python solution runs in about 44 ms and the C solution in about 2.5 ms, a roughly 17x speedup, which is typical for this class of tight numerical loop. Neither memoisation in Python nor its absence in C matters practically at L = 10000. For future problems involving divisor sums at much larger scales I should reach for the sieve-style approach: iterate each candidate divisor d and add it to all its multiples, accumulating sums in an array in O(L log L) time, which is asymptotically superior to the per-number sqrt loop used here.

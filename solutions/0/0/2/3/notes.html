Problem Analysis

The problem hinges on the notion of an abundant number: a positive integer whose proper divisors (every divisor except the number itself) sum to more than the number. That sum of proper divisors is called the aliquot sum, and it is the central quantity I have to compute. The problem hands me a crucial mathematical fact: every integer greater than 28123 can be expressed as the sum of two abundant numbers. This bounds an otherwise infinite search into the finite range [1, 28123], which is what makes the problem tractable at all. Without that bound there would be no terminating computation.

With the range fixed, the problem decomposes into two clean phases. Phase one classifies every number up to the limit as abundant or not by computing its aliquot sum. Phase two enumerates all sums of two abundant numbers and marks which targets in the range are reachable; the answer is the sum of the unmarked targets. The classification phase can be done either by per-number trial division in O(N√N) total time, or by a sieve in O(N log N) time. The pairwise-sum phase costs O(A²) where A is the number of abundant numbers below the limit - roughly 6965, giving about 24 million pair checks, which a modern machine dispatches in milliseconds. Recognising the two-phase structure is the key to optimising each part independently.

Solution Approach

Algorithm 0: Sieve-Based Aliquot-Sum Construction (s0.py and s0.c)

My first approach computes the aliquot sum for the entire range in a single sieve pass, inspired by the Sieve of Eratosthenes. Instead of asking "what are the divisors of n?" for each n, I invert the question: for each divisor i from 1 to limit/2, I add i to every multiple of i starting at 2*i. Each divisor value is processed exactly once and contributes to all the numbers it divides. After the pass, div_sums[n] holds the sum of all proper divisors of n. The outer loop runs limit/2 times, the inner loop runs about limit/(2i) times per step, and the total work is proportional to the harmonic series, giving O(N log N).

Once the sieve completes, I collect abundant numbers with a single comparison div_sums[i] > i, naturally producing them in ascending order. The pairwise-sum phase iterates over all pairs (i, j) with j >= i, marking each sum that lands within the limit in a flat boolean array, and finally sums the integers whose mark was never set.

The two language versions differ in how the inner loops are expressed. In C I use plain nested loops with direct array indexing - cache-friendly and idiomatic. In Python I lean on NumPy: the entire inner sieve loop becomes a single vectorised slice assignment div_sums[2*i::i] += i, abundant numbers are extracted with np.flatnonzero(div_sums > np.arange(limit + 1)), and even the sum-marking loop uses vectorised slicing and boolean indexing. The Python interpreter only drives the outer loops; the heavy lifting runs in compiled NumPy code.

Algorithm 1: Trial Division per Number (s1.py and s1.c)

My second approach computes each number's aliquot sum independently by trial division. For a given n, I start the accumulator at 1 (every n > 1 has 1 as a proper divisor) and iterate i from 2 up to √n. Whenever i divides n I add both i and its cofactor n/i, taking care not to double-count when i == n/i (the case of a perfect square). This costs O(√n) per number and O(N√N) overall. The divisor-pairing trick - that divisors come in pairs straddling the square root - is what keeps the per-number cost at √n rather than n.

The pairwise-sum phase is identical to algorithm 0: a sorted list of abundant numbers, symmetric pair iteration, a boolean marking array, and a linear final scan. Only the classification phase differs. The C version outperforms the Python version dramatically because the tight square-root loop runs in native code; even so, it is slower than the sieve-based C solution because trial division repeats divisibility work that the sieve shares across multiples.

Programming Techniques

Sieve-Style Divisor Accumulation

The decisive performance idea in algorithm 0 is inverting the divisor question. Rather than factoring each number, I let each divisor i push a contribution onto all its multiples. This is the same structural trick as the Sieve of Eratosthenes, and it reduces the classification phase from O(N√N) to O(N log N). Whenever you need a multiplicative or additive property for every number in a contiguous range, ask whether a sieve formulation exists before reaching for per-number computation.

Divisor Pairing Around the Square Root

In the trial division solutions I only loop up to √n because divisors come in pairs: if i divides n, so does n/i, and one of each pair is always at or below the square root. Adding both at once collapses the search space. The single guard i != n/i handles the perfect-square case where the two cofactors coincide. This is the standard way to enumerate divisors efficiently for a single number.

NumPy Vectorised Slice Assignment

In s0.py the inner sieve loop becomes div_sums[2*i::i] += i. The extended slice 2*i::i selects every i-th element starting at 2i, and the addition runs entirely inside compiled NumPy. The interpreter only runs the outer loop over i (at most 14061 iterations). Pushing the hot inner loop into vectorised array operations is the canonical way to write fast numerical Python with NumPy without dropping to C.

Direct-Address Boolean Flag Array

Rather than store pairwise sums in a hash set, I allocate a flat boolean array indexed directly by value. Marking is an O(1) random write, testing is an O(1) read, and the final summation is one linear scan. Total memory is about 28 KB - negligible. When the value domain is known and densely packed, a direct-address table beats a hash set because it eliminates hashing, comparison, and pointer indirection.

Early Loop Termination on Sorted Input

Because the list of abundant numbers is ascending, once abundant[i] + abundant[j] exceeds the limit, every later j will too, so the inner loop breaks immediately. Exploiting monotonicity to short-circuit a scan is a reusable technique any time your data is sorted and you are accumulating up to a threshold.

Symmetric Pair Iteration (j >= i)

Since a + b == b + a, I start the inner index at i rather than 0, enumerating each unordered pair exactly once and halving the work. Starting at i rather than i+1 correctly includes the a + a case. Always check whether order matters before writing a nested enumeration.

calloc for Zero-Initialised Arrays in C

For arrays that must start at zero - the sieve accumulator and the boolean marking array - I use calloc instead of malloc plus memset. calloc allocates and zeroes in one call and may use OS-level zero pages without physically writing them, so it is at least as fast. I reserve malloc for the abundant array, which I overwrite entirely.

Key Takeaways

The core insight is the two-phase decomposition: compute the aliquot sum for the whole range, then enumerate pairwise sums against a bounded marking table. The provided upper bound of 28123 is what makes the search finite, and the sieve formulation of the classification phase is the decisive algorithmic win.

The benchmarks make the trade-offs concrete. The sieve-based C solution (p0023_s0_c) at 5.7 ms is fastest, with the trial division C solution (p0023_s1_c) at 10.4 ms - roughly 1.8x slower, exactly the gap you expect from O(N log N) versus O(N√N) in the classification phase. On the Python side, the NumPy sieve (p0023_s0.py) at 55 ms is about 9.7x faster than the pure-Python trial division version (p0023_s1.py) at 536 ms, a vivid demonstration that keeping the inner loop in compiled NumPy code rather than the interpreter recovers nearly an order of magnitude.

  • When you need a property for every number in a range, prefer a sieve (O(N log N)) over per-number trial division (O(N√N)).
  • To enumerate divisors of a single number, loop only to its square root and add both members of each divisor pair.
  • A direct-address boolean array indexed by value is the fastest set-membership structure for dense, bounded integer ranges.
  • Exploit sorted order to break out of loops early once a running sum passes the target.
  • For unordered pairs, start the inner index at the outer index to enumerate each pair exactly once.
  • In Python, move hot inner loops into NumPy vectorised operations to gain one to two orders of magnitude.
  • Use calloc in C whenever you need a zeroed block; reserve malloc for memory you will fully overwrite.
  • Decompose a problem into independent phases and optimise each one separately.

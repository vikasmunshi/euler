Problem 23: Non-Abundant Sums [Level 2]

Problem Statement

A perfect number is a number for which the sum of its proper divisors is exactly equal to the number. For example, the sum of the proper divisors of $28$ would be $1 + 2 + 4 + 7 + 14 = 28$, which means that $28$ is a perfect number.

A number $n$ is called deficient if the sum of its proper divisors is less than $n$ and it is called abundant if this sum exceeds $n$.

As $12$ is the smallest abundant number, $1 + 2 + 3 + 4 + 6 = 16$, the smallest number that can be written as the sum of two abundant numbers is $24$. By mathematical analysis, it can be shown that all integers greater than $28123$ can be written as the sum of two abundant numbers. However, this upper limit cannot be reduced any further by analysis even though it is known that the greatest number that cannot be expressed as the sum of two abundant numbers is less than this limit.

Find the sum of all the positive integers which cannot be written as the sum of two abundant numbers.

Mathematical Complexity

The core concept is the aliquot sum - the sum of all proper divisors of a number. A number is abundant when its aliquot sum strictly exceeds the number itself. The problem provides a proven upper bound of 28123: every positive integer above this threshold is expressible as the sum of two abundant numbers, so I only need to examine a fixed finite range. Classifying each number up to 28123 costs either O(sqrt(n)) per number via trial division, giving O(N sqrt(N)) total, or O(N log N) via a sieve that adds each integer i as a divisor contribution to every proper multiple 2i, 3i, 4i, ... up to the limit, exploiting the harmonic series structure.

After identifying the roughly 6,965 abundant numbers below 28123, the problem reduces to marking all pairwise sums as reachable. A naive double loop over A abundant numbers is O(A^2), but because the abundant list is sorted the inner loop can break early as soon as the running sum exceeds the limit, making the practical cost considerably lower than the worst case. The final answer is the sum of all indices in the range that were never flagged as an abundant-number sum.

Solution Approaches

s0 - Aliquot Sieve with Vectorised Sum Marking

My s0 approach builds the full array of aliquot sums in a single sieve pass. For each integer i from 1 to limit/2, I add i to every slot that is a proper multiple of i. In Python I use NumPy slice assignment ( div_sums[2*i::i] += i), which keeps the entire classification step inside NumPy's C layer and runs in O(N log N). Once abundant numbers are identified with np.flatnonzero, I mark achievable sums by taking a slice of the abundant array from index i onward, adding the scalar abundant[i], and writing True into the boolean flag array in one vectorised operation. The C version uses the same sieve logic with plain nested loops and an inner break once the sum exceeds the limit. The Python version finishes in about 0.081 seconds; the C version finishes in about 0.0091 seconds.

s1 - Trial Division with Sorted Double Loop

My s1 approach computes aliquot sums individually using a sum_proper_divisors helper that walks factors up to sqrt(n) and accumulates both members of each factor pair, avoiding double-counting the square root case. All abundant numbers up to the limit are collected into a sorted list, then a nested double loop marks reachable sums with an early-exit break in the inner loop once the partial sum exceeds 28123. The logic in Python and C is essentially identical; the only structural difference is that the C version uses char arrays instead of Python lists for the boolean flag array. The Python version takes about 0.82 seconds; the C version takes about 0.015 seconds.

Learnings

The key unlock was recognising that the stated upper bound of 28123 is a gift from the problem setter. It converts what might feel like an open-ended number-theory search into a bounded brute-force exercise. Once I accepted the fixed search space, both the sieve classification and the double-loop sum marking became obviously tractable with straightforward enumeration.

Comparing benchmarks across both algorithms and both languages is instructive. In Python, the sieve (s0) is roughly 10x faster than trial division (s1) - 0.081 s vs 0.82 s - because the NumPy sieve avoids the Python interpreter loop almost entirely for both the classification and marking phases. In C the gap is much smaller: s0_c takes 0.0091 s while s1_c takes 0.015 s. The sieve is still faster in C for the aliquot-sum phase, but compiled trial division is already very fast and the sieve's irregular memory stride pattern is slightly less cache-friendly at this problem size. The s0 Python solution is therefore the most elegant overall - it exploits NumPy's vectorised operations to stay close to C speed without leaving Python.

Key takeaways worth remembering for future problems:

  • A divisor-sum sieve ( div_sums[2*i::i] += i) achieves O(N log N) aliquot-sum classification and is the natural NumPy analogue of the prime sieve. It gives a large speedup in Python but only a modest one in compiled C.
  • Sorting the abundant list before the double loop enables a cheap break that eliminates a large fraction of the O(A^2) inner work, making even the slower Python variant finish in under one second.
  • Whenever a problem statement provides an explicit numerical upper bound backed by mathematical analysis, treat it as a hard constraint that defines the complexity class - it frequently makes an otherwise daunting search space tractable by simple enumeration.
  • The performance gap between Python and C is typically 10x-100x for numeric loops; pushing inner loops into C via NumPy vectorisation can close most of that gap without leaving Python at all.

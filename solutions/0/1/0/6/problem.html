Problem 106: Special Subset Sums [Level 13]

Problem Statement

Let $S(A)$ represent the sum of elements in set $A$ of size $n$. We shall call it a special sum set if for any two non-empty disjoint subsets, $B$ and $C$, the following properties are true:

  1. $S(B) \ne S(C)$; that is, sums of subsets cannot be equal.
  2. If $B$ contains more elements than $C$ then $S(B) \gt S(C)$.

For this problem we shall assume that a given set contains $n$ strictly increasing elements and it already satisfies the second rule.

Surprisingly, out of the $25$ possible subset pairs that can be obtained from a set for which $n = 4$, only $1$ of these pairs need to be tested for equality (first rule). Similarly, when $n = 7$, only $70$ out of the $966$ subset pairs need to be tested.

For $n = 12$, how many of the $261625$ subset pairs that can be obtained need to be tested for equality?

NOTE: This problem is related to Problem 103 and Problem 105 .

Mathematical Complexity

The problem asks how many equal-sized disjoint subset pairs from an n-element sorted set cannot be ruled out by positional dominance alone and therefore must be explicitly tested for sum equality. Because rule 2 (larger set implies larger sum) is assumed to hold, only pairs of equal size k matter, and only those where neither subset positionally dominates the other. Given 2k elements in sorted order a_1 < a_2 < ... < a_{2k}, a split into sets B and C of size k is "safe" when B_i > C_i for all i (or vice versa) in sorted order, which guarantees S(B) != S(C) from the ordering property alone. The count of such safe unordered pairs is the Catalan number C_k = C(2k, k) / (k+1). So the unsafe (must-test) unordered pairs for a fixed 2k-element pool is C(2k,k)/2 - C(2k,k)/(k+1) = C(2k,k)*(k-1)/(2*(k+1)). Summing over all choices of 2k elements from n gives a closed-form answer computable in O(n) arithmetic operations with no search required.

The Catalan number connection is the central insight: the number of ways to partition 2k ranked items into two equal groups such that one group completely dominates the other positionally follows a Catalan recurrence. Recognising this turns what initially sounds like an exponential enumeration into a handful of binomial coefficient evaluations. The overall complexity is O(n) loop iterations, each involving O(k) multiplications for the binomial coefficients, giving O(n^2) total arithmetic - effectively instant for any n of practical interest.

Solution Approach

Closed-form Catalan Subtraction (s0 - Python and C)

I implemented the same algorithm in both Python ( p0106_s0.py) and C ( p0106_s0.c). The core loop runs k from 2 to n/2 and accumulates:

  • C(n, 2k) - ways to choose 2k elements from the full set of n
  • multiplied by C(2k, k)*(k-1) / (2*(k+1)) - unsafe unordered pairs within those 2k elements

The numerator factor (k-1) equals zero when k=1, so pairs of size 1 vs 1 automatically contribute nothing - they are always ruled safe by the positional argument (a single-element set always has a definite order). The Python version uses math.comb and integer floor division; the C version computes binomial coefficients iteratively with the standard multiplicative formula, using integer division at each step to keep intermediate values exact. Both return identical results across all test cases including the large extra case n=32.

Learnings

The key "aha moment" was recognising that "positional dominance" (every element of B is greater than the corresponding sorted element of C) is exactly what the Catalan numbers count. The total unordered equal-size pairs is C(2k,k)/2; subtracting the Catalan number C(2k,k)/(k+1) leaves precisely the pairs that cannot be decided by ordering alone. Factoring C(2k,k)/2 - C(2k,k)/(k+1) = C(2k,k)*(k-1)/(2*(k+1)) makes the cancellation algebraically clean and avoids any floating-point issues because the result is always an integer.

I only used one algorithmic approach here since the closed form is both obvious once the Catalan connection is found and already O(n) - there is nothing meaningful to optimise further. Both the Python and C solutions run in effectively zero time (well under a millisecond even for n=32), so benchmarking does not distinguish them in any useful way. The C solution exists mainly for consistency with the rest of the archive. For future problems, this problem is a good reminder to look for Catalan numbers whenever counting non-crossing or dominance-preserving partitions of sorted sequences.

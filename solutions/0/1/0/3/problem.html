Problem 103: Special Subset Sums [Level 11]

Problem Statement

Let $S(A)$ represent the sum of elements in set $A$ of size $n$. We shall call it a special sum set if for any two non-empty disjoint subsets, $B$ and $C$, the following properties are true:

  1. $S(B) \ne S(C)$; that is, sums of subsets cannot be equal.
  2. If $B$ contains more elements than $C$ then $S(B) \gt S(C)$.

If $S(A)$ is minimised for a given $n$, we shall call it an optimum special sum set. The first five optimum special sum sets are given below.

  • $n = 1$: $\{1\}$
  • $n = 2$: $\{1, 2\}$
  • $n = 3$: $\{2, 3, 4\}$
  • $n = 4$: $\{3, 5, 6, 7\}$
  • $n = 5$: $\{6, 9, 11, 12, 13\}$

It seems that for a given optimum set, $A = \{a_1, a_2, \dots, a_n\}$, the next optimum set is of the form $B = \{b, a_1 + b, a_2 + b, \dots, a_n + b\}$, where $b$ is the "middle" element on the previous row.

By applying this "rule" we would expect the optimum set for $n = 6$ to be $A = \{11, 17, 20, 22, 23, 24\}$, with $S(A) = 117$. However, this is not the optimum set, as we have merely applied an algorithm to provide a near optimum set. The optimum set for $n = 6$ is $A = \{11, 18, 19, 20, 22, 25\}$, with $S(A) = 115$ and corresponding set string: 111819202225.

Given that $A$ is an optimum special sum set for $n = 7$, find its set string.

NOTE: This problem is related to Problem 105 and Problem 106 .

Mathematical Complexity

A special sum set of size n requires two properties for every pair of non-empty disjoint subsets B and C: their sums must differ, and if B has strictly more elements than C then S(B) must exceed S(C). Verifying condition 1 naively means enumerating all 2^n - 1 non-empty subsets and checking that no two share the same sum - an O(n * 2^n) operation. Condition 2 admits a much cheaper O(n^2) check: for a sorted set, the hardest case for the size-ordering rule is always the k smallest elements versus the j largest. If that extremal pair satisfies the inequality for every valid (k, j) pair with k + j <= n, then all other disjoint pairs of those sizes automatically satisfy it too, so only prefix and suffix sums over the sorted set need comparing.

The search space in principle grows exponentially with n, but the problem hints that each optimum set lies close to a structured perturbation of the previous one. I exploit this by forming a near-optimal candidate using the middle-element rule described in the problem statement - prepend the middle element b of the previous optimum and add b to every existing element - then restricting the depth-first search to a window of +/-5 around each position of that candidate. Each position then has at most 11 candidate values, giving roughly 11^n leaf nodes before pruning. In practice a lower-bound sum cutoff prunes the tree drastically, but the search remains exponential and becomes slow around n = 9 or 10 without careful ordering of validity checks.

Solution Approaches

Index 0 - Condition-1-first DFS with neighborhood search (p0103_s0.py and p0103_s0.c)

Both files share the same strategy. I hard-code the known optima for n = 1 through n = 6, then for larger n derive the near-optimal candidate from the previous result using the middle-element rule. I set +/-5 search bounds around each position of the candidate and run a depth-first search over sorted integer tuples within those bounds. Any branch whose lower-bound sum - current sum plus the minimum achievable sum for the remaining positions - already meets or exceeds the best known total is pruned immediately. The warm-start from evaluating the candidate first keeps the pruning bound tight from the very beginning of the search.

Inside is_special I check condition 1 first: I enumerate all non-empty subsets grouped by size and record their sums in a boolean array, returning false as soon as a duplicate appears. Only if condition 1 passes do I check condition 2 using prefix and suffix sums. The C version additionally allocates the seen-sums array and the combination-index array on the heap inside each call to is_special, adding malloc/free overhead on every validity test. At n = 7 the C version runs in about 44 ms; at n = 8 it takes roughly 797 ms. The Python version takes 876 ms at n = 7 and 13.7 s at n = 8.

Index 1 - Condition-2-first DFS with stack-allocated inner loop (p0103_s1.py and p0103_s1.c)

Structurally identical to index 0, with one critical change inside is_special: I check condition 2 before condition 1. Because condition 2 is O(n^2) and most candidates encountered early in the DFS violate the size-ordering rule, flipping the check order means the vast majority of invalid sets are rejected immediately without ever paying the O(n * 2^n) cost of subset enumeration. In the C version I also replaced the heap-allocated combination-index array from index 0 with a fixed-size stack array int idx[20], removing per-call malloc/free overhead in the inner combination loop.

The Python versions show a speedup from 876 ms to 526 ms at n = 7 and from 13.7 s to 8.1 s at n = 8. The C improvements are even more dramatic: 44 ms versus 8.5 ms at n = 7, 797 ms versus 155 ms at n = 8, and 13.2 s versus 2.8 s at n = 9 - roughly a 4-5x improvement across the board from one reordering of two independent checks. Only p0103_s1_c completes n = 10 within the time limit (22.6 s); all other versions time out at that size.

Learnings

The central insight was that condition 2 is both cheaper to evaluate and more frequently violated than condition 1, especially for the mostly-invalid candidates that dominate the early branches of the DFS tree. Reordering two independent validity checks - moving the O(n^2) test in front of the O(n * 2^n) test - delivered a 5x speedup in C and nearly 2x in Python without changing the algorithm at all. This is a general principle worth remembering: when a function has multiple independent rejection criteria, always evaluate them from cheapest to most discriminating first.

A second important technique was seeding the DFS with a concrete upper bound before the search begins. By evaluating the near-optimal candidate immediately and recording its sum as the initial best_sum, the pruning condition is tight from the very first branch. Without this warm start the DFS would explore far more of the space before encountering any valid set, and the lower-bound pruning would be ineffective early on.

The +/-5 neighborhood assumption is empirical. It holds for all n through at least 8 based on the benchmark results and the confirmed correct answers, but I have no proof it continues to hold for larger n. The deviation between the rule-derived candidate and the true optimum appears to grow slowly, and widening the window to +/-6 or +/-7 is probably sufficient for n up to 12 at the cost of significantly longer search times. For future problems of this class I would document the window size clearly and note that it is an unproven heuristic assumption.

One practical note about the C output: for large n the set string formed by concatenating multi-digit element values grows too long to fit in a 64-bit signed integer, so I print the result directly as a string by concatenating with snprintf and strncat rather than converting through atoll. This avoids silent overflow in the output formatting, which would produce a wrong answer even when the algorithm itself finds the correct set.

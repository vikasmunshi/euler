Problem 107: Minimal Network [Level 7]

Problem Statement

The following undirected network consists of seven vertices and twelve edges with a total weight of 243.


The same network can be represented by the matrix below.

However, it is possible to optimise the network by removing some edges and still ensure that all points on the network remain connected. The network which achieves the maximum saving is shown below. It has a weight of 93, representing a saving of 243 − 93 = 150 from the original network.


Using network.txt (right click and 'Save Link/Target As...'), a 6K text file containing a network with forty vertices, and given in matrix form, find the maximum saving which can be achieved by removing redundant edges whilst ensuring that the network remains connected.

Mathematical Complexity

This problem reduces directly to the classical Minimum Spanning Tree (MST) problem from graph theory. Given a connected, weighted, undirected graph, the MST is the spanning subgraph that connects all vertices with the minimum possible total edge weight and no cycles. The maximum saving is simply the difference between the total weight of all original edges and the weight of the MST. For a graph with V vertices and E edges, Kruskal's algorithm runs in O(E log E) time (dominated by sorting the edges), and the Union-Find operations with path compression add only nearly O(E alpha(V)) time, where alpha is the inverse Ackermann function - effectively constant. The 40-vertex problem file has at most 40*39/2 = 780 possible edges, so even a naive approach would be fast.

The key mathematical insight is that a spanning tree on V vertices always has exactly V-1 edges, and any connected subgraph with fewer than V-1 edges is disconnected while any connected subgraph with more than V-1 edges contains a cycle. Removing all edges not in the MST therefore achieves the maximum possible saving while preserving full connectivity.

Solution Approach

Kruskal's Algorithm with Union-Find (s0 - Python and C)

I chose Kruskal's algorithm as the single approach for both the Python and C implementations. The algorithm proceeds as follows:

  • Parse the adjacency matrix, reading only the upper triangle (i < j) to avoid counting each edge twice, accumulating the total edge weight as I go.
  • Sort all collected edges by weight in ascending order.
  • Use a Union-Find (disjoint-set) structure with path compression to greedily add the cheapest edge that does not form a cycle, until V-1 edges have been accepted.
  • Return total - mst as the maximum saving.

In Python, I implemented find as a nested function using path halving (assigning parent[x] = parent[parent[x]] before moving up). The matrix parsing uses a list comprehension over str.splitlines() and str.split(','). In C, I implemented the same logic with a static par[] array and a manual iterative find function with the same path-halving trick. The C version also includes a file-loading helper that tries several path conventions relative to the executable directory before falling back to a hardcoded default matrix for the dev test case. Both versions accept either an empty file URL (triggering the built-in 7-node example) or a path to the 40-node network file.

Learnings

The key "aha moment" here was recognising immediately that "maximum saving by removing redundant edges while keeping the network connected" is precisely the definition of MST savings - there is no need for any custom search or pruning. Once I framed it that way, Kruskal's algorithm was the obvious choice because the input is given as an edge list (after parsing the matrix), making sorting and greedy selection straightforward.

I only implemented one algorithmic approach (Kruskal with Union-Find) for this problem, in both Python and C. The benchmark results show the Python solution solving the main case in about 1.4 ms and the C solution in about 0.3 ms - roughly a 5x speedup for C, which is typical for this kind of simple graph algorithm where Python's interpreter overhead on small inputs is noticeable but not severe. Both are fast enough that the choice of language is purely a matter of preference here.

A technique worth remembering: path compression in Union-Find (even the simple path-halving variant rather than full two-pass compression) is sufficient to make the structure essentially O(1) per operation in practice. Also, parsing only the upper triangle of a symmetric adjacency matrix is an easy way to deduplicate edges without any extra bookkeeping.

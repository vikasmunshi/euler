Problem 108: Diophantine Reciprocals I [Level 6]

Problem Statement

In the following equation $x$, $y$, and $n$ are positive integers.

$$\dfrac{1}{x} + \dfrac{1}{y} = \dfrac{1}{n}$$

For $n = 4$ there are exactly three distinct solutions:

$$\begin{align} \dfrac{1}{5} + \dfrac{1}{20} &= \dfrac{1}{4}\\ \dfrac{1}{6} + \dfrac{1}{12} &= \dfrac{1}{4}\\ \dfrac{1}{8} + \dfrac{1}{8} &= \dfrac{1}{4} \end{align} $$

What is the least value of $n$ for which the number of distinct solutions exceeds one-thousand?

NOTE: This problem is an easier version of Problem 110 ; it is strongly advised that you solve this one first.

Mathematical Complexity

The key algebraic step is to rewrite 1/x + 1/y = 1/n by multiplying through and rearranging to obtain (x - n)(y - n) = n². Every positive integer solution then corresponds to a way of writing n² as a product of two positive factors a * b with a <= b, by setting x = n + a and y = n + b. The number of such unordered factor pairs is exactly (tau(n²) + 1) / 2, where tau is the divisor-count function. The problem therefore reduces to finding the smallest n for which this expression is at least 1000, which is equivalent to requiring tau(n²) >= 1999.

Because tau is multiplicative, if n = prod(p_i ^ a_i) then tau(n²) = prod(2*a_i + 1). This product is maximised for a given magnitude of n by choosing n to be a product of small primes with decreasing exponents - the class of highly composite numbers. A brute linear scan costs roughly O(n_answer * sqrt(n_answer)) in total; the DFS approach over smooth-number candidates is far faster and visits only a tiny fraction of all integers.

Solution Approach

Linear Scan with Trial Division (s0)

I implemented this as a straightforward loop over all n starting from 1. For each candidate I trial-divide by all integers up to sqrt(n) to extract the prime factorisation, accumulate tau(n²) = prod(2*exp + 1), and check whether (tau(n²) + 1) / 2 >= min_solutions. Any factor remaining after trial division must be a prime with exponent 1, contributing a multiplicative factor of 3. The loop stops as soon as the threshold is crossed. I wrote both a Python and a C version with structurally identical logic. The C build is about 18x faster on the main case (0.045 s vs 0.822 s), which is the typical interpreter-overhead gap for tight arithmetic inner loops.

DFS over Smooth-Number Candidates (s1)

I noticed that the answer must be a product of small primes with non-increasing exponents, because any prime factor larger than necessary only inflates n without helping tau(n²). I therefore fixed a list of the first 15 primes and ran a depth-first search over assignments of exponents to those primes, always enforcing that exponents are non-increasing (to avoid counting permutations of the same factorisation as different candidates). At each node I maintain the running product n and the running product tau. I prune a branch immediately if even assigning the current exponent to every remaining prime slot cannot reach the target tau, and also prune if n already exceeds the current best. Whenever tau >= target I update the best answer. This collapses an enormous search space: the DFS visits only a handful of candidates compared to the 180180 iterations of the linear scan. The Python version solves the main case in 0.0005 s and the C version in essentially zero measurable time.

Learnings

The central insight was the substitution (x - n)(y - n) = n², which converts what looks like a Diophantine reciprocal problem into a straightforward divisor-counting problem. Once I had that, the formula tau(n²) = prod(2*a_i + 1) followed directly from the multiplicativity of tau. These two steps together reduce the problem to a purely arithmetic search.

The benchmark comparison between s0 and s1 is striking. The linear scan (s0) in C takes 0.045 s; the DFS (s1) in Python takes 0.0005 s - nearly 100x faster despite being interpreted, because it visits only smooth-number candidates. The s1 C build is so fast it does not register. This makes clear that algorithmic improvements dwarf language-level optimisations when the search space can be dramatically reduced. The linear scan is adequate for a threshold of 1000 but would be hopeless for Problem 110 (threshold of one million), whereas the DFS generalises immediately.

  • The substitution (x - n)(y - n) = n² is the universal first step for Diophantine reciprocal problems.
  • The identity tau(n²) = prod(2*a_i + 1) is worth memorising; it reappears directly in Problem 110.
  • When maximising a multiplicative function over integers below a bound, the optimum is always a smooth number with non-increasing prime exponents. Restricting search to this family with a DFS and pruning is orders of magnitude faster than a linear scan.
  • The non-increasing exponent constraint (so that p_1^{a_1} <= p_1^{a_2}... wait, non-increasing exponents assigned to increasing primes) eliminates redundant orderings and keeps the DFS tree small.

Problem 102: Triangle Containment [Level 3]

Problem Statement

Three distinct points are plotted at random on a Cartesian plane, for which $-1000 \le x, y \le 1000$, such that a triangle is formed.

Consider the following two triangles:

$$\begin{gather} A(-340,495), B(-153,-910), C(835,-947)\\ X(-175,41), Y(-421,-714), Z(574,-645) \end{gather}$$

It can be verified that triangle $ABC$ contains the origin, whereas triangle $XYZ$ does not.

Using triangles.txt (right click and 'Save Link/Target As...'), a 27K text file containing the co-ordinates of one thousand "random" triangles, find the number of triangles for which the interior contains the origin.

NOTE: The first two examples in the file represent the triangles in the example given above.

Mathematical Complexity

The core challenge is a point-in-triangle test, a classical computational geometry problem. Given a triangle with vertices A, B, C and a query point (here the origin O), I need to determine whether O lies strictly inside the triangle. Both approaches I implemented reduce this to computing 2D cross products (equivalently, twice the signed areas of sub-triangles formed by O with each edge), which are simple integer arithmetic operations. There is no iteration or recursion involved; each test is O(1), and the full problem is O(N) where N is the number of triangles in the file (1000 here).

The key mathematical fact underpinning both algorithms is that the signed area of a triangle with one vertex at the origin and the other two at points P and Q is (Px*Qy - Py*Qx) / 2 - the 2D cross product of the two position vectors. This quantity is positive or negative depending on the orientation (counter-clockwise or clockwise), which is exactly what encodes which side of an edge the origin lies on.

Solution Approach

Algorithm 0 - Cross-product same-sign test (p0102_s0.py and p0102_s0.c)

I chose what I think of as the "winding" or "same-sign" test. For the origin to be inside triangle ABC, it must lie on the same side of each directed edge A->B, B->C, and C->A. I compute three cross products:

  • d1 = ax*by - ay*bx (cross product of OA and OB, i.e. twice signed area of triangle OAB)
  • d2 = bx*cy - by*cx (twice signed area of OBC)
  • d3 = cx*ay - cy*ax (twice signed area of OCA)

The origin is strictly inside the triangle if and only if all three values are strictly positive or all three are strictly negative (covering both orientations of the triangle). The Python and C implementations are essentially identical in logic. The C version uses long long arithmetic throughout to avoid overflow, since coordinates can reach 1000 and products can reach 10^6 in magnitude.

Algorithm 1 - Area decomposition test (p0102_s1.py and p0102_s1.c)

My second approach uses the area-sum identity: the origin lies inside triangle ABC if and only if the sum of the areas of sub-triangles OAB, OBC, and OCA equals the total area of ABC. Using the same cross-product quantities d1, d2, d3 as above (twice the signed sub-triangle areas), this condition becomes:

  • |d1| + |d2| + |d3| == |d1 + d2 + d3|

When the origin is outside, at least one sub-triangle has the opposite orientation, so its absolute value adds rather than cancels, making the left side strictly larger than the right. This formulation elegantly handles both triangle orientations without explicit sign checks. The C version for this algorithm also adds an absolute-value macro and computes the same three cross products. The logic is slightly different from algorithm 0 in its conditional structure but computationally equivalent.

Learnings

The key insight was recognising that point-in-triangle reduces to checking whether three 2D cross products all share a sign. Once I saw that the signed area of a triangle with a vertex at the origin is just a cross product, the rest followed naturally. Both algorithms compute exactly the same three values (d1, d2, d3) and differ only in how they test them - algorithm 0 checks signs explicitly, algorithm 1 uses the absolute-value identity. I find algorithm 1 slightly more elegant mathematically, because it generalises directly to the statement "sub-areas sum to total area", but algorithm 0's sign-check is arguably more transparent in code.

Looking at the benchmark results, all four solutions are extremely fast: the C solutions run in around 0.0001 seconds and the Python solutions in around 0.002 seconds, a roughly 20x speedup for C. The difference between algorithms 0 and 1 within the same language is negligible - both are pure O(N) with the same three multiplications and one branch per triangle. If I had to pick a favourite, I would call algorithm 0 (the sign-check version) slightly more readable for future reference, and algorithm 1 (the area-sum version) a useful trick worth remembering: checking whether a point is inside a convex polygon by verifying that sub-region areas sum to the total is a technique that scales beyond triangles.

For future problems involving geometry, I will keep in mind that the 2D cross product Px*Qy - Py*Qx is the workhorse for orientation tests, area computations, and containment checks, and that sticking to integer arithmetic avoids all floating-point precision pitfalls when the input coordinates are integers.

Problem 104: Pandigital Fibonacci Ends [Level 4]

Problem Statement

The Fibonacci sequence is defined by the recurrence relation:

$F_n = F_{n - 1} + F_{n - 2}$, where $F_1 = 1$ and $F_2 = 1$.

It turns out that $F_{541}$, which contains $113$ digits, is the first Fibonacci number for which the last nine digits are $1$-$9$ pandigital (contain all the digits $1$ to $9$, but not necessarily in order). And $F_{2749}$, which contains $575$ digits, is the first Fibonacci number for which the first nine digits are $1$-$9$ pandigital.

Given that $F_k$ is the first Fibonacci number for which the first nine digits AND the last nine digits are $1$-$9$ pandigital, find $k$.

Mathematical Complexity

The core challenge here splits into two independent sub-problems. Checking the last nine digits of a Fibonacci number is straightforward: because only the last nine digits matter, I can reduce every Fibonacci term modulo 10^9 and work entirely in that space. Checking the first nine digits requires a different technique: Binet's formula tells us that for large k, F_k is approximately phi^k / sqrt(5), so the leading digits are determined by the fractional part of k * log10(phi) - 0.5 * log10(5). Extracting the first nine digits then reduces to a floating-point computation using fmod and pow. The two checks are essentially O(1) per candidate, and the iteration runs until k = 329468, making the overall complexity O(k) with a very small constant per step.

The pandigital check itself is O(1) because it operates on a fixed nine-digit window. The Fibonacci recurrence modulo 10^9 is maintained with only two integers, so memory usage is O(1) as well. The problem is fundamentally a linear search through Fibonacci indices, and there is no known closed form that would let us jump directly to the answer, so the O(k) bound is essentially tight for this approach.

Solution Approach

Modular tail tracking combined with logarithmic leading-digit extraction (s0)

I chose a single unified algorithm implemented in both Python and C. The two implementations share the same design:

  • I maintain two integers a and b representing F_k and F_{k+1} reduced modulo 10^9. This keeps the trailing nine digits exact at every step without ever touching big integers.
  • For the trailing check, _is_pandigital_9 (Python) / is_pandigital_9 (C) verifies that the value is exactly nine digits and that the digit set is {1,...,9} - no zero, no repeats. In C I walk the digit array explicitly; in Python I use a frozenset comparison against the precomputed constant _PANDIGITAL.
  • For the leading check, I apply the logarithmic approximation: the fractional part of k * log10(phi) - 0.5 * log10(5) gives the base-10 mantissa of F_k. Multiplying 10 raised to that fraction by 10^8 yields the first nine digits as an integer, which I then pass through the same pandigital test. I use math.fmod rather than the % operator to keep precision for large k values.
  • Both checks are applied only together: I short-circuit by testing the tail first, since that is cheaper (no transcendental math), and only call the leading-digit function when the tail already passes.

In Python I iterate with itertools.count and keep the loop body minimal. The C version is a plain for loop. The C build recomputes log_phi and log_sqrt5 inside first_nine_pandigital on every call; strictly those could be hoisted to module-level constants, but because the function is only called when the tail is already pandigital (a rare event), the overhead is negligible.

Learnings

The key insight was recognizing that the two ends of a Fibonacci number can be tracked with entirely separate techniques: modular arithmetic handles the tail exactly and indefinitely, while Binet's formula plus logarithms handles the head cheaply without ever computing the full number. Trying to iterate on full arbitrary-precision Fibonacci numbers would be orders of magnitude slower and was unnecessary. The "aha moment" was realizing I could apply the logarithm trick - the same one used for problems involving leading digits of powers - directly to the Fibonacci recurrence via its closed form.

I only implemented one algorithm (s0), so there is no alternative approach to compare. The benchmark results show the C version running in about 6.8 ms versus 162 ms for Python - roughly a 24x speedup - which is typical for this style of tight numerical loop. The Python solution is already fast enough at well under a second, so the C version is an optimization of convenience rather than necessity.

Techniques worth remembering: using math.fmod instead of % for floating-point modular reduction at large arguments; the general pattern of extracting leading digits of any sequence defined by a recurrence with a known closed-form via fractional parts of logarithms; and the habit of checking the cheaper condition first to minimize expensive function calls inside a tight loop.

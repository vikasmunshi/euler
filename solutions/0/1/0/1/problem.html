Problem 101: Optimum Polynomial [Level 8]

Problem Statement

If we are presented with the first $k$ terms of a sequence it is impossible to say with certainty the value of the next term, as there are infinitely many polynomial functions that can model the sequence.

As an example, let us consider the sequence of cube numbers. This is defined by the generating function,
$u_n = n^3$: $1, 8, 27, 64, 125, 216, \dots$

Suppose we were only given the first two terms of this sequence. Working on the principle that "simple is best" we should assume a linear relationship and predict the next term to be $15$ (common difference $7$). Even if we were presented with the first three terms, by the same principle of simplicity, a quadratic relationship should be assumed.

We shall define $\operatorname{OP}(k, n)$ to be the $n$ th term of the optimum polynomial generating function for the first $k$ terms of a sequence. It should be clear that $\operatorname{OP}(k, n)$ will accurately generate the terms of the sequence for $n \le k$, and potentially the first incorrect term (FIT) will be $\operatorname{OP}(k, k+1)$; in which case we shall call it a bad OP (BOP).

As a basis, if we were only given the first term of sequence, it would be most sensible to assume constancy; that is, for $n \ge 2$, $\operatorname{OP}(1, n) = u_1$.

Hence we obtain the following $\operatorname{OP}$s for the cubic sequence:

$\operatorname{OP}(1, n) = 1$ $1, {\color{red}\mathbf 1}, 1, 1, \dots$
$\operatorname{OP}(2, n) = 7n - 6$ $1, 8, {\color{red}\mathbf{15}}, \dots$
$\operatorname{OP}(3, n) = 6n^2 - 11n + 6$ $1, 8, 27, {\color{red}\mathbf{58}}, \dots$
$\operatorname{OP}(4, n) = n^3$ $1, 8, 27, 64, 125, \dots$

Clearly no BOPs exist for $k \ge 4$.

By considering the sum of FITs generated by the BOPs (indicated in red above), we obtain $1 + 15 + 58 = 74$.

Consider the following tenth degree polynomial generating function: $$u_n = 1 - n + n^2 - n^3 + n^4 - n^5 + n^6 - n^7 + n^8 - n^9 + n^{10}.$$

Find the sum of FITs for the BOPs.

Mathematical Complexity

The core mathematical idea is polynomial interpolation: given k values of an unknown sequence, the "simplest" model is the unique polynomial of degree at most k-1 passing through those k points. Evaluating that polynomial at the next index k+1 gives the First Incorrect Term (FIT) whenever the true generating polynomial has degree higher than k-1. The problem reduces to summing those FITs for k = 1 up to the degree of the true polynomial (here degree 10, giving 10 BOPs). Three classical interpolation frameworks all apply - Lagrange's formula, Newton's forward difference formula, and the barycentric form of Lagrange interpolation - and a fourth elegant shortcut exploits the structure of the finite difference table directly. Because the degree d is fixed at 10, every algorithm is O(d^2) in arithmetic operations, making all solutions essentially instantaneous.

The key mathematical fact that makes the correction approach work is that the k-th finite difference Delta^k[u_1] is exactly the leading-coefficient contribution that a degree-(k-1) polynomial misses. Specifically, OP(k, k+1) = sum_{j=0}^{k-1} C(k,j) * Delta^j[u_1] by Newton's forward difference formula evaluated at t = k (i.e. x = k+1 = 1 + k*1). Since the true value u_{k+1} includes the additional term Delta^k[u_1], the FIT satisfies FIT_k = u_{k+1} - Delta^k[u_1], which is the foundation of the s2 approach.

Solution Approaches

s0 - Classical Lagrange Interpolation

I first generate the sequence values u[1]...u[degree+1] by direct evaluation of the given polynomial. For each k from 1 to degree, I fit the unique polynomial of degree k-1 through the k points (1, u[1]), ..., (k, u[k]) and evaluate it at x = k+1 using the standard Lagrange basis formula: OP(k, k+1) = sum_{i=1}^{k} u[i] * prod_{j != i} (k+1 - j)/(i - j). The Python version uses fractions.Fraction to keep arithmetic exact throughout. The C version implements its own Frac struct with GCD-based reduction and uses __int128 for intermediate products to avoid overflow. This is the most direct translation of the mathematical definition, though it involves O(k^2) multiplications per FIT.

s1 - Newton Forward Difference Interpolation

I precompute the full triangular forward difference table: dtable[j][i] = Delta^j[u_{i+1}]. The Newton forward difference formula evaluated at x = k+1 (displacement t = k from the first node) gives OP(k, k+1) = sum_{j=0}^{k-1} C(k, j) * Delta^j[u_1]. I accumulate the binomial coefficients incrementally (C(k,j) to C(k,j+1) in one integer multiply-divide) to avoid calling a comb function inside the inner loop. Because the intermediate differences are always integers, no rational arithmetic is needed at all - every value stays in long long. This makes s1 the cleanest approach algorithmically and entirely avoids fractions.

s2 - Finite Difference Correction Formula

I recognised that I do not need the full difference table. Observing that FIT_k = u_{k+1} - Delta^k[u_1], I compute Delta^k[u_1] by iteratively applying one round of forward differences to an in-place array. After k rounds the first element is Delta^k[u_1]. This is the most compact implementation: a single array that shrinks by one element per outer iteration. Like s1, it requires no rational arithmetic, and the Python version is a tight six-line solve function. The C version similarly uses only integer arrays.

s3 - Barycentric Lagrange Interpolation

I use the barycentric form of Lagrange interpolation, which rewrites the classical formula as a ratio: L(x) = (sum_i w[i]*y[i]/(x-xi)) / (sum_i w[i]/(x-xi)), where the barycentric weights for equally-spaced integer nodes 1..k are w[i] = (-1)^i * C(k-1, i). Because the evaluation point x = k+1 satisfies x - xi = k - i (always a positive integer), no zero-division arises and the computation stays clean. The Python version uses fractions.Fraction for the rational accumulation and math.comb for weights. The C version uses the same Frac rational type as s0 plus __int128 guard. This approach is numerically superior to classical Lagrange in floating-point contexts because it avoids computing the full product in the denominator separately for each basis polynomial, though for this problem both forms are exact under rational arithmetic.

Learnings

The key insight was recognising that all four approaches are equivalent routes to the same answer, differing only in which representation of the interpolating polynomial they exploit. The "aha moment" was seeing the correction formula FIT_k = u_{k+1} - Delta^k[u_1]: the k-th order finite difference of the sequence starting at u_1 is precisely the term that a degree-(k-1) polynomial cannot reproduce, making s2 the most elegant once that identity is clear.

All solutions are effectively instantaneous on a degree-10 polynomial (the benchmark shows sub-millisecond times for every variant in both languages). Among the Python solutions, s1 and s2 are fastest because they avoid fractions.Fraction entirely - s0 and s3 both use exact rational types and show slightly higher elapsed times (0.0005 s and 0.0003 s respectively on the main case). Among the C solutions all four report 0.0000 s, so there is no meaningful difference at this scale. For elegance and simplicity I consider s2 the winner: the correction formula requires no table, no binomial coefficients, and no fractions, and the Python solve function is fewer than ten lines. s1 is a close second - the Newton forward difference approach is clean and instructive.

Techniques worth remembering from this problem include:

  • The Newton forward difference formula OP(k, n) = sum_{j=0}^{k-1} C(n-1, j) * Delta^j[u_1] is the natural way to think about polynomial extrapolation from tabulated data.
  • The correction identity FIT_k = u_{k+1} - Delta^k[u_1] lets you compute the sum of BOPs without ever explicitly constructing the interpolating polynomial.
  • For equally-spaced nodes the barycentric weights are simply alternating binomial coefficients, making the barycentric form easy to implement exactly.
  • When intermediate results are guaranteed integers (as here with difference tables), avoiding rational arithmetic entirely is both faster and simpler - there is no need to reach for Fraction or a custom Frac struct.

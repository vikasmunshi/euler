Problem 124: Ordered Radicals [Level 4]

Problem Statement

The radical of $n$, $\operatorname{rad}(n)$, is the product of the distinct prime factors of $n$. For example, $504 = 2^3 \times 3^2 \times 7$, so $\operatorname{rad}(504) = 2 \times 3 \times 7 = 42$.

If we calculate $\operatorname{rad}(n)$ for $1 \le n \le 10$, then sort them on $\operatorname{rad}(n)$, and sorting on $n$ if the radical values are equal, we get:

Let $E(k)$ be the $k$-th element in the sorted $n$ column; for example, $E(4) = 8$ and $E(6) = 9$.

If $\operatorname{rad}(n)$ is sorted for $1 \le n \le 100000$, find $E(10000)$.

Mathematical Complexity

The radical function rad(n) is a multiplicative arithmetic function defined as the product of the distinct prime factors of n. Computing it efficiently for all integers up to a limit is the central task. The key insight is that this is structurally identical to a prime sieve: any prime p contributes itself as a factor to every multiple of p, so I can adapt the Sieve of Eratosthenes to compute rad(n) for all n up to max_n in a single linear-ish pass. The sieve runs in O(N log log N) time, matching the standard prime sieve complexity.

Once the radical values are known, the problem reduces to a sort: build an array of (rad(n), n) pairs and sort by radical first, then by n as the tiebreaker, then read off the k -th element. The sort dominates at O(N log N). Overall the algorithm is O(N log N) in both time and O(N) in space.

Solution Approach

Radical Sieve plus Sort (s0 - Python and C)

I implemented the same algorithm in both Python and C. The approach has two phases:

  • Radical sieve: I initialise an array rad[0..max_n] to all ones. For each index p starting from 2, if rad[p] is still 1 then p is prime (no smaller prime has touched it yet). I then iterate over every multiple of p and multiply rad[multiple] by p. After the sieve, rad[n] holds exactly the product of the distinct prime factors of n.
  • Sort and index: I build a list of all n from 1 to max_n and sort it using (rad[n], n) as the comparison key. The answer is the element at zero-based position k - 1.

In Python I used a list comprehension and sorted() with a lambda key. In C I defined a RadEntry struct holding rad and n, populated an array of these structs, and sorted with qsort using a custom comparator that first compares rad and falls back to n. Both detect primality via the same sieve condition ( rad[p] == 1 before processing p) rather than maintaining a separate boolean sieve array, which keeps memory usage tidy.

Learnings

The key insight that made this problem straightforward was recognising that the radical sieve is almost identical to the standard Sieve of Eratosthenes. Instead of marking composites as non-prime, I accumulate prime factor products. The primality test falls out for free: if rad[p] is still 1 when I reach index p, no smaller prime has divided it, so p must be prime. This avoids the need for a separate boolean sieve array.

The benchmark results illustrate the typical Python-vs-C performance gap on numerical work. For the main problem ( max_n = 100000, k = 10000) the C solution runs in about 8.8 ms versus 99 ms for Python - roughly an 11x speedup. At the larger extra case ( max_n = 200000) the ratio is similar (18.7 ms vs 214 ms). The algorithmic work is identical; the difference is entirely language overhead in the sieve loop and sort. For future problems involving multiplicative functions over a range, the sieve-style accumulation pattern is worth remembering - it generalises cleanly to Euler's totient, the sum-of-divisors function, and related arithmetic functions, all computable in a single O(N log log N) pass.

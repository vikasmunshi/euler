Problem 125: Palindromic Sums [Level 4]

Problem Statement

The palindromic number $595$ is interesting because it can be written as the sum of consecutive squares: $6^2 + 7^2 + 8^2 + 9^2 + 10^2 + 11^2 + 12^2$.

There are exactly eleven palindromes below one-thousand that can be written as consecutive square sums, and the sum of these palindromes is $4164$. Note that $1 = 0^2 + 1^2$ has not been included as this problem is concerned with the squares of positive integers.

Find the sum of all the numbers less than $10^8$ that are both palindromic and can be written as the sum of consecutive squares.

Mathematical Complexity

The core challenge here sits at the intersection of two properties: palindromicity and representability as a sum of at least two consecutive squares of positive integers. The sum of consecutive squares from i to j has a closed form via the formula for sum of squares, but I found it simpler to accumulate the running total incrementally. The key bound is that the starting index i satisfies i^2 + (i+1)^2 < max_limit, which constrains the outer loop to at most O(sqrt(max_limit)) iterations. For each starting index the inner loop runs at most until the partial sum reaches max_limit, so the total work across all pairs is roughly O(max_limit) in the worst case, though in practice the inner loops are short for large starting values. Palindrome checking on a number with d digits is O(d), which is effectively constant here since d <= 9. A set (or hash table in C) is used to deduplicate results - the same palindrome can arise as a consecutive-square sum starting from multiple different bases, so deduplication is essential for a correct answer.

The overall complexity is approximately O(max_limit) with a very small constant, dominated by the double loop over starting and ending indices. For max_limit = 10^8 this completes in well under a second.

Solution Approach

Double-loop enumeration with hash-set deduplication (s0)

I implemented the same algorithm in both Python and C. The outer loop iterates over each possible starting square index i from 1 upward, terminating early once the minimum two-term sum i^2 + (i+1)^2 would already reach or exceed max_limit. The inner loop extends the run one term at a time, accumulating the total, and checks each intermediate sum for palindromicity. Any palindromic total is inserted into a set so duplicates are collapsed. The final answer is the sum of all unique members of that set.

In Python I used a built-in set and string reversal ( s == s[::-1]) for palindrome detection. In C I wrote a simple open-addressing hash table with a power-of-two capacity (65536 buckets, more than adequate for the density of palindromes below 10^9) and a string-comparison palindrome check using snprintf. One language-specific detail in C is that I used a goto to skip the insertion step when a duplicate is already present in the hash table - not elegant, but straightforward and correct. The C version is roughly 3x faster than Python on the main test case (0.024 s vs. 0.073 s) and similarly faster on the extended billion-limit case (0.094 s vs. 0.325 s).

Learnings

The key insight was recognising that many different consecutive-square runs can produce the same palindrome, so the result must be accumulated in a set rather than a simple running sum. Missing this deduplication step gives an inflated answer.

The early-termination condition on the outer loop ( i^2 + (i+1)^2 >= max_limit) is worth noting: the minimum two-term sum grows quadratically in i, so the outer loop only runs to about sqrt(max_limit / 2), keeping the search space manageable even for max_limit = 10^9.

  • Palindrome checking by string reversal is idiomatic and fast enough at these digit lengths.
  • Running totals updated incrementally avoid recomputing prefix sums from scratch on every inner iteration.
  • A fixed-capacity open-addressing hash table in C is a lightweight alternative to a dynamic structure when an upper bound on the number of distinct results is known to be small.
  • For future problems requiring deduplication of results from a two-dimensional search, reaching for a set immediately saves debugging time on subtly overcounted sums.

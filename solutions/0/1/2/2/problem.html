Problem 122: Efficient Exponentiation [Level 15]

Problem Statement

The most naive way of computing $n^{15}$ requires fourteen multiplications: $$n \times n \times \cdots \times n = n^{15}.$$

But using a "binary" method you can compute it in six multiplications:

$$\begin{align} n \times n &= n^2\\ n^2 \times n^2 &= n^4\\ n^4 \times n^4 &= n^8\\ n^8 \times n^4 &= n^{12}\\ n^{12} \times n^2 &= n^{14}\\ n^{14} \times n &= n^{15} \end{align}$$

However it is yet possible to compute it in only five multiplications:

$$\begin{align} n \times n &= n^2\\ n^2 \times n &= n^3\\ n^3 \times n^3 &= n^6\\ n^6 \times n^6 &= n^{12}\\ n^{12} \times n^3 &= n^{15} \end{align}$$

We shall define $m(k)$ to be the minimum number of multiplications to compute $n^k$; for example $m(15) = 5$.

Find $\sum\limits_{k = 1}^{200} m(k)$.

Mathematical Complexity

The core concept here is the addition chain . An addition chain for a positive integer k is a sequence starting at 1 where each subsequent term is the sum of two earlier terms (not necessarily distinct), ending at k. The minimum length of such a chain, minus one, equals m(k) - the minimum number of multiplications needed to compute n^k. Finding the shortest addition chain for a single number is already an NP-hard problem in general, and there is no known closed-form formula. The binary (repeated squaring) method gives an upper bound of roughly 2 log2(k) multiplications, but optimal chains are often shorter, as the n^15 example illustrates. For the range up to 200, the minimum chain lengths are at most around 11 or 12, which makes exhaustive search with aggressive pruning tractable.

Because we need optimal m(k) for all k from 1 to max_limit simultaneously, I exploit the fact that a single DFS tree over addition chains naturally visits many targets at once. The search space grows exponentially in chain depth, so the effective complexity is hard to bound tightly, but the pruning reduces it dramatically in practice. The iterative-deepening structure ensures correctness: the first time a value is reached at a given depth, that depth is provably minimal.

Solution Approach

Iterative-Deepening DFS over Addition Chains (s0 - Python and C)

I implemented the same algorithm in both Python ( p0122_s0.py) and C ( p0122_s0.c). The approach is iterative-deepening depth-first search (IDDFS) over addition chains. Starting from the chain [1], at each step I extend the chain by appending chain[j] + chain[-1] for every index j, trying pairs in descending order (largest sums first, so promising branches are explored before weaker ones). For each candidate value reached at depth d that still appears in the "remaining" set, I record m[val] = d and remove it from the remaining set.

Two pruning rules keep the search manageable:

  • Reachability pruning: if the current chain tip, even when doubled at every future step, cannot reach the smallest remaining target, the branch is abandoned immediately.
  • Depth pruning: the candidate next value is skipped if even repeated doubling from that value for the remaining steps cannot reach 2 (i.e., the value is effectively useless).

The outer loop increments max_depth from 1 up to 14 (sufficient for all values up to 400). Once the remaining set is empty, the search terminates early. Using a sorted array for remaining values in C enables binary search to check membership, while the Python version uses a set for O(1) average lookup. The C version also uses memmove to remove found values from the sorted array. Both versions share identical algorithmic logic; the C translation is a straightforward port without structural changes.

Learnings

The key insight is that computing m(k) for all k up to a limit at once is much more efficient than solving each target independently. A single IDDFS tree over addition chains touches many targets naturally, so the per-target overhead is amortized across the whole search. The "batch IDDFS with a shrinking remaining set" pattern - where any value first encountered at depth d is immediately locked in as optimal and removed from future consideration - is the design decision that makes this approach work cleanly.

The benchmark results show a dramatic performance gap between the two implementations: the C version solves the main case ( max_limit=200) in about 2 ms versus 122 ms for Python, and the extra case ( max_limit=400) in 7.5 ms versus 900 ms - roughly a 60x and 120x speedup respectively. This gap is larger than typical for simpler problems because the inner DFS loop is very tight and call-heavy, exactly the kind of workload where Python's interpreter overhead compounds most severely. For future problems involving exhaustive search over combinatorial structures with pruning, porting to C is especially worthwhile when the Python solution already works but takes seconds rather than milliseconds.

Addition chains are worth remembering as a concept: they bridge exponentiation algorithms and shortest-path problems in a non-obvious way. The greedy "always double" heuristic (binary exponentiation) is easy and good, but not always optimal - the slack between greedy and optimal is exploitable by allowing arbitrary pairs from the current chain rather than just doubling the last element.

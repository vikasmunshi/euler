Problem 110: Diophantine Reciprocals II [Level 11]

Problem Statement

In the following equation $x$, $y$, and $n$ are positive integers.

$$\dfrac{1}{x} + \dfrac{1}{y} = \dfrac{1}{n}$$

It can be verified that when $n = 1260$ there are $113$ distinct solutions and this is the least value of $n$ for which the total number of distinct solutions exceeds one hundred.

What is the least value of $n$ for which the number of distinct solutions exceeds four million?

NOTE: This problem is a much more difficult version of Problem 108 and as it is well beyond the limitations of a brute force approach it requires a clever implementation.

Mathematical Complexity

The key algebraic step is rewriting 1/x + 1/y = 1/n as (x-n)(y-n) = n². Each distinct solution pair corresponds to a factorisation of n², and because the equation is symmetric in x and y, the number of distinct solutions is ceil(tau(n²) / 2), where tau is the divisor-counting function. For n = p1^a1 * p2^a2 * ... we have tau(n²) = prod(2*ai + 1). The problem therefore reduces to: find the smallest positive integer n whose prime factorisation yields prod(2*ai + 1) >= 2K - 1, where K is the target solution count. Because the optimal n is always a product of small primes in non-increasing order of exponent (any other arrangement would be dominated), the search space is a tree over the first handful of primes with bounded, decreasing exponent sequences.

The asymptotic complexity is hard to state cleanly because the search tree size depends on the target threshold. In practice the depth is bounded by the number of primes needed (at most 17 primes up to 59 suffice for targets up to four million solutions), and aggressive branch-and-bound pruning keeps the explored nodes tiny. The dominant cost is effectively constant for any fixed target of this magnitude.

Solution Approach

Branch-and-Bound DFS over Prime Exponent Sequences (s0 - Python and C)

I implemented a depth-first search over exponent assignments to the first 17 primes. The state at each step is the current prime index, the accumulated product n, the accumulated tau value prod(2*ai + 1), and the maximum exponent allowed at this depth (enforced non-increasing to avoid counting the same n twice via different orderings). Two pruning rules fire at every node:

  • Upper-bound prune on n: if the partial product already meets or exceeds the current best answer, the branch is abandoned immediately.
  • Lower-bound prune on tau: if - even assuming every remaining prime is assigned the same current exponent e - the total tau still cannot reach the target, then e and all smaller values for this slot are skipped with a break.

When tau >= target_tau is first satisfied, n is recorded as a candidate best and the subtree is not explored further. I seed best with a conservative upper bound (the product of the first k primes where k is just enough to exceed the target with all exponents equal to 1) so that pruning is effective from the very first DFS call.

The Python and C implementations follow identical logic. The C version handles potential overflow explicitly with pre-multiply checks against best and a ceiling of 9e18, while Python's arbitrary-precision integers avoid that concern entirely. Both produce the same result; the C version simply runs roughly 100x faster in wall-clock time.

Learnings

The central "aha moment" was recognising that counting solutions to the reciprocal equation is purely a divisor-counting problem on n², and that minimising n subject to a tau lower bound is equivalent to choosing a prime factorisation whose exponents are non-increasing and whose product of (2*ai + 1) terms is large enough. Once I saw that, the DFS with branch-and-bound was a natural fit - exactly the same pattern as Problem 108 but with a target four orders of magnitude larger, which is why brute force fails and the pruning is essential.

The benchmark comparison is instructive. The Python solution solves the main case in about 54 ms and the C solution in about 0.4 ms - a roughly 135x speedup. For a problem the note explicitly warns requires a "clever implementation", even the Python solution is fast enough because the algorithmic work is genuinely small; the C version is just a nice-to-have for archival completeness. The technique of bounding the remaining tau contribution with factor^remaining * current_tau is worth remembering for any future problem that requires minimising a multiplicative function over exponent vectors - it turns an exponential search into something effectively linear in practice.

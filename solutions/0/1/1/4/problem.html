Problem 114: Counting Block Combinations I [Level 8]

Problem Statement

A row measuring seven units in length has red blocks with a minimum length of three units placed on it, such that any two red blocks (which are allowed to be different lengths) are separated by at least one grey square. There are exactly seventeen ways of doing this.

How many ways can a row measuring fifty units in length be filled?

NOTE: Although the example above does not lend itself to the possibility, in general it is permitted to mix block sizes. For example, on a row measuring eight units in length you could use red (3), grey (1), and red (4).

Mathematical Complexity

This problem asks for the number of ways to tile a row of length n using grey squares and red blocks of minimum length m, where any two red blocks must be separated by at least one grey square. The key insight is that this is a classic combinatorial tiling problem, best approached with dynamic programming rather than direct combinatorics. The state is simply the position along the row, and the recurrence captures two choices at each position: place a grey square and advance by one, or place a red block of some length k (where k ranges from m to the remaining row length) followed by a mandatory separating grey square (unless the block reaches the very end).

The recurrence fills a table of size O(n), but at each position the inner loop over block lengths k runs up to O(n) iterations, giving an overall time complexity of O(n^2) and space complexity of O(n). For the problem's target of n = 50 this is entirely trivial, but the approach scales gracefully to much larger rows.

Solution Approach

Backward Dynamic Programming (s0 - Python and C)

I defined f[i] as the number of valid arrangements for positions i through n-1, where position i is free to begin either a grey square or a red block. The base case is f[n] = 1, representing the single empty arrangement once all positions are filled. I compute the table from right to left:

  • Grey at position i: contributes f[i+1], moving one step right.
  • Red block of length k starting at position i (for k from m to n-i): if the block ends exactly at n, it contributes 1 (no trailing grey needed); otherwise it contributes f[i+k+1], skipping one mandatory grey square after the block.

The answer is f[0]. I implemented this identically in both Python and C. The C version uses calloc for the dp array and reads n and m from command-line arguments, while the Python version receives them as keyword arguments via a solve function. There are no algorithmic differences between the two implementations - the C version is simply a direct translation. Both pass the dev case (n=7, m=3 giving 17) and the main case (n=50, m=3 giving 16475640049).

Learnings

The key insight for me was recognising that the mandatory one-grey-square gap between red blocks can be absorbed naturally into the recurrence by jumping over it when placing a block: after a block of length k at position i, the next free position is i+k+1, not i+k. This elegantly encodes the separation rule without any special-case logic. The only other edge case to handle carefully is when a block reaches exactly to the end of the row, which does not require a trailing grey square - contributing 1 directly rather than looking up f[n+1].

Since I only implemented one algorithm (s0), there is no comparison between approaches. The benchmark results confirm both the Python and C solutions are extremely fast at this scale - Python finishes in under 0.1 ms and C is effectively instantaneous. For future tiling and placement problems, this backward-DP pattern - defining f[i] as ways to complete the arrangement from position i onward, with choices that skip ahead by variable amounts - is a versatile template worth keeping in mind.

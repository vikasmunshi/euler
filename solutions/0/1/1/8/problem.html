Problem 118: Pandigital Prime Sets [Level 14]

Problem Statement

Using all of the digits $1$ through $9$ and concatenating them freely to form decimal integers, different sets can be formed. Interestingly with the set $\{2,5,47,89,631\}$, all of the elements belonging to it are prime.

How many distinct sets containing each of the digits one through nine exactly once contain only prime elements?

Mathematical Complexity

The core challenge is partitioning the digit set {1, 2, ..., 9} into non-empty subsets such that every subset, when its digits are arranged as a decimal integer in some order, forms a prime. Because sets are unordered, I needed a canonical way to count each partition exactly once. The natural choice is to impose a strictly increasing order on the prime values chosen across successive subsets during the search. The digit space is tiny - nine digits, 511 non-empty bitmasks - so the precomputation phase is cheap. Primality of each candidate number is tested by trial division, which is fast enough here because the largest possible number is a 9-digit permutation of {1,...,9}, well within the range where trial division up to the square root is practical.

The main combinatorial task is enumerating all set partitions of a 9-element set whose parts each yield a prime. The number of set partitions of a 9-element set (the Bell number B(9)) is 21,147, so the search space is manageable. Using bitmask DP to track which digits have been used and sub-setting the remaining bits efficiently reduces the backtracking to essentially a depth-first traversal over valid partial partitions. The total work is dominated by the precomputation step, which runs through all permutations of each subset (at most 9! = 362,880 permutations for the full set, far less for small subsets), making the overall complexity roughly O(sum over subsets of k! * sqrt(max_value_k)) where k is the subset size.

Solution Approach

Bitmask Precomputation with Backtracking (s0 - Python and C)

My approach has two phases. In the precomputation phase I iterate over all 511 non-empty bitmasks of the digits 1-9. For each mask I collect the corresponding digit values, enumerate every permutation of those digits, form the decimal integer, and test it for primality. The results are stored as a sorted list of distinct primes keyed by bitmask. In Python I used a set to deduplicate before sorting; in C I do an explicit linear scan before inserting into a dynamically grown array, then qsort at the end.

The backtracking phase maintains a used_mask (bits for digits already placed) and a min_prime value (the largest prime chosen so far). At each step I enumerate all non-empty subsets of the remaining digits using the standard bit trick sub = (sub - 1) & remaining, look up each subset in the precomputed table, and recurse for every prime in that subset that is strictly greater than min_prime. This strictly increasing requirement ensures each unordered partition is counted exactly once. When used_mask reaches 511 (all nine bits set) I count it as one valid set.

The Python and C implementations are algorithmically identical. The C version uses a flat array of 512 dynamically allocated integer arrays instead of a dictionary, and the primality check and backtracker are plain iterative/recursive C functions. The C version runs in about 0.21 seconds compared to 4.45 seconds for Python - a roughly 21x speedup - driven mainly by the cost of generating and testing permutations in the precomputation loop and the overhead of Python recursion in the backtracker.

Learnings

The key insight was recognising that the "distinct sets" requirement maps directly onto a set-partition problem, and that imposing a canonical ordering (strictly increasing prime values) on the elements as they are chosen during backtracking eliminates all duplicate counting with zero extra bookkeeping. Without this trick one would need to sort and deduplicate sets after the fact, which is more complicated.

A second useful idea is separating the problem into offline precomputation (all primes per bitmask) and online search (backtracking over partitions). The precomputation is done once over a tiny space (511 masks, at most a few thousand permutations each), and the backtracker then only ever looks up precomputed tables rather than re-testing primality at every node.

The bitmask subset enumeration idiom sub = (sub - 1) & remaining is worth remembering for any problem that requires iterating over all subsets of a given set. It visits every non-empty subset of remaining in decreasing order and terminates cleanly when sub reaches zero. This pattern appears frequently in bitmask DP problems and is far more efficient than checking all 512 masks and filtering.

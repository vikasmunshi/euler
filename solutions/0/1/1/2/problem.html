Problem 112: Bouncy Numbers [Level 2]

Problem Statement

Working from left-to-right if no digit is exceeded by the digit to its left it is called an increasing number; for example, $134468$.

Similarly if no digit is exceeded by the digit to its right it is called a decreasing number; for example, $66420$.

We shall call a positive integer that is neither increasing nor decreasing a "bouncy" number; for example, $155349$.

Clearly there cannot be any bouncy numbers below one-hundred, but just over half of the numbers below one-thousand ($525$) are bouncy. In fact, the least number for which the proportion of bouncy numbers first reaches $50\%$ is $538$.

Surprisingly, bouncy numbers become more and more common and by the time we reach $21780$ the proportion of bouncy numbers is equal to $90\%$.

Find the least number for which the proportion of bouncy numbers is exactly $99\%$.

Mathematical Complexity

The classification of a positive integer as increasing, decreasing, or bouncy reduces to a simple left-to-right scan of its digit sequence. An increasing number has each digit no greater than the one to its right; a decreasing number has each digit no less than the one to its right. A number satisfying neither condition is bouncy. Equal adjacent digits satisfy both monotonicity conditions simultaneously, so single-digit and two-digit numbers are always classifiable as one or the other, which is why no bouncy numbers exist below 100. The problem asks for the first integer n such that exactly target_percentage / 100 of all integers from 1 to n are bouncy - a density condition with no closed-form solution, so a linear scan is required.

Classifying a single number costs O(d) where d is its number of digits, so the overall complexity of scanning up to the answer N is O(N log N) since d = O(log N). For the 99% target the answer is 1,587,000, which makes this completely tractable. The density of bouncy numbers increases monotonically toward 1 as numbers grow, guaranteeing the target percentage is always eventually reached and that the scan terminates.

Solution Approach

Linear Scan with Per-Number Digit Classification (s0)

I implemented one algorithm in both Python and C. The approach iterates n from 1 upward, classifies each number as bouncy or not, increments a running counter, and after each step checks whether bouncy_count * 100 == target_percentage * n. I chose this integer cross-multiplication form deliberately to avoid any floating-point comparison, since the problem asks for exact equality rather than a threshold crossing.

In the Python version ( p0112_s0.py) I convert n to a string and use all() with generator expressions over adjacent character pairs to test the increasing and decreasing conditions. This is concise and idiomatic Python. The function accepts target_percentage as a keyword argument, and the script reads it from argv[1], making all three test cases (50%, 90%, 99%) runnable without code changes.

In the C version ( p0112_s0.c) I extract digits into a small fixed-size array by repeatedly taking n % 10, then reverse the array to restore left-to-right order, then scan adjacent pairs with two boolean flags. As soon as both flags are false I return early without finishing the digit scan. The target percentage is read from argv[1] via atoi, mirroring the Python interface. The --runs flag is stripped before passing arguments to solve so the benchmarking harness does not interfere with argument parsing.

Learnings

The key insight I found most satisfying was the integer-equality check. The naive floating-point formulation bouncy_count / (double)n == target_percentage / 100.0 can silently miss the exact crossing point due to rounding. Cross-multiplying to bouncy_count * 100 == target_percentage * n costs nothing extra and is provably exact for the integer values involved. This is a pattern worth applying whenever a problem asks for an exact fractional equality rather than a threshold.

The C solution is roughly 70x faster on the main case, solving it in under 20 ms compared to about 1.4 seconds for Python. The Python time is still acceptable for a Level 2 problem. A workflow habit reinforced here: parameterising the target percentage as a command-line argument from the start made it trivial to verify the two intermediate checkpoints given in the problem statement (538 for 50%, 21780 for 90%) before committing to the expensive 99% run. Any problem that supplies intermediate numerical checkpoints in its statement should be treated as an opportunity to build that kind of sanity-check into the solution interface from day one.

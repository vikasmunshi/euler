Problem 115: Counting Block Combinations II [Level 9]

Problem Statement

NOTE: This is a more difficult version of Problem 114 .

A row measuring $n$ units in length has red blocks with a minimum length of $m$ units placed on it, such that any two red blocks (which are allowed to be different lengths) are separated by at least one black square.

Let the fill-count function, $F(m, n)$, represent the number of ways that a row can be filled.

For example, $F(3, 29) = 673135$ and $F(3, 30) = 1089155$.

That is, for $m = 3$, it can be seen that $n = 30$ is the smallest value for which the fill-count function first exceeds one million.

In the same way, for $m = 10$, it can be verified that $F(10, 56) = 880711$ and $F(10, 57) = 1148904$, so $n = 57$ is the least value for which the fill-count function first exceeds one million.

For $m = 50$, find the least value of $n$ for which the fill-count function first exceeds one million.

Mathematical Complexity

This problem is a direct extension of Problem 114. The fill-count function F(m, n) counts the number of ways to tile a row of length n using red blocks of minimum length m, with at least one black square separating any two red blocks. The counting naturally decomposes by examining the rightmost position: either it is black (reducing to F(n-1)), or it is the right end of a red block of length k where m ≤ k < n (requiring a mandatory black separator to the left, reducing to F(n-k-1)), or the entire row is one red block of length n (contributing exactly 1). This recurrence is clean enough to evaluate with memoization, and each F(n) depends on O(n) previously computed values, giving an overall time complexity of O(n^2) to compute all values up to the answer. In practice, for m=50 the answer is n=168, so the table is tiny and runs in well under a millisecond.

The problem is really asking for the smallest n such that F(m, n) exceeds a threshold - a simple linear search on n starting from m, with each step requiring an O(n) recurrence evaluation. The dominant computational concern is avoiding redundant recomputation of subproblems, which both my Python and C solutions handle through memoization or a cache array respectively.

Solution Approach

Memoized Recurrence (Python and C, index 0)

Both my Python and C solutions implement the same algorithm: a top-down memoized recursion for F(n) combined with a linear search for the smallest n exceeding the given limit. The recurrence I use is:

  • If n < 0, return 0 (base case for the separator offset going negative).
  • If n < m, return 1 (only the all-black arrangement fits; no red block of length m can be placed).
  • Otherwise: F(n) = F(n-1) + sum over k from m to n-1 of F(n-k-1) + 1.

The final +1 accounts for the single arrangement where a red block covers the entire row. In Python I used functools.lru_cache for zero-boilerplate memoization. In C I maintained a flat array cache[] and a parallel boolean array cache_valid[] to replicate the same top-down approach. The C version takes m and the limit as command-line arguments (as integers), mirroring the Python keyword arguments m and limit. Both versions are functionally identical in logic.

I only implemented one algorithmic approach (index 0) in both languages. An equivalent bottom-up iterative DP would also work and would avoid any recursion overhead, but given how small n ends up being (168 for m=50), the top-down approach is perfectly sufficient.

Learnings

The key insight was recognising that the recurrence from Problem 114 generalises directly by treating m as a parameter rather than a hardcoded constant. Once I saw that F(m, n) decomposes cleanly by the character of the last occupied position, the recurrence wrote itself. The "aha moment" was confirming that the boundary case n < m correctly collapses to 1 (just the all-black row), which anchors the recursion.

Because I only tried one algorithmic approach, there is no meaningful comparison between solution indices. The Python solution runs in about 0.7 ms for the main case (m=50, limit=1000000), which is already fast enough that porting to C brings no practical benefit here - I included the C version primarily for consistency with the rest of my archive. The technique of parametrising a tiling recurrence by a minimum block length is worth remembering: it comes up repeatedly in combinatorial tiling problems and the top-down memoized form is easy to adapt quickly.

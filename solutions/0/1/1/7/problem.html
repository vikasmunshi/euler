Problem 117: Red, Green, and Blue Tiles [Level 8]

Problem Statement

Using a combination of grey square tiles and oblong tiles chosen from: red tiles (measuring two units), green tiles (measuring three units), and blue tiles (measuring four units), it is possible to tile a row measuring five units in length in exactly fifteen different ways.

How many ways can a row measuring fifty units in length be tiled?

Mathematical Complexity

This problem is a direct application of linear recurrence via dynamic programming. The number of ways to tile a row of length n using tiles of sizes 1, 2, 3, and 4 satisfies the recurrence dp[i] = dp[i-1] + dp[i-2] + dp[i-3] + dp[i-4], with base case dp[0] = 1 (one way to tile an empty row). This is essentially a generalised Fibonacci-style recurrence with a window of four previous terms. The key insight is that including the grey unit tile (size 1) - unlike Problem 116 which forbids it - means every position can be filled by at least a grey square, making the recurrence straightforward: at each position we simply consider which tile ends at that position and sum the ways the prefix before that tile could be tiled.

The algorithm runs in O(n) time and O(n) space (reducible to O(1) space if only the last four values are kept). For n = 50 the answer fits comfortably in a 64-bit integer, but for larger values such as n = 500 the result grows to roughly 140 decimal digits, requiring arbitrary-precision arithmetic.

Solution Approach

Dynamic Programming with Optional Big Integer Support (s0)

I implemented the same recurrence in both Python and C. In both cases I allocate a dp array of length n + 1, set dp[0] = 1, and fill it forward using the four-term sum. The transition at each step checks bounds to avoid out-of-range accesses for the first few indices.

In Python, arbitrary-precision integers are native, so the same code handles both the n = 50 and n = 500 test cases without any modification. In C, a 64-bit long long suffices for n = 50, but I wanted the C solution to also be correct for larger inputs, so I implemented a simple big-integer type backed by an array of unsigned long long chunks in base 10 18 . Each BigInt struct holds up to 16 such chunks, which is enough to represent values with roughly 288 decimal digits - well beyond the 140 digits needed for n = 500. Addition is performed chunk by chunk with carry propagation. Converting to a decimal string is done by printing the most significant chunk without leading zeros and all remaining chunks with exactly 18 leading zeros via %018llu format. The C solution also parses a --runs=N flag and strips --show before forwarding arguments to the solver, following the same harness pattern used across my solution archive.

Learnings

The key insight that unlocked this problem immediately was recognising that including the unit grey tile collapses the combinatorial structure into a simple linear recurrence - the same recurrence that counts compositions of n using parts from the set {1, 2, 3, 4}. Once that framing is clear, the DP writes itself in a few lines. This is a useful pattern to remember: tiling problems where all integer sizes from 1 up to some maximum are allowed reduce directly to a sum-of-previous-terms recurrence.

The benchmark results show both the Python and C solutions are effectively instantaneous for n = 50, since the problem size is tiny. The main practical difference between the two implementations is that Python's built-in big integers make the n = 500 extra test case trivial, whereas the C version required me to write a minimal big-integer library. That was a useful exercise - using base 10 18 (the largest power of 10 fitting in a 64-bit unsigned integer) keeps the chunk count small and makes decimal output straightforward without needing a division-based digit extraction loop.

  • Tiling with tile sizes {1, 2, ..., k} is equivalent to counting compositions, giving a k-term recurrence.
  • In C, big-integer output is simpler when the base is a power of 10, since chunks print directly as decimal.
  • Python's arbitrary precision makes it ideal for quick verification of large-input test cases before porting to C.

Problem 116: Red, Green or Blue Tiles [Level 8]

Problem Statement

A row of five grey square tiles is to have a number of its tiles replaced with coloured oblong tiles chosen from red (length two), green (length three), or blue (length four).

If red tiles are chosen there are exactly seven ways this can be done.

If green tiles are chosen there are three ways.

And if blue tiles are chosen there are two ways.

Assuming that colours cannot be mixed there are $7 + 3 + 2 = 12$ ways of replacing the grey tiles in a row measuring five units in length.

How many different ways can the grey tiles in a row measuring fifty units in length be replaced if colours cannot be mixed and at least one coloured tile must be used?

NOTE: This is related to Problem 117 .

Mathematical Complexity

This problem reduces to a straightforward combinatorial recurrence. For each fixed tile length L (2, 3, or 4), I need to count the number of ways to fill a row of n grey unit squares by placing any number of same-coloured oblong tiles of length L, with the remaining positions kept as grey unit tiles. Because colours cannot be mixed, each colour is treated as a completely independent sub-problem. The key insight is that at each position i in the row, either a single grey tile occupies that position (extending the arrangement from position i-1), or a coloured tile of length L ends exactly at position i (extending from position i-L). This yields the recurrence dp[i] = dp[i-1] + dp[i-L] with dp[0] = 1, which is a linear recurrence similar in spirit to a generalised Fibonacci sequence. Subtracting 1 from dp[n] removes the all-grey case. The total answer is the sum of the three independent counts.

The asymptotic complexity is O(n) time and O(n) space per tile colour, giving O(n) overall since there are only three fixed colours. For n = 50 this is trivially fast. For larger inputs such as n = 500, the numbers themselves grow very large (over 100 decimal digits), so arbitrary-precision arithmetic dominates the cost in the C implementation, while Python handles big integers natively.

Solution Approach

Dynamic Programming with per-colour recurrence (Python and C)

Both the Python ( p0116_s0.py) and C ( p0116_s0.c) solutions implement the same algorithm. I iterate over the three tile lengths L in {2, 3, 4} and for each one build a DP table where dp[i] counts the number of tilings of a row of length i using grey unit tiles and at least zero coloured tiles of length L. The recurrence is:

  • dp[0] = 1 (one way to tile an empty row)
  • dp[i] = dp[i-1] + dp[i-L] for i >= L, otherwise dp[i] = dp[i-1]

After filling the table, I subtract 1 from dp[row_length] to exclude the all-grey arrangement, then accumulate the result into a running total across all three colours.

In Python, big integer arithmetic is built in, so the implementation is compact and straightforward. In C, I implemented a simple arbitrary-precision library using base-10^9 limb arrays. The C version allocates a BigNum array of size row_length + 1 for each colour, with each BigNum holding up to 64 limbs (sufficient for 576 decimal digits, covering row lengths up to 500). The C code also accepts a command-line argument for the row length, making it easy to verify all test cases including the large extra cases.

Learnings

The key insight that unlocked this problem was recognising that colours cannot be mixed, which decouples the three colours into three independent sub-problems. Each sub-problem is a simple linear recurrence of the form dp[i] = dp[i-1] + dp[i-L] - placing either a grey unit tile or a coloured tile of the fixed length. This is far simpler than Problem 117 where all colours can be mixed simultaneously.

For the extra test cases with row lengths of 100 and 500, the answers have 21 and 103 digits respectively, which means any C solution for this family of problems must include big-integer support. I chose a compact base-10^9 limb representation that is easy to implement and sufficient for the range needed. Python's native arbitrary-precision integers make the Python version much more concise for verifying correctness across all test case sizes.

A technique worth remembering here is that counting arrangements that include "at least one special tile" is most cleanly handled by computing all arrangements (including the all-blank case) via DP, then subtracting the one forbidden all-grey case at the end, rather than trying to enforce the "at least one" constraint inside the recurrence itself.

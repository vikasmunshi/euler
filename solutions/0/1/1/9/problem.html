Problem 119: Digit Power Sum [Level 7]

Problem Statement

The number $512$ is interesting because it is equal to the sum of its digits raised to some power: $5 + 1 + 2 = 8$, and $8^3 = 512$. Another example of a number with this property is $614656 = 28^4$.

We shall define $a_n$ to be the $n$th term of this sequence and insist that a number must contain at least two digits to have a sum.

You are given that $a_2 = 512$ and $a_{10} = 614656$.

Find $a_{30}$.

Mathematical Complexity

The problem asks for numbers where the digit sum raised to some integer power reproduces the original number. The naive approach of scanning every integer and testing this property would be impractical for large terms. The key mathematical inversion is to recognise that any qualifying number N can be written as base^exp where base = digit_sum(N). Since the digit sum of an integer with at most d digits is bounded by 9*d, and since we are searching up to roughly 10^18 (an 18-digit ceiling), the base can never exceed about 162. The exponent starts at 2 (because a one-digit number cannot have a "sum of its digits" in the intended sense) and is limited because 2^exp overflows the ceiling well before exp = 65. The total number of (base, exp) pairs to check is therefore tiny - at most around 200 * 60 = 12 000 - making the problem computationally trivial once the inversion is understood.

Asymptotically, the work is O(B * E) where B is the maximum base (bounded by the digit-sum ceiling) and E is the maximum exponent for the smallest base. Both are small constants relative to the magnitude of the answer, so the algorithm runs in effectively constant time for any reasonable n. A final sort over the collected candidates adds an O(K log K) term where K is the small number of qualifying values found.

Enumeration by Base and Exponent (s0 - Python and C)

I implemented this single algorithm in both Python and C. The approach is identical in both: I iterate base from 2 up to 200 and, for each base, repeatedly multiply to form base^exp starting at exp = 2. For every power that is at least 10 (two or more digits), I compute its digit sum and check whether it equals base. Qualifying values are collected into a list, which is sorted at the end; the answer is the n th element (1-indexed).

In the Python version I used sum(int(d) for d in str(power)) to extract the digit sum, relying on Python's arbitrary-precision integers to avoid overflow entirely. I set the search ceiling at 10**18 and the base limit at 200, which comfortably covers all 30 terms needed. In the C version I used a digit_sum helper that divides out decimal digits in a loop, and I used long long arithmetic with an explicit overflow guard ( power > max_val / base) before each multiplication step. The candidate array is grown dynamically with realloc, and a nested comparator function is passed to qsort. Both versions produce identical results and pass all test cases including the two provided reference values ( a_2 = 512, a_10 = 614656) and the target a_30 = 248155780267521.

Learnings

The key insight that unlocked this problem was inverting the search direction. Instead of asking "does this number equal (its digit sum) raised to some power?", I asked "for this base and exponent, does the resulting power have the right digit sum?". This collapses an unbounded scan into a small bounded double loop. The upper bound on the base follows directly from the fact that digit sums grow only logarithmically with the number itself.

The benchmark results show the C solution running in under a microsecond while the Python solution takes about 2 ms - a roughly 100x speedup typical for tight arithmetic loops. However, even the Python version is essentially instantaneous for this problem size. The most memorable technique here is the overflow-safe multiplication check power > max_val / base used in the C code, which avoids undefined behaviour from signed 64-bit overflow without needing wider types. This pattern is worth keeping in mind whenever iterating powers in C up to a known ceiling.

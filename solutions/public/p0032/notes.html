Problem Analysis

We are asked to find every multiplication identity a * b = c whose three numbers, written side by side, use each digit from 1 to 9 exactly once - a property called being pandigital - and then to sum the distinct products. A naive search over wide ranges of a and b would waste enormous effort, so the first thing I did was reason about digit counts to shrink the search space before writing any enumeration code.

The pivotal observation is a digit-count identity. The concatenation of a, b, and c must be exactly 9 characters long, so the digit-lengths of the three numbers sum to 9. Multiplication, in turn, almost preserves additivity of length: the product of an m-digit number and an n-digit number has either m+n-1 or m+n digits. This follows directly from logarithms, since the digit-count of x is roughly log10(x) + 1 and logarithms turn products into sums. Combining "lengths sum to 9" with "product length is about the sum of operand lengths" leaves only two consistent shapes for (len(a), len(b), len(c)): (1, 4, 4) and (2, 3, 4). Every other arrangement - a 3x3 product, a 1x3 product, and so on - fails this arithmetic and can be discarded without ever computing a product. This combinatorial pruning step is the heart of the solution.

With the two splits fixed, I enumerate ordered selections of digits for a and then for b from the digits a did not use, compute c = a * b, and verify pandigitality. The number of candidates is a small fixed constant (on the order of a couple of thousand), so the runtime is O(1) - it does not grow with any input, because the input is fixed at the digits 1 through 9.

Solution Approach

Constrained Permutation Enumeration with Digit-Split Filtering (Python and C)

Both the Python and C solutions implement the same algorithm; they differ only in how they generate the permutations. The steps are:

  • Fix the two valid splits (a_len, b_len) as (1, 4) and (2, 3). In each, the product has 4 digits, giving 1+4+4=9 or 2+3+4=9.
  • For each split, generate every ordered selection of a_len digits from {1,...,9} to form a, then every ordered selection of b_len digits from the unused digits to form b.
  • Compute c = a * b and check that the concatenation of all three is 1-through-9 pandigital. Because a and b already draw distinct digits from the pool, this check effectively verifies that c supplies exactly the remaining digits.
  • Insert each valid product into a set so a product reachable through more than one (a, b) pair is counted only once, then sum the set.

The Python version expresses the whole search as a single set comprehension. It loops over the two splits, uses itertools.permutations(nine_digits, a_len) for a, and feeds a generator (d for d in nine_digits if d not in a) into a second itertools.permutations(..., b_len) call so that b draws only from unused digits. The walrus operator (:=) assigns a_str, b_str, and c inside the filter condition so they are computed once and reused. Wrapping the generator in set() deduplicates, and sum() finishes the job. The helper is_nine_pandigital confirms both that the decimal string is 9 characters long and that its character set equals {1,...,9}.

The C version reproduces the same logic with explicit nested loops, since there is no permutation library to lean on. For the (1, 4) split it builds a rem[] array of the eight digits not used by a, then runs four nested loops with collision guards (if (q1 == q0) continue) to enumerate the ordered 4-selections. The (2, 3) split uses two outer loops for a and three inner loops for b. The pandigital test formats the three integers into a buffer with snprintf, rejects immediately if the combined length is not 9, then marks each digit in a seen[10] array, rejecting any repeat or any zero. Uniqueness is enforced by set_add, which linearly scans the small products[] array before inserting.

Programming Techniques

Filtering at Construction Time Instead of Post-Filtering

Neither solution generates all 9! = 362,880 permutations and then discards the bad ones. Instead they build only the prefix for a at the correct length and draw the suffix for b from the complement of the already-used digits. In Python this is the inline generator passed to the second permutations call; in C it is the explicitly constructed rem[] array. The principle - apply constraints as early as possible in a search so branches are pruned before they expand - is a cornerstone of efficient enumeration and carries over directly to backtracking, constraint solving, and tree search in general.

The Walrus Operator for Single-Pass Reuse (Python)

The walrus operator lets the code compute a_str, b_str, and c once, inside the if condition, and then reuse c as the value the comprehension yields. Without it I would either compute these values twice or break the comprehension into a loop with temporaries. Reach for an assignment expression whenever a value is needed both in a filter test and in the produced result and you want to keep the pipeline in one place.

Deduplication via a Set

Because the problem warns that a product can arise from several identities, both solutions collect into a set. Python uses the built-in set, a hash table giving amortized O(1) inserts. C uses a linear-scan array because the number of distinct valid products is only a couple of dozen - at that scale a linear scan is simpler, allocates nothing, and stays friendly to the cache. Matching a data structure's sophistication to the actual scale of the data is an engineering judgment worth practising.

Format-String Digit Extraction (C)

The C pandigital check uses snprintf(buf, sizeof(buf), "%d%d%d", a, b, c) to both concatenate the three numbers and obtain their combined digit count from the return value. This trades repeated division by 10 for a single formatting call and gives the length for free, enabling the early exit if (len != 9) return 0 before any character is inspected. Whenever you need both the textual form of a number and its length together, this idiom is concise and hard to get wrong.

Loop Unrolling for Fixed, Small Permutation Depths (C)

Rather than write a general recursive permutation generator, the C code unrolls the enumeration into explicit nested loops - four deep for the (1,4) split, three deep for (2,3) - with inline collision guards enforcing distinctness. When the recursion depth is a known small constant, unrolling avoids call-stack overhead and makes the distinctness logic visible and auditable. This is a good default for fixed-depth combinatorial enumeration where both performance and clarity matter.

Key Takeaways

The insight that unlocks this problem is the digit-count identity: the operand and product lengths must sum to 9, and only the (1,4,4) and (2,3,4) splits are consistent with how multiplication transforms digit-lengths. That single deduction turns a sprawling brute force into a tightly bounded search.

Both solutions are correct and effectively instant. The C implementation runs in about 0.0019 s and the Python in about 0.0112 s - roughly a 6x speedup for C, driven by the absence of interpreter overhead and by avoiding per-candidate string allocation. At this scale the difference is immaterial: the Python version is markedly more concise and readable, while the C version shows how to express the same algorithm with explicit loops when no permutation library is at hand. The algorithmic pruning matters far more than the language choice.

  • Derive structural constraints (digit counts, parity, modular relations) first and use them to eliminate whole classes of candidates before evaluating any single one.
  • When the answer requires distinct values, collect into a set from the start rather than deduplicating afterward.
  • Build constrained permutations by drawing each element from a shrinking pool of unused values instead of generating everything and filtering.
  • For fixed, small enumeration depths, explicit nested loops are faster and easier to verify than a general recursive generator.
  • An assignment expression lets you compute a value once and use it in both a test and a result, keeping pipeline code in a single expression.
  • Match data-structure complexity to the actual data size - a linear scan can beat a hash set when the collection is tiny.

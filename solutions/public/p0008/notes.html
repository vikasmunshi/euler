Problem Analysis

This problem asks for the maximum product of any run of length consecutive digits inside a fixed 1000-digit number. The structure is a classic sliding window problem: I slide a window of fixed width across the sequence, compute a quantity — here the product of the digits — for each position, and track the best value seen. There is no number theory or combinatorics hidden here; the difficulty is purely about doing the obvious thing correctly while watching for two pitfalls: integer overflow and the presence of zero digits.

The search space is tiny. For a string of length N and a window of width L there are exactly N - L + 1 window positions, and each costs up to L multiplications, giving O(N · L) complexity — the product of input size and window width. With N = 1000 and L = 13 that is under 13,000 operations, effectively instantaneous, so a plain brute-force search over every window is entirely sufficient. The one structural fact worth exploiting is that any window containing the digit '0' has a product of zero, which can never beat a window of non-zero digits. The other thing I must reason about up front is the magnitude of the answer: thirteen nines multiply to 9^13 = 2,541,865,828,329, which exceeds the range of a 32-bit integer, so the accumulator must be at least 64 bits wide.

Solution Approach

Sliding-Window Scan (Python and C)

Both the Python and C solutions implement the same sliding window scan: visit every window position, compute its product, and keep the maximum. The two versions differ only idiomatically.

I store the 1000-digit number as a compile-time string constant in both languages — a Python str literal and a C static const char *. In C, adjacent string literals are concatenated by the compiler at no runtime cost, so I can break the number across many lines for readability while still producing one contiguous string. There is no I/O, no parsing, and no allocation for the data itself.

For each starting index i from 0 to N - length inclusive, the algorithm does two things. First it checks whether the window contains a '0', skipping it if so. Second, for surviving windows it multiplies the length digits together and updates the running best. In C I extract each digit with the idiom number[j] - '0', which converts an ASCII character to its numeric value because the C standard guarantees the digits '0' through '9' have consecutive codes. The product accumulator and the return type are both long long to hold the worst-case 9^13 safely.

The Python version expresses the same logic in a single generator expression: each window substring number[i:i+length] is produced lazily, a guard clause if "0" not in sequence filters out zero-containing windows, math.prod over (int(d) for d in sequence) computes each product, and the outer max(...) selects the largest. Python's arbitrary-precision integers mean overflow is never a concern, so no type reasoning is needed.

In both solutions the window length is a parameter rather than a hard-coded constant — read from argv in C and passed as a positional argument in Python. This lets the development case (length = 4, expected 5832) and the main case (length = 13, expected 23514624000) run against the same untouched source.

Programming Techniques

The Zero Annihilator Guard

Before computing any product, both solutions test whether the window contains a zero. Zero is a multiplicative annihilator — an absorbing element — so any product including it collapses to zero, which cannot be the maximum whenever a non-zero window exists. Testing this cheap disqualifying condition before the expensive inner work is a general optimization pattern: when about to perform a costly computation, ask first whether a quick check can rule the candidate out. Here the multiplication is trivially cheap, but the habit pays off where the inner work is genuinely expensive.

ASCII Digit Arithmetic in C

The expression number[j] - '0' turns a digit character into its integer value with one subtraction. This works because the C standard mandates that the ASCII codes for '0' to '9' are contiguous and increasing. I prefer it over a lookup table or a library call because it involves no branching and no memory indirection. Reach for this idiom whenever numeric data is held as ASCII text in C.

Overflow-Aware Type Selection in C

I computed the worst-case product (9^13 ≈ 2.5 × 10^12) before choosing a type and selected long long, which is guaranteed to be at least 64 bits wide. Silent integer overflow in C produces wrong answers that are hard to diagnose, so whenever the upper bound of a computation is calculable, do the calculation and pick the smallest type that safely holds it. Python is exempt because its integers grow arbitrarily large.

Compile-Time String Constants

Both solutions embed the fixed input directly in the source as a string constant. In C the compiler joins adjacent string literals automatically, so the line breaks I use for readability cost nothing at runtime. When data is fixed and known ahead of time, embedding it as a literal avoids file I/O, parsing, and allocation — it is both simpler and faster than reading it at runtime.

Generator Expressions with math.prod

The Python solution composes a generator expression with math.prod to express "map each character to a digit, then multiply them all" in one line. The generator (int(d) for d in sequence) yields digit values lazily, so no intermediate list is built. I chose math.prod over functools.reduce(operator.mul, ...) because it reads more clearly. This compose-a-generator-into-a-reduction pattern is idiomatic Python for any map-then-fold operation.

Parameterising the Window Length

Rather than hard-coding 13, both solutions accept the window width as a runtime parameter. This turns the development test (length 4) and the main test (length 13) into two invocations of the same untouched code. Whenever a problem has a natural scale parameter — a window size, a threshold, an iteration count — make it a variable from the outset rather than a magic number buried in a loop.

Key Takeaways

The core insight is that estimating the search space first tells you a brute-force search is sufficient: under 13,000 operations is nothing, so there is no need for a cleverer scheme such as maintaining a running product by dividing out the departing digit. That division trick is not only unnecessary here but fragile — it breaks exactly when a zero enters the window, the case requiring the most care. The simple scan-and-skip approach is correct, obvious, and trivially fast.

The recorded benchmarks bear out how light this problem is. Averaged over 21 runs, the C solution comes in at roughly 0.0000037 s on the development case and 0.0000051 s on the main case, while the Python solution takes about 0.00032 s and 0.00028 s respectively — all effectively instantaneous. The gap reflects C's lack of interpreter overhead, but at this size the distinction is academic; the Python version, expressing the whole algorithm in one generator expression, is arguably the more elegant.

  • Estimate the size of the search space before choosing an algorithm; if a brute-force search fits the time budget, prefer it for its simplicity and verifiability.
  • Identify multiplicative annihilators such as zero early and skip candidates containing them; it is a cheap guard that also clarifies intent.
  • In C, compute the worst-case value of any product or sum and choose the smallest numeric type — here long long — that holds it safely to avoid silent overflow.
  • Embed fixed, compile-time data as string or array constants rather than reading it at runtime; it is faster and removes a parsing step.
  • Use adjacent string-literal concatenation in C to break long embedded data across lines at no runtime cost.
  • In Python, compose a generator expression with reductions like math.prod and max to express map-then-fold logic in one clear line.
  • Parameterise any number that defines the problem's scale so different test cases run against the same untouched source.

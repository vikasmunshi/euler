Problem Analysis

The problem asks me to count reduced proper fractions n/d with 1 <= n < d whose numerator and denominator share no common factor other than 1, for every denominator up to one million. Two integers whose greatest common divisor is 1 are called coprime. The decisive observation is that for a fixed denominator d, the number of valid numerators is exactly the count of integers in 1..d-1 that are coprime to d - and that quantity is precisely Euler's totient function, written phi(d). So the answer to a question about coprime fractions collapses into a single sum: the running total of phi(d) from d = 2 up to the limit. I start at 2 rather than 1 because phi(1) = 1 would count 1/1, which is not a proper fraction with n < d.

The remaining work is computing phi(d) for every d efficiently. The totient is a multiplicative function given by phi(d) = d * product over distinct primes p dividing d of (1 - 1/p). Because this is a product over prime factors, I can compute all totients up to N simultaneously with a sieve modelled on the Sieve of Eratosthenes: seed each phi[d] = d, then for every prime p apply the factor (p-1)/p to each multiple of p. The cost is dominated by the inner loops, which run N/2 + N/3 + N/5 + ... times over the primes - a sum that grows like N log log N, the same near-linear bound as the Sieve of Eratosthenes itself. Memory is O(N). Both are entirely practical even at N = 10^7.

Solution Approach

Euler Totient Sieve (Python and C)

Both files implement the same totient sieve; the C version exists purely for speed. The algorithm proceeds as follows:

  • Allocate an array euler_totients of length max_d + 1 and set euler_totients[d] = d. This is the "no prime factors applied yet" identity value, matching the leading d in the product formula. In Python I write this compactly as list(range(max_d + 1)); in C it is a single initialisation loop.
  • Iterate n from 2 upward. If euler_totients[n] == n, no smaller prime has yet touched this cell, so n must itself be prime. This doubles as the primality test - I need no separate boolean sieve.
  • When n is prime, I walk every multiple j = n, 2n, 3n, ... and apply euler_totients[j] = euler_totients[j] / n * (n - 1). Each distinct prime factor of j is applied exactly once this way, so once all primes <= j have been processed, the cell at j holds its true totient.
  • I accumulate result += euler_totients[n] in the same outer loop. This is correct because every prime factor of n is at most n, so the totient of n is fully finalised by the time the outer loop reaches index n.

The two implementations differ only in their host language. The C version uses malloc for the heap array and long long to hold the running total (about 3.0e11 at max_d = 1000000, and roughly 3.0e13 at 10^7 - both comfortably inside a 64-bit integer). The Python version relies on arbitrary-precision integers and a built-in list, so there is no overflow concern at all. Both run one linear outer pass with nested per-prime inner passes.

Programming Techniques

Sieve Initialisation with Identity Values

I seed the array with euler_totients[i] = i, the value that survives unchanged when zero prime corrections are applied. This mirrors how the Sieve of Eratosthenes seeds a boolean array as "prime until proven otherwise." The general pattern - initialise with the uncorrected state, then let the sieve apply multiplicative corrections - works for any multiplicative arithmetic function you need to tabulate across a whole range.

In-place Primality Detection

The test euler_totients[n] == n detects primes without a separate flag array. Any composite n would already have had a smaller prime factor reduce its cell below n, so an untouched cell can only be prime. This is a reusable idea: whenever a sieve modifies cells monotonically in one direction, "has this cell been touched?" can serve as a derived predicate, saving an entire parallel array and the memory traffic that goes with it.

Divide-before-Multiply to Avoid Overflow

I deliberately write the update as euler_totients[j] / n * (n - 1) rather than the algebraically equal euler_totients[j] * (n - 1) / n. At the moment of update, euler_totients[j] is guaranteed divisible by n - I am applying n as a fresh prime factor - so the division is exact and loses nothing. Dividing first keeps the intermediate value smaller, which protects against overflow in C's fixed-width arithmetic. The general rule: when a division is known to be exact, perform it before any multiplication.

Accumulation Inside the Sieve Loop

Rather than running the sieve and then summing in a second pass, I add euler_totients[n] to the running total within the same loop. This is correct because each entry's totient is finalised before the outer loop reaches its index. It removes a whole pass over the array and improves cache locality - the value just written is immediately read back from a warm cache line.

Heap Allocation for Large Runtime-sized Arrays (C)

The C code uses malloc and free for the totient array. At max_d = 10^7 that is roughly 80 MB of long long values, far beyond a typical stack limit. Heap allocation is the right tool when an array's size is known only at runtime and may be large, and I pair each malloc with a free so repeated benchmark runs stay leak-free. I also check the allocation result and fail gracefully rather than dereferencing a null pointer.

Key Takeaways

The unlocking insight is the identity "number of reduced proper fractions with denominator exactly d" equals phi(d). Once you see that, a coprime-counting problem becomes a totient sum, and a multiplicative sieve computes all the needed totients in near-linear time.

Both solutions use the same algorithm, so the benchmark is a pure language comparison. On the main case (max_d = 1000000) the C build runs in about 0.0113 s versus 0.5135 s for Python - roughly 45x faster - and on the extra case (10^7) it is 0.2836 s versus 6.6851 s, about 24x faster. The gap reflects the interpreter overhead and dynamic-typing cost Python pays on tight arithmetic loops. Both are correct and elegant; I reach for C when latency matters and Python when clarity and arbitrary precision matter.

  • When counting integers coprime to a fixed modulus, reach for Euler's totient - it is exactly that count.
  • Any multiplicative arithmetic function can be tabulated for all values up to N with one Eratosthenes-style sieve in O(N log log N).
  • Initialise a sieve with identity (uncorrected) values and detect primes by checking whether a cell is still untouched - no extra boolean array required.
  • In integer arithmetic, divide before you multiply whenever the division is known to be exact, to keep intermediates small and avoid overflow.
  • Accumulate results inside the computing loop when each entry is finalised before its index is visited - it saves a pass and improves cache locality.
  • Use heap allocation for large, runtime-sized arrays, pair every allocation with a release, and check for failure to stay both leak-free and robust.

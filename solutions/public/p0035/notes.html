Problem Analysis

A circular prime is a prime number whose every cyclic rotation of decimal digits is also prime. For 197 the rotations 197, 971, and 719 are all prime, so it qualifies. The task is to count circular primes below one million. The naive approach - take every prime, generate its rotations, and test each rotation for primality - is correct but asks the same primality questions over and over. Two observations turn this into a quick search.

The first is a divisibility argument that prunes almost the entire candidate set. If a multi-digit number contains any of the digits 0, 2, 4, 5, 6, or 8, then some rotation places that digit in the units position. A number ending in an even digit is divisible by 2; a number ending in 0 or 5 is divisible by 5. Either way that rotation exceeds 5 and is composite, so the original cannot be a circular prime. Therefore every multi-digit circular prime is built only from the digits 1, 3, 7, and 9 - a tiny fraction of all integers. The single-digit primes 2, 3, 5, and 7 are trivially circular and are handled as a special case. The second observation is that primality is queried many times - once per rotation per surviving candidate - so I should precompute it. The standard tool is a Sieve of Eratosthenes, which marks all composites up to a limit n in O(n log log n) time and O(n) space, after which a primality test is an O(1) array lookup. Sieve construction dominates the runtime; the rotation phase adds only a small constant amount of work per surviving candidate (at most 7 rotations below 10^7).

Solution Approach

Hand-rolled Sieve (Eratosthenes in C, Sundaram in Python)

Both index-0 files build their own sieve, then apply the digit filter and rotation check. The C version (p0035_s0.c) uses a classic Sieve of Eratosthenes over a calloc-zeroed byte array where sieve[i] = 1 means composite. It iterates i from 2 up to sqrt(max_num), and for each prime marks multiples starting at i*i - everything smaller has already been marked by a smaller factor, so this avoids redundant work. The Python version (p0035_s0.py) uses a Sieve of Sundaram: it marks integers of the form i + j + 2*i*j as composite, then each unmarked index k maps to the odd prime 2k+1, with 2 added explicitly. Sundaram works only over odd numbers, halving the working array at the cost of a slightly less cache-friendly access pattern. Both are correct ways to enumerate primes.

Having the primes, both iterate over them. Single-digit primes are counted immediately. For larger primes, has_bad_digit (a scan for the characters "024568") rejects the overwhelming majority. The survivors have their rotations generated by string slicing - str_num[i:] + str_num[:i] in Python, two sequential copy loops in C - and each rotation is tested for primality. A subtle correctness detail: a rotation can numerically exceed max_limit (rotating 199999 yields 999991), so any rotation above the sieve range is rejected. That is safe, because such a value lies outside both the range I was asked to count and the sieve, so it can never be a valid in-range circular prime.

External primesieve Library

The index-1 files keep the same digit-filter and rotation logic but replace the hand-rolled sieve with the tuned primesieve library, which implements a fast segmented sieve in native code. The Python version (p0035_s1.py) imports primes and gets back a list of primes up to the limit. The C version (p0035_s1.c) calls primesieve_generate_primes to obtain an array, then builds its own lookup array where sieve[i] = 1 means prime - the inverted polarity relative to the index-0 C sieve, so the rotation check reads rot > max_limit || !sieve[rot]. The C version also allocates the lookup array up to max(max_limit, 1000000), a floor that keeps the small-limit cases consistent. The algorithmic structure is otherwise identical; only prime generation is delegated to a faster implementation.

Programming Techniques

Cheap-guard-first Filter Ordering

Every solution runs the inexpensive has_bad_digit scan before the more expensive rotation generation and lookup. Because almost every multi-digit prime contains a forbidden digit, this short string pass eliminates the vast majority of candidates cheaply. Ordering predicates from cheapest to most expensive is a universal optimisation - spend the least work to discard the most candidates. Reach for it whenever multiple conditions must all hold and they differ in cost.

Set-based and Array-based O(1) Membership

The Python solutions convert the prime list into a set, turning "is this rotation prime?" into an average O(1) hash table lookup rather than an O(n) linear scan. The C solutions get the same effect with a direct-indexed flag array, O(1) by construction. Whenever you need many membership queries against a fixed integer collection, build a set or indexed array once instead of searching repeatedly.

String-based Digit Rotation

Rather than computing rotations with division, modulo, and powers of ten, both languages convert the number to its decimal string and form rotations by slicing (Python) or two sequential copy loops (C). The conversion cost is negligible for small numbers, and the code is far easier to read and verify than the arithmetic equivalent. Working in string space is often the clearest correct way to manipulate individual decimal digits.

Documenting Boolean Array Polarity at Declaration

The two C files use opposite conventions: the index-0 sieve stores 1 for composite, while the index-1 lookup stores 1 for prime. Each annotates the meaning beside the array. Mixing conventions would silently invert every test, so stating the polarity at the declaration is essential. Whenever a flag array's truth value is not self-evident, comment it immediately.

calloc for Zero-initialised Arrays

Both C solutions allocate the sieve with calloc, which allocates and zeroes in one call - shorter than malloc plus memset, and potentially faster since the OS may hand back pre-zeroed pages. For any flag array whose initial state must be all-zero, calloc is the idiomatic choice.

Wrapping a Native High-performance Library

The index-1 solutions replace hand-written prime generation with primesieve. When a hotspot is raw arithmetic throughput, delegating to a battle-tested native implementation usually beats hand-tuning your own loop - at the cost of an external dependency. The Python speedup here is dramatic, and even in C the library outperforms a textbook Eratosthenes sieve.

Key Takeaways

The insight that unlocks this problem is the digit filter: no multi-digit circular prime can contain an even digit or a 5, because some rotation would push that digit into the units place and produce an obvious composite. This collapses the candidate set to numbers built from {1, 3, 7, 9}, turning a primality-heavy search into a quick scan with a handful of O(1) lookups. Precomputing primality with a sieve makes every rotation test instant.

All four solutions agree on the answers, and the benchmarks show that the choice of sieve dominates. At the main 10^6 limit, the library-backed C version (p0035_s1_c, 0.0026 s) edges out the hand-rolled C sieve (p0035_s0_c, 0.0044 s), and both crush the Python versions. Among Python solutions, primesieve (p0035_s1.py, 0.0327 s) is roughly 5x faster than the pure-Python Sundaram sieve (p0035_s0.py, 0.1813 s). The gap widens at 10^7: 0.0234 s and 0.0494 s for the C versions versus 0.3196 s and 2.0520 s for Python. The consistent pattern - C beats Python, and the tuned native sieve beats the hand-rolled one - confirms that prime generation, not rotation checking, is the bottleneck. For a Python user, p0035_s1.py is the most elegant balance: nearly identical code with an order-of-magnitude speedup from a single import.

  • Apply parity and divisibility constraints to prune candidates before any expensive test - structural filters often eliminate most of the search space for free.
  • Order multiple predicates cheapest-first so the least work discards the most candidates.
  • Precompute primality with a sieve when you will query it many times; pay the O(n log log n) setup once, then enjoy O(1) lookups.
  • Use a set or a direct-indexed array for repeated membership queries instead of scanning a list.
  • String conversion is frequently the clearest way to manipulate individual decimal digits.
  • Always document the truth-meaning of a boolean or flag array next to its declaration.
  • Prefer calloc for zero-initialised arrays in C.
  • Delegate hot arithmetic loops to a tuned native library when one exists; the readability and speed gains usually outweigh the dependency.
  • Guard against generated values that fall outside your computed range - a rotation can exceed the original limit.

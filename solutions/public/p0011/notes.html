Problem Analysis

This problem has essentially no number theory in it - it is a search problem over a small, fixed two-dimensional array. The task is to find the largest product of four consecutive values along any of the eight compass directions in a 20x20 grid. The first key observation is that of those eight directions, only four are genuinely distinct: a run going left is the same set of numbers as the matching run going right, just multiplied in the opposite order, and multiplication is commutative. So I only need to examine horizontal, vertical, and the two diagonal orientations. This immediately halves the work.

The second observation is what keeps the code clean: every one of those four orientations can be anchored at a single cell and walked in a single forward direction. If I pick an anchor cell and restrict it to positions where a run of num_adjacent cells fits both rightward and downward, then all four runs stay inside the grid. This turns the whole problem into a brute-force search: enumerate every valid anchor, compute four products, take the maximum. For a grid of side N and a run length k there are (N - k + 1)^2 valid anchors, and at each anchor I do O(k) multiplications, giving an overall complexity of O((N - k + 1)^2 * k), which simplifies to roughly O(N^2 * k). For N = 20 and k = 4 that is only a few thousand operations - trivially fast, so there is no incentive to look for anything cleverer.

Solution Approach

Single-Pass Anchor Scan (Python and C)

Both files implement the identical algorithm; only the language differs. I iterate over every valid anchor cell (x, y), where "valid" means x and y each range from 0 to dim - num_adjacent inclusive. That bound is the crucial correctness detail: it guarantees that a run of length num_adjacent never walks off the right or bottom edge in any of the four directions.

For each anchor I maintain four running products, all initialised to 1, and update all of them inside a single loop over the offset i from 0 to num_adjacent - 1:

  • Horizontal: grid[x][y + i] - fixed row, advancing column.
  • Vertical: grid[x + i][y] - fixed column, advancing row.
  • Main diagonal: grid[x + i][y + i] - top-left to bottom-right.
  • Anti-diagonal: grid[x + num_adjacent - 1 - i][y + i] - starts at the bottom row of the run and walks upward as the column advances, covering the bottom-left to top-right direction.

The anti-diagonal needs the starting row offset of num_adjacent - 1, which is still in bounds precisely because x itself was capped at dim - num_adjacent. After the inner loop finishes, I compare all four products against the running best and keep the largest. The C version does this with explicit if (... > best) comparisons and four long long accumulators; the Python version expresses the same logic as a generator expression fed to the built-in max().

The most interesting language-specific difference is how Python handles the four accumulators inside a generator expression. Because a generator expression cannot contain assignment statements, I "reset" the four products by iterating once over a single-element tuple for horizontal, vertical, diagonal_1, diagonal_2 in ((1, 1, 1, 1),), and then re-bind those same names on each step of i by iterating over another single-element tuple holding the updated products. The C version simply declares four local variables and multiplies them in place - the more natural form in an imperative language.

Programming Techniques

Parallel Accumulation in a Single Inner Loop

Rather than writing four separate loops - one per direction - I update all four products inside one loop over i. This keeps the four closely related computations side by side, so a reader can verify all of them at a glance, and it avoids re-traversing the same index range four times. In the C version this also helps the compiler: the four accumulators have no data dependency on one another, so the CPU can compute them in parallel via instruction-level parallelism. Reach for this pattern whenever several independent quantities are derived from the same iteration.

Simulating Local Variables in a Generator Expression

Python generator expressions are pure expressions and forbid assignment statements. To introduce mutable named intermediates I iterate over a single-element tuple: for a, b, c, d in ((expr_a, expr_b, expr_c, expr_d),). The loop runs exactly once, binding the four names. This is a niche idiom, but it lets you keep an entirely expression-based generator expression when you genuinely need named local state - useful when splitting into a helper function would obscure the logic more than it clarifies it.

Bounding Loop Ranges Instead of Guarding Array Accesses

Both loops range only over valid anchors (0 .. dim - num_adjacent), so no index inside the inner loop can ever go out of bounds. This concentrates all correctness reasoning at the loop headers rather than scattering if checks across every array access. The result is both faster (no per-access branch) and easier to audit. As a general rule, prefer tightening loop limits over branching inside the hot path.

Hardcoding Fixed Input as a Compile-Time Constant

The grid never changes, so both solutions embed it directly in source - a Python tuple-of-tuples and a C static array. This removes all file I/O from the critical path. In C the static array is placed in read-only data and stays cache-warm, giving the compiler full freedom over memory layout. When a problem's input is small and known ahead of time, hardcoding it is a legitimate and practical choice.

Choosing an Integer Type by Worst-Case Magnitude

For k = 4 the largest possible product is 99^4 = 96,059,601, which fits comfortably in a 32-bit int. I still declared the C accumulators as long long because the run length is a runtime parameter, and a larger k would overflow a 32-bit type silently and produce a wrong answer with no error. The principle generalises: before accumulating products, compute the worst-case magnitude and pick an integer width that can hold it.

Key Takeaways

The unlocking insight is recognising that the eight named directions collapse to four distinct ones (thanks to commutativity), and that all four can be anchored at one cell and walked forward if the anchor range is bounded so runs never leave the grid. Once you see that, the problem is a tiny, fully enumerable brute-force search.

Both solutions compute the same answer of 70600674. The benchmark shows the C version at roughly 0.0000016 s versus about 0.00024 s for Python - the C build is on the order of 150 times faster, which is unsurprising given that the work is a few thousand integer multiplications dominated by interpreter overhead in Python. The Python version is the more elegant expression of the idea (a single max() over a generator expression), while the C version is the one to reach for if raw speed ever mattered; here, both are effectively instant.

  • When the search space is small and bounded, full enumeration is usually the clearest and most maintainable solution - add cleverness only when profiling demands it.
  • Use symmetry to eliminate redundant work: commutative operations mean opposite directions need not be searched twice.
  • Accumulate independent quantities in a single loop pass to keep related logic together and to expose parallelism to the compiler.
  • Move bounds-checking into loop limits rather than guarding each access; it is faster and centralises correctness reasoning.
  • Hardcode small fixed inputs as compile-time constants to keep I/O off the critical path.
  • Always size integer types against the worst-case magnitude of any product; overflow in C is silent and corrupts results.

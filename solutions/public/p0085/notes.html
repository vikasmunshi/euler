Problem Analysis

The heart of this problem is a counting question: how many distinct sub-rectangles fit inside an H x W grid? The naive mental model is to slide every possible smaller rectangle across every position, but there is a far cleaner way to see it. A grid of height H and width W is bounded by H + 1 horizontal lines and W + 1 vertical lines. Any rectangle is uniquely determined by choosing two of the horizontal lines (its top and bottom edges) and two of the vertical lines (its left and right edges). Because the choice along each axis is completely independent of the other, the total count factors into a product. This is the structural insight that turns a two-dimensional search into something far smaller.

The number of ways to choose 2 items from n is the binomial coefficient C(n, 2) = n*(n-1)/2. Choosing 2 lines from H + 1 lines therefore gives (H+1)*H/2, which is exactly the triangular number T(H) = H*(H+1)/2 - the running sum 1 + 2 + ... + H. So the total rectangle count for an H x W grid is T(H) * T(W). With 3 x 2 this gives T(3) * T(2) = 6 * 3 = 18, matching the worked example. This factored form is the key observation: instead of enumerating all rectangles, we only need to find the pair of triangular numbers whose product is closest to two million. Since the triangular numbers form a strictly increasing sequence, we can iterate over one axis and use a binary search on the other, giving an O(max_side * log(max_side)) algorithm. The brute-force alternative that literally sums over every sub-rectangle position is O(max_side^2) in the outer loop with an additional O(H*W) inner cost per pair - correct, but far slower, and useful mainly for illustrating the definition.

Solution Approach

Algorithm 0 - Brute-Force Position Counting (s0)

This approach counts rectangles exactly as the definition suggests. In num_rectangles(height, width) I sum (height - h + 1) * (width - w + 1) over every sub-rectangle dimension (h, w). The term (height - h + 1) is the number of vertical positions a sub-rectangle of height h can occupy, and likewise for width; multiplying and summing over all (h, w) yields the total. I then iterate over every grid shape (height, width) and track the count closest to the target.

The Python version (p0085_s0.py) uses itertools.product to generate the grid shapes and stores every result in a dictionary keyed by count, then selects the global best with a single min(..., key=delta) call. The C version (p0085_s0.c) performs the same nested summation but keeps only a running best_area and best_delta, avoiding the memory overhead of a dictionary. Both versions break out early once a count falls within max_error of the target; in C this early exit uses goto done to escape both loops cleanly. This algorithm is instructive but expensive - the inner double loop recomputes the count from scratch for every grid shape.

Algorithm 1 - Triangular-Number Binary Search (s1)

This approach exploits the factored formula count(H, W) = T(H) * T(W). I precompute an array numbers where numbers[i] = T(i+1) for lengths from 1 up to max_side - 1. Because triangular numbers increase monotonically, this array is sorted and supports binary search. In the Python version num_rectangles_along_axis computes T(length) by the explicit running sum, while the C version uses the closed form length*(length+1)/2; both yield the same sequence.

For each width w I read off num_width = T(w) and ask: what height H makes T(H) * num_width closest to the target? That ideal satisfies T(H) ~= target / num_width. I locate the insertion point of target / num_width in the sorted array using bisect.bisect_left in Python or a hand-rolled bisect_left in C. The closest triangular number must be one of the two neighbours of that insertion point, so I evaluate both candidates, compute their products with num_width, and update the running best. The same early-exit on max_error applies, but now it triggers far sooner because each inner step is O(log n) rather than a linear scan. The C version frees its allocated array before returning, and uses integer division for target / num_width - acceptable here because the array neighbours bracket the true closest value regardless of the small rounding from truncation.

Programming Techniques

Closed-Form Counting via Separable Axes

Rather than simulate placements, I recognised that the count factors into independent per-axis contributions: T(H) * T(W). Whenever a counting problem on a grid or lattice separates into independent choices along each dimension, you can compute each dimension once and combine the results. This collapses a two-dimensional search into one precomputed list plus a one-dimensional scan, and it is almost always the first structural simplification worth looking for.

Binary Search for the Nearest Value

Because the triangular numbers are sorted, I use a binary search (bisect_left) to find where target / num_width would be inserted in O(log n) time. The standard idiom for finding the nearest value (not an exact match) is to check both neighbours around the insertion point: the closest element is guaranteed to be one of numbers[j-1] or numbers[j]. The Python solution leans on the standard-library bisect module; the C solution implements the same lower-bound search by hand. Reach for this pattern any time you need the closest entry in a sorted sequence.

functools.partial for Reusable Key Functions

I defined a standalone delta_func(kv, *, target_num) and bound the target with functools.partial, producing a single-argument callable suitable for use as min(..., key=delta). This is cleaner than a lambda capturing a variable by reference, since the bound value is fixed at partial-creation time and the intent is explicit. Use this whenever a scoring or comparison function depends on a runtime parameter.

Walrus Assignment Inside an Expression

In both Python solutions I write results[(num := ...)] = (...), using the assignment expression (the walrus operator :=) to capture the computed count into num while simultaneously using it as a dictionary key. This avoids a redundant separate statement and keeps the freshly computed value available for the immediate max_error check. It is a tidy idiom when a value is needed in two places within one logical step.

goto for Multi-Level Loop Exit in C

The C s0 solution uses goto to break out of both loops the moment the error threshold is met. A boolean flag would require an extra check on every outer iteration, and factoring into a helper would obscure what is conceptually one search. A forward goto to a label in the same function is the canonical, readable C idiom for escaping nested loops.

Running Best vs. Result Registry

The two s0 implementations illustrate a deliberate trade-off. The Python version builds a dictionary of every result and computes the minimum afterwards - readable, but it costs memory. The C version tracks only best_area and best_delta as it goes. When the search space is small and clarity matters, a registry is fine; when memory or performance is the concern, maintain a running best inside the loop.

Key Takeaways

The decisive insight is that the rectangle count factors as a product of two triangular numbers, one per axis. Once that is recognised, the problem reduces to finding the pair of triangular numbers whose product is closest to two million, and the binary-search optimisation follows naturally from the sequence being sorted.

The benchmarks make the payoff vivid. On the main case, brute-force Python (p0085_s0.py) runs in about 0.1499 s, while the binary-search Python (p0085_s1.py) finishes in roughly 0.0002 s - a speedup of several hundred times purely from algorithmic insight. The C brute-force (p0085_s0_c) takes about 0.0008 s and the C binary-search (p0085_s1_c) registers as effectively instantaneous at 0.0000 s. Strikingly, the right algorithm in Python (0.0002 s) beats the wrong algorithm in C (0.0008 s): mathematical structure matters more than the language. The s1 binary-search approach is both the fastest and the most elegant.

  • When counting objects in a 2-D grid, look for separability - if the count factors along each axis independently, handle each axis separately and combine.
  • Triangular numbers T(n) = n*(n+1)/2 appear whenever you count ways to choose 2 items from an ordered set; learn to recognise them in line-selection problems.
  • To find the closest value in a sorted sequence, binary-search for the target and inspect the two neighbours of the insertion point.
  • Bind fixed parameters into key functions with functools.partial rather than relying on closures over mutable variables.
  • In C, a forward goto to a label is the clean way to exit nested loops on an early-termination condition.
  • Invest in understanding the underlying formula before optimising code - algorithmic insight usually beats language-level micro-optimisation by a wide margin.

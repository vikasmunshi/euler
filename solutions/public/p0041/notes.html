Problem Analysis

An n-digit number is pandigital if it uses each of the digits 1 through n exactly once. I want the largest such number that is also prime. The brute-force formulation is clear: for every length from 1 to 9, generate every permutation of the digit set, test each for primality, and take the maximum. That is feasible - the largest length, 9, has only 9! = 362880 arrangements - but a single piece of number theory collapses almost all of the work before I write a search loop.

The decisive observation is the divisibility rule for 3: a number is divisible by 3 exactly when the sum of its digits is divisible by 3. For an n-digit pandigital number, the digit sum does not depend on the arrangement - it is the triangular number n(n+1)/2. Evaluating it: n=9 gives 45, n=8 gives 36, n=6 gives 21, n=5 gives 15, n=3 gives 6, n=2 gives 3 - all divisible by 3, so every pandigital of those lengths is composite. Only n=1 (sum 1, but the value 1 is not prime), n=4 (sum 10), and n=7 (sum 28) survive. That single analytic step cuts nine candidate lengths down to two. What remains is to enumerate the permutations of {1,2,3,4} and {1,2,3,4,5,6,7} - 4! = 24 and 7! = 5040 candidates - and test each for primality. Since the largest candidate is below 8 million, trial division up to the square root (about 2800) is far more than fast enough. The overall complexity is O(k! * sqrt(N)) with k at most 7 - microseconds in practice.

Solution Approach

Descending Permutation Search (C and Python)

Both files implement the same algorithm: enumerate the permutations of each viable digit set in descending numeric order, length 7 before length 4, and return the first arrangement that passes a primality test. Because I walk from the largest value downward, the first prime I encounter is necessarily the largest pandigital prime - there is no need to collect every prime and take a maximum. This turns "find the maximum element with property P" into "stop at the first match", which holds the result count at one and avoids any sorting.

In the C version I store the surviving lengths as the literal array {7, 4}, encoding the digit-sum pruning conclusion directly in the code. For each length I initialise the digit array in descending order (7,6,5,4,3,2,1) and then call prev_permutation after every test. prev_permutation is the mirror image of the classic next_permutation: it finds the rightmost position where the array descends, swaps that element with the largest element smaller than it in the tail, and reverses the tail - producing every arrangement in strictly descending lexicographic order, in O(n) time and O(1) extra space per step. (The file also carries a forward next_permutation for symmetry, though the search itself only uses the descending walk.) The helper digits_to_number folds the digit array into an integer with the standard num = num * 10 + digit accumulation, keeping the number-building step cleanly separate from the primality test. The test itself, is_prime, handles 2 specially, rejects evens, then trial-divides by odd numbers up to the square root, guarding the multiplication with long long to avoid overflow.

The Python version achieves the same descending order by reversing the digit tuple before handing it to itertools.permutations. Since permutations yields its results in the lexicographic order of its input, feeding it ("7","6",...,"1") produces numbers in descending order. The candidates are formed with int("".join(digits)) and filtered lazily inside a generator expression; next() pulls the first prime and abandons the rest. The is_prime routine is identical in spirit to the C one, and fast_is_prime is a thin alias kept only for naming parity with other solutions.

Programming Techniques

Digit-Sum Divisibility Pruning

Before writing any search loop I used the divisibility rule for 3 to eliminate seven of the nine possible lengths analytically, hard-coding the survivors as {7, 4} in C and (7, 4) in Python. The general lesson: when generating a large class of candidates, look for a cheap arithmetic invariant that proves whole subclasses unviable. Here the invariant is that the digit sum of a pandigital is a fixed triangular number, independent of arrangement, so divisibility by 3 is decided per length rather than per permutation.

In-Place prev_permutation

The C code implements the descending variant of the standard lexicographic permutation algorithm directly on a small integer array. Generating one arrangement at a time in O(n) with no allocation is ideal for an early-exit search - I can stop the instant I find a prime without ever materialising the remaining arrangements. Knowing both the ascending and descending forms lets me iterate in whichever direction matches the answer I am looking for.

Lazy Generator Pipelines

The Python solution chains a generator expression over itertools.permutations with an if fast_is_prime(number) guard and drives it with next(). No list is ever built; each permutation is produced, converted, and tested exactly once, and the pipeline is abandoned at the first hit. This is the idiomatic Python way to express "find the first element satisfying a predicate", and it is both memory-light and concise. Reach for it whenever you only need a prefix or the first match of a sequence.

Separating Number Formation from Primality

Both solutions keep building an integer from digits distinct from testing it - digits_to_number in C, int("".join(digits)) in Python. Isolating these concerns makes each step independently readable and testable, and avoids tangling array or string manipulation with arithmetic logic.

Descending Search Order as an Optimisation

By iterating from the largest permutation downward I turned "collect all primes, take the maximum" into "stop at the first prime". This holds the result count at one and avoids any sort. The same trick applies to any "find the largest or smallest element with property P" problem: iterate in the direction of the answer and exit on the first match.

Key Takeaways

The insight that unlocks this problem is the digit-sum argument that reduces nine candidate lengths to two before any searching begins. Combined with descending enumeration and early exit, the search finishes effectively instantly with no precomputed prime table. The benchmarks bear this out: the C solution measures at 0.0000 s and the Python solution at 0.0001 s, both correct with answer 7652413. The C version is marginally faster owing to native arithmetic and the in-place permutation walk, but the gap is negligible here - the analytic reduction dominates, and the Python version is arguably more elegant thanks to its generator pipeline. When the search space is this small, language choice barely matters; the number theory is what counts.

  • Apply divisibility rules (digit sum mod 3, last-digit parity, and so on) before generating candidates - eliminating whole classes analytically is always cheaper than testing them.
  • When you need the maximum or minimum element with a property, iterate in the direction of the answer and stop at the first match rather than collecting all matches.
  • In-place next/prev permutation gives every arrangement in O(n) time and O(1) space per step, with the freedom to stop early - prefer it to building a full list.
  • Generator pipelines driven by next() implement "find first" searches without materialising the whole sequence; this is idiomatic and efficient Python.
  • Trial division up to the square root is adequate for primality when candidates are small (below ~10^7); reserve sieves or probabilistic tests for larger or more numerous inputs.
  • Hard-code analytic conclusions as named constants or small literals - it makes your reasoning visible and removes dead work.

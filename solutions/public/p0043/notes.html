Problem Analysis

A pandigital number over the digits 0 to 9 is a ten-digit arrangement that uses each digit exactly once. There are 10! = 3,628,800 such arrangements, since each is a permutation of the ten digits. The brute-force route would generate every permutation and test each against the seven divisibility rules. That is computationally feasible but profoundly wasteful: nearly every permutation fails the first constraint, so almost all of the work is discarded immediately.

The structural insight that makes this problem genuinely easy is that the seven three-digit windows overlap by two digits. The window d2 d3 d4 shares d3 d4 with d3 d4 d5, which shares d4 d5 with d4 d5 d6, and so on down to d8 d9 d10. Whenever constraints share variables, they can be propagated: once I fix one window, the adjacent window is almost completely determined - only one new digit is free, and it must simultaneously satisfy a divisibility rule and avoid repeating any digit already placed. This reduces the search from an exhaustive scan of permutations to a narrow backtracking walk with aggressive constraint propagation. The worst-case asymptotic bound is still O(10!), but the real cost collapses to a few thousand node visits, completing in microseconds. The first digit d1 carries no divisibility constraint at all, so it is simply whichever digit remains unused once the other nine positions are settled.

Solution Approach

Right-to-left Constrained Backtracking over the Divisor Chain

Both the C file and the Python file implement the same algorithm and differ only in language idiom. The core is a recursive descent over the divisor chain 17 -> 13 -> 11 -> 7 -> 5 -> 3 -> 2, building the number from right to left one digit at a time.

I seed the search with a precomputed constant table of the 44 valid three-digit multiples of 17 that have no repeated digit (such as "017", "289", "986"). These are exactly the legal values for the rightmost window d8 d9 d10. Starting from this table eliminates one full recursion level and makes the entry conditions explicit: I have already discharged the divisibility-by-17 constraint before recursion even begins.

From each seed I call the recursive generator with the current partial string and the next divisor to satisfy. At each step I iterate over the digits '0' through '9', skip any digit already present in the partial string, prepend the candidate, and check whether the leading three characters of the new string are divisible by the current divisor. If they are not, I prune that branch immediately. If they are, I recurse with the next divisor in the chain. When the chain is exhausted (after satisfying 2), the partial string holds positions d2 through d10; I locate the single unused digit, prepend it as d1, convert the ten-character string to an integer, and accumulate it.

I chose to build right to left rather than left to right because the divisibility window always sits at the first three characters of the partial string. Prepending keeps the validation at a fixed offset, so the index arithmetic never changes as the string grows - this removes a whole class of off-by-one errors. The C version encodes the divisor transition in a switch statement (next_divisor) and accumulates into a global total_sum using a fixed-size character buffer and memcpy, so it never touches the heap. The Python version expresses the same transition as a dictionary lookup and uses yield from recursion, so the top-level aggregation consumes results lazily. Functionally they are identical; the C avoids allocation entirely, while the Python reads more declaratively.

Programming Techniques

Precomputed Seed Table for the Deepest Recursion Level

Rather than rediscovering the valid d8 d9 d10 triples at runtime by scanning all multiples of 17, I hard-coded the 44 valid strings as a constant array. This is a broadly useful pattern whenever the base level of a recursive search has a small, fixed, enumerable structure: precomputing it removes a recursion level, makes the starting conditions auditable, and shaves a redundant scan off every run. The cost is a few dozen lines of data, which is cheap relative to the clarity gained.

Prepend-and-check Incremental Construction

Each recursive step grows the candidate by adding one digit to the left and immediately validating the leading three-digit window. Constructing the object in the direction that keeps every new check at a fixed position - here, the front of the string - simplifies the code and eliminates index drift. When building combinatorial objects under sliding-window constraints, choose the build direction that makes each new choice immediately testable.

Generator Functions for Lazy Enumeration (Python)

The Python solution uses a generator with yield from to stream valid numbers one at a time. The caller aggregates with sum(), so no intermediate list is ever built. This keeps memory usage minimal and cleanly separates concerns: the generator's job is "produce valid numbers", the caller's job is "add them up". I reach for generators whenever a recursive enumeration yields results that can be consumed incrementally.

Switch-based State Machine for the Divisor Chain (C)

In C, the sequence of divisors is encoded as a small finite-state machine: next_divisor maps each divisor to its successor via a switch, returning 0 to signal the end of the chain. This is self-documenting and avoids pointer indirection. The Python equivalent is a dictionary .get() call. Both are appropriate; the choice follows the natural idiom of each language.

Digit-presence Check by Linear Scan

To test whether a candidate digit is already used, I scan the partial string linearly - a loop in C, a substring membership test in Python. With at most nine characters, this is effectively O(1) and far simpler than a bitmask or hash set. For small, fixed-size alphabets, a linear scan often beats more sophisticated structures in both speed and clarity. Asymptotic complexity only matters at scale; at tiny fixed sizes, the simplest correct code wins.

Key Takeaways

The unlocking insight is that the seven divisibility windows overlap by two digits each, forming a dependency chain. Fixing the rightmost window constrains the digit to its left, which constrains the next, and so on. Recognising this turns an exhaustive search over 3.6 million permutations into a narrow backtracking walk that completes effectively instantly.

Both files implement the identical algorithm and both are essentially instantaneous in the benchmarks: the C version records 0.0000 s and the Python version 0.0002 s. The C build is marginally faster, attributable to lower runtime overhead and zero heap allocation, but the search is so heavily pruned that the difference is negligible. The Python version is arguably more elegant, since its lazy generator composes directly with sum(). With a search this aggressively pruned, algorithmic choice dwarfs language choice.

  • When constraints share variables, propagate them eagerly: fix one end and grow incrementally, pruning the instant any constraint is violated.
  • Precomputing a small seed set for the deepest level of a recursive search can eliminate a whole recursion level without affecting correctness.
  • Build combinatorial objects in the direction that keeps each new check at a fixed offset - this removes off-by-one risk entirely.
  • For small, fixed-size alphabets, a linear membership scan is usually faster and simpler than any hash-based structure; do not over-engineer at small scale.
  • Generator functions let recursive enumerations compose naturally with aggregation functions, keeping concerns separated and memory usage minimal.
  • Once a search is heavily pruned, clarity and correctness should dominate design choices, since constant-factor and asymptotic differences become irrelevant.

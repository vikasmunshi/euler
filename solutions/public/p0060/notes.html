Problem Analysis

The structure hiding beneath this problem is a clique in an undirected graph. I treat each prime as a node and draw an edge between two primes p and q exactly when both concatenations - pq and qp - are themselves prime. With that model, "find five primes such that any two concatenate to a prime" becomes "find a clique of size five," and "lowest sum" becomes "the clique whose node sum is minimal." This reframing matters because clique search comes with well-understood pruning strategies, and naming the structure tells me at once how to organise the search.

Two number-theoretic facts make the search tractable. First, I need fast primality testing: concatenating a k-digit and an m-digit prime yields a (k+m)-digit candidate, so for a set of five the candidates stay small enough that trial division up to the square root is plenty fast - that test is O(sqrt(n)) per call. Second, I bound the candidate set: every prime in the answer for a set of length L turns out to be below 10^(L-1), so I generate that prime list once with a Sieve of Sundaram (roughly 10,000 primes for L=5). The dominant cost is the pairwise primality predicate, which I memoize so each pair is tested at most once. Clique search is NP-hard in general, so the worst case is exponential, but aggressive pruning over a small prime set keeps the real running time well under a tenth of a second.

Solution Approach

Cached Depth-First Clique Search (Python and C)

Both files implement the same algorithm. I describe it once and note language differences at the end.

Step 1 - Build the prime set. I sieve all primes up to 10^(set_length-1) with the Sieve of Sundaram. Sundaram marks an odd number 2i+1 as composite whenever its index can be written as i + j + 2ij; the unmarked indices map back to the odd primes, and I prepend 2. I picked Sundaram for variety - it is O(N log log N) like the Sieve of Eratosthenes, and either would serve. The output is returned sorted ascending, which is essential for the canonical search below.

Step 2 - Memoized pair predicate. concatenate_is_prime(a, b) tests that both concatenations are prime. This predicate is expensive relative to the recursion overhead and the same pair is queried across many partial cliques, so I cache it. In Python that is a single functools.lru_cache decoration. In C I hand-build a hash table with separate chaining, canonicalising each key as (min(a,b), max(a,b)) to halve the entries (the predicate is symmetric), folding a multiplicative-XOR mix into 2^20 buckets.

Step 3 - Depth-first extension in ascending order. I begin with each prime as the first clique element, then recursively extend: a candidate may join only if it is strictly greater than the current last element and pairs correctly with every element already chosen. I check the pairing predicate before recursing, so branches that cannot grow into a valid clique are never explored. Requiring strictly increasing values means each unordered set is visited exactly once in sorted form, which both removes duplicate orderings and guarantees the first complete clique found has the minimum sum.

Python specifics. The Python version expresses the nested depth-first search as a chain of generator expressions - solution_pairs feeds extend_solution, which feeds another extend_solution - one nesting level per clique element beyond the first pair. A single next() on the outermost generator pulls only the first complete solution and lazily short-circuits the entire chain.

C specifics. The C version uses a recursive dfs with a global found flag; once set, every active frame returns immediately. It keeps the partial clique in a fixed stack array sol[10] and passes the current depth and the next start index explicitly. The loop bound i <= num_primes - need (where need = target - sol_size) prunes the tail when fewer primes remain than slots left to fill.

Programming Techniques

Memoization with lru_cache

functools.lru_cache(maxsize=None) wraps a pure function so the first call with a given argument tuple computes and stores the result, and later calls return it instantly. I used an unbounded cache because evicting entries would re-trigger expensive primality scans. This is my first reach whenever a deterministic function is called repeatedly with the same inputs inside a combinatorial search.

Generator expressions for lazy, early-terminating search

Generator expressions yield values one at a time instead of building the full list. By nesting them and calling next() exactly once, the search stops at the first complete clique without ever constructing the rest of the candidate space. This is idiomatic Python whenever you need only the first match from a large implicit search space.

Ascending-order enumeration to canonicalise sets

Requiring each new element to exceed the previous one visits every k-element combination once, in its unique sorted form, removing the k! redundant orderings. The same trick - always extend in strictly increasing value or index order - applies to any search over unordered combinations and, as a bonus here, makes the first hit the minimum-sum hit.

Custom hash table with canonical keys in C

C has no built-in associative map, so I wrote a hash table with separate chaining (each bucket is a linked list), which handles collisions without open-addressing's resize logic. Canonicalising the key as (min, max) exploits the predicate's symmetry to halve storage. Canonical key plus hash table is the standard replacement for lru_cache when porting a search to C.

Early-exit DFS with a global flag

Instead of threading a sentinel return value up every frame, the C code sets a global found flag checked at the top of dfs; once set, all frames unwind immediately. This is a simple, readable early-termination idiom for single-solution recursive search. The cost is a non-reentrant function, acceptable in this single-threaded context.

Remaining-slots pruning guard

The loop bound i <= num_primes - need skips positions from which not enough primes remain to complete the clique. One comparison per iteration buys the ability to cut many doomed iterations near the end of the prime list - a cheap, general DFS prune.

Sieving once against a known bound

Because every answer prime is below 10^(set_length-1), I know the candidate range up front and produce the whole prime list with one Sieve of Sundaram pass rather than testing candidates individually. Bulk sieving is amortised cheaply across thousands of primes and beats repeated trial division over the same range. Reach for a sieve whenever you need many primes within a known bound.

Key Takeaways

The unlocking insight is to model a "pairwise property over a set" problem as a graph and search for a clique: once "compatible pair" is an edge, the standard pruning falls out naturally. The second insight is that enumerating in ascending order both eliminates duplicate orderings and makes the first solution the minimum-sum one, so no sorting or post-processing is needed.

The two solutions implement the identical algorithm, so the benchmark gap is pure runtime overhead. On the main case (set_length 5) the C solution runs in about 0.0346 s versus 0.1052 s for Python - roughly a 3x advantage, driven by raw integer arithmetic and the custom cache avoiding interpreter and big-integer overhead. On the trivial dev cases (set_length 3 and 4) Python is actually the faster pair (0.0000 s and 0.0001 s against C's 0.0012 s and 0.0013 s), because its startup and tiny-workload path beats C's sieve and hash-table allocation; the C advantage only appears once the search does real work. Either is fast enough at this scale, but C is the choice if the search is pushed to set_length 6 or beyond.

  • When a problem asks for a set with a pairwise property, model it as a graph and look for a clique; the standard pruning strategies apply at once.
  • Enumerate combinations in sorted order during clique search - it removes permutation duplicates for free and makes the first solution found the minimal one.
  • Cache any expensive deterministic predicate evaluated repeatedly on the same inputs; one decorator or hash table can cut runtime substantially.
  • Prefer generator expressions plus a single next() when you need only the first result from a large search space.
  • A global "found" flag is a pragmatic way to short-circuit deep recursion once a solution exists, avoiding sentinel propagation.
  • A remaining-slots lower-bound check inside a DFS loop is a cheap, broadly applicable prune.
  • Sieve primes upfront when the upper bound is known; per-candidate trial division wastes time re-testing numbers the sieve handles in bulk.
  • Compiled-language wins only materialise once the workload dwarfs startup and allocation costs - measure before assuming C beats Python on small inputs.

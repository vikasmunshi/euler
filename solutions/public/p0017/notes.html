Problem Analysis

This problem requires no number theory in the usual sense - there are no primes, divisors, or modular arithmetic to reason about. Instead the challenge is one of structured decomposition: converting an integer into its written English form and counting only the alphabetic characters. The central observation is that English number words are built hierarchically from a small, fixed vocabulary. Any number from 1 to 999 decomposes into at most three positional fragments - a hundreds part, a tens part, and a units part - and each fragment contributes a word whose length is fixed and known in advance. Because the letter count is fully determined by the digit structure rather than the numeric value, the whole problem reduces to assembling strings from a lookup table and tallying their letters.

The second key observation is that the vocabulary is finite and tiny: nineteen atomic words (one through nineteen), eight tens-words (twenty through ninety), plus "hundred", "and", and "thousand". Every number in the range is a combination of these. The only subtlety is the British convention requiring "and" between the hundreds and the remainder (e.g. "three hundred and forty-two"). Since the input range is bounded at 1000, the algorithm runs in linear time, O(N) where N = 1000, with each number requiring only a constant amount of work to build and measure. The problem is therefore dominated by correctness concerns - careful bookkeeping of the language rules - rather than by performance.

Solution Approach

Hierarchical Word Construction with Memoization

Both the C and Python files implement the same algorithm: build the English word representation of each number, then count its letters. I chose to construct the actual word strings rather than computing letter counts analytically, because building strings makes the language rules - especially the British "and" - explicit and easy to verify. Computing counts directly would save a little work but would obscure the logic and invite off-by-one errors.

The decomposition has two layers. The inner function, number_triplet_in_words, handles any value from 0 to 999. It first checks whether the number is directly representable - numbers below 20 and exact multiples of ten are served straight from the lookup table. For values from 21 to 99 that are not multiples of ten, it joins a tens-word and a units-word with a hyphen ("twenty-one"). For values of 100 and above, it prepends the hundreds word and "hundred", then recursively appends the remainder, inserting "and" only when the remainder is non-zero. This recursion is shallow - at most one level deep within a triplet - so it costs nothing.

The outer function, convert_number_to_words, layers the thousands scale on top. In the Python version this is handled generically: the number is formatted with comma separators via f"{number:,}", split on commas into triplet groups, and each group is paired with its scale suffix ("thousand", "million", and so on) from a pre-defined tuple. The British "and" prefix is then applied to the final group when appropriate. This design makes the Python version extensible to arbitrarily large numbers with no extra arithmetic. The C version handles 1000 as an explicit special case (it is the only four-digit value in range) but also includes a general thousands decomposition for robustness.

To avoid recomputing the same triplet repeatedly - the suffix "forty-two" appears in 42, 142, 242, and so on - both solutions cache triplet results. Python uses functools.lru_cache, which transparently memoizes the function. The C version uses a pair of parallel arrays: triplet_cache[1000][64] holds the strings and triplet_cached[1000] records which indices have been filled. Both caches are reset at the start of every solve() call - Python via cache_clear(), C via memset - so the timed benchmark loop measures the full word-construction cost on each run rather than reusing a cache left warm by a previous run. Letter counting is done by walking the string and skipping spaces and hyphens (C), or by stripping those characters with str.translate before measuring length (Python).

Programming Techniques

Lookup Tables for Fixed Vocabularies

Both solutions store the atomic words in a direct-access structure: a C array indexed by integer value, a Python dictionary keyed by integer. I prefer this to a long if/elif chain because it separates data from logic - adding or correcting a word means editing the table, not the algorithm. In the C array I place NULL entries between the tens-words (positions 21-29, 31-39, and so on) so that each tens-word sits at exactly its numeric index. This wastes a handful of pointers but allows O(1) lookup using nothing more than integer division and modulo. Whenever a mapping is small and stable, a table is easier to audit and test than branching code.

Memoization of Pure Sub-problems

The triplet function is a pure function - its output depends only on its input - which makes it a perfect candidate for caching. Python's @functools.lru_cache() is the zero-effort idiom: a single decorator turns the function into a memoized one; in C there is no built-in equivalent, so I manage the cache by hand with a value array and a boolean flag array. Because the harness re-runs solve() to benchmark it, I clear the cache at the top of each call (cache_clear() in Python, memset in C) so a run never inherits a predecessor's warm cache and the measured time reflects the real per-call work. For N = 1000 the saving from caching is minor, but the pattern generalises powerfully: whenever a computation revisits the same sub-problem many times, memoization reduces redundant work and can turn repeated exponential recomputation into linear time.

Bounded String Formatting in C

Every string assembly in C uses snprintf(buf, 64, ...) rather than the unbounded sprintf. The size argument guarantees the write never overflows the buffer regardless of input. Each triplet buffer is 64 bytes - comfortably larger than the longest possible triplet, "nine hundred and ninety-nine" (31 characters including separators). Sizing buffers conservatively and always passing a length to formatting functions is a basic defensive habit in C that eliminates an entire class of memory-corruption bugs at no runtime cost.

Comma Formatting to Split Numeric Triplets (Python)

In the Python solution I use f"{number:,}".split(",") to decompose a number into three-digit groups. Python's integer formatter does the grouping arithmetic for me, so I get ["1", "000"] for 1000 without manually dividing by powers of 1000. Zipping these groups with a tuple of scale suffixes then assembles the full word. This is a neat idiom worth remembering for any task that needs digit grouping.

Translation Tables for Bulk Character Filtering (Python)

str.translate with a deletion mapping built as {ord(c): None for c in (" ", "-")} removes the chosen characters in a single C-level pass. This is faster than a regular expression and cleaner than a manual loop, and it makes the intent - "strip these characters before measuring" - obvious. Reach for it whenever you need to delete or replace a fixed set of characters in a tight loop.

Lazy Aggregation with Generator Expressions (Python)

The solve function passes a generator expression directly to sum: each converted string is built, measured, and discarded before the next is requested, so no intermediate list is ever materialised. This keeps memory constant in the number of iterations. Using generators with sum, max, or min is the idiomatic way to aggregate over a sequence without paying for storage you do not need.

Key Takeaways

The insight that unlocks this problem is that written numbers are built from a tiny fixed vocabulary by a well-defined recursive decomposition - units, tens, hundreds, thousands. Once that decomposition is implemented correctly, paying close attention to the British "and" rule, the summation is trivial. Algorithmic sophistication matters far less here than meticulous attention to the language rules.

Comparing the benchmark results on the main case (N = 1000), the C solution runs in about 0.00009 s while the Python solution takes roughly 0.0017 s - the C version is around 20 times faster. This gap reflects the usual compiled-versus-interpreted overhead for tight loops over small data, not a difference in algorithm; the two implement identical logic. The Python solution trades that raw speed for greater generality and readability: the comma-split triplet idiom and lru_cache make it naturally extensible to arbitrarily large numbers with almost no extra code.

  • Separate data from logic: store fixed mappings in tables, not in conditional branches, so they are easy to audit and extend.
  • Memoize any pure function called repeatedly with the same arguments - lru_cache in Python, a manual cache array in C - but reset the cache between timed runs so a benchmark never measures a cache warmed by an earlier run.
  • In C, always prefer snprintf over sprintf; the length argument costs nothing and prevents buffer overflows.
  • Use str.translate to delete or replace characters in bulk - it is faster than regex and clearer than a loop.
  • Pass generator expressions to sum, max, or min to aggregate lazily and keep memory usage constant.
  • When the input space is small and bounded, optimise for correctness and clarity first - performance tends to take care of itself.
  • Constructing the explicit representation of a value is often easier to verify than computing a derived quantity directly, even if it does slightly more work.

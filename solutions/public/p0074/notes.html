Problem Analysis

The engine of this problem is a single deterministic rule: from any positive integer n, the successor is the sum of the factorials of its decimal digits, so 145 -> 1! + 4! + 5! = 145. Iterating this rule traces a path through the integers, and because every integer has exactly one successor, the structure is a functional graph - a directed graph in which every node has out-degree exactly one. A fundamental property of any functional graph on a finite set is that every trajectory eventually enters a cycle: there are only finitely many reachable values, so by the pigeonhole principle some value must repeat, and once it does the path loops forever. The chain the problem asks about is the count of distinct values visited before the first repetition. Our task is to count how many starting numbers below one million produce a chain of exactly sixty distinct terms.

Two observations make this tractable. First, the state space is tightly bounded. The digit factorials are fixed constants - the largest is 9! = 362880 - so any 7-digit number maps to at most 7 x 362880 = 2540160. Every value we ever encounter, even for starting numbers up to ten million, stays below about 2.5 million, so a flat array of three million slots covers the entire universe of reachable values. Second, because each value has a single fixed successor, two paths that ever meet stay merged forever. This means each value's chain length can be computed once and reused - the defining condition for memoization. Without caching, the cost is roughly O(N x L), where N is the limit and L the maximum chain length (sixty here); with caching, the amortised cost per starting number falls toward O(1), giving an effectively linear O(N) algorithm.

Solution Approach

Algorithm 0 - Walk-and-cache storing only chain lengths (p0074_s0.py and p0074_s0.c)

This is the most direct memoization strategy and the fastest of the three. For each starting number I walk forward along the successor map, appending each value to a seen list, and stop as soon as one of two things happens: either the current value is already in seen (the walk has closed a cycle entirely within itself), or the current value is already in the cache (we have reached a value whose chain length is already known). At that point the total length of the chain beginning at seen[0] is fully determined: it is len(seen) plus the cached length of the value we stopped on, or just len(seen) if we closed a cycle.

The decisive step is the back-propagation pass. Once I know the chain length of the first node, every node on the walk has a known length too: seen[i] has length total - i, because it sits exactly i steps further along the same chain. A single backward loop fills the cache for every node visited, so the work this walk performed benefits every future walk that ever touches one of these nodes. This is precisely the flattening idea behind path compression in union-find structures: collapse a whole traversal into the cache in one go.

The Python version (p0074_s0.py) uses two dictionaries: chain_length_cache mapping a value to its chain length, and graph mapping a value to its precomputed successor so the digit-factorial sum is never recomputed. The C version (p0074_s0.c) replaces both dictionaries with flat int arrays of size three million, using 0 as the "not yet computed" sentinel. Cycle detection during the walk is a simple linear scan over seen; this looks like an O(L) operation, but L is bounded near sixty, so it costs at most a few dozen comparisons per step - cheaper in practice than maintaining a hash set. At N = 1,000,000 the C version runs in 0.0160 s and the Python version in 0.9436 s.

Algorithm 1 - Walk-and-cache storing full chain lists (p0074_s1.py and p0074_s1.c)

Algorithm 1 stores, in Python, the actual list of values that make up each chain rather than just its length. The function add_chains builds a current_chain list and a visited set. When the walk reaches a value already in the cache, it splices the cached list onto the current chain with list.extend; when it reaches a value already in visited, it has found a self-contained cycle. Each previously unseen node is then assigned the appropriate tail slice of current_chain. This is conceptually clean and supports visualisation - you can print an entire chain - but it allocates a fresh list slice for every cached node, which is expensive. In Python it runs in 2.6839 s at N = 1,000,000, roughly three times slower than Algorithm 0's 0.9436 s.

The C translation (p0074_s1.c) recognises that storing the full list is unnecessary: only the lengths are needed for the answer. It therefore keeps a single chain_length array, exactly like Algorithm 0, and reconstructs the same length information from each value's position index in current_chain. The one structural difference from s0 is that s1's C code uses a boolean visited_flag array - cleared after each walk via a tracked visited_list - for O(1) cycle-membership testing, instead of s0's linear scan. This extra bookkeeping is why s1_c is marginally slower than s0_c (0.0185 s vs 0.0160 s at N = 1,000,000).

Algorithm 2 - Visualisation variant of Algorithm 1 (p0074_s2.py and p0074_s2.c)

Algorithm 2 is functionally identical to Algorithm 1; it lives in a separate file purely to add optional coloured terminal output gated behind the --show flag. The Python file (p0074_s2.py) adds a ColorCodes enum of ANSI escape sequences and a print_chains function that renders each chain with its cycle entry highlighted. The C translation (p0074_s2.c) drops the visualisation entirely and is therefore essentially identical to s1_c. Its benchmarks track s1 closely across every input size (0.0214 s vs 0.0185 s in C, 2.6254 s vs 2.6839 s in Python at N = 1,000,000).

Programming Techniques

Memoization with a flat array

In the C solutions I allocate a contiguous int array of three million entries, initialised to zero by calloc, using zero as the "not computed" sentinel. Because the array index is the value itself, every cache lookup is a single dereference - no hashing, no collision handling. This relies on the key being a bounded non-negative integer, which the problem's mathematics guarantees (every reachable value stays below 2.5 million). Whenever your keys are integers confined to a small known range, a flat array beats a hash table on both raw speed and cache locality. Reach for it as the default; fall back to a hash map only when the key space is genuinely sparse or unbounded.

Path-compression back-propagation

Rather than recomputing each chain length from scratch, I collect the whole walk and fill the cache for every node on it in a single backward pass, assigning seen[i] the length total - i. This amortises the cost so that the next traversal touching any of those nodes is served instantly. It is the same flattening idea that makes union-find with path compression run in near-constant time. The general principle: when one traversal reveals the answer for many intermediate states, write them all back at once.

Precomputed lookup table for digit factorials

Both languages define a ten-element constant array holding 0! through 9!. Computing a factorial inside the innermost loop, which runs millions of times, would be wasteful when there are only ten possible inputs. The table fits permanently in L1 cache, so each digit costs one indexed read. Any function with a small fixed domain that is called repeatedly inside a hot loop is a candidate for this substitution of arithmetic by a table read.

Linear scan justified by a bounded constant

Algorithm 0 detects cycles by scanning the seen list, an O(L) operation. For unbounded L this would be a red flag, but here L is capped near sixty, so the scan is a tiny constant that avoids the allocation and hashing overhead of a set. The benchmark confirms the call: s0_c, with its linear scan, beats s1_c, which maintains a boolean-flag set, even though the latter is asymptotically "better" (0.0160 s vs 0.0185 s at N = 1,000,000). Knowing a problem-specific bound can justify a simpler design that the textbook would reject - measure rather than assume.

Decoupling the benchmark harness from the solver

The C solve functions receive a clean argument vector and concern themselves only with the algorithm; the surrounding runner strips out framework flags such as --runs= and --show before invoking them. This keeps the algorithmic core ignorant of timing and presentation concerns, making it independently testable. Separating measurement infrastructure from the code being measured is good practice for any program that must run both interactively and under a benchmark.

Python StrEnum for ANSI codes

Algorithm 2's Python file defines its terminal colours with enum.StrEnum. Members of a StrEnum behave as plain strings, so they drop directly into f-strings without needing .value. This gives meaningful names to magic escape sequences while keeping zero-overhead string semantics - a tidy way to name a set of related string constants.

Key Takeaways

The unlocking insight is to recognise the successor map as a functional graph: every component contains exactly one cycle, every trajectory falls into it, and - because the map is deterministic - each value's chain length can be computed once and cached. The supporting observation that all reachable values stay below about 2.5 million lets us swap a hash map for a flat array, which simplifies the code and speeds it up at the same time.

Comparing the three approaches, Algorithm 0 is the clear winner. At N = 1,000,000 it runs in 0.0160 s in C and 0.9436 s in Python, versus 0.0185 s / 2.6839 s for Algorithm 1 and 0.0214 s / 2.6254 s for Algorithm 2. The Python gap is the most dramatic: storing only an integer per node (Algorithm 0) avoids the per-node list-slice allocations that dominate Algorithms 1 and 2. In C the three converge because the C translations of s1 and s2 discard list storage and reduce to the same integer-per-node scheme; s0_c stays marginally ahead thanks to its lighter cycle-detection scan. The same ordering holds at the largest inputs - at N = 10,000,000, s0.py finishes in 9.3755 s against roughly 27-28 s for the others. The lesson is that the minimal representation - one integer per node - is both faster and simpler once the algorithm is understood; full chain lists are worth their cost only for debugging and visualisation.

  • When a problem iterates a deterministic function, identify its functional-graph structure and memoize: compute each node's value once and reuse it.
  • Back-propagate cache entries across an entire traversal in one pass - this is path compression and makes future lookups effectively free.
  • When keys are bounded non-negative integers, prefer a flat array over a hash table for both speed and simplicity.
  • Replace repeated computation over a small fixed domain with a precomputed lookup table.
  • A linear scan can outperform an asymptotically superior hash set when the scanned length is a small known constant - let measured bounds, not textbook asymptotics, guide the choice.
  • Store the minimal representation needed for the answer; richer intermediate structures cost memory and time and belong only where you need them for inspection.
  • Keep the algorithmic core free of harness and presentation concerns so it stays testable and easy to reason about.

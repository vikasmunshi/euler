Problem Analysis

The Fibonacci sequence is defined by the recurrence relation F(n) = F(n-1) + F(n-2). A fundamental fact about this sequence is that it grows exponentially: F(n) is asymptotically proportional to phi^n, where phi = (1 + sqrt(5)) / 2 ≈ 1.618 is the golden ratio. Because the value grows exponentially, the number of decimal digits grows only linearly with the index n - roughly n * log10(phi) digits. This means reaching 1000 digits takes about 1000 / log10(phi) ≈ 4785 terms, so the loop is short. The real challenge is arithmetic precision: Fibonacci numbers exceed the range of a 64-bit integer after only about the 93rd term, so to reach 1000 digits we need arbitrary-precision arithmetic (big integers).

The observation that makes the problem tractable is that we never need the full sequence - only the two most recent terms suffice to compute the next. So memory is constant in the number of live big integers (just two or three). Likewise, checking whether a number has reached 1000 digits does not require fully expanding it: we can compare against a power of ten, or inspect only the most significant chunk of a decimal representation. The overall complexity is O(N * D), where N ≈ 4782 is the target index and D is the number of digit-groups (limbs) in each number - each addition touches every limb once, and there are N additions.

Solution Approach

Iterative Big-Integer Fibonacci (Python and C, index 0)

Both solutions share the same algorithm: start from F(1) = F(2) = 1, repeatedly apply (a, b) = (b, a + b), and stop as soon as b has at least n decimal digits. An index counter advances alongside the values, and its final value is the answer.

In Python the algorithm is almost trivial because the language has native big integers. I keep two variables a and b, both initialised to 1, and an index i = 2 pointing at b. The loop condition b < 10 ** (n - 1) exploits a clean threshold test: a number has at least n digits exactly when it is greater than or equal to 10**(n-1). Inside the loop, the parallel assignment a, b = b, a + b advances both values in one step. Python grows the integers automatically, so there is no manual memory management.

The C solution has no native big integer, so I hand-rolled a minimal one. I chose a base-109 limb representation: each entry of the limbs array holds up to 9 decimal digits packed into a 64-bit unsigned integer. I picked a decimal base rather than a binary one specifically because the problem asks about decimal digits - with decimal limbs, counting digits requires inspecting only the top limb (counting its digits by repeated division by 10) and adding 9 * (len - 1) for the fully-packed lower limbs. This avoids a base conversion on every iteration. The base 109 is the largest power of ten that lets two limbs plus a carry fit comfortably inside a 64-bit accumulator without overflow. bigint_add walks both numbers from the least significant limb upward, propagating a carry exactly as in grade-school addition. I pre-allocated MAX_LIMBS = 200 limbs (1800 digits), far beyond the requirement, so no dynamic resizing is ever needed. The two live values are a and b, with a third tmp holding the sum before memcpy shuffles the values into place.

Programming Techniques

Parallel (Tuple) Assignment for State Advancement

In Python, a, b = b, a + b updates both state variables at once. Python evaluates the entire right-hand side before binding any names, so the old value of a is still available when computing a + b. Without this parallel assignment, a temporary variable would be needed to avoid clobbering a before use. Reach for this idiom whenever the next state of two variables each depends on the current state of the other - it is concise and eliminates a class of aliasing bugs.

Digit-Count via Threshold Comparison

Rather than converting a number to a string and measuring its length, the Python solution compares b < 10 ** (n - 1). A number has fewer than n digits exactly when it is below 10**(n-1). This avoids repeated string allocation in the hot loop. The C code achieves the same end differently: because limbs are decimal, it counts digits in the top limb and scales by 9 for the rest. Both avoid a full decimal expansion on every iteration.

Base-109 Decimal Limb Representation

Most big-integer libraries use a binary base (232 or 264) because binary maps directly onto machine arithmetic. I deliberately chose base 109 here because the question is about decimal digits. With a decimal base, the digit-count check is O(1) per iteration instead of a base conversion. The trade-off - addition is marginally less efficient than in a binary base - is irrelevant because we only ever add, never multiply or divide. Match your internal representation to the operation that dominates your work.

Fixed-Size Buffer with Pre-Allocation

The BigInt struct embeds a fixed array of 200 limbs directly, so the whole value lives on the stack with no heap allocation. Avoiding malloc inside a tight loop eliminates allocation overhead and fragmentation. The cost is a compile-time cap on magnitude, but since 1800 digits vastly exceeds the 1000-digit requirement, the bound is safe. Pre-sizing a buffer from a known upper bound is a standard technique in competitive programming and embedded systems.

Shallow Struct Copy with memcpy

Advancing the C state is conceptually a = b; b = tmp. Because BigInt is a plain value type with no pointer members, a bitwise copy via memcpy is correct and fast - the compiler can often lower a fixed-size copy to a few vector moves. This idiom is reliable for value-type structs; it breaks only when a struct owns heap memory, in which case a deep copy is required.

Key Takeaways

The core insight is that this problem reduces to iterating the Fibonacci recurrence relation with big-integer arithmetic until a digit-count threshold is crossed - and that threshold check is cheapest when the internal representation is already decimal. The Python solution is elegant because the language handles big integers natively, compressing the whole algorithm into a few lines. The C solution is faster (0.0006 s versus 0.0113 s on the main case) because it avoids interpreter overhead and object allocation, at the cost of a hand-written big-integer type. For a one-off answer the Python version is the obvious choice; the C version pays off only where raw speed matters.

  • When a problem counts decimal digits, consider a decimal-base internal representation - it turns an O(D) digit count into O(1).
  • Parallel assignment removes temporary variables when two state variables update from each other's current values.
  • Match your number representation to the operation that dominates: add-only workloads favour simplicity over binary efficiency.
  • Pre-allocating a fixed buffer from a known upper bound avoids heap allocation in tight loops whenever the bound is computable in advance.
  • To detect when a value first reaches n digits, comparing against 10**(n-1) beats converting to a string.
  • Plain structs with no pointer members can be copied safely and efficiently with memcpy; reserve deep copies for types that own heap memory.
  • Prototype big-number problems in Python; port to C only when measurements show the performance is worth the extra code.

Problem Analysis

This problem asks for the minimum-cost path from the top-left to the bottom-right of an N-by-N grid, where movement is permitted in all four directions - up, down, left, and right. The cost of a path is the sum of the values of the cells it visits, including the starting cell. The crucial observation is that allowing all four directions turns this into a genuine shortest-path problem on a weighted graph, rather than a grid-restricted DP variant. I model each cell as a node and place a directed edge from every cell to each of its four neighbours, with the edge weight equal to the value of the destination cell. The goal is then the cheapest route from node (0,0) to node (N-1, N-1).

Why does this distinction matter? In the right- and down-only variants (Problems 81 and 82), the grid forms a directed acyclic graph (DAG) - it has no cycles, so I can solve it with dynamic programming in a single fixed-order pass. The moment upward and leftward moves are allowed, cycles become possible and there is no longer a valid evaluation order for a one-pass DP. I need a general shortest-path algorithm. Because every cell value is positive - so every edge weight is non-negative - Dijkstra's algorithm applies and yields an exact answer. Dijkstra maintains a tentative shortest distance to each node, repeatedly finalises the closest unfinalised node, and "relaxes" (improves) the distances of its neighbours. With V = N^2 nodes and a linear scan to find the next node to finalise, the cost is O(V^2) = O(N^4). For N = 80 that is about 40 million comparisons - comfortably fast.

Solution Approach

Dijkstra's Algorithm with Linear Minimum Scan (C and Python)

Both files implement the same algorithm - Dijkstra's algorithm with a linear scan to choose the next node to finalise - so I describe them together and flag language differences as they arise.

Step 1 - Parse the matrix. In Python I build a list of lists with a comprehension, splitting each non-empty line on commas: [[int(n) for n in line.split(",")] for line in content.splitlines() if line]. In C I scan the raw buffer character by character, using atoi to read each integer and advancing a pointer past spaces, commas, and line breaks. The C parser is more verbose but avoids any intermediate allocation, writing directly into a fixed int matrix[80][80].

Step 2 - Initialise distances. Every node's tentative distance starts at a sentinel I call "infinity", computed as the sum of all cell values plus one. This is guaranteed to exceed any real path cost, because no minimal path visits any cell more than once, so the cost cannot exceed the sum of all cells. The start node (0,0) is initialised to its own cell value, since the starting cell's weight is part of the total.

Step 3 - The main loop. While the target has not been finalised, I scan all unvisited nodes for the one with the smallest tentative distance, mark it visited, and relax its four neighbours: if dist[current] + value(neighbor) improves the neighbour's recorded distance, I update it. Crucially, I stop the instant the target is finalised. Dijkstra's algorithm finalises nodes in non-decreasing distance order, so once the target is extracted its distance is provably optimal and no further work is needed.

Why a linear scan rather than a priority queue? With V = 6400 nodes, the O(V^2) scan is about 40 million operations - fast enough that the constant-factor and code-complexity cost of a binary heap is not justified. A heap would give O((V+E) log V), worthwhile for much larger or sparser graphs, but here simplicity wins.

Language-specific differences. The Python solution keys its distances dictionary and unvisited set on (row, col) tuples and finds the minimum with min(unvisited, key=lambda node: distances[node]). The C solution flattens each node to a single integer index i = row * size + col and stores dist[] and visited[] as contiguous one-dimensional arrays. This flat layout is far more cache-friendly and avoids hash-table overhead, which is the chief reason C runs in 0.0228 s on the full matrix while Python takes 1.5129 s. Both produce the correct answer of 425185.

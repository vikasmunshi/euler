Problem Analysis

This problem asks for the minimum-cost path from the top-left to the bottom-right of an N-by-N grid, where movement is permitted in all four directions - up, down, left, and right. The cost of a path is the sum of the values of the cells it visits, including the starting cell. The crucial observation is that allowing all four directions turns this into a genuine shortest-path problem on a weighted graph, rather than a grid-restricted DP variant. I model each cell as a node and place a directed edge from every cell to each of its four neighbours, with the edge weight equal to the value of the destination cell. The goal is then the cheapest route from node (0,0) to node (N-1, N-1).

Why does this distinction matter? In the right- and down-only variants (Problems 81 and 82), the grid forms a directed acyclic graph (DAG) - it has no cycles, so I can solve it with dynamic programming in a single fixed-order pass. The moment upward and leftward moves are allowed, cycles become possible and there is no longer a valid evaluation order for a one-pass DP. I need a general shortest-path algorithm. Because every cell value is positive - so every edge weight is non-negative - Dijkstra's algorithm applies and yields an exact answer. Dijkstra maintains a tentative shortest distance to each node, repeatedly finalises the closest unfinalised node, and "relaxes" (improves) the distances of its neighbours. With V = N^2 nodes and a linear scan to find the next node to finalise, the cost is O(V^2) = O(N^4). For N = 80 that is about 40 million comparisons - comfortably fast.

Solution Approach

Dijkstra's Algorithm with Linear Minimum Scan (C and Python)

Both files implement the same algorithm - Dijkstra's algorithm with a linear scan to choose the next node to finalise - so I describe them together and flag language differences as they arise.

Step 1 - Parse the matrix. In Python I build a list of lists with a comprehension, splitting each non-empty line on commas: [[int(n) for n in line.split(",")] for line in content.splitlines() if line]. In C I scan the raw buffer character by character, using atoi to read each integer and advancing a pointer past spaces, commas, and line breaks. The C parser is more verbose but avoids any intermediate allocation, writing directly into a fixed int matrix[80][80].

Step 2 - Initialise distances. Every node's tentative distance starts at a sentinel I call "infinity", computed as the sum of all cell values plus one. This is guaranteed to exceed any real path cost, because no minimal path visits any cell more than once, so the cost cannot exceed the sum of all cells. The start node (0,0) is initialised to its own cell value, since the starting cell's weight is part of the total.

Step 3 - The main loop. While the target has not been finalised, I scan all unvisited nodes for the one with the smallest tentative distance, mark it visited, and relax its four neighbours: if dist[current] + value(neighbor) improves the neighbour's recorded distance, I update it. Crucially, I stop the instant the target is finalised. Dijkstra's algorithm finalises nodes in non-decreasing distance order, so once the target is extracted its distance is provably optimal and no further work is needed.

Why a linear scan rather than a priority queue? With V = 6400 nodes, the O(V^2) scan is about 40 million operations - fast enough that the constant-factor and code-complexity cost of a binary heap is not justified. A heap would give O((V+E) log V), worthwhile for much larger or sparser graphs, but here simplicity wins.

Language-specific differences. The Python solution keys its distances dictionary and unvisited set on (row, col) tuples and finds the minimum with min(unvisited, key=lambda node: distances[node]). The C solution flattens each node to a single integer index i = row * size + col and stores dist[] and visited[] as contiguous one-dimensional arrays. This flat layout is far more cache-friendly and avoids hash-table overhead, which is the chief reason C runs in 0.0228 s on the full matrix while Python takes 1.5129 s. Both produce the correct answer of 425185.

Programming Techniques

Computed Sentinel Instead of INT_MAX or float('inf')

Rather than relying on INT_MAX or float('inf'), I compute a concrete upper bound - the sum of all cell values plus one. In C this matters: writing INT_MAX + cell_value during relaxation would risk integer overflow and wrap to a negative number, silently corrupting the comparison. A real, finite bound keeps every arithmetic operation well-defined. In Python it also avoids mixing integers with floating-point infinity. This pattern - derive a provable bound from the problem's data - is a robust habit whenever you need a "larger than anything possible" value.

Row-Major Flattening of a 2D Grid in C

The C solution maps each cell to a single index with i = row * size + col, recovering coordinates with row = i / size and col = i % size. This row-major layout lets per-node arrays be plain contiguous blocks rather than arrays of pointers, eliminating an indirection and improving locality during the inner scan. Reach for this whenever you need per-node bookkeeping over a grid in a low-level language and want predictable, cache-friendly memory access.

Direction Vectors for Neighbour Enumeration

Both solutions enumerate neighbours using displacement arrays - drow = {-1, 1, 0, 0} and dcol = {0, 0, -1, 1} in C, and explicit tuples in Python. Looping over these vectors replaces four near-identical branches with one, and bounds-checking happens once inside the loop. The same pattern extends trivially to diagonal (8-directional) connectivity by enlarging the displacement arrays. This is a standard, readable idiom for any grid-traversal code.

Early Termination on Target Finalisation

Because Dijkstra's algorithm finalises nodes in order of increasing distance, the search can stop the moment the target node is finalised. For a corner-to-corner query this avoids exploring large regions of the grid that lie beyond the optimal path. Both solutions check this (while target in unvisited in Python, while (!visited[target]) in C, plus an inner break in C when the popped node is the target). Use this optimisation whenever you need a single shortest path rather than distances to all nodes.

Dictionaries and Sets for Sparse Bookkeeping in Python

The Python solution uses a dict for distances and a set for the unvisited frontier. Membership tests (neighbor in unvisited) and removals (unvisited.remove(current)) are O(1) on average, and min(unvisited, key=...) performs the linear scan idiomatically. This is clear and expressive, at the cost of tuple-hashing overhead that makes it materially slower than the flat C arrays.

Runtime Heap Allocation in C

The C code allocates dist and visited with malloc and frees them at the end of path_sum_four_ways. Although 6400 elements would fit on the stack, allocating runtime-sized buffers on the heap is good discipline and matters when the function is invoked repeatedly in a benchmark loop - the matching free calls prevent leaks across runs.

Key Takeaways

The unlocking insight is recognising that four-directional movement destroys the acyclic structure that made Problems 81 and 82 solvable by simple dynamic programming. Once cycles exist, you reach for a general shortest-path algorithm, and since all weights are positive, Dijkstra's algorithm is both correct and efficient. Even its naive O(V^2) linear-scan form handles a 6400-node grid comfortably.

Both solutions implement the identical algorithm, so the benchmark gap is purely a language-and-data-structure effect. On the full 80-by-80 matrix the C solution runs in 0.0228 s while the Python solution takes 1.5129 s - roughly 66 times faster. The difference comes from C's flat contiguous arrays versus Python's tuple-keyed hashing, compiled arithmetic versus interpreted bytecode, and the absence of per-object overhead. For N = 80 the Python version is perfectly usable; as the O(N^4) work scales into larger N, the C version's memory layout becomes decisive.

  • When grid movement allows cycles, abandon single-pass DP and use a general shortest-path algorithm.
  • Dijkstra's algorithm is exact and efficient precisely when all edge weights are non-negative.
  • A naive O(V^2) Dijkstra with a linear minimum scan is often good enough for graphs up to tens of thousands of nodes.
  • For single-target queries, stop as soon as the target is finalised to avoid unnecessary exploration.
  • Derive a concrete "infinity" from the data rather than using INT_MAX, to sidestep overflow and mixed-type arithmetic.
  • Flatten 2D grids to row-major 1D indices in low-level code for better cache behaviour and simpler memory management.
  • Encode neighbour offsets as direction vectors to keep traversal code compact and easy to extend.
  • High-level dictionaries and sets are ergonomic, but inner-loop-heavy algorithms pay a large constant-factor penalty for their convenience.

Problem Analysis

I am asked to find every number that equals the sum of the factorials of its own decimal digits, the canonical example being 145 = 1! + 4! + 5!. Any search problem of this shape begins with the same question: what bounds the search space? Without an upper limit the candidate set is infinite, so I need an analytic argument to make it finite. The argument is a counting one. The largest digit factorial is 9! = 362880, so a d-digit number can produce a digit factorial sum of at most d * 362880. For d = 8 that ceiling is 2,903,040 - a number with only seven digits. An eight-digit number can therefore never equal its own digit factorial sum, because the sum is too small to reach eight digits. This proves that I only need to examine numbers with at most seven digits, capping the brute-force search at roughly ten million candidates.

The second and more powerful observation is one of symmetry: the digit factorial sum depends only on the multiset of digits - the collection of digit values with their multiplicities but without regard to order - not on the sequence in which they appear. The numbers 145, 514, and 451 all share the identical digit factorial sum. So rather than scanning all ten million ordered numbers I only need to enumerate distinct digit multisets. Enumerating multisets is precisely the operation called combinations with replacement: choosing k digits from {0,...,9} where repetition is allowed but order is ignored. For length k there are C(k + 9, k) such multisets; summed over k = 2 through 7 the total is under 20000 candidates - roughly 500x fewer than the naive scan. The complexity is therefore O(sum over k of C(k+9, k)) candidate evaluations, each costing O(k) work to compute and verify - polynomial in the digit count and trivially fast in practice.

Solution Approach

Combinations-with-Replacement Filter (Python and C)

Both files implement the same algorithm: generate each digit multiset once, compute its factorial sum, then verify whether that sum is itself a number whose digits reproduce the multiset.

Step 1 - precompute the factorials. The ten digit factorials 0! through 9! are stored in a lookup table: a dictionary keyed by character in Python, a plain int[10] indexed by digit value in C. Both give O(1) factorial access and avoid recomputing factorials inside the inner loop.

Step 2 - enumerate multisets. For each length k from 2 to 7, I generate every non-decreasing sequence of length k over {0,...,9}. A non-decreasing sequence is the canonical representative of a multiset, so each multiset is visited exactly once with no deduplication pass needed. Python calls itertools.combinations_with_replacement directly. The C code drives the enumeration by hand: it starts with all positions at 0, then to advance it finds the rightmost position that is not yet 9, increments it, and sets every later position to that same value. This walks through all non-decreasing sequences in order.

Step 3 - compute the factorial sum of the digits in the combination.

Step 4 - apply three guards, cheapest first. Guard A: the factorial sum must have exactly k digits, otherwise it cannot be a k-digit candidate. Guard B: every digit in the combination must appear in the decimal representation of the factorial sum - a necessary (though not by itself sufficient) condition for the two digit multisets to coincide. Guard C: the definitive test - recompute the digit factorial sum of the sum itself and confirm it equals the sum. Guard C is what actually proves the curiosity property; the earlier guards exist only to discard the overwhelming majority of candidates cheaply before reaching it.

The differences are purely in representation. Python converts the sum to a string and tests digit membership with the in operator. C uses snprintf to render the sum, then counts digit frequencies in two int[10] arrays (need from the combination, have from the string) and verifies need[d] <= have[d] for every digit - a stricter multiplicity check than Python's plain membership test, though both reach the same answer because guard C is exact regardless. C's definitive recomputation in digit_factorial_sum uses arithmetic (modulo 10, divide by 10) rather than string conversion.

Programming Techniques

Precomputed Lookup Tables

Both solutions store the ten digit factorials once before searching. Recomputing a factorial by repeated multiplication inside the inner loop would waste cycles on a value that has only ten possible inputs. A direct array or dictionary lookup is O(1) with a single memory access. Whenever a function has a small, fixed domain and is called many times, precomputing it into a table is the first micro-optimisation I reach for.

Multiset Enumeration via Non-Decreasing Sequences

Generating non-decreasing sequences is the standard way to visit each multiset exactly once without a hash set or a separate deduplication pass. Python's combinations with replacement packages this into one readable call. The C version implements the "advance the rightmost non-maximal position" rule by hand. Understanding this rule frees me from depending on a library: it generates combinations with replacement in any language and lets me interleave generation with early termination.

Generator Expressions and Expression-Level Binding (Python)

The Python solve is a single sum() wrapping a generator expression with nested for clauses and filter conditions. A generator expression yields candidates one at a time without materialising a full list in memory. To bind the intermediate string of the factorial sum without a separate assignment statement, the code uses the idiom for num in (str(...),) - iterating over a one-element tuple. This is a clean way to introduce a named intermediate inside a comprehension where assignment is not otherwise permitted.

Progressive Filtering (Guard Chains)

The three guards are ordered from cheapest-and-coarsest to most-expensive-and-exact: a single integer comparison (digit count), a linear scan over at most ten digits (subset check), then a full digit decomposition (definitive recomputation). Because each guard is only reached when all earlier ones pass, the total cost is dominated by applying the cheapest guard to every candidate. Ordering tests by the ratio of cost to discriminating power is a principle that recurs everywhere from search loops to database query planning.

Choosing String vs Arithmetic Digit Extraction

The Python solution converts numbers to strings to test digit membership; C uses snprintf for the subset check but plain modulo/divide arithmetic for the final recomputation. The right representation depends on the downstream operation: membership and frequency tests favour strings or counting arrays, while accumulating a sum favours arithmetic. Picking the representation that makes the next operation natural avoids needless conversions.

Key Takeaways

The insight that unlocks this problem is recognising that a digit factorial sum is order-independent, which turns a sequential scan of around ten million numbers into an enumeration of under 20000 multisets - a combinatorial saving from a single observation about symmetry. Pairing that with an analytically derived upper bound (no candidate exceeds seven digits) keeps the search both finite and tiny.

Both solutions implement the identical algorithm and are extremely fast. The C version runs in about 0.0003 s and the Python version in about 0.0098 s - roughly 30x faster - reflecting the overhead of Python's dynamic dispatch and object allocation even in a compact implementation. The Python version is the more elegant to read thanks to combinations with replacement and the single generator expression form; the C version is the faster to run.

  • When a property depends only on a multiset of values and not their order, enumerate multisets instead of sequences - the saving is combinatorial.
  • Derive an upper bound on the search space analytically before writing any search code; it often shrinks the problem dramatically.
  • Precompute any function over a small fixed domain into a lookup table and never recompute it inside a loop.
  • Order filter conditions from cheapest-and-coarsest to most-expensive-and-exact, so each layer is only reached by candidates that survived the cheaper ones.
  • Generate combinations-with-replacement by walking non-decreasing sequences; the manual "advance rightmost non-maximal position" rule works in any language.
  • In Python, iterating a one-element tuple (for x in (expr,)) cleanly binds an intermediate result inside a comprehension.
  • Choose string or arithmetic digit handling based on the next operation: membership and counting favour strings, accumulation favours arithmetic.

Problem Analysis

The task is to find the smallest prime number that belongs to an "eight-prime family." A family is built by taking a number, choosing one digit value that appears in it, and replacing every occurrence of that value simultaneously with each of the digits 0 through 9. This yields up to ten candidate numbers; if eight of them are prime, the family qualifies. The crucial modelling insight is that I replace a digit value (for example, all the asterisk-marked 1s in 56**3), not an arbitrary subset of positions. That distinction collapses what could otherwise be a search over all subsets of digit positions - exponential in the number of digits - into a tiny linear scan: for each prime I need only try a handful of digit values to replace.

A second observation prunes the search further. If a digit value d is the one being replaced, then the largest family that can start with d as its smallest member contains 10 - d numbers - I can replace d with d, d+1, ..., 9. To reach a run of prime_run primes I therefore need 10 - d >= prime_run, that is d <= 10 - prime_run. For an eight-prime family the replaced digit can only be 0, 1, or 2. I iterate primes in ascending order, so the first match is automatically the smallest, and for each prime I try only the permitted low replacement digits. The overall complexity is roughly O(P * D) where P is the count of primes below 10^num_digits and D is the digit length - completely tractable for six-digit numbers.

Solution Approach

Sieve Generation Plus Digit-Replacement Enumeration (Python and C)

Both files implement the same three-phase algorithm and differ only in language idioms.

Phase 1 - prime generation via the Sieve of Sundaram. I produce every prime up to 10^num_digits. The Sundaram sieve marks composite odd numbers: an odd value 2k+1 is composite exactly when k = i + j + 2ij for integers 1 <= i <= j. Every unmarked index i then yields the prime 2i+1, and I prepend 2 manually. I chose Sundaram as a compact alternative to the Sieve of Eratosthenes; either gives a sorted prime list, which is all I need to scan in order.

Phase 2 - candidate enumeration. For each prime I convert it to its decimal digit string. I loop over each permitted replaced digit (characters '0' through '0' + (10 - prime_run), exclusive). If the digit does not appear in the string I skip it - no replacement is possible. Otherwise I substitute every digit from the replaced value up to '9' into all occurrences, building a new number string. I discard any result with a leading zero (which would be a shorter number and not a valid same-length family member), and I count only results that are both prime and >= prime. Counting only forward - members at or above the current prime - ensures families are never double-counted: any smaller member would have triggered the match earlier in the ascending scan.

Phase 3 - primality check. Each candidate is tested with simple trial division up to its square root. Because I test only individual candidates - not a second full sieve - and the numbers are at most six digits, this is more than fast enough. When the forward count equals prime_run, I return the current prime immediately.

Language-specific differences. The Python version expresses the replacement family as a generator inside a tuple comprehension and uses str(prime).replace(...) for substitution - concise and idiomatic. The C version uses snprintf to stringify, builds the replaced string with a manual character loop, and uses atoi to convert back. C stores sieve primes in a heap-allocated array (via calloc/malloc) and frees them after use. The algorithm and pruning are identical; only the mechanics differ.

Programming Techniques

Byte-Array Sieve

Both solutions mark composites in a flat array of bytes - Python's bytearray and C's unsigned char * from calloc. A byte-per-element layout needs no bit masking, so the inner marking loop stays simple. The cost is 8x more memory than a bit array, which is irrelevant for limits in the low millions. Reach for a byte sieve when clarity matters and memory is not the constraint.

Digit-as-Character Manipulation

The replacement step treats numbers as strings of digit characters rather than as arithmetic values. "Replace all occurrences of digit d" becomes a direct string substitution rather than repeated modulo/division to extract digits and reconstruct the number. In Python this is one call to str.replace; in C it is a short character-copy loop. I convert back to an integer only when I need to test primality. Whenever an operation is naturally about digit symbols, model the number as text first.

Constraint-Bounded Loop Ranges

Rather than trying all ten replaced digits and filtering, I bake the bound d <= 10 - prime_run directly into the loop limit ("0123456789"[:10 - prime_run] in Python, rd < 10 - prime_run in C). Encoding a mathematical constraint as the loop range, rather than as an early-exit branch inside the body, keeps the inner loop tight and eliminates impossible iterations before they start.

Scan-and-Stop on Sorted Input

Because primes are produced in ascending order and I want the smallest qualifying one, I return on the first match. There is no need to collect candidates and minimise afterward. This "find the first element of an ordered sequence satisfying a predicate" pattern is optimal in output cost whenever the search order already matches the desired result order.

Forward-Only Counting to Avoid Duplicates

When sizing a family, I count only members >= the current prime. Smaller members belong to the same family but would have been discovered - and the search stopped - at an earlier prime. Counting forward only is what guarantees the algorithm returns the smallest family member, not a later one.

Generator Expression Inside a Comprehension (Python)

The Python solution builds the qualifying family as a tuple from a generator with two if filters and a walrus operator (:=) to bind new_prime once and reuse it in both the comparison and the result. This keeps the construction declarative and avoids a separate temporary variable. Use the walrus when you need a value computed in a filter to also appear in the produced item.

Parameterised Solver

Both versions accept num_digits and prime_run as inputs (positional arguments in both languages), so the same code runs the 6-prime and 7-prime warm-up cases as well as the 8-prime main case. Parameterising the search turns the smaller known-answer cases into a built-in regression suite rather than hard-coding constants.

Key Takeaways

The unlocking insight is that replacing a digit value is far more constrained than replacing arbitrary positions: the family is determined entirely by which value you swap, turning an exponential subset search into a trivial linear one. Combined with the bound that an eight-prime family can only start by replacing a 0, 1, or 2, the search runs in milliseconds.

Both files implement the identical algorithm, so the benchmark gap is purely a language effect. On the main case (6 8) the C build runs in 0.0100 s versus 0.7614 s for Python - roughly 75x faster - and the same ratio holds on the dev cases (for example 0.0021 s versus 0.1498 s on 6 6). The C version wins on raw speed thanks to compiled native code and direct memory access; the Python version wins on brevity and readability, and at well under a second it is perfectly adequate for a one-off puzzle. Choose Python for fast development when the limit is small, and reach for C when you need to push the search space much higher.

  • When a problem involves replacing "a digit," model the number as a digit string first - substitution is cleaner than arithmetic extraction.
  • Translate mathematical constraints into loop bounds rather than in-loop branches, so impossible cases never execute.
  • If the answer is the minimum of a set, iterate candidates in ascending order and return on the first hit - no collection or post-sort needed.
  • A byte-per-element sieve is simpler than a bit sieve and fast enough for limits in the millions; prefer it unless memory is scarce.
  • When counting members of a group while scanning in order, count only forward to avoid rediscovering the same group from a later starting point.
  • Parameterise solvers over their key inputs and validate on smaller known-answer cases before running the full problem.

Problem Analysis

The cipher here is a repeating-key XOR cipher - a byte-level form of the Vigenere cipher. Every plaintext byte is combined with one byte of a short, cyclically repeating key using the bitwise XOR operation. XOR has a property that makes the scheme both convenient and breakable: it is its own inverse. Because (p ^ k) ^ k == p, decryption is the very same operation as encryption - you reapply the key. That is exactly why 65 XOR 42 = 107 and 107 XOR 42 = 65.

The observation that makes the key recoverable without knowing it is the periodicity of a repeating key. With a key of length k, the byte at position j is always encrypted with key byte j mod k. So key position 0 only ever touches ciphertext bytes 0, k, 2k, ...; key position 1 touches 1, k+1, 2k+1, ...; and so on. If I partition the ciphertext into k interleaved sub-sequences (one per key position), each sub-sequence was encrypted with a single constant byte - that is, each is a one-byte XOR cipher. I can break each independently. Because the problem tells me the key is three lowercase letters, every position has only 26 candidate values. Instead of searching the combinatorial space of 26^k full keys, I solve k separate one-dimensional problems of 26 trials each. That is the difference between exponential and linear work. To pick the right candidate per slice I use frequency analysis: natural English is dominated by e, t, a, and n, so the candidate byte that produces the most of these is almost certainly correct. The overall complexity is O(26 * n) for n ciphertext bytes - negligible.

Solution Approach

Frequency-Scoring Slice Attack (Python and C, index 0)

Both files implement the same algorithm and differ only in language idiom. It proceeds in four steps.

Step 1 - Parse the ciphertext. The input is a comma-separated list of decimal ASCII codes. Python uses a list comprehension, [int(x) for x in ...split(",")]. C scans the buffer character by character, accumulating digit runs into integers and pushing them into a dynamically grown array. Both fetch the file through a shared get_text_file helper.

Step 2 - Build interleaved slices. Given key length k, slice i holds ciphertext bytes at positions i, i+k, i+2k, ... Python expresses this directly with extended slice notation: encrypted[i::key_length]. C uses a two-pass approach: a first j % key_length pass counts each slice's size, then it allocates each slice to its exact size and a second pass routes elements in. This avoids over-allocation and per-slice reallocation.

Step 3 - Score each candidate byte per slice. For each candidate byte from 97 ('a') to 122 ('z'), and each slice, I XOR every element with the candidate and count how many results appear in the supplied list of common letters (eEaAnNtT). The highest-scoring candidate becomes that slice's key byte. I deliberately chose a plain count against a short alphabet rather than a full chi-squared test: the ciphertext is long and the per-slice key space is tiny, so the common-letter count is more than discriminating enough, and far simpler to reason about.

Step 4 - Compute the answer. With the per-position key known, I XOR each ciphertext byte back to plaintext and sum all ASCII values. Python caches the decrypted slice strings while scoring and reuses them; C runs a final XOR pass over each slice. The Python version also prints the recovered plaintext under --show - a valuable sanity check, since the bare answer 129448 gives no confidence the decryption is real English.

Programming Techniques

Interleaved Slice Extraction

Python's encrypted[i::key_length] is extended slice syntax: start at index i and take every k-th element. It is a one-line, allocation-light way to extract a periodic sub-sequence, replacing a manual modular-index loop. Whenever a problem decomposes into periodic strides, this should be the first tool you reach for. In C the equivalent is a j % key_length dispatch - more verbose, but it exposes the same structure explicitly.

The Walrus Operator for Compute-Once Comparison

The expression (_score := sum(...)) > score[i] uses the assignment expression (walrus) operator. It captures the computed score in a variable while comparing it, so the sum is evaluated exactly once yet remains available to store on the next line. This is the idiomatic way to avoid recomputing a value you both test and keep.

Generator Expressions for Aggregation

sum(_decrypted.count(x) for x in most_common_letters) and sum(ord(_char) for ...) feed lazy generator expressions straight into sum without building an intermediate list. This keeps memory flat and the code compact. Reach for them any time you aggregate (sum, max, any, all) over a transformed sequence.

Structure-of-Arrays State

Both solutions hold parallel arrays - key, score, and in Python decrypted - indexed by key position, rather than one array of records. This structure-of-arrays layout keeps the comprehensions clean in Python and avoids defining a one-off struct in C. It is the natural choice when each attribute is accessed uniformly across all positions in a tight loop.

Dynamic Array Growth by Doubling in C

The C parser does not know the element count in advance, so it starts at capacity 8192 and doubles via realloc on overflow. Doubling guarantees amortised O(n) total copying - each element is relocated only a constant number of times. Never grow by a fixed additive amount; that degrades to O(n^2).

Two-Pass Exact Allocation

For the slices, C counts sizes first, then allocates each slice to its exact size, then fills. This trades one extra pass for zero wasted memory and no per-slice reallocation. When the final size is cheaply computable up front, a counting pass followed by an exact allocation is cleaner than incremental growth.

Argument-Driven Parameterisation

Neither solution hard-codes the key length or the scoring alphabet; both take them as arguments. The same binary or script then handles any cipher file, key length, or language assumption without editing - and is trivially testable with alternative vectors. Keeping tunable values external costs nothing.

Separating I/O, Solver, and Benchmarking

Both files factor file reading into get_text_file, the algorithm into solve, and timing into the runner. This separation keeps each concern independently readable and testable, and it is what lets the harness run the solver repeatedly for benchmarking without the algorithm knowing anything about it.

Key Takeaways

The unlocking insight is that a repeating-key XOR cipher is not one hard cipher but k easy ones, interleaved. Once you partition the ciphertext by position modulo the key length, each slice is a single-byte XOR that yields to a 26-candidate frequency analysis scan. That structural decomposition turns a 26^k search into O(26 * n) work.

Both solutions reach 129448. The C build runs in about 0.0001 s and the Python script in about 0.0018 s - roughly an order of magnitude apart, consistent with interpreter overhead on tight per-byte loops. Both are effectively instant; the C version is faster, while the Python version is more concise and gains a free correctness aid through --show. For a problem this size the language choice is purely a matter of taste.

  • When a transformation uses a repeating key of length k, split the data into k interleaved slices and solve each independently - this collapses exponential search into linear.
  • XOR is its own inverse; encryption and decryption are the same operation, which both enables the attack and simplifies the code.
  • Frequency analysis needs only a short list of common symbols and a count, not a full statistical test, when the data is long and the per-position key space is small.
  • Prefer extended slice notation seq[start::step] for periodic sub-sequences over manual modular-index loops.
  • Feed generator expressions directly into sum or max to aggregate without allocating intermediates.
  • Capture a computed value once with the walrus operator when you both test and reuse it.
  • In C, double an array's capacity on overflow for amortised O(n) growth, or count-then-allocate when the final size is known cheaply.
  • Keep tunable parameters external as arguments, and separate I/O, algorithm, and benchmarking - the discipline pays off the moment you need to test or time the code.

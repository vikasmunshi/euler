Problem Analysis

The probability of drawing two blue discs in succession, without replacement, from a box of n total discs containing b blue discs is (b/n) * ((b-1)/(n-1)). Setting this equal to 1/2 and clearing denominators gives 2 * b * (b - 1) = n * (n - 1). Because both quantities count physical discs, only integer solutions matter, which makes this a Diophantine equation - a polynomial equation for which we seek integer solutions only. A naive search over all n up to 10^12 is hopeless, so the task is really to find hidden structure in this relation.

The decisive observation comes from completing the square on both sides. Multiplying out and rearranging yields (2n - 1)^2 - 2 * (2b - 1)^2 = -1. With the substitution x = 2n - 1 and y = 2b - 1 this is exactly x^2 - 2*y^2 = -1, a classic Pell equation. A Pell equation of the form x^2 - D*y^2 = ±1 has infinitely many integer solutions, all generated from a small fundamental solution by a fixed linear recurrence relation. Once this structure is recognised, the open-ended search collapses into a direct iteration: each step produces the next valid arrangement. Because the solutions grow geometrically (by a factor of roughly 3 + 2*sqrt(2) ≈ 5.83 per step), only about 40 iterations suffice to cross 10^12. The overall complexity is O(log T) in the threshold T, with each iteration performing constant-time arithmetic.

Solution Approach

Iterating the Pell equation recurrence (Python and C)

Both files implement the same algorithm: iterate the linear recurrence that generates successive solutions of the Pell equation x^2 - 2*y^2 = -1. The fundamental solution is (x, y) = (1, 1), which corresponds to the trivial arrangement n = 1, b = 1. From any solution the next is obtained by composing with the fundamental solution (3, 2) of the companion equation X^2 - 2*Y^2 = +1. Worked out in matrix form, this gives the recurrence relation:

x' = 3x + 4y
y' = 2x + 3y

I start at (x, y) = (1, 1) and apply the step in a loop. After each step I recover the disc counts with n = (x + 1) / 2 and b = (y + 1) / 2. These divisions are exact because x and y stay odd at every step - a parity that the recurrence preserves from odd starting values (each new value is an even multiple plus an odd multiple of two odd numbers, hence odd). The loop stops and returns b as soon as n first meets or exceeds the target total.

The two implementations are line-for-line equivalent. The chief language-specific difference is integer handling: Python uses arbitrary-precision integers automatically, while C declares long long (64-bit signed) explicitly. Since the answer 756872327473 and its corresponding total comfortably fit within 64 bits, the C version is safe with no big-integer machinery. The second difference is in how the recurrence step is written: Python uses a single tuple assignment, whereas C introduces two temporaries (nx, ny) so that the new x does not clobber the old x before y is computed.

Programming Techniques

Parallel (tuple) assignment for recurrence state

In Python I write x, y = (3*x + 4*y, 2*x + 3*y) as one statement. Python fully evaluates the right-hand tuple before binding any name, so both new values derive from the old x and y. This is the idiomatic, error-free way to advance a multi-variable recurrence relation without a temporary. In C, where assignment is eager, I must introduce nx and ny to capture the new values before overwriting the old - forgetting this is a classic bug when translating recurrences. Whenever new state depends on the whole old state, reach for parallel assignment (Python) or explicit temporaries (C).

Carrying transformed variables that evolve linearly

Rather than tracking n and b directly - which would force solving a quadratic each step - I carry the Pell equation variables x = 2n - 1 and y = 2b - 1, which satisfy a simple linear recurrence relation, and recover n and b with a single divide-by-two at the end of each step. This change-of-variable trick is broadly useful: when the natural quantities obey a nonlinear relation but a simple function of them obeys a linear one, work in the linear domain and invert cheaply at the boundaries.

Infinite loop with an internal exit condition

Both solutions use while True: / while (1) with a return inside. The number of iterations is data-dependent, and bounding it in advance would itself require a logarithm, so an unbounded loop with an explicit early return most honestly expresses "iterate until the threshold is crossed." Reach for this pattern when the iteration count is unknown up front but termination is guaranteed - here by the geometric growth of the solutions.

Command-line parameterisation for incremental testing

The target total is passed as an argument rather than hard-coded. This lets the same program run the small developer checks (21 and 120 discs, whose answers are verifiable by hand) and the full 10^12 case through one identical code path, with no recompilation. Parameterising by input is a habit that pays off on harder problems where the final answer cannot be checked directly but a code path already validated on small inputs can be trusted.

Key Takeaways

The unlocking insight is purely mathematical: the innocuous probability condition rearranges into a Pell equation, and Pell equations come with a built-in generator for all their solutions. That turns what looks like a brute-force search over 10^12 possibilities into a handful of constant-time arithmetic steps. Recognising a special equation form is worth more here than any amount of loop optimisation.

Both implementations run in unmeasurable time (reported as 0.0000 s) on every test case, because the loop executes only a few dozen times. With such a short, geometrically growing iteration the choice between C and Python is irrelevant to performance; the Python version is marginally more elegant thanks to tuple assignment, while the C version is a faithful, equally fast translation. Neither is meaningfully faster than the other.

  • When a probability or combinatorial condition reduces to a polynomial equation in two integer unknowns, look for Pell or other special structure before attempting a search.
  • A Pell equation x^2 - D*y^2 = ±1 always has a fundamental solution from which all others follow by a fixed linear recurrence, turning enumeration into iteration.
  • Work in transformed variables that evolve linearly, then recover the natural quantities by cheap arithmetic, to avoid repeated nonlinear computation inside a loop.
  • Advance multi-variable recurrences with parallel assignment or explicit temporaries so you never read partially updated state.
  • Use an unbounded loop with an internal exit when the iteration count is data-dependent but termination is guaranteed.
  • Parameterise solutions by their inputs so small, hand-verifiable cases and the full problem share one trusted code path.
  • An O(log T) loop body makes runtime negligible even for thresholds as large as 10^12 or beyond.

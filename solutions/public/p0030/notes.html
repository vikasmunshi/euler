Problem Analysis

I want every integer with more than one digit that equals the sum of the fifth powers of its own digits. Numbers with this self-referential property are called narcissistic numbers (also "Armstrong numbers") for the chosen exponent. The single-digit case is excluded because 1 = 1^5 is a value, not a sum. At first glance the search looks unbounded - there are infinitely many integers - so the first and most important job is to derive a finite ceiling beyond which no solution can exist.

The bound comes from a counting argument. For exponent n, the largest digit-power sum a k-digit number can produce is k * 9^n (every digit a 9). But the smallest k-digit number is 10^(k-1). Once k * 9^n has fewer digits than k itself, no k-digit number can equal its own digit-power sum, and the search must terminate. Taking the base-10 logarithm of n * 9^n and rounding up gives the maximum number of digits any candidate can have; for n = 5 this works out to 6 digits. The second key observation is that the digit-power sum depends only on the multiset of digits, not their order, because addition is commutative. So rather than scanning up to a million integers, I enumerate only sorted digit combinations using combinations with replacement. The number of such multisets of length k from 10 digits is C(10+k-1, k) - a few thousand for k = 6 rather than a million. The complexity is therefore O(C(10+k-1, k) * k), which is trivially small.

Solution Approach

Combinations-with-replacement enumeration (Python and C)

Both files implement the same algorithm, so I describe it once and note language differences at the end.

Step 1 - compute the search width. I evaluate k = ceil(log10(n * 9^n)) to fix the number of digits I will enumerate. I use exactly this length and no fewer because any narcissistic number with fewer digits is still discovered: its missing leading positions simply appear in the multiset as zeros, and 0^n = 0 contributes nothing to the sum. Enumerating a single fixed length keeps the loop structure clean - I do not need a separate pass per digit count.

Step 2 - enumerate multisets. I walk over every non-decreasing sequence of length k drawn from {0, ..., 9}. In Python this is handed to me directly by itertools.combinations_with_replacement(range(10), k). In C, lacking such an iterator, I keep an integer array indices[0..k-1] and advance it lexicographically: find the rightmost position that is not yet 9, increment it, and flood-fill every position to its right with that same value. The flood-fill preserves the non-decreasing invariant and keeps each multiset in exactly one canonical form, so no combination with replacement is visited twice.

Step 3 - candidate sum and verification. For each multiset I sum the n-th powers of its elements to get a candidate num. A multiset's sum is only a genuine narcissistic number if the digits of num itself reproduce that sum. So I verify num > 9 (to drop the trivial single-digit cases) and that the digit-power sum of num equals num. In Python I re-expand via str(num) and sum the powers of each character; in C the helper digit_power_sum peels digits off with repeated % 10 and / 10.

Language differences. The Python solve is a single sum(...) over a generator expression, using the walrus operator to bind num inline. The C version uses explicit loops, a manually advanced combination array, and an integer power helper instead of pow() in the hot path. The algorithm is identical; the C build is roughly 50x faster on the main case purely because of compiled native code versus interpreter overhead.

Programming Techniques

Combinations with replacement to eliminate redundant work

Rather than looping over a dense integer range, I loop over unordered multisets of digits. Because the digit-power sum is symmetric in its inputs, many distinct integers map to the same sum; enumerating only the canonical sorted form computes each sum once. I reach for combinations with replacement any time the quantity I am computing is invariant under permutation of its inputs - the savings are combinatorial.

Analytic upper bound before any loop

The closed-form ceiling ceil(log10(n * 9^n)) converts a seemingly unbounded search into a finite one. The base-10 logarithm counts the number of digits of a value, so comparing it against k directly answers "can a k-digit number still reach its own digit-power sum?". Whenever a problem looks infinite, I ask what the maximum possible value of the tracked quantity is; the answer often gives a tight, provable bound for free.

Walrus operator for inline binding (Python)

num := sum(x**n for x in digits) computes the candidate and binds it in one step inside the generator's filter, so the sum is not computed twice. I use the walrus operator whenever a value must be both tested and reused without breaking out of an expression.

Short-circuit ordering in conjunctions

The filter num > 9 and num == sum(...) places the cheap comparison first. Short-circuit evaluation in Python (and C) stops at the first false term, so the expensive re-expansion only runs when the cheap guard passes. I always put the cheapest, most-discriminating test first in an and chain.

Manual lexicographic iterator in C

C has no combinations iterator, so I encode the current multiset in a fixed-length array and define a "find the rightmost non-maximal slot, increment, flood-fill right" advance rule, looping until it cannot advance. This produces every multiset in lexicographic order exactly once. This state-array-plus-next-state pattern is the standard way to enumerate any combinatorial structure in a language without a rich standard library.

Integer power helper instead of floating-point pow (C)

ipow multiplies in a loop using long long, avoiding the floating-point rounding risk of math.h's pow() on integer arguments. For exact integer exponentiation with small fixed exponents, an integer loop is both faster and correct. I do still use pow/log10 once for the bound, where a tiny rounding margin is harmless.

Argument-driven parameterisation

Both solutions take the exponent n as an argument rather than hard-coding 5, which is why the benchmark can exercise both n = 4 and n = 5. Parameterising costs nothing and makes the code reusable across the whole family of narcissistic-number problems while keeping automated testing simple.

Key Takeaways

The unlock is recognising that a digit-power sum ignores digit order, which collapses a loop over a million integers into a loop over a few thousand multisets, plus the analytic bound that tells us where to stop. The verification step then ensures only genuine narcissistic numbers are counted.

Both files run the same algorithm. On the main case (n = 5) the C build averages about 0.0001 s versus 0.0056 s for Python - roughly 50x faster - while the dev case is too fast to measure meaningfully (0.0000 s for C, 0.0018 s for Python). The difference is entirely compilation versus interpretation; the algorithm itself is so cheap that either language finishes in well under 10 ms. I find the Python version the more elegant expression of the idea, and the C version the better demonstration of how to build a combinatorial iterator by hand.

  • When a function of its inputs is order-independent, enumerate multisets (combinations with replacement) rather than ordered tuples or whole integer ranges.
  • Derive a finite upper bound analytically before writing a loop; an "unbounded" search often has a tight closed-form ceiling.
  • Prefer exact integer arithmetic over floating-point pow() when computing integer powers in a verification path.
  • Place the cheapest, most-discriminating guard first in an and chain so short-circuiting skips the expensive work.
  • Use the walrus operator when a value must be both tested and reused inside a single expression.
  • In languages without a combinations iterator, encode the state in an array and define an explicit "next state" rule - a pattern reusable for any combinatorial enumeration.
  • Parameterise constants like the exponent so one implementation serves an entire problem family and is easy to test.

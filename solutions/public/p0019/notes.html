Problem Analysis

This problem asks us to count how many times the first day of a month landed on a Sunday over a 100-year window. The essential mathematical tool is modular arithmetic applied to the day-of-week cycle. The days of the week form a cycle of length 7: if today is a Monday, then 365 days later is whatever day you get by computing 365 mod 7 = 1 step forward. Because the day of week is inherently cyclic with period 7, we never need to track absolute date counts - only their residues modulo 7. Given a single known anchor date (1 Jan 1900 was a Monday), we can compute the day of week for any future date by accumulating the number of intervening days and reducing modulo 7.

The only subtlety is the Gregorian leap-year rule: a year is a leap year if divisible by 4, except for century years, which must additionally be divisible by 400. This rule decides whether February contributes 28 or 29 days and therefore how many days separate consecutive month boundaries. The problem is tractable because the search space is tiny: the 20th century spans only 1200 months. A simple linear scan over months suffices, giving complexity O(Y x 12) = O(Y) where Y is the number of years in the range - in practice an instantaneous constant-time computation. No combinatorics or dynamic programming is required.

Solution Approach

Day-of-week Simulation (C and Python)

Both files implement the same conceptual algorithm - anchor to a known date and advance month by month - but the C version computes the day-of-week from first principles while the Python version delegates to the standard library.

Step 1 - Establish a reference point. The problem states that 1 January 1900 was a Monday. In the C solution I encode Monday as 0, Tuesday as 1, ..., Sunday as 6. This zero-based scheme makes the modulo-7 arithmetic clean, since the residue is exactly the day index.

Step 2 - Advance to the start year. The count begins in 1901, not 1900. Rather than hard-code the day of week for 1 January 1901, I advance the counter from the known anchor by adding 366 for each leap year and 365 otherwise, reducing modulo 7 at each step. This keeps the code general: it accepts any start year as input.

Step 3 - Iterate over every first-of-month. For each year from start_year to end_year, and for each month 1 through 12, I first check whether the current day-of-week is 6 (Sunday) and increment the count if so, then advance the counter by the number of days in that month. The days_in_month helper encodes the standard month-length table and applies the leap-year correction to February. Crucially, I check before advancing, so dow always represents "the day of week on the 1st of the current month."

Language differences. The C solution does everything manually with loops and a static array. The Python solution lets datetime.date(y, m, 1).isoweekday() do the work - it returns 7 for Sunday, so Python never tracks the day-of-week or the leap-year rule itself. This is more concise and less error-prone, at the cost of a function call per date, which is negligible at this scale. The Python solution sums a generator expression, idiomatically counting matches without building an intermediate list.

Programming Techniques

State-variable pattern (rolling day-of-week counter)

The C solution maintains a single integer dow and updates it with dow = (dow + days_in_month(y, m)) % 7 after each month. This is the state-variable pattern: one variable carries forward all the information needed to take the next step, so we never recompute from scratch. Reach for this pattern whenever a quantity changes by a known delta at each step - it underlies rolling windows, running totals, and incremental hashing.

Modular arithmetic for cyclic quantities

The day of week wraps with period 7, so applying % 7 after each addition keeps the counter in [0, 6] automatically and eliminates any conditional resets. Any time a quantity wraps around - days of week, clock positions, indices in a circular buffer - modular arithmetic is the natural representation.

Delegating to the standard library (Python)

Python's datetime.date already encodes the full Gregorian calendar, including the leap-year rule and day-of-week computation. By calling isoweekday() I avoid reimplementing tested logic. The principle: check whether the standard library already solves a sub-problem before writing custom code. The trade-off is per-call overhead; for a hot path, a hand-rolled approach (as in C) may be preferable.

Generator expression with sum()

The Python solution writes sum(predicate for ... for ...). A generator expression is a lazy sequence that yields values one at a time without materialising a list, keeping memory usage O(1). Summing booleans exploits the fact that Python treats True as 1 and False as 0, so sum(predicate(x) for x in iterable) is the idiomatic way to count items satisfying a condition.

Command-line parameterisation

Both solutions read end_year and start_year as arguments rather than hard-coding the range. This separates logic from data, making the solutions testable across arbitrary ranges without recompilation - a small, free engineering win.

Key Takeaways

The core insight is that tracking the day of week requires nothing more than incrementing a counter modulo 7 - no date conversions or calendrical formulas are needed. Anchoring to a known reference date and advancing from there is simple, general, and correct. Both solutions produce 171 and run effectively instantly: the C version at about 0.0000039 s and the Python version at about 0.00023 s. In relative terms the C version is roughly 60x faster because it avoids constructing a date object per month, but the absolute difference is well under a millisecond and irrelevant at 1200 iterations; the Python version is the more elegant of the two, leaning on a well-tested standard library to express the same idea in a single statement.

  • For cyclic quantities, represent them modulo their period - it removes conditionals and keeps arithmetic clean.
  • The state-variable (rolling counter) pattern avoids redundant recomputation: carry just enough state to take the next step.
  • Check the standard library before writing custom calendar, math, or string logic - it is tested, correct, and expressive.
  • Counting items that satisfy a predicate is cleanly expressed as sum(predicate(x) for x in iterable) in Python.
  • Anchor-and-advance is a general technique: find one position you know, then walk forward incrementally rather than recomputing absolute positions.
  • Parameterise inputs through arguments rather than hard-coding them; it costs nothing and makes testing and reuse trivial.

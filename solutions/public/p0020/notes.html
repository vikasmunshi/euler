Problem Analysis

The task is to compute the sum of the decimal digits of 100!. The factorial of n is the product of all positive integers up to n, and the wrinkle is purely one of scale: 100! has 158 decimal digits, which vastly exceeds the roughly 19 digits a 64-bit integer can hold. The problem is therefore an exercise in arbitrary-precision arithmetic — computation on numbers larger than a native machine word, where the value is spread across many storage cells rather than held in a single register.

The observation that makes this tractable is that we never need n! as an abstract number — we only ever need its decimal digits. That reframes the work as two routine sub-tasks: repeatedly multiply a large number by a small integer (at most 100), then add up the resulting digits. Multiplying a D-digit number by a small scalar is ordinary long multiplication — walk each digit, multiply, propagate a carry — an O(D) pass. Doing this for every factor from 2 to n gives O(n · D(n)), where D(n) is the digit count of n!. By Stirling's approximation D(n) grows as O(n log n), so the whole algorithm is about O(n² log n) — utterly negligible at n = 100.

Solution Approach

Big-Integer Digit Array (C) and Built-in Arbitrary Precision (Python)

Both solutions (index 0) implement the same logical algorithm — form n!, then sum its decimal digits — but they diverge entirely in how they represent the large number.

The C solution builds the big integer by hand. I store n! as a little-endian array of single decimal digits: digits[0] is the least significant digit, digits[1] the next, and so on. I chose little-endian ordering deliberately, because carry propagation runs from least significant to most significant — the same direction as increasing array index — so the inner loop always walks forward. Starting from 1, I multiply by each i from 2 to n using long multiplication: for each i I walk every occupied digit, compute digits[j] * i + carry, store the result modulo 10, and carry the quotient forward; any carry remaining past the top digit appends new positions one digit at a time. A final pass sums every digit. I sized the buffer at 512 digits, which comfortably covers the 158 digits of 100!.

The Python solution is a single expression: sum(int(d) for d in str(math.factorial(n))). Python's int is arbitrary-precision arithmetic natively, so math.factorial returns the exact value with no manual digit handling; I convert it to a string and sum the integer value of each character. The representations differ fundamentally: the C code works in base 10 throughout, whereas CPython stores the integer in a binary internal form and converts to a decimal string only at the end.

Programming Techniques

Little-Endian Digit Arrays for Manual Big Integers

When you write big-integer arithmetic in C without a library, digit ordering has real consequences. I stored digits little-endian because the multiplication carry always moves from the lowest digit upward, matching natural forward iteration over an array; this keeps the inner loop simple and cache-friendly and avoids a final reversal. Big-endian storage would force backward scans or a reversal at the end. For any multiply-by-scalar bignum routine, little-endian is the right default.

Fixed-Size Stack Buffers with an Analytical Bound

Rather than calling malloc, I declared a fixed 512-element array on the stack, sizing it by reasoning about the worst case first: Stirling's approximation bounds 100! at 158 digits, so 512 is more than twice the requirement. A fixed stack buffer avoids allocation overhead, removes any chance of an allocation failure in the hot path, and keeps the code shorter. Reach for this whenever you can derive a tight, modest upper bound on a structure's size analytically.

Generator Expressions for Map-then-Fold in Python

The Python solution feeds a generator expression straight into sum(). A generator expression is a lazy iterable: it yields one value at a time without materialising an intermediate list, so memory stays constant. Here it converts each character to an integer on the fly and folds the results into a running total — the idiomatic Python way to express "transform each element, then aggregate", and the pattern to reach for instead of an explicit accumulation loop.

Leveraging the Standard Library for Big Numbers

Python's math.factorial is implemented in C inside CPython and uses efficient internal techniques — including Karatsuba multiplication for large operands — so delegating to it yields a correct, tested, fast routine for free and reduces the solution to one line. The principle generalises: check what the standard library offers before writing an algorithm by hand. The manual C version is invaluable for understanding the mechanics, but the library version is almost always the better production choice — and, as the timings below show, often the faster one too.

Key Takeaways

The unlocking insight is representational: by storing each decimal digit explicitly and computing in base 10 from the start, a "huge number" problem collapses into routine array manipulation with trivial carry arithmetic — no library, no floating point, no mid-computation base conversion.

The recorded averages are revealing. On the main case (100!) Python runs in 8.9 µs versus 12.2 µs for C, and on the dev case (10!) C runs in 0.45 µs versus 1.26 µs for Python. The crossover is instructive: for the tiny input C wins on raw overhead, but for 100! CPython's Karatsuba multiplication overtakes my hand-written long multiplication, whose O(D) per-multiply cost the library improves on. The Python solution is thus both the more elegant and the faster at the target size; the C version's value is pedagogical, exposing the mechanics the library hides. For far larger inputs, the C digit array would itself need a higher-radix representation or a faster multiplication algorithm to keep pace.

  • When a problem asks about the digits of a large number, represent it in base 10 from the outset to eliminate a base-conversion step.
  • Little-endian digit arrays align with carry propagation and are the default for hand-written big-integer multiplication in C.
  • Prefer a fixed stack buffer over heap allocation whenever you can bound the size analytically and the bound is modest.
  • Combine generator expressions with built-in aggregators like sum() to express map-then-fold cleanly and without intermediate storage.
  • Check the standard library before implementing an algorithm yourself — its routines are tested, optimised, and often asymptotically better than a naive hand-rolled version.
  • A microbenchmark can overturn intuition: "C is faster than Python" held only for the trivial input here — always measure rather than assume.

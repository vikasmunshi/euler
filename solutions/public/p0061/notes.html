Problem Analysis

The surface of this problem is number theory: figurate (polygonal) numbers generated by simple closed-form formulas such as the triangle numbers n(n+1)/2 and the octagonal numbers n(3n-2). But the real structure is combinatorial. I need an ordered set of six 4-digit numbers, one drawn from each of six polygon families, in which the last two digits of each number equal the first two digits of the next, and the chain closes back on itself. The phrase "last two digits link to the first two digits" is the decisive clue: this is a graph problem in disguise. If I treat each 4-digit figurate number as a node and draw a directed edge from A to B whenever A mod 100 == B div 100, then a valid answer is a cycle of length six in this graph whose members collectively cover all six polygon types.

Two observations make the problem tractable. First, the candidate set is tiny: each family contributes only a few dozen values between 1000 and 9999, so the whole graph has fewer than 200 nodes and the average out-degree (the number of successors per node) is small. Second, because each node carries one or more polygon-type labels and I want exactly one node per type, the search is naturally bounded to paths of length six. A depth-first search (DFS) that extends paths node by node, pruning when it would revisit a node, suffices. The worst-case bound is O(VL) for paths of length L in a graph of V nodes, but the short successor lists collapse this to a near-instant search in practice - the recorded times are 0.0009 s in C and 0.0233 s in Python on the full six-family case. The one subtlety that demands care is that some numbers belong to more than one polygon family (every hexagonal number is also triangular, for instance), so the type-coverage check is not simply "six distinct types appear."

Solution Approach

Graph-Based DFS with Polygon-Type Verification (C and Python)

Both files implement the same algorithm. I describe it once and note the language-specific differences afterwards.

Step 1 - generate figurate numbers. For each active polygon type I enumerate every 4-digit value its formula produces. In C I iterate n from 1, skip forward until the value reaches 1000, then collect values until they exceed 9999. In Python I do the same but seed the loop at the right index by inverting the formula: for triangle numbers, P(n) = n(n+1)/2 inverts to n = (-1 + sqrt(1+8P))/2, so I begin the generator at the floor of that value rather than from 1. The ranges are so small that the inverse is a refinement of style rather than a necessity, but it is the principled way to start a monotone sequence at a threshold.

Step 2 - build the successor graph. I collect all distinct 4-digit figurate values into a node list, then add a directed edge from node A to node B whenever A % 100 == B / 100. I additionally require the shared prefix to be at least 10: a two-digit prefix below 10 (like 05) cannot be the first two digits of a genuine 4-digit number, so those links would be spurious. C stores the graph as a fixed-size 2D array successors[i][] with a parallel succ_counts[] array; Python uses a dict mapping each number to a list of successors.

Step 3 - DFS for closed cycles. Starting from every node in turn, I walk the graph depth-first, maintaining the current path and a visited marker. When the path reaches the target length I check whether the final node links back to the start (cur % 100 == start / 100); if so I have a closed cycle. The visited marker guarantees all members of the path are distinct, which the problem requires. The marker is set on entry and cleared on exit - the standard backtracking discipline.

Step 4 - verify type coverage. A candidate cycle is valid only if each polygon type is represented by exactly one member, accounting for numbers that belong to several families. My check first counts, for each type, how many cycle members belong to it, and rejects the cycle if any type has zero members. Then it builds the reverse map - for each cycle member, which types it belongs to - and flags a conflict if any multi-type number is the sole representative of every one of its types simultaneously (which would mean those types cannot all be filled). The C version expresses this with small explicit loops; the Python version builds two dictionaries (type-to-members and member-to-types) and applies the same test.

Step 5 - deduplicate and report. Because the DFS starts from every node, the same unordered set is discovered once per starting point. I canonicalise each cycle by sorting its members, then store only unique sorted sets - Python uses a set of sorted tuples for O(1) deduplication, while C sorts each candidate with an insertion sort and linearly scans previously stored results. The final output reports how many valid sets were found and their sums.

Language differences. Python leans on generators, an enum for the polygon types, a decorator for input validation, and dict/set comprehensions, all of which make the code expressive and self-documenting. C uses static arrays sized by #define constants and an array of function pointers to drive a single generation loop over all six formulas. The algorithms are identical, so the roughly 25x speed gap (0.0009 s versus 0.0233 s on the main case) is pure interpreter overhead.

Programming Techniques

Modeling a linking constraint as a directed graph

The decisive move is recognising the "last two digits become the first two digits" rule as an edge relation. Once the data is a directed graph, the question "is there a closed chain of six members covering all types" becomes "is there a length-six cycle with a coverage property," which standard graph traversal answers directly. Whenever a problem describes objects that connect to each other under a deterministic rule, ask whether modeling those connections as edges turns a tangled search into a textbook graph algorithm.

Adjacency list for a sparse graph

With under 200 nodes each having only a handful of successors, I store the graph as an adjacency list rather than a dense matrix. Iterating successors[i] visits only real edges, which keeps the DFS inner loop short. An adjacency list is the natural choice whenever the graph is sparse - most pairs of nodes have no edge - and you mostly need to enumerate the neighbours of a given node.

DFS with a visited set and backtracking

The search maintains a visited structure - a boolean array in C, a set in Python - that is marked on entry to a node and unmarked on exit. This enforces the distinctness requirement and prevents infinite loops. The mark/recurse/unmark pattern is the canonical shape of any exhaustive path enumeration via backtracking; once you internalise it, every "find all paths/cycles satisfying P" problem follows the same skeleton.

Canonical form for deduplication

Starting DFS from every node finds each unordered cycle once per member. To count each set once I reduce every discovered cycle to a canonical form - its members sorted in ascending order - before storing it. In Python a sorted tuple in a set gives near-free deduplication; in C a sort plus a linear scan does the same. The general principle: when objects have several equivalent representations, define one canonical form and convert everything to it before comparing or storing.

Inverse formulas to initialise a loop at a threshold

The Python generators start iteration at the smallest index that produces a 4-digit value by inverting each monotone polygonal formula and taking the floor. This is a reusable trick: whenever a closed-form, monotonically increasing sequence must begin at some threshold, solving the formula for the index lets you jump straight to the starting point instead of iterating and discarding.

Function pointer arrays (C) versus a dict of callables (Python)

C stores the six generators in gen_func generators[6] so one loop, indexed by type, handles all of them. Python achieves the equivalent with dictionaries mapping enum members to functions (p_funcs, c_funcs). Both let you parameterise behaviour by a runtime selector and write the driving logic once. Reach for a function pointer array (or a dict of callables) whenever you have a fixed family of behaviours that differ only in body and need to be chosen by index or key.

Enum for a fixed set of named domain values

The Python FigurateNumber enum names the six polygon types and stores the side count as the value, so iterating FigurateNumber(n) for n in range(3, length+3) selects the active types and dictionary keys read as FigurateNumber.HEXAGONAL rather than a bare 6. Enums make code self-documenting and guard against passing the wrong integer. Use them whenever a fixed, symbolic set of constants would otherwise be raw ints or strings.

Decorator for precondition enforcement

The n_is_positive_integer decorator wraps each nth-value function and raises if the argument is not a positive integer. This separates the guard from the math and applies it uniformly, so adding a new formula needs only the formula body. Decorators are the right tool when the same cross-cutting check must apply consistently across many functions.

Key Takeaways

The insight that unlocks this problem is the reframing: a chain of numbers connected by a digit rule is a directed graph, and the answer is a cycle in it. After that, a plain DFS with a visited set and a post-search type-coverage check finds the unique six-family cycle almost instantly. The only genuinely tricky correctness detail is the coverage check, because numbers that belong to multiple polygon families can make a naive "all six types appear" test wrong.

Both solutions run the identical algorithm, so the benchmark gap is entirely language overhead. On the main length-6 case the C binary takes 0.0009 s while the Python program takes 0.0233 s - roughly a 25x difference driven by interpreter dispatch and dynamic data structures. For a one-off search of this size Python is more than fast enough and considerably more readable; C earns its keep only when the search is repeated thousands of times or grows much larger.

  • When objects connect under a deterministic rule, model the connections as graph edges - the reformulation often turns a hard-looking search into a standard traversal.
  • For small combinatorial searches with bounded path length, DFS with a visited set and backtracking is simple, correct, and fast enough; do not over-engineer.
  • Use an adjacency list for sparse graphs where you mostly enumerate a node's neighbours.
  • Define a canonical form (such as sorted order) and convert all equivalent representations to it before storing, so you count each solution once.
  • Watch for elements that belong to multiple categories; verification logic that assumes a unique category is a common and subtle source of bugs.
  • Invert a monotone closed-form sequence to start a loop at a threshold instead of iterating and discarding.
  • Function pointer arrays in C and dicts of callables in Python are equivalent ways to parameterise behaviour by a selector - use whichever is idiomatic.
  • Enums and precondition decorators make intent explicit and enforce invariants once, keeping the core logic clean.

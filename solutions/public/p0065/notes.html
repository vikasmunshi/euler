Problem Analysis

A continued fraction represents a number as a nested expression a0 + 1/(a1 + 1/(a2 + ...)), where the integers a_k are the partial quotients. If you truncate the expansion after k levels you get a rational number called the k-th convergent; these are provably the best rational approximations to the underlying real number for their denominator size. For the constant e the partial quotients follow a clean rule: [2; 1, 2, 1, 1, 4, 1, 1, 6, ...], where every third quotient is an even number 2k and all the others are 1. The task asks for the numerator of the 100th continued fraction convergent of e, then the sum of that numerator's decimal digits.

The key observation is that I never need to evaluate the nested fraction by hand. The numerators h(k) and denominators of successive convergents obey the standard three-term convergent recurrence h(k) = a(k)*h(k-1) + h(k-2), seeded with h(0) = 1 and h(1) = a(1). This turns an awkward nested division into a plain linear iteration. The one complication is magnitude: the numerator grows roughly like e^n, so by n = 100 it has about 40 decimal digits - already past the range of a 64-bit integer - and by n = 10000 it spans thousands of digits. Since each multiply-add touches all O(n) digits of the running value, the whole computation costs O(n^2) digit-operations: quadratic, but easily fast enough.

Solution Approach

Three-Term Convergent Recurrence (Python and C)

Both files compute the same quantity via the same convergent recurrence; they differ only in how they manage the large numerator. Each starts with a small helper, e_denominator(n), that encodes the partial-quotient pattern directly: it returns 2 for n == 1, 2*(n/3) when 3 divides n, and 1 otherwise. I chose to express the rule as a tiny function rather than a precomputed table, because the function is trivially correct for any n and generalises immediately without resizing anything.

The C solution iterates the recurrence directly. It seeds h_prev2 = 1 (the artificial h(0) that makes the first multiply-add come out right) and h_prev1 = 2 (which is a(1)). For each k from 2 to n it computes h_curr = a(k)*h_prev1 + h_prev2 and rotates the three buffers so only the two most recent numerators survive. After the loop, h_prev1 is the numerator of the n-th convergent, and a digit-sum pass produces the answer.

The Python solution expresses the same mathematics recursively. nth_convergent_of_e descends to depth n and then unwinds, repeatedly forming a + 1/sub-result using fractions.Fraction for exact rational arithmetic. This is a literal transcription of the definition of a continued fraction, so it is very easy to verify by inspection. Its cost is that it carries both numerator and denominator through every level and recurses to depth n - which is why it raises sys.setrecursionlimit - making it slower than the pure-numerator iteration at large n. The digit sum falls out of sum_digits, a repeated mod-10 / divide-10 loop.

The C version has no built-in big integers, so it supplies its own. It stores a number as an array of unsigned long long limbs in base 10^9 (the BigInt type). The workhorse is bigint_mul_add(dst, a, scalar, b), which computes a*scalar + b in a single pass: at each limb position it forms av*scalar + bv + carry, stores val % BASE, and propagates val / BASE. I deliberately picked base 10^9 for two reasons. First, the product of a limb (under 10^9) and a small scalar (at most about 6667 at n = 10000), plus a carry, stays comfortably within 64 bits, so no overflow can occur. Second, because the base is a power of ten, each limb is itself a block of decimal digits, so the final digit sum is a free byproduct rather than requiring a base conversion.

Programming Techniques

Sliding-Window State for Linear Recurrences

The numerators satisfy f(n) = expr(f(n-1), f(n-2)), so at any moment I only need the last two values. The C loop keeps exactly three BigInt objects and recycles them, giving O(1) auxiliary storage no matter how large n is. This is the same pattern used to compute Fibonacci numbers iteratively. Whenever you recognise a fixed-order linear recurrence, reach for a small set of state variables rather than storing the whole history.

Pointer Rotation Instead of Copying

Advancing the recurrence means "previous-1" becomes "previous-2" and the freshly computed value becomes "previous-1". Rather than memcpy multi-kilobyte BigInt structs each step, the C code swaps the three pointers and reuses the now-stale buffer as the destination for the next computation. This avoids per-iteration copying of large arrays and is the standard idiom for double- or triple-buffering in tight loops.

Limb-Based Big Integers with a Decimal Base

When native integer types are too narrow, a number can be held as an array of limbs, each carrying one digit-chunk in some base. My choice of base 10^9 (rather than a power of two) trades a little space efficiency for a free decimal digit sum: each limb is decoded with a simple v % 10; v /= 10 loop, with no base conversion. Choose your big-integer base to match the questions you will ask of the result.

Fused Multiply-Add

The recurrence's core is exactly coeff * prev + prev2. Instead of computing the product into a temporary and adding it in a second pass, bigint_mul_add fuses both into one loop over the limbs. This halves the memory traffic and avoids allocating an intermediate. Whenever a hot loop computes a*scalar + b, consider fusing the two steps.

Arbitrary-Precision Integers and Exact Rationals in Python

Python integers grow without bound, so the recurrence could be written with bare * and + and no big-integer library at all. The Python solution goes further and uses fractions.Fraction to model the nested fraction exactly, eliminating any floating-point error. For moderate inputs this is the quickest route to a verified-correct answer; the cost only becomes visible at very large n.

Encoding a Coefficient Rule as a Function

The partial-quotient pattern of e is a simple modular rule, so I encoded it as e_denominator(n) rather than as a lookup table. A function is easier to verify than a hand-typed array, costs nothing to maintain, and works for any n without bounds management. Prefer a small generating function over a table whenever the rule is concise.

Key Takeaways

The unlocking insight is that continued fraction convergents satisfy a simple three-term recurrence, so the numerator can be built up with elementary arithmetic instead of nested-fraction evaluation. Both solutions share this recurrence; they differ only in how they cope with numbers too large for a machine word.

The benchmarks make the trade-off concrete. For the actual problem (n = 100) both are effectively instantaneous - C at 0.0000 s, Python at 0.0002 s. The gap widens with scale: at n = 1000 Python takes 0.0096 s versus 0.0001 s for C, and at n = 10000 Python takes 6.3311 s while the C limb-based big integer finishes in 0.0119 s - roughly a 500x speedup. The Python version is the more elegant and self-documenting, mirroring the mathematical definition directly; the C version is the one to reach for once n grows into the thousands.

  • A fixed-order linear recurrence needs only a fixed number of state variables - never store the full history when a sliding window suffices.
  • Rotate pointers instead of copying large buffers when advancing a multi-buffer loop.
  • Choose a big-integer base that matches the final query: a power-of-10 base makes decimal digit sums free.
  • Fuse a*scalar + b into a single pass when it sits in an inner loop.
  • Encode a simple modular coefficient rule as a small function, not a table - it is easier to verify and generalises immediately.
  • Reach for a hand-written big integer only when profiling shows the language's native arbitrary precision is too slow; for moderate inputs, exact-arithmetic convenience wins.

Problem Analysis

The spiral picture is a deliberate distraction. What actually matters is a single arithmetic observation: each time I wrap one new layer around the centre, I add exactly four corner cells - one at each corner of the larger square - and these corners lie precisely on the two diagonals. If I index layers as k = 1, 2, 3, ..., the side length at layer k is 2k+1, an odd number, and the bottom-right corner is always the odd square (2k+1)^2. This is because the spiral fills clockwise to the bottom-right and the largest value on a side-(2k+1) spiral is (2k+1)^2. The remaining three corners are reached by repeatedly subtracting 2k (the side length minus one), since each edge of the square spans exactly that many cells between consecutive corners. So the entire diagonal sequence is a closed-form function of k - I never need to build or store the grid.

The real work is counting how many diagonal entries are prime and watching when the running ratio of primes to total diagonal entries first drops below 10%. I test primality with trial division: for a candidate n, I check divisibility by odd numbers up to sqrt(n), costing O(sqrt(n)) per test. The largest corner at the answer (side length 26241) is about 26241^2 ~ 6.9 * 10^8, so each test does at most ~13000 divisor checks. I process roughly n/2 layers, and a corner near layer k costs O(k) work, so the total is O(n^2) in the side length n - comfortably within a single linear pass, which is why this sits at Level 2. The bottom-right corner is an odd square (and thus never prime for k >= 1), so I could skip it, but testing it uniformly is harmless.

Solution Approach

Layer-by-Layer Diagonal Scan (Python and C)

Both files implement the same algorithm; only the language idioms differ. The core loop walks layers from k = 1 upward. For each layer I compute side_length = 2k+1 and the step side_length - 1 = 2k. From the bottom-right corner bottom_right = side_length^2 I derive the other three by successive subtraction:

  • bottom_left = bottom_right - step
  • top_left = bottom_left - step
  • top_right = top_left - step

I keep two running accumulators: num_prime_diagonals (starting at 0) and num_diagonal_elements (starting at 1, to count the centre cell 1, which lies on both diagonals but is not prime). Each layer adds 4 to the total and adds 1 per prime corner. After updating, I check whether the ratio has dropped below the threshold and, if so, return the current side length.

In Python I expressed layer enumeration as a generator that yields a 5-tuple (side_length, bottom_right, bottom_left, top_left, top_right), and the solve function consumes it with a for loop. This separates "how to produce corners" from "when to stop". The C version is a direct port: an unbounded for (int layer = 1; ; layer++) with the same arithmetic. The crucial difference is type width. The corner values exceed the 32-bit signed range (~2.1 * 10^9) well before the answer, so the C code uses long long throughout the corner arithmetic to avoid silent overflow - note in particular (long long)side_length * side_length, which casts before multiplying so the squaring itself does not overflow. Both programs accept the threshold as a command-line argument so they can be re-run at other ratios.

Programming Techniques

Infinite Generator with Early Termination

The Python solution wraps layer production in a generator using yield. A generator produces values lazily - each iteration resumes execution until the next yield - which is exactly what I want when the stopping point is unknown in advance. The consumer iterates and returns as soon as its condition is met, so neither side needs to know about the other's logic. Reach for a generator whenever you have an open-ended sequence and want to decouple value production from the termination test.

The Walrus Operator for Loop State

I increment and test the layer counter in one expression with while layer := (layer + 1), the assignment expression operator. Because layer is always positive after the first increment, the condition is always truthy and the loop runs forever - exactly the unbounded behaviour the generator wants. This keeps the increment and the loop condition in one place rather than scattering a separate layer += 1 at the bottom of the body.

Deriving Values by Arithmetic Instead of Storing a Grid

Rather than allocate a 2-D spiral, I compute the four corners from a closed-form relationship to the layer index. This keeps memory use at O(1) no matter how many layers I scan. Whenever the elements you actually need from a structure have a formula, prefer the formula over materialising the whole structure - it saves both the memory and the time spent building data you never read.

Running Accumulators Instead of Recomputing

I carry num_prime_diagonals and num_diagonal_elements forward and update them with a constant amount of work per layer, then perform one division to test the ratio. Recomputing the totals from scratch each iteration would add a factor of n to the runtime. This state-variable pattern is the standard way to keep a scan linear: update aggregates incrementally rather than re-traversing.

Boolean-as-Integer Counting

The line num_prime_diagonals += is_prime(corner) exploits Python's coercion of True to 1 and False to 0, replacing an if branch with plain arithmetic. The C version does the same because is_prime returns an int of 0 or 1. This is concise and avoids a branch; reach for it any time a predicate's result feeds directly into a count.

Parameterising via Command-Line Arguments

Both solutions read the threshold from arguments (float(args[0]) in Python, atof(argv[1]) in C) rather than hard-coding 0.1. Parameterising the entry point makes it trivial to re-run experiments at other ratios without editing or recompiling - good hygiene for any exploratory or benchmarking tool.

Key Takeaways

The unlocking insight is that the diagonal values are an arithmetic sequence keyed to the layer index, so a problem that looks two-dimensional collapses into a one-dimensional scan with O(1) bookkeeping per step (aside from the primality test). Choosing trial division was deliberate: a Sieve of Eratosthenes would need a fixed upper bound, which we do not have until the answer is found, whereas trial division works candidate-by-candidate with no preallocation.

Comparing the two implementations, the C version runs in about 0.13 s versus the Python version's 2.17 s - roughly 17 times faster. The gap comes entirely from the trial-division inner loop, a tight numeric kernel where native compiled arithmetic dominates. The algorithm and structure are identical; only the constant factor differs.

  • Look for closed-form formulas for the elements you care about in grid or spiral problems - you rarely need to store the whole structure.
  • Maintain running accumulators rather than recomputing aggregates each iteration; this keeps a scan linear.
  • Use trial division for primality when the candidate range is unknown in advance; a sieve needs a fixed upper bound.
  • Treat boolean predicates as 0/1 integers when counting occurrences - it removes a branch and reads cleanly.
  • Pair an infinite generator with an early-return consumer to separate value production from stopping logic.
  • In C, cast to 64-bit before multiplying when intermediate values can exceed the 32-bit range, to avoid silent overflow.
  • Parameterise the entry point and read inputs from the command line to make exploration and benchmarking effortless.

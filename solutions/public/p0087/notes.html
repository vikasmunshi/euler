Problem Analysis

We are asked to count the distinct integers below N = 50,000,000 that can be written as p2 + q3 + r4, where p, q, and r are each prime numbers. The word "distinct" is the crux of the problem: many different triples (p, q, r) can produce the same sum, so we cannot simply multiply together the counts of valid bases - we must deduplicate. The fact that makes this tractable is that the upper bound N tightly constrains how large each prime can be in each role. Since p2 alone must be below N, the largest prime base we could ever need is at most the square root of N, roughly 7071. The cube and fourth-power roles are far more restrictive: r4 < N means r is at most the fourth root of N (about 84), and q3 < N means q is at most the cube root (about 368). The single most demanding constraint is the square term, so one prime list up to sqrt(N) is a superset of every prime we need for all three roles.

The algorithm is a triple nested loop over (fourth-power prime, cube prime, square prime), computing each sum and inserting it into a deduplication structure. Naively this is O(P4 · P3 · P2), where Pk is the number of primes whose k-th power fits under N; by the prime number theorem each Pk is approximately N1/k / ln N. Because the prime lists are sorted in ascending order, we can break out of each inner loop the moment a partial sum exceeds the budget, which prunes the vast majority of iterations. Empirically the number of valid triples is only a few million, so the dominant costs end up being sieve construction and the deduplication structure, both linear in their respective sizes.

Solution Approach

Triple-loop enumeration with sieve-based prime generation (Python and C)

Both files implement the identical algorithm in two languages, so I describe them together and note the language-specific differences where they matter.

Step 1 - Generate primes with the Sieve of Sundaram. I generate all primes up to sqrt(N) using the Sieve of Sundaram. This sieve marks integers of the form i + j + 2ij (for positive integers i, j ≥ 1) in a half-sized array; any position k that remains unmarked corresponds to the odd prime 2k+1, and I prepend 2 by hand. I chose it over the more familiar Sieve of Eratosthenes largely as a variation - both run in similar time and produce the same primes. The single sqrt(N) bound serves all three roles, as argued above.

Step 2 - Pre-compute early-exit thresholds. Before looping I compute max_quadruple_cube = max_num - 4 (reserving room for the smallest square 22 = 4) and max_quadruple = max_quadruple_cube - 8 (reserving room additionally for the smallest cube 23 = 8). These constants become the break conditions at the outer two loop levels, so I exit as soon as further iterations could not possibly yield a valid sum.

Step 3 - Triple nested loop. The outermost loop iterates over fourth powers, the middle over cubes, and the innermost over squares. At each level I test the running partial sum against its threshold and break (not continue) when it is exceeded. This is correct precisely because the prime lists are ascending: once a partial sum overshoots, every later term in that loop overshoots too.

Step 4 - Deduplication. In Python I add each valid sum to a set and return its length. In C I allocate a calloc-zeroed unsigned char array of length max_num used as a presence table, incrementing a counter only the first time a given index is set. The C array gives guaranteed O(1) lookup with no hashing overhead at the cost of about 50 MB of memory; the Python set uses memory proportional only to the number of distinct sums (~1 million) but pays hash-table overhead per insertion.

Programming Techniques

Generator functions for lazy prime-power sequences

In Python, prime_powers is a generator using yield - a function that produces values one at a time on demand rather than building a full list up front. Because every loop breaks early, a list would pre-compute powers we never consume. Whenever an inner loop exits on a threshold, making the source lazy avoids that wasted work and states the intent clearly. The C equivalent is a plain for loop over the primes array computing the power inline.

Early-exit break versus continue in sorted loops

The decision to use break rather than continue when a partial sum exceeds a threshold is subtle but decisive. It is valid only because the prime lists are sorted ascending: once quadruple + cube > max_quadruple_cube, every subsequent cube is at least as large and equally doomed, so visiting them is pointless. Using continue would scan the entire list each time, restoring the full O(P3) cost per outer iteration. Any iteration over a monotonically increasing sequence with an upper bound should prefer break to prune the tail.

Boolean flag array for deduplication in C

The calloc-ed byte array in C is a direct-addressed presence table: index number holds 0 if unseen and 1 once counted. Compared to a hash table it offers guaranteed O(1) reads and writes, no collision resolution, and excellent cache locality. The trade-off is memory proportional to the value range (50 MB), not to the number of distinct results. This is the right call when the range is known, fixed, and not astronomically large.

Python set for deduplication

Python's built-in set stores only the values actually inserted, so its memory cost scales with the number of distinct sums rather than the range. For an unknown or enormous range this is the idiomatic choice; here it costs hash overhead per insert, which is the main reason the Python version is slower.

Single sieve covering all exponent roles

I compute one Sieve of Sundaram up to sqrt(N) and reuse the resulting prime list for the square, cube, and fourth-power roles. The square term imposes the weakest bound on the prime base (the largest primes appear there), so its prime list is a superset of what the cube and fourth-power roles require. Reusing a single data structure rather than running three sieves at different limits cuts both code complexity and computation time.

Walrus operator for compute-once subexpressions in Python

The expressions quadruple_cube := (quadruple + cube) and number := (quadruple_cube + square) use the walrus operator (:=, assignment expressions, Python 3.8+). This computes a value inside the loop condition, binds it to a name for the threshold test, and reuses that name in the loop body - avoiding both a separate assignment statement and a recomputation.

Incremental partial sums across loop levels

Rather than recomputing the full triple sum at the innermost level, I carry quadruple into the cube loop, form quadruple_cube there, and add only the square at the bottom. Building results incrementally keeps each loop's threshold test to a single addition and comparison, which matters when the inner body executes millions of times.

Widening casts to avoid integer overflow in C

In the C version I cast each prime to long long before multiplying, as in (long long)primes[i4] * primes[i4] * primes[i4] * primes[i4]. A prime near sqrt(N) raised to the fourth power would vastly exceed the range of a 32-bit int; forcing the first factor wide promotes the entire expression, so the multiplication is carried out in 64-bit arithmetic and cannot overflow. This is a habit worth keeping whenever a product of small values can grow large.

Key Takeaways

The unlocking insight is that bounding the partial sum at each loop level - and breaking, not continuing, when the bound is exceeded - turns a nominally cubic search into one that visits only a few million productive triples. The sorted order of the primes is exactly what makes early breaking correct. Counting distinct results then reduces to pushing each sum into a deduplication structure and returning its size.

Both solutions share the algorithm; the difference is purely in deduplication mechanics. On the main input the C version runs in about 0.041 s and the Python version in about 0.303 s - roughly a 7x speedup. That gap comes entirely from the C boolean array's collision-free O(1) writes and direct integer arithmetic versus Python's hash-set overhead and interpreter cost, not from any algorithmic difference. The C approach is the fastest; the Python approach is the more memory-frugal and arguably the more readable.

  • When iterating over a sorted sequence under an upper bound, use break rather than continue to prune the entire tail.
  • One sieve sized to the weakest constraint can serve every role; avoid running multiple sieves at different limits.
  • When the range of output values is fixed and manageable, a direct-addressed boolean array beats a hash set for deduplication on both speed and cache behaviour.
  • Compute partial sums incrementally across loop levels and test them against pre-computed thresholds so early-exit checks stay cheap.
  • Reach for generator functions when an inner loop may exit early - they avoid materialising values you will never consume.
  • Widen integer operands before a product that can grow large, so the arithmetic happens in a type that cannot overflow.
  • For "how many distinct values satisfy a combinatorial condition" problems, the pattern is: enumerate combinations, insert each result into a dedup structure, return its size.

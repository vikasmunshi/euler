Problem Analysis

The n-th triangular number is T(n) = n(n+1)/2, the sum of the first n natural numbers. We want the first such number with more than 500 divisors. The key tool is the divisor-counting function d(m): if m has the prime factorization m = p1^a1 * p2^a2 * ... * pk^ak, then d(m) = (a1+1)(a2+1)...(ak+1). The reasoning is combinatorial: every divisor is built by choosing, independently, an exponent from 0 to ai for each prime, so the product of the (ai+1) choice-counts gives the total number of divisors. This is far cheaper than enumerating divisors directly.

To factor an individual number we use trial division up to the square root: any composite m has a factor no larger than sqrt(m), so testing divisors d while d*d <= m suffices, with any residue greater than 1 being a prime that occurs once. This costs O(sqrt(m)) per number. The structural observation that makes the search fast is that n and n+1 are coprime (consecutive integers share no common factor), and the divisor function is multiplicative: d(a*b) = d(a)*d(b) when gcd(a,b)=1. Since exactly one of n, n+1 is even, T(n) splits into two coprime halves, and we can count divisors on numbers roughly half the magnitude of T(n) itself.

Solution Approach

Algorithm 0: Multiplicative split into coprime halves (p0012_s0.py, p0012_s0_c)

This approach exploits multiplicativity directly. At each step I look at the parity of i. If i is even, T(i) = (i/2)*(i+1); if i is odd, T(i) = i*((i+1)/2). In both cases the two factors are coprime, so d(T(i)) = num_factors(first) * num_factors(second). The dividing-by-two folds the /2 of the triangular formula into whichever factor is even, keeping both operands exact integers and roughly half the size of T(i). The loop advances i and recomputes triangle_number = i*(i+1)/2 at the bottom, so the value returned when the divisor count first exceeds the target is correct. Factoring two smaller numbers instead of one large one is the entire payoff: trial division cost scales with the square root, so two halves are cheaper than the whole.

The Python and C versions are logically identical. C uses long long for the triangle number as a defensive choice against overflow, and folds the leftover-prime case into result *= 2 (an unconsumed residue is a prime with exponent 1, contributing a factor of 2). Python's prime_factorization returns explicit (prime, exponent) tuples, which num_factors then reduces to the product of (exp+1).

Algorithm 1: Direct trial division per triangular number (p0012_s1.py, p0012_s1_c)

This is the straightforward baseline. For each T(i) I factor the whole number by trial division and compute d(T(i)) from the exponents, advancing through triangular numbers until the count exceeds the target. There is no coprime split; each triangular number is treated as an independent factorization problem. The code is simpler - the loop condition is just num_factors(triangle_number) < num_divisors - but it factors a number twice as large as either half in Algorithm 0, doing more work per step.

Programming Techniques

Multiplicativity of arithmetic functions

A function f is multiplicative when f(a*b) = f(a)*f(b) for coprime a, b. The divisor count is such a function. Whenever you must evaluate a multiplicative function on a product, check whether the factors are coprime; if so, split the work into smaller pieces. This is a reusable lever throughout number theory.

Coprimality of consecutive integers

gcd(n, n+1) = 1 always, because any common divisor would divide their difference, 1. This elementary fact lets us factor T(n) into coprime halves and unlock multiplicativity. When you see products of consecutive or near-consecutive integers, ask whether coprimality simplifies the computation.

Trial division bounded by the square root

Both solutions share a factorization helper that iterates d while d*d <= n, peeling off each prime's full exponent in an inner loop, then treating any residue above 1 as a final prime. The sqrt bound turns an O(n) scan into O(sqrt(n)) and is the default tool for factoring individual integers in this range.

Separating the factorization helper from the search loop

The Python solutions expose prime_factorization and num_factors as standalone functions. This keeps the search loop readable - it reads almost like a specification - and makes the helpers independently testable. Clean separation of concerns pays off in any language.

Parallel assignment in Python

Both Python solutions initialize i, triangle_number = (1, 1) in one statement. Python evaluates the right side fully before binding names, so paired state is set atomically and clearly. Reach for this when two variables conceptually belong together.

long long as a defensive default in C

The C solutions declare the triangle number as long long even though the answer fits in 32 bits. Integer overflow in C is silent, and a wider type costs nothing here, so defaulting to it for any accumulating or multiplied quantity is a cheap safety habit.

Key Takeaways

The core insight is that T(n) = n(n+1)/2 factors into two coprime numbers, and the divisor function being multiplicative means d(T(n)) is the product of the two halves' divisor counts - each computed on a number half the size. The benchmark confirms the gain: on the 500-divisor case the split solution s0 runs in about 2.1 ms in C versus 8.3 ms for the naive s1, and 31 ms versus 132 ms in Python - roughly a 4x speedup in both languages. The simpler s1 is still fast enough here, but s0 is the principled, scalable choice.

  • When a value is a product of coprime factors, a multiplicative function over it factors too - always check coprimality before factoring a composite.
  • Consecutive integers are always coprime; this small fact frequently unlocks larger simplifications.
  • Trial division up to sqrt(n) is the workhorse for factoring single integers; memorize the exponent-peeling inner loop and the leftover-prime check.
  • Extract reusable helpers (factorization, divisor count) so the main loop reads like a specification.
  • In C, default to long long for anything multiplied or accumulated - overflow is silent and the cost is negligible.
  • Recognizing multiplicative functions (divisor count, Euler's totient) lets you decompose large problems into smaller coprime subproblems.

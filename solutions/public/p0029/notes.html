Problem Analysis

The task is to count how many distinct values appear in the set of powers ab as a and b each range over an interval. If every pair (a, b) produced a different number, the answer would just be the count of pairs. It does not, because different bases and exponents can coincide - the canonical small example is 24 = 42 = 16. To get the right count I must detect and discard these collisions.

The reason collisions occur, and the key to characterising them cheaply, is the fundamental theorem of arithmetic: every integer above one has a unique factorisation into primes. If a base a factors as p1e1 · p2e2 · ..., then ab = p1e1b · p2e2b · ... Two powers are equal precisely when these prime-exponent maps match. That gives me a canonical signature - the tuple of (prime, scaled-exponent) pairs - that encodes equality of two potentially enormous numbers using only small integers. The point matters because 100100 already has roughly 200 decimal digits and 10001000 about 3000, so direct comparison demands either arbitrary-precision arithmetic or a compact encoding.

In either representation, the algorithm is the same: generate every power, reduce it to a hashable key (the raw big-integer value or its prime signature), insert each key into a hash set, and report the set's size. There are roughly (amax - amin + 1) × (bmax - bmin + 1) insertions, each expected O(1), plus a factorisation cost per base. Factoring a single base by trial division is O(√a). So overall I expect O(amax · bmax) for the insertions plus O(amax · √amax) for factoring - and at these sizes the insertion term dominates.

Solution Approach

Algorithm 0 - Direct big-integer set, with prime-signature fallback (hand-rolled factoriser)

Both p0029_s0.py and p0029_s0.c branch on min(a_max, b_max) against a threshold of 100. The idea is to use the simplest correct method for small bounds and the memory-efficient method for large ones.

Small-range path. In Python this is a one-liner: len({a**b for a in range(...) for b in range(...)}). Python integers are arbitrary-precision, so there is no overflow, and the built-in set handles hashing and equality of the big values. In C there is no native big integer, so I built one: numbers live in arrays of 32-bit limbs in base 109, least significant limb first. For each base a I compute ab_min by repeated multiplication, then obtain each successive power by a single multiply by a. Each value is serialised to a decimal string and inserted into a custom string-keyed hash set; the final element count is the answer.

Large-range path. Here I factorise each base a by trial division (the inner loop tries divisors d while d · d ≤ n), producing (prime, exponent) pairs. For each exponent b I scale every exponent by b to form the signature of ab. In Python the signature is a tuple of (prime, power * b) pairs added to a set; in C it is encoded as a comma-separated string like "2:6,5:3" and inserted into the same hash table. This path never builds a large number - it works entirely with small integers - which is why it scales gracefully to a = b = 1000.

Algorithm 1 - Same dual-path strategy with a sieve-backed factoriser

p0029_s1.py and p0029_s1.c are algorithmically identical to Algorithm 0. The only difference is the factorisation step: instead of trial-dividing against every integer, they first generate all primes up to √amax using the primesieve library (the primes function in Python, primesieve_generate_primes in C) and trial-divide only against that prime list. This is asymptotically the same as trial division against all integers but avoids wasted attempts on composite divisors. The reason for keeping both variants is to let the benchmark answer whether factorisation is the bottleneck - and as the timings show, it is not, because hash-set insertion dominates.

Programming Techniques

Canonical signatures instead of raw values

Rather than comparing two huge numbers, I reduce each ab to its prime-factorisation signature and compare those. This is an instance of a general principle: when you must test equality of composed mathematical objects (powers, polynomials, factored forms), find a canonical form - a representation that is equal exactly when the objects are equal - and work with that. It is frequently both simpler and faster than evaluating the objects, and here it lets me avoid 3000-digit integers entirely while relying on the fundamental theorem of arithmetic to guarantee correctness.

Set comprehension for deduplication

The small-range Python path uses len({a**b for ...}). A set comprehension is the most direct expression of "collect all values, drop duplicates, count what remains". Python's set is a hash table with expected O(1) insertion and lookup, and arbitrary-precision integers hash and compare effectively in constant time at these sizes. Reach for this idiom whenever the objects have natural equality and the collection fits in memory.

Incremental big-integer exponentiation

In the C small-range path I compute ab_min once, then obtain ab_min+1 by a single multiply by a, and so on across the inner loop. This turns what would be O(b_max) independent re-exponentiations (each itself O(b_max) multiplies) into a single chain of O(b_max) multiplies per base. The pattern - initialise once, then apply a cheap update per step - is "incremental" or "online" computation, and it pays off whenever successive outputs are related by a small, cheap update.

Base-109 big-integer representation in C

Large integers are stored as arrays of int in base 109, least significant limb first - a hand-rolled form of arbitrary-precision arithmetic. Base 109 is chosen so each limb fits in a 32-bit int, the product of two limbs plus a carry fits in a 64-bit long long (avoiding overflow during multiplication), and serialisation to decimal is trivial: print the top limb unpadded, then each remaining limb zero-padded to exactly nine digits. This is a standard, portable way to do big-integer arithmetic without 128-bit types or external libraries.

String-keyed chained hash table in C

Because C has no built-in hash set, I built one as an array of linked-list buckets - a hash table with separate chaining. I use the djb2 hash (h = h*33 + c) for good distribution on short strings, and 222 buckets so the bucket index is a single bitwise AND with HASH_MASK rather than a modulo. Keys are duplicated with strdup and released in hash_clear, so repeated benchmark runs leak no memory. Choosing power-of-two table sizes for cheap masking is a recurring trick in performance-sensitive hashing.

Runtime algorithm selection by input size

Both solutions branch on a size threshold (min(a_max, b_max) > 100). The direct method is simplest and fast for small bounds; the signature method is necessary and memory-frugal for large bounds. Guarding a fast special case with a threshold and falling back to a general method is a pragmatic, readable form of adaptive algorithm selection - it keeps the common case clean without sacrificing correctness or scalability on the hard case.

Library-backed prime generation versus hand-rolled trial division

Algorithm 1 swaps the inline divisor loop for a precomputed prime list from primesieve. Trial-dividing only against primes up to √n is asymptotically the same but avoids work on composite divisors, and using a vetted library removes a class of off-by-one bugs. The lesson is to measure before assuming the library buys you anything - here the gain is within measurement noise because factoring is not the bottleneck.

Key Takeaways

The unlocking insight is that equality of powers is equality of prime-factorisation signatures, which sidesteps thousand-digit numbers altogether. For the canonical bounds (a, b ≤ 100) the numbers are small enough that a direct set of arbitrary-precision integers is trivially correct, which is why this is a Level 1 problem; the signature trick is what makes the extra case (a = b = 1000) tractable in memory.

The benchmarks tell a clear two-regime story. On the dev and main cases the Python solutions (about 2.3 ms on main) beat the C solutions (about 11 ms) by roughly 5x, because the C code spends its time on big-integer serialisation and hash-table bookkeeping while Python delegates both to highly optimised built-in types. On the extra case (a = b = 1000), where the signature path runs and the per-operation count dominates, the situation reverses: the C solutions (about 0.36-0.37 s) are roughly 4x faster than Python (about 1.6 s), the crossover where C's lower per-operation cost outweighs interpreter overhead. Between Algorithm 0 and Algorithm 1 there is no meaningful difference in either language (within measurement noise), confirming that hash-set insertion, not factorisation, is the dominant cost.

  • When testing equality of composed mathematical objects, compare a canonical form rather than the evaluated value - it is often both simpler and faster.
  • Incremental computation, updating a running value one step at a time, can remove a factor equal to the inner-loop length from the total work.
  • Python's built-in set with arbitrary-precision integer keys can beat a hand-rolled C hash table for small-to-medium inputs; measure before assuming C wins.
  • For big integers in C without library support, base-109 limbs in a 64-bit accumulator are portable, overflow-safe, and easy to serialise.
  • Guard a fast special case behind a size threshold and fall back to a general algorithm to stay both readable and scalable across the input range.
  • Choose power-of-two hash-table sizes so the bucket index is a cheap bitwise AND rather than a modulo.
  • Profile before optimising a subroutine: the obvious candidate (factorisation) was not the bottleneck here.

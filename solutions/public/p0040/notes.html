Problem Analysis

Champernowne's constant is formed by concatenating the positive integers into an endless decimal string. The naive way to find the millionth digit - actually building that string - is wasteful: it would require gigabytes of memory and linear time in the position. The decisive observation is that the string is not random at all; it is a stack of regular bands, where each band collects all the integers with the same number of digits. The 1-digit numbers (1 through 9) contribute 9 * 1 = 9 characters; the 2-digit numbers (10 through 99) contribute 90 * 2 = 180 characters; and in general the band of d-digit numbers holds 9 * 10^(d-1) integers, each contributing d characters, for a band length of d * 9 * 10^(d-1) characters.

Because each band's length has a closed-form formula, I can locate any digit by accumulating band lengths - a running prefix sum - until the total first reaches or exceeds the target position. That tells me which band the position falls in; two integer operations (floor division and modulo) then pinpoint the exact integer and the exact digit within it. The number of bands to examine is bounded by the number of digits in the target position, so each lookup costs O(log n) integer operations. Since the problem asks for only a handful of digits (positions 10^0 through 10^6), the total work is a few dozen integer operations and the runtime is effectively immeasurable.

Solution Approach

Arithmetic band-walk (Python and C)

The Python file p0040_s0.py and the C file p0040_s0.c implement the same algorithm. The core lookup is get_nth_digit_champernowne_s_constant in Python and get_nth_digit_champernowne in C.

Step 1 - find the band. I keep two running totals: length_till_num_digits, the number of characters contributed by all bands strictly before the current one, and length_with_num_digits, the same total after including the current band. On each iteration I increment num_digits, copy the old running total into length_till_num_digits, then add the band length num_digits * 9 * 10^(num_digits-1). I stop as soon as length_with_num_digits reaches or exceeds n. At that point num_digits is the digit-length of the numbers in the band containing position n, and the two totals bracket that band's start and end. This is the prefix sum comparison made concrete.

Step 2 - find the offset within the band. I compute offset_of_number = n - length_till_num_digits - 1. Subtracting length_till_num_digits measures how far into the band we are; the extra -1 converts the problem's 1-based digit numbering into a 0-based index. Then floor division by num_digits gives the index of the number within the band, and the modulo gives the index of the digit within that number.

Step 3 - recover the digit. The first number in the d-digit band is 10^(d-1), so the target number is 10^(num_digits-1) + offset_of_number // num_digits. In Python I render the number with str and index the requested character, converting back to int. In C I format the number into a small stack buffer with snprintf and subtract '0' from the indexed character.

The driver loops over k from 0 to i inclusive, querying position 10^k each time and accumulating the product. Python expresses this declaratively with functools.reduce folding a multiplication over a generator. C uses a plain loop, multiplying the running pos by 10 on each iteration to avoid recomputing the power. Both compute 10^(num_digits-1) and the band length with short explicit multiplication loops in C, which sidesteps the floating-point inaccuracy that pow would introduce.

Programming Techniques

Band-boundary arithmetic instead of string materialisation

The central technique is recognising the hierarchical structure of the sequence and navigating it arithmetically rather than building it. Each band is fully described by two numbers - its start position and its length - both derivable from a formula. This is the general pattern for positional lookup in a structured sequence: when a sequence is a concatenation of groups whose sizes follow a known formula, find the n-th element in O(depth) time using prefix-sum comparisons and integer division. Reach for this whenever a problem asks for the n-th element of an enumerated or concatenated sequence that would be wasteful to build in full.

Index decomposition with floor division and modulo

Inside the correct band I use the classic decomposition: floor division (offset // width) yields the item index and modulo (offset % width) yields the position within the item. This is precisely the arithmetic that converts a flat array index into row and column coordinates of a 2-D grid. I apply the -1 conversion from 1-based to 0-based indexing once, at the single point where I compute the offset - keeping this conversion explicit and localised is the simplest defence against off-by-one errors.

functools.reduce for product accumulation (Python)

The Python solution folds a product over a generator using functools.reduce with a lambda and an explicit initial value of 1. This declarative style removes the need for a mutable accumulator and states the intent plainly - collapse a sequence with an associative binary operation. The explicit initial value also makes the fold well-defined for an empty sequence. Reach for reduce whenever you need products, GCDs, or other folds that the standard library does not provide a dedicated function for.

snprintf for digit extraction (C)

In C the safest way to extract the k-th decimal digit of a large integer is to format it into a stack buffer with snprintf and index the character directly. The alternative - peeling digits with repeated division and modulo - requires reversing the order and tracking lengths, which is more error-prone. Because the function runs only a few times, the formatting cost is irrelevant and clarity wins. I sized the buffer at 32 bytes, comfortably above the roughly 20 characters a 64-bit integer can require.

Integer power by short multiplication loop (C)

Rather than calling pow from math.h - which operates in floating point and can return values like 999.9999 that truncate incorrectly - the C code computes 10^(num_digits-1) and the band length with small explicit multiplication loops over long long. This guarantees exact integer arithmetic. Whenever you need an integer power for indexing or sizing, prefer exact integer multiplication over a floating-point power function.

Harness separated from algorithm (C)

The C solve entry point receives a clean argc/argv pair from the runner, which strips framework flags before calling it. This keeps the algorithmic core readable in isolation and lets the Python and C solutions share an identical command-line interface, which simplifies automated testing.

Key Takeaways

The unlocking insight is that an infinite, irregular-looking string is actually a stack of regular bands whose lengths follow a closed-form formula. Recognising that structure turns a hopeless string search into a few integer operations per query, giving an O(log n) lookup with a tiny constant. Both solutions are correct across all test cases, including i = 11, where one queried digit is 0 and the product collapses to 0 - a satisfying edge case the arithmetic handles without any special treatment.

On the benchmark both implementations report 0.0000 s for every input, which is expected: the total work is at most a few dozen integer operations regardless of the query size. With the timings indistinguishable, the choice between them is one of context rather than speed - the Python version is the more concise and expressive thanks to functools.reduce and big-integer literals, while the C version demonstrates the explicit integer-arithmetic discipline needed to avoid floating-point pitfalls in a lower-level language.

  • When a sequence is a concatenation of groups with a predictable size formula, locate any position with prefix-sum comparisons and integer division instead of building the sequence.
  • The pair offset // width (item index) and offset % width (position within item) recurs everywhere you flatten a structured sequence into a linear index - internalise it.
  • Convert between 1-based and 0-based indexing at a single, clearly named point to stop off-by-one errors from spreading.
  • Prefer functools.reduce with an explicit initial value over a mutable accumulator when folding a sequence to one value.
  • In C, compute integer powers with explicit multiplication rather than the floating-point pow to keep results exact for indexing and sizing.
  • Formatting a number into a stack buffer and indexing it is often cleaner than manual digit peeling, especially for code called infrequently.
  • Separating the benchmarking and argument harness from the algorithmic core keeps both easier to read, test, and port across languages.

Problem Analysis

This problem asks me to search a family of quadratic expressions of the form n^2 + an + b, with integer coefficients bounded by |a| < 1000 and |b| <= 1000. For each pair (a, b) I feed in consecutive non-negative integers n = 0, 1, 2, ... and count the longest unbroken run of prime numbers the formula produces before the first non-prime value. The answer is the product a * b for the pair with the longest run.

The single observation that makes this tractable is a boundary condition. At n = 0 the formula collapses to just b, so if b is not prime the run has length zero. Only prime values of b can ever begin a meaningful run. Since there are only 168 primes below 1000, this insight shrinks the candidate set for b from 2001 integers to roughly 168 magnitudes (each usable with either sign). The remaining work is a brute-force scan over the ~1000 values of a. Each primality test is done by trial division at O(sqrt(V)) cost per value. With P primes, A values of a, runs of length at most R (~80), and values up to V_max (~2 * 10^6), the overall cost is roughly O(P * A * R * sqrt(V_max)) - large in the abstract, but fast in practice because almost all runs collapse after a handful of steps.

Solution Approach

Sieve-constrained brute-force search

The Python and C files share index 0: they implement the same algorithm, with the C version being a direct translation that benefits from compiled arithmetic in the hot inner loop. I describe the algorithm once and note the language-specific differences as they arise.

Step 1 - Generate prime candidates for b. I generate all primes up to the limit (1000) using the Sieve of Sundaram. This sieve operates entirely on odd numbers: it marks every integer of the form i + j + 2ij (for positive integers i, j) as composite within an array of size (N-1)/2; each surviving index i corresponds to the odd prime 2i + 1, and the prime 2 is added explicitly. I chose Sundaram over the more familiar Sieve of Eratosthenes mainly for variety; at this problem size both have equivalent cost, and the Sundaram array is half the size since it skips even numbers.

Step 2 - Restrict the parity of a. For every odd prime b the code steps a in increments of 2 starting from 1; only b = 2 uses step 1 from 0. The reasoning is a parity argument. With odd b, consider n = 1: the value is 1 + a + b. For this to be prime (and not 2, which would force a very short run for these magnitudes) it must be odd, so a must be odd. Exploring only odd magnitudes of a therefore covers all coefficient pairs that can produce a long run while halving the iteration count - a cheap arithmetic filter applied before the expensive primality tests.

Step 3 - Enumerate sign combinations. Rather than looping a from -999 to 999 and b over both signs, I loop a from 0 upward and b over positive primes only, then evaluate all four signed pairs explicitly: (+a, +b), (+a, -b), (-a, -b), (-a, +b). Their products are +ab, -ab, +ab, -ab respectively, computed inline so no extra multiplication is needed. These four cover every distinct signed combination for a given pair of magnitudes.

Step 4 - Measure the prime run. The helper prime_run(a, b) increments x from 0 and tests whether |x^2 + ax + b| is prime, stopping at the first failure and returning the count. I take the absolute value before the test because negative integers are never prime, and a negative output simply ends the run - taking the magnitude lets me detect both signs uniformly. In C the value is held in a long long to avoid integer overflow, since the quadratic can reach roughly 2 * 10^6.

Step 5 - Track the best result. In Python I build a list of (run_length, product) tuples and call max(), which compares lexicographically: the longest run wins, and ties break on the larger product. The C version replicates this exactly with explicit comparisons (run > best_run, then products[c] > best_product on a tie).

Programming Techniques

Boundary-condition pruning

Before writing any loop, I asked what the formula must satisfy at its first input. Since n = 0 yields b, only prime b can begin a run. This cut the search space for b by roughly a factor of six. Whenever you face an exhaustive search, inspect the constraints at a fixed boundary point first - they often eliminate most candidates for free.

Sieve as a one-time preprocessing step

I generate the prime candidate set once with the Sieve of Sundaram rather than testing each b for primality individually. A sieve amortizes the cost of producing a dense set of primes up to a known bound. Reach for a sieve whenever you need the complete set of primes below a fixed limit; the bytearray (Python) or unsigned char array (C) keeps it cache-friendly.

Trial division for sparse, large-value primality queries

I deliberately did not sieve up to V_max (~2 * 10^6) for the run-length checks. The values produced are sparse relative to that range, and each run is at most ~80 steps, so per-call trial division at O(sqrt(V)) is cheaper than building and storing a giant membership table. The general rule: sieve when queries are dense within the range, use trial division when they are sparse or the range is large relative to the query count.

Sign-combination enumeration instead of negative loops

By iterating non-negative magnitudes and applying the four sign variants at the call site, I keep loop variables simple and halve the iteration count for a. The trade-off is four cheap function calls per magnitude pair. This decompose-into-magnitude-plus-signs pattern is broadly useful whenever a problem has sign symmetry.

Parity pruning as early rejection

Stepping a by 2 for odd b is an early-rejection filter: a trivial parity test removes half the candidates before any primality work. Applying cheap arithmetic filters ahead of an expensive dominant test is a recurring optimization technique.

Tuple comparison for lexicographic maximization

Python compares tuples element by element, so max() over (run_length, product) pairs selects the best run and breaks ties on the product in a single expression - no manual if/else chain. Encoding primary and secondary sort keys as a tuple is the idiomatic way to do ranked selection with tie-breaking in Python; the C code makes the same lexicographic logic explicit because the language has no built-in tuple ordering.

Separating the pure solver from the timing harness

Both files isolate solve() from a main() that parses arguments and times runs. Keeping the solver a pure function makes it testable and reusable in isolation, at zero algorithmic cost.

Key Takeaways

The unlocking insight is boundary-condition pruning: because n = 0 forces the value to equal b, the search for b reduces to primes only. Combined with parity pruning and sign-combination enumeration, this makes brute force fast enough that no cleverer algorithm is needed.

Both solutions implement the same algorithm and produce the same answer (-59231). The C version runs in about 0.0177 s versus roughly 0.3293 s for Python - a speedup of about 19x. The difference is entirely the cost of the tight primality inner loop: compiled integer arithmetic dwarfs CPython's interpreted bytecode. When the algorithm is fixed and the work is compute-bound, translating the hot path to C is the most direct route to lower runtime.

  • Inspect the constraints at a fixed boundary point before writing any search loop - they often eliminate most candidates immediately.
  • Use a sieve to build a fixed set of primes once, then choose trial division for sparse primality queries over a large value range.
  • Apply cheap arithmetic filters (parity, modular checks) before expensive tests to reject candidates early.
  • Decompose signed search ranges into non-negative magnitudes plus explicit sign variants to keep loops simple and cut iteration counts.
  • Encode multi-key selection as tuples and use max() / min() for clean, correct tie-breaking in Python.
  • Keep the solver a pure function separate from the timing harness so it stays testable and reusable.
  • When the algorithm is fixed and work is compute-bound, a compiled language can deliver an order-of-magnitude speedup over an interpreted one.

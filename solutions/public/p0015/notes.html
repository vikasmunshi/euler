Problem Analysis

The problem asks how many monotone lattice paths exist from the top-left corner to the bottom-right corner of an n x n grid, where each step can only move right or down. The crucial observation is that the geometry of the grid is irrelevant to the count: any valid path consists of exactly 2n unit steps, of which n must go right and n must go down. A path is fully determined by the order in which we interleave these two kinds of moves. Counting paths therefore reduces to counting the ways to choose which n of the 2n positions in the move sequence are "right" moves (the remaining n are automatically "down" moves). That count is precisely the binomial coefficient C(2n, n) = (2n)! / (n! * n!), known here as the central binomial coefficient. For n = 20 the answer is C(40, 20) = 137,846,528,820, which fits in a signed 64-bit integer; for the extra test case n = 200 the answer is a 120-digit number, which forces us to use arbitrary-precision ("big integer") arithmetic.

This reframing - from "simulate or enumerate paths" to "evaluate one closed-form expression" - is what makes the problem trivial. There is no need for dynamic programming over the grid, no recursion, no path-counting table. The remaining concern is purely arithmetic: how to compute the central binomial coefficient without producing absurdly large intermediate values. Computing (2n)! and (n!)^2 separately works but creates enormous factorials. A better route is the incremental product form C(2n, n) = product over i from 1 to n of (n + i) / i, which interleaves multiplication and division to keep partial values small. Either way the cost is O(n) arithmetic operations on integers whose size grows as O(n log n) bits, giving an overall bit-complexity of about O(n^2 log n) - entirely negligible for the sizes here.

Solution Approach

Central Binomial Coefficient (Python and C)

Both the Python and C solutions compute the same quantity, the central binomial coefficient C(2n, n), and differ only in how they manage arbitrary-precision arithmetic.

Python (p0015_s0.py): I wrote the answer in a single line, math.factorial(2 * lattice_size) // math.factorial(lattice_size) ** 2. Python integers are arbitrary-precision by default, so there is no overflow to worry about and no custom arithmetic to implement. I used floor division (//) rather than true division (/) because the binomial coefficient is always an exact integer and I want to avoid floating-point rounding entirely. math.factorial is implemented in C inside the interpreter, so even for n = 200 this is effectively instantaneous. This is the simplest correct solution available in the language.

C (p0015_s0.c): C has no native big integers, so I implemented a minimal arbitrary-precision integer type and used the product form rather than factorials. The BigInt struct stores the number as an array of base-10^9 "limbs" in little-endian order (index 0 holds the least significant chunk). I chose base 10^9 because a limb fits inside a 32-bit value, yet I store each limb in a 64-bit unsigned long long, leaving ample headroom for the intermediate product limb * multiplier + carry in a multiply step - the multiplier is at most 2n = 400, far below the 64-bit overflow threshold.

The solve function initialises the result to 1 with bi_set, then loops i from 1 to n, calling bi_mul_ll(&result, n + i) followed by bi_div_ll(&result, i). The ordering is deliberate: the running value after step k equals C(n + k, k), which is always an integer, so each division by i is exact and never discards a remainder. bi_mul_ll propagates a carry upward through the limbs (natural for little-endian storage), and bi_div_ll sweeps from the top limb downward, carrying a remainder into lower limbs - the standard long division direction. Finally bi_to_str prints the most significant limb with no leading zeros and every subsequent limb as exactly nine zero-padded decimal digits, which reconstructs the full decimal number directly from the base-10^9 representation.

Programming Techniques

Python Built-in Arbitrary-Precision Integers

Python integers grow in memory as needed and have no fixed maximum, so the factorial expression just works for any n, returning an exact result. This eliminates an entire class of bugs - overflow and silent truncation - with zero extra code. Reach for Python's native arbitrary-precision integers whenever correctness on large numbers matters more than raw speed; they trade a little performance for complete freedom from overflow reasoning.

Base-10^9 Limb Representation for Big Integers in C

When C needs big integers, the practical pattern is an array of limbs, each holding a chunk of the number in some base. I picked base 10^9 (one billion) for two reasons. First, it leaves room for multiplication: a limb under 10^9 times a multiplier under 400, plus a carry, stays well within 64 bits. Second, it makes decimal output trivial - each limb maps to exactly nine decimal digits, so no repeated divide-by-ten loop is needed to print the number. Use a power-of-ten base when decimal output dominates; switch to a power-of-two base when bitwise operations or binary I/O are the main concern.

Little-Endian Limb Order

I store limbs from least significant (index 0) to most significant. This little-endian layout matches the natural direction of carry propagation: multiplication carries flow upward (index 0 to higher), and division remainders flow downward (top index to lower), keeping both carry loops simple and free of awkward index arithmetic. It is also the convention used by GMP, the de facto standard C big-integer library, so code written this way is easy to compare against a reference.

Interleaved Multiply-then-Divide to Control Intermediate Size

Instead of accumulating the full numerator and denominator and dividing once at the end, I divide out the known factor i immediately after multiplying by (n + i). Because each partial product is a genuine binomial coefficient, the division is always exact, so no information is lost and the intermediate values stay as small as the final answer rather than ballooning to factorial size. This "build a product incrementally and cancel known factors as you go" pattern is a staple of combinatorics code and applies any time you know the final result is an integer but the raw numerator and denominator would be enormous.

Minimal Self-Contained Big-Integer Module

Rather than linking a full library such as GMP, I implemented only the three operations I actually needed: set from a 64-bit value, multiply by a 64-bit value, and divide by a 64-bit value. This keeps the C file dependency-free and easy to audit. When writing competitive or embedded solutions, a purpose-built arithmetic module is often preferable to a heavy general-purpose dependency, provided the required operation set is small and well defined.

Key Takeaways

The unlocking insight is recognising that lattice path counting is a thin disguise over a binomial coefficient: every path is one arrangement of n right moves and n down moves, so the answer is the central binomial coefficient C(2n, n). Once you see this, all simulation and dynamic programming disappear and the problem collapses to evaluating a single formula. Both solutions handle arbitrarily large n correctly - Python through native big integers, C through a compact custom implementation.

The recorded benchmark averages reveal a genuine crossover rather than a tie. On the small inputs C is the faster of the two - 0.48 microseconds versus 1.35 microseconds on the dev case (n = 2), and 0.77 microseconds versus 2.05 microseconds on the main case (n = 20) - because its work is a short loop with negligible allocation, whereas CPython pays fixed interpreter and object-creation overhead per call. On the extra case (n = 200), however, the order reverses: Python finishes in 8.87 microseconds against C's 13.19 microseconds. Once the numbers grow to 120 digits the cost is dominated by the big-integer multiplications themselves, and CPython's highly optimised native integer routines outpace my straightforward schoolbook base-10^9 limb arithmetic. The practical reading is that the Python one-liner is both the simplest and the faster choice at the scale that actually stresses the arithmetic; the C version wins only on tiny inputs where per-call overhead, not arithmetic, dominates, and earns its place as a teaching example of building arbitrary-precision arithmetic from scratch.

  • When a counting problem maps to a known combinatorial quantity (a binomial coefficient, a Catalan number, and so on), use the closed-form formula - it is faster and simpler than any simulation.
  • Compute binomial coefficients via the incremental product form rather than full factorials; it keeps intermediate values small and makes exact integer division automatic.
  • Cancel known integer factors immediately after each multiplication to stop intermediate values from growing far beyond the final answer.
  • For big integers in C, a base-10^9 limb array gives trivially simple decimal output and no external dependencies - a reliable fallback when numbers outgrow 64 bits.
  • Reach for Python's built-in arbitrary-precision integers whenever correctness on large numbers is the priority; at scale their optimised native routines can even beat a hand-rolled C implementation.
